## IT 보안이란

민감 정보를 지키기 위해 시스템, 네트워크, 데이터를 공격과 무단 접근으로부터 보호.

- **IT 보안 (Tech)**: 컴퓨터 네트워크를 통해 디지털 데이터를 보안하는 작업. 방화벽, Nmap 등
- **정보 보안 (InfoSec, 정책)**: 민감한 기업 정보를 침입으로부터 보호할 프로세스와 툴
- 두 개는 서로 유기적으로 연결되어 보안을 설계

## IT 보안을 위협하는 요인

### Malware (악성코드)
- **Ransomware**: 데이터 암호화 후 몸값 요구
- **Spyware**: 사용자 정보 몰래 수집
- **Virus**: 파일에 감염되어 복제·전파
- **RAT (Remote Access Trojan)**: 원격 접속 트로이 목마

### RAT 동작 원리와 위험성

RAT는 원격 접속 트로이 목마. 팀뷰어 같은 정상 원격 지원 툴과 발상이 같지만, 동의 없이 몰래 깔린다는 점이 다르다.

```
화면을 스크린샷으로 계속 찍어서 전송 → 받는 쪽에선 동영상처럼 보임
캡처 시점 = OS 인터럽트 발생 순간 (마우스 움직임 등)
→ 움직임이 있을 때만 찍어 보내니 효율적이고 들키기 어려움
```

> 디지털 데이터는 한 번 복제·유포되면 회수가 안 된다. → 화면에 띄운 순간 다 털린다는 전제로 대비해야 함.
> 이걸 막으려고 **망분리 · 접근통제 · 로그 관리**가 필요한 것.

### 악성코드 유포 실태 (크랙 버전)

```
크랙 버전 = 정품 인증을 우회한 불법 버전
→ MS 오피스 크랙: 프로덕트 키를 인지하는 레지스트리 값을 없애서 인증 우회
→ 이 과정에서 악성코드가 함께 심김 (2024년 기사지만 지금도 동일)
```

- 크랙 버전으로 유포되는 대표 악성코드: **Updater(업데이트), OrcusRAT(원격 제어), XMRig(코인 채굴)**
- 안랩 ASEC 분석: **작업 스케줄러가 치료되지 않으면** 시스템에서 악성코드가 지속적으로 재설치됨
- ⚠️ 악성코드만 지워도 작업 스케줄러에 등록된 재설치 루틴이 남아있으면 또 깔린다 → 스케줄러까지 봐야 함

#### CDK(제품 키) 추출 프로그램

```
정품 키가 레지스트리에 등록되어 있음
→ 경로의 레지스트리 값만 읽어오면 CDK가 그대로 나옴
→ 추출된 정보는 해커 서버(C2 / CNC)로 전송
```

> PC방에서도 한동안 CDK 추출 프로그램이 유행했음. "레지스트리에 평문으로 있는 값"이 노출 포인트.

### 키로깅의 위험성

```
키보드 입력을 전부 텍스트로 가로채 외부로 전송
→ 개인적인 대화까지 다 나감 → 그걸 미끼로 피싱·사회공학 공격 유입
→ 데이터가 나갈 때 흔적(로그)이 남음 → 역추적의 단서가 되기도 함
```

### 내부 위협
- 직원, 협력사 직원
- 해커보다 침투가 수월함 (해커가 하는 어려운 작업을 보다 쉽게 수행)

## 웹 공격 기법

### Directory Listing Attack
- 웹 URL에 접근 제한이 없는 파일에 무작위 침투
- 내부 정보 및 버전 정보 유출
- F12 → Network → Header에서 서버 정보 확인 가능

### SQL Injection
```
SELECT * FROM member WHERE id='입력값' AND pw='입력값'
→ id에 admin 입력, pw에 ' or 1=1 -- 입력
→ '로 이전 명령 닫고, 1=1 참값, -- 뒤 주석처리
→ 비밀번호 없이 최고 권한 획득
```

### XSS (Cross-Site Scripting)
```html
<script>alert('XSS')</script>
```
- 게시판 등에 악성 스크립트 삽입
- 사용자 클릭 시 쿠키 탈취 또는 악성코드 실행

### Webshell Attack
- 해커가 악성 파일(.php)을 웹에 업로드해 서버 동작 탈취
- 파일 업로드가 되는 지점까지 갔으면 서버 조작 가능
- 리버스쉘 이용 시 공격자 PC에서 명령어 수행까지 가능

## 계정 탈취 공격

### 브루트포스 (Brute Force)
- 무차별 패스워드 대입 공격
- 대량의 로그가 남아 탐지 비교적 쉬움
- 단일 로그가 아닌 **여러 로그의 상관관계**를 봐야 함

```
access.log 인증 실패 급증
+
443/80 세션 연결 수 급증 (Established)
= 브루트포스 공격 의심
```

#### Hydra 사전공격(Dictionary Attack) 실습 — 6/19

Kali에서 **Hydra**로 패스워드 목록(wordlist) 파일을 돌려 웹서버 SSH 비밀번호를 무차별 대입했다.

```
순수 브루트포스 = 가능한 모든 조합을 0000, 0001 … 식으로 전부 시도
사전공격(Dictionary) = 자주 쓰는 패스워드 "목록 파일"을 대입 (Hydra가 이 방식)
 → 둘 다 넓게 보면 브루트포스. Hydra는 wordlist 기반이라 "사전공격"이 정확한 표현
```

> 💡 음차 정정: 수업노트의 *"디렉터리 공격"* 은 **딕셔너리(dictionary) 공격**의 오기다. Hydra가 디렉터리를 터는 게 아니라 **패스워드 사전(wordlist)** 을 대입하는 것.

⚠️ **서버 입장에서 비밀번호 실패 로그를 남기는 건 둘 중 하나뿐이다** — ① 진짜 관리자가 오타 낸 것, ② 크리덴셜을 노린 공격자. 그래서 **실패 로그가 급증하면 거의 무조건 공격**으로 본다. (Linux-Commands `/var/log/auth.log`·`journalctl`로 SSH 접근 실패 추적)

```
실습 흐름
 Kali ── Hydra(wordlist) ──▶ Web 서버 SSH(포트)
 Web 서버 ── auth 로그에 "Failed password ..." 무더기 기록 ──▶ 탐지
```

> 🔗 포트가 열려 있어야 공격이 시작된다 — **포트포워딩(DNAT)으로 외부에 연 포트가 곧 nmap 포트스캔에 걸리는 공격 표면**이다. 그래서 "안 쓰는 포트는 닫고, 연 포트는 IPS/IDS로 본다"가 한 세트. (Lab 2026-06-19 참고)

### 크리덴셜 스터핑 (Credential Stuffing)
- 유출된 아이디/패스워드 쌍을 다른 서비스에 무작위 시도
- 계정당 1회만 시도 → 로그가 적어 탐지 어려움
- 회피 기법: VPN으로 IP 변조, 접속 시간 분산, User-Agent 매 요청마다 변경

## 포트 스캔 (nmap) — 외부에서 "어느 문이 열렸나" 정찰

공격의 첫 단계는 정찰이다. **외부에서 내부의 어떤 포트(문)가 열려 있는지** 훑는 게 포트 스캔. 6/19엔 "열린 포워딩 포트만 잡힌다"는 사실만 봤고, 오늘(6/22)은 **스캔 방식의 차이**를 와이어샤크로 뜯어봤다.

### 열림 / 닫힘을 무엇으로 판단하나

```
포트 열림  → 서버가 SYN+ACK 응답 (3-way를 받아줄 준비됨)
포트 닫힘  → 서버가 RST 응답 ("그런 서비스 없음, 끊어")
필터됨     → 아무 응답 없음 (방화벽이 drop) → 스캔에 안 뜨거나 오래 걸림
```

> 클라이언트가 ACK까지 보내 **ESTABLISHED가 되면 그 포트는 살아있는 것** → nmap 결과에 "open"으로 올라온다.

### -sT (TCP Connect 스캔) vs -sS (SYN / 스텔스 스캔)

```bash
nmap -sT 192.168.1.100 -p 1-65535     # TCP Connect: 3-way를 끝까지 맺음
nmap -sS 192.168.1.100 -p 1-65535     # SYN(Stealth): SYN→SYN+ACK까지만, ACK 안 보내고 RST로 끊음
```

| 구분 | -sT (Connect) | -sS (SYN / Stealth) |
|---|---|---|
| 핸드셰이크 | **3-way 완성** (SYN→SYN+ACK→ACK→…→RST) | **반만** (SYN→SYN+ACK→RST) |
| 연결 성립 | ESTABLISHED 됨 → 서버에 **세션·로그 남음** | ESTABLISHED 안 됨 → **연결 로그 안 남기 쉬움** |
| 권한 | 일반 권한으로 가능(OS 소켓 사용) | **raw socket 필요(root/관리자)** |
| 흔적 | 많음 (정상 연결처럼 보이지만 다 기록) | 적음 → 그래서 "스텔스" |

### 스텔스 스캔의 원리 — "흔적을 안 남기려고 연결을 안 맺는다"

```
해커가 ACK까지 보내면 ESTABLISHED가 되어 자기 IP가 세션/로그에 박힘
 → 그래서 SYN+ACK(포트 열림 확인)를 받는 순간 ACK 대신 RST로 끊어버림
 → "열렸다"는 정보만 챙기고 연결은 맺지 않음 → 자기 흔적 최소화
 = 이게 스텔스(SYN) 스캔
```

> 비유: 가게 문이 열렸나 **손잡이만 살짝 돌려보고(SYN) 열린 걸 확인하면 바로 손 떼는(RST)** 것. 안에 들어가 자리에 앉으면(ESTABLISHED) CCTV에 찍히니까, 들어가진 않는다.

⚠️ 닫힌 포트는 어느 방식이든 **RST**가 돌아온다. 와이어샤크로 보면 ① 열린 포트=SYN+ACK, ② 닫힌 포트=RST, ③ 필터=무응답 — 이 세 반응이 명확히 갈린다.

> 🔗 그래서 방어는 ① **안 쓰는 포트는 닫고**(스캔 표면 축소, Lab 6/15 samba·dns 아웃바운드 차단), ② 연 포트는 **IPS/IDS로 본다**. SYN만 잔뜩 뿌리는 스캔/SYN Flood는 IPS가 `flags:S` 임계치로 잡는다. (OSI-7Layer.md "TCP 플래그", Lab 2026-06-19/06-22)

## 네트워크 공격

### MAC 스푸핑
- 허용된 MAC 주소를 복사해서 망 진입 시도
- NAC가 있으면 막힘 (인증서 확인, 미등록 장비 탐지, MAC 충돌 탐지)

### DNS 스푸핑
- 임의로 DNS 캐시 정보 변조해 공격자 사이트로 유도
- hosts 파일 변조 (`C:\Windows\System32\drivers\etc\hosts`)

### ARP 스푸핑
- ARP Reply 패킷의 MAC 정보를 **검증하지 않는** 점을 악용
- 공격자가 가짜 ARP 정보("이 IP의 MAC은 나야")를 빠르게 뿌림
- 상대방이 공격자 MAC으로 통신하게 됨 → 트래픽 가로채기(중간자 공격)

> 근본 원인은 ARP 프로토콜 자체가 응답을 신뢰해버리는 구조 → OSI-7Layer.md "ARP 동작 원리" 참고.

### Lateral Movement (수평 이동)
- 1차 방화벽을 뚫고 들어온 공격자가 내부에서 다른 서버로 이동
- 1차 방화벽은 이미 통과됐으니 내부 이동을 못 탐지 → TP모드 방화벽이 잡는 영역

#### 침투 → 수평 이동 → 반출 시나리오

```
1. 해커가 외부에서 DMZ를 통해 진입 (이미 아이디/패스워드 확보)
2. 세팅 값(접속 경로·계정)을 그대로 들고 다님 → 최종 목적은 DB
3. DB가 나올 때까지 서버에서 서버로 수평 이동
4. 탈취 데이터를 C2 서버로 반출(아웃바운드)
```

- ⚠️ 파일에 패스워드나 접속 경로를 평문으로 저장해두면, RAT 한 번에 회사가 통째로 털린다 → 내부 파일 보안·관리 방안이 핵심
- **방화벽에서 아웃바운드를 막아두면** 데이터를 C2로 못 빼냄 → 공격자가 마지막 단계에서 막힌다
- 과정1에서 L3를 자꾸 잘게 나눈 이유도 바로 이 **수평 이동을 막기 위함**

## DDoS (분산 서비스 거부) — 6/23 심화

```
DDoS = 집단 폭행(다구리)
→ 정상 트래픽을 압도하는 대량 트래픽으로 서비스를 마비
→ 공격 기법 자체는 단순함 (그냥 숨이 멎을 때까지 때리는 것)
```

> 음식점 비유: 정상 손님 2명은 문제없는데, 갑자기 200명이 몰려오면 매장 포화 → 정상 손님도 서비스 못 받음 = 자원 고갈.

### DoS → DDoS : "맞짱"에서 "다구리"로 진화한 이유

```
DoS  = 1:1 맞짱   → 전제조건: 상대(서버)가 나보다 약해야 함
                    옛날엔 공격자 PC가 성능 우위 → 혼자서도 서버를 죽일 수 있었다
서버 성능 상승  → 1:1로는 더 이상 안 통함
공격자의 결론   → "혼자선 안 되겠다. d(분산=다구리)를 하나 더 붙이자" → n:1
DDoS = n:1 집단폭행 → 다수의 좀비 PC(봇넷)를 동원해 한 대상을 동시 타격
```

⚠️ **결론부터: DDoS는 사실상 "못 막는다."** 세상엔 좀비 PC가 너무 많기 때문.

```
내 서버 캐파 10 → 공격 20으로 옴
캐파 1000 → 공격 10000으로 옴
서버는 전 지구인을 상대로 싸울 수 없다
캐파 늘리려면 → 방화벽 교체 + 시스템 증설 + 회선 증설 … 돈·한계 끝없음
그 이상의 트래픽이 오면 → game over
```

> 그래서 DDoS는 "완전 차단"이 아니라 **"완화(mitigation) + 골든타임 안에 끊기"** 가 현실적 목표다.

### DDoS 3대 유형 — 공격 계층이 다르면 막는 장비도 다르다

| 유형 | 노리는 곳 | 대표 기법 | 계층 | 단위 |
|---|---|---|---|---|
| **대역폭 고갈** | 회선(파이프) 자체 | UDP / ICMP / Fragment Flooding | L3/L4 | **bps** (bit per second) |
| **자원 고갈** | 보안장비·세션테이블 | SYN Flood, 단편화, 80/443 인젝션 폭주 | L3/L4 | pps / 세션 수 |
| **응용 계층** | 웹 서버·DB·앱 엔진 | HTTP GET/POST Flooding | L7 | rps (req/s) |

> 🔗 이게 6-5 아키텍처(DDoS → F/W → IPS → … → Server)에서 맨 앞단이 **Anti-DDoS**인 이유다. 앞에서부터 굵은 트래픽을 흘려 거르고, 뒤로 갈수록 정밀 검사. (Architecture.md "보안 장비 배치", Security-Solutions.md "Anti-DDoS와 IPS를 한 라인에 몰지 마라")

### ① 대역폭 고갈 공격 (Volumetric) — 회선(수도관)을 터뜨린다

```
과도한 트래픽으로 회선 대역폭 자체를 고갈시켜
정상 트래픽이 들어올 자리를 없애는 공격
→ 큰 패킷을 만들려 위조된 큰 사이즈 패킷 사용 + 출발지 IP도 위조
→ 주요 프로토콜: UDP, ICMP (3-way 없이 일방적으로 밀어넣을 수 있어서)
```

> 수도관 비유: 회선이 1Gbps 파이프인데 평소 실사용자는 600~700Mbps만 쓴다 → 1Gbps로 충분. 근데 **2Gbps짜리 디도스**가 들어오면? 수도관이 터진다 = 동맥경화. 방화벽으로 들어오기도 전에 회선이 이미 막혀버림.

**해커의 정찰 방식 — 좀비 PC도 "자산"이라 아껴 쓴다**

```
회선이 1Gbps인 걸 모르니까
→ 200Mbps 같은 작은 양부터 며칠에 걸쳐 천천히 올려보며 대역폭을 측정
→ "이쯤이면 죽는다" 지점을 찾으면 그 양으로 한 방에 고갈
왜 아끼나? → 좀비 PC는 한 번 쓰면 노출·차단되는 소모 자산이라
```

⚠️ **골든타임** = 회선/CPU가 맥스를 치기 직전 구간. DDoS가 오면 CPU·대역폭이 들쑥날쑥하다가 풀로 차면 손쓸 수가 없다. → **대역폭과 자원을 평소부터 모니터링**하고 있어야 맥스 전에 끊는다.

> 🔗 출발지 IP를 위조(spoofing)하니 차단 IP 목록(블랙리스트)이 무력 → 그래서 뒤의 **Ingress 필터링(BCP38)** 과 **1:1 방화벽 정책**이 나온다.

#### UDP Flooding — 와이어샤크 실관찰 (6/23)

와이어샤크 캡처에서 본 패턴:

```
Source IP            Destination       Proto  ...        Length
75.186.27.63    →    192.168.100.50    UDP    1861→80    Len=1400
229.141.146.27  →    192.168.100.50    UDP    1862→80    Len=1400
16.166.31.80    →    192.168.100.50    UDP    1863→80    Len=1400
…(소스 IP가 전부 제각각, 목적지만 동일, 큰 Len)
```

- **소스 IP가 매 패킷마다 다르다** → 전부 위조됨 → "이 IP만 막으면 돼"가 불가능
- 외부에서 **3-way를 맺고 오는 게 아니다** (UDP라 핸드셰이크 없음) → 정상 연결처럼 안 보임
- 방어: **1:1 방화벽 정책**(필요한 출발지↔목적지·포트만 명시 허용)으로 좁히고, 서버 대역폭도 어느 정도 여유 확보

#### Fragment Flooding — 조각만 보내고 끝을 안 맺는다

```
MTU보다 큰 패킷은 잘게 쪼개져(단편화) 전송 → 받는 쪽이 다시 조립
공격: MF(More Fragment)=1 조각만 잔뜩 보내고 마지막(MF=0) 조각을 안 보냄
→ 받는 쪽은 "다음 조각" 기다리며 버퍼에 쌓아둔 채 조립 못 끝냄 → 자원 고갈
```

> 🔗 6-22 와이어샤크에서 `ping -l`로 MF·Fragment Offset 직접 봤던 그 단편화 메커니즘을 **악용**한 게 이것. (OSI-7Layer.md "IPv4 헤더 / 단편화")

### ② 자원 고갈 공격 — 회선은 멀쩡한데 "장비"가 죽는다

대역폭이 여유 있어도, **보안 장비(IPS 등)의 CPU·메모리·세션테이블**을 말려 죽이는 류.

```
상황: 회선 여유 / F/W 여유 / 그런데 IPS CPU 70~80%, 메모리 80%
공격: 80·443 TCP로 인젝션·XSS 취약점 패턴 + MF값 1 무한 전송 등
      → IPS가 패킷마다 딥 인스펙션하려다 자원을 풀로 소모
결과: 회선은 살아있는데 IPS(어플라이언스)가 뻗음
```

⚠️ **이때는 하드웨어 바이패스도 HA도 못 구한다.**

```
바이패스 무력 → 장비가 "죽은(down)" 게 아니라 자원이 풀로 차서 "지연(delay)"된 상태
              → 바이패스는 down을 감지해 우회하는 건데, 안 죽고 느려진 거라 트리거 안 됨
HA 무력      → Active가 죽어 Standby로 넘겨도, 그 세션이 그대로 넘어가니
              → 넘겨받은 장비도 똑같이 자원 고갈로 뻗음 (도긴개긴)
```

> 🔗 그래서 Security-Solutions.md의 **"Anti-DDoS와 IPS를 한 라인에 몰지 마라"** 가 핵심. IPS에 DDoS 막이를 떠넘기면 이렇게 같이 죽는다. 고스루풋 Anti-DDoS가 앞단에서 굵은 트래픽을 먼저 흘려야 한다.

### "미완성 상태"로 자원 고갈시키는 DoS (Half-open / 단편화) — 6/22

플러딩(다구리)과 달리, **연결이나 조각을 일부러 끝맺지 않아** 받는 쪽이 무한 대기하게 만드는 류. 트래픽이 많지 않아도 자원이 마른다.

```
SYN Flood        → SYN만 보내고 ACK 안 줌 → half-open 연결이 세션 테이블을 꽉 채움
단편화 DoS(Teardrop류) → MF=1 조각만 보내고 마지막(MF=0)을 안 보냄
                  → 받는 쪽이 "다음 조각" 기다리며 버퍼에 쌓아둔 채 자원 고갈
```

> 🔗 위 Fragment Flooding과 **같은 원리** — "끝나지 않는 미완성"으로 상대 자원을 잠근다. IPS·방화벽이 **불완전 세션·비정상 단편**을 임계치(Threshold)로 끊어내는 게 방어 포인트.

#### SYN Flooding 실증 — Snort 임계치 룰로 끊기 (6/24)

6/22엔 와이어샤크로 half-open을 "관찰"만 했고, 6/24엔 hping3로 직접 쏴보고 Snort 룰로 끊어봤다.

**왜 SYN Flood가 먹히나 — 숫자로**
```
OS의 backlog queue(반쪽 연결 대기열)는 보통 300~400개 수준
→ SYN만 계속 쏴서 이 큐를 가득 채우면, 정상 SYN도 큐에 못 들어가 거부
→ 큐를 늘려도(튜닝) 결국 한계에 봉착 → "양으로 미는" 싸움
```
- ⚠️ 그래서 SYN Flood는 **CPS(Connection Per Second)가 힘들어야** 효과가 난다 — 초당 연결 시도를 폭발적으로 올려 큐가 비워질 틈을 안 줘야 한다. 느릿느릿 쏘면 OS가 타임아웃으로 비워내며 버틴다.

**위조 IP가 "실제 존재하는 IP"면? (교재 그림 3-56)**
```
공격자가 SRC IP를 2.2.2.2(실존 IP)로 위조해 SYN을 보냄
→ 서버는 2.2.2.2로 SYN-ACK를 보냄
→ 근데 2.2.2.2는 SYN을 보낸 적이 없음 → "이게 뭐지?" 하고 RST로 끊어버림
→ RST가 오면 서버는 backlog에서 그 연결을 삭제 → 자원이 회복됨
```
> 💡 즉 **SYN Flooding이 성공하려면 SYN 이후 아무 응답도 안 와야** 한다. 그래서 보통은 **응답이 안 오는(라우팅 불가) 위조 IP**를 쓰거나, 응답이 와도 무시되도록 한다. 실존 IP를 위조하면 그쪽이 RST를 날려줘서 오히려 방어가 되는 셈.

**Snort 차단 룰 핵심 — `flags:S` + `threshold`** (룰 자체는 Security-Solutions.md "SYN Flood 임계치 룰" 참고)
```
flags:S          → SYN 플래그만 세팅된 패킷을 매칭 (S=SYN, A=ACK, R=RST, P=PSH)
threshold        → 정상 트래픽엔 안 걸리고 "과도할 때만" alert/drop 되도록 임계치 설정
  type both      → 같은 조건이 임계치를 넘을 때 (limit+threshold 결합)
  track by_src   → 출발지 IP 기준으로 카운트
  count 20       → 20회
  seconds 2      → 2초 안에  → "2초에 20개 넘는 SYN" = 공격으로 간주
```

> 🔗 결국 6-23에서 본 **"골든타임 안에 끊기"** 의 실전 구현이다. threshold가 곧 "정상은 이 정도, 그 선을 넘으면 공격" 이라는 애노멀리 판단선. (Security-Solutions.md "Threshold(임계치)")

#### 프로토콜을 맞춰야 서버에 "도달"한다 (6/24 SPAN 실관찰)

DDoS 연장 실습에서, 칼리를 ESXi 2번 어댑터에 **SPAN(미러)** 으로 물려 오가는 패킷을 와이어샤크로 봤다. 거기서 확인한 핵심:
```
pfSense 포트포워딩 53번(DNS)으로 WAN 접근 → 내부 DNS 서버까지 정상 도달
  → 프로토콜(UDP/53)을 정확히 맞췄기 때문에 응용프로그램(BIND)까지 부하가 닿음
ICMP(핑)로 같은 서버를 때리면? → 서버까지 도달 못 함 (포워딩된 건 53뿐)
```
> 💡 교훈: **"열려있는 포트/프로토콜에 정확히 실어야" 부하가 응용계층까지 닿는다.** 아무 프로토콜이나 쏜다고 서버가 아프지 않다 — 방화벽 정책에서 안 여는 프로토콜은 앞단에서 그냥 버려진다. 이게 6/15에 "안 쓰는 포트 닫기"가 곧 공격 표면 축소인 이유. (Lab 2026-06-24)

### ③ 응용 계층 공격 (L7) — 가장 까다롭다. "정상 손님"으로 위장하니까

회선·장비를 다 키워서 잔챙이 공격을 막아내도, **마지막에 웹 서버가 문제**다.

```
공격자가 Real IP(진짜 자기 IP)로 정상 3-way handshake까지 정상 수행
→ 그 후 대량의 HTTP GET / POST 요청을 퍼부음
→ 웹 엔진(JAVA 등)은 세션을 정상으로 받아들이지만, 서버가 그 양을 감당 못 함
→ 응용 프로그램·연결된 DB까지 부하 → 다운
```

> 비유: 회선(100Gbps)도 IPS(고성능)도 다 키웠다. 그런데 웹 서버는 1대고, 콘텐츠가 많아 **정상 사용자 200~1000명만 들어와도 느려지기 시작**한다. 공격자는 "정상처럼 보이는" 요청을 그 한계 위로 쏟아붓는다.

- ⚠️ **Real IP라서 더 무섭다** — 위조 IP가 아니라 진짜 IP로 정상 핸드셰이크를 하니, 출발지 필터링·블랙리스트로 못 거른다. **트래픽 양도 L3/L4 플러딩만큼 크지 않아** 대역폭 경보에도 안 걸린다.
- 방어: L7까지 읽는 **Anti-DDoS / WAF**, 요청 빈도 기반 Rate Limiting, CAPTCHA, 행위 기반 탐지가 필요
- 🔗 한국 대형 사례: **2009년 7·7 DDoS 대란**, **2011년 3·4 DDoS 대란** — 봇넷으로 정부·금융·포털을 동시 타격해 사회적 혼란 유발.
  > 💡 정정: 수업노트의 *"2010년 3.1절 사이버 전쟁"* 은 연도/명칭 혼동이다. 한국의 대표 DDoS 대란은 **7·7(2009)** 과 **3·4(2011)**.

### DDoS 대응 전략 — "막기"가 아니라 "흡수·분산·우회"

#### (1) 클라우드 / 오토스케일링 — DDoS에 가장 강한 형태

```
평소 트래픽은 일정한데 순간적 급증을 감당해야 하는 문제
→ 클라우드 오토스케일링: 트래픽 급증 시 서버를 자동 증설해 흡수
→ DDoS에 가장 강한 구조가 결국 Cloud
대표: AWS Shield (+ CDN도 대용량 트래픽 완화 역할)
```

> 🔗 온프레미스 어플라이언스는 캐파가 고정이라 한계가 명확 → 탄력적으로 늘어나는 클라우드가 본질적 우위. (Security-Solutions.md "CDN도 DDoS 완화")

#### (2) DNS는 방화벽 "밑"이 아니라 "위/바깥"에 둬라

```
해커는 UDP로 자기 IP를 숨겨 밀어넣음 → 대표적으로 열려 있는 DNS(53/UDP)로 들어옴
→ DNS가 방화벽 안쪽(밑)에 있으면, DNS 노린 플러딩이 방화벽까지 끌고 들어옴
→ DNS는 상단(외부 존/별도)에 두어야 안전 (또는 외부 DNS 서비스 위탁)
```

#### (3) Anycast + BGP 라우팅 — 한 IP를 전 세계에 뿌려 분산

```
Anycast = 같은 IP 대역을 여러 지역 네트워크에서 동시에 광고(Announce)
→ 사용자는 BGP 라우팅이 골라준 "가장 가까운(Shortest Path)" 노드로 접속
→ 속도 개선 + 트래픽이 여러 노드로 자연 분산되는 효과
→ 한 노드가 DDoS로 죽어도, 사용자는 다른 근거리 정상 노드로 접속해 통신 유지
```

> 💡 정정: 수업노트의 *"bgt 프로토콜"* 은 **BGP(Border Gateway Protocol)** 의 오기. 빠른 경로로 네트워크 라우팅을 골라주는 게 BGP.

#### (4) Ingress / Egress 필터링 — 위조 출발지 IP를 원천 차단 (BCP38)

```
Ingress 필터링 = 들어오는 패킷의 출발지 IP가 "정상 대역인지" 검사
Egress  필터링 = 나가는 패킷의 출발지 IP가 "내부 대역이 맞는지" 검사 (위조 송출 방지)
→ ACL로 라우터/방화벽에 설정 (각 구간 #1·#2·#3에 적용)
→ 위조 IP(spoofing)를 못 쓰게 만들면, 응용계층(Real IP) 빼고
  대역폭·자원 고갈류의 위조 기반 공격을 대거 예방
```

> 🔗 출발지 IP 위조가 잦은 이유 = 같은 출발지로 단시간에 많이 보내면 금방 차단당하니까. ISP 구간/내 라우터에서 이걸 걸러주면 위조 자체가 무력화된다. 이게 **BCP38**(네트워크 위조 방지 권고)의 핵심.

#### (5) 점검 도구 — hping3

```
hping3 = 패킷을 직접 생성하는 도구. TCP·ICMP·UDP 등 다양한 패킷 제작 가능
→ TCP Flags를 임의 조합해 Flooding 트래픽을 생성, 네트워크 점검(부하 테스트)에 사용
주요 옵션:
  -c 1000          전송할 패킷 수
  -d 120           패킷 데이터 크기(byte)
  -S               SYN 플래그
  -w 64            TCP 윈도우 크기
  -p 80            목적지 포트
  --rand-source    출발지 IP를 랜덤 위조 (← UDP Flooding 캡처에서 본 그 "제각각 IP")
  --flood          가능한 최고 속도로 전송
```

⚠️ DoS 시뮬레이션도 가능한 도구라 **반드시 허가된 점검·실습 환경에서만** 사용. (`hping3 --help`로 옵션 확인)

### 봇넷 / C&C(C2) 구조 — DDoS의 "병력 동원" 체계

```
공격자 → C&C(Command & Control) 사이트 구축 (보통 탈취한 서버)
       → 악성코드로 다수의 좀비 PC(봇넷) 감염
       → C&C로 일괄 명령 하달 → 좀비들이 동시에 대상 타격 = DDoS
```

> 🔗 좀비 PC는 위 RAT·크랙 악성코드로 감염된 PC들이다. 그래서 **내 PC가 털리면 = 가해의 도구가 된다** — 출발지 IP에 내 IP가 박힌다. (이 문서 상단 "RAT / 크랙 악성코드" 참고) IPS가 C&C 통신·DDoS 가담을 아웃바운드에서 끊는 것도 같은 맥락. (Security-Solutions.md)

## 탐지 우회 (시그니처 난독화)

해커는 IPS/IDS의 시그니처에 안 걸리도록 공격 패턴을 변형한다.

```
IPS가 "위키" 문자열을 차단한다는 걸 알면
→ "위 키" 처럼 띄어쓰기로 변형
→ 또는 숫자·인코딩 코드로 변환해 난독화
→ 시그니처에 안 걸리게 우회
```

방어 측(보안 회사)이 하는 일:

```
→ 문자열 "조합" 능력으로 시그니처를 만든다
→ 오탐 없이 정확히 잡는 조합을 찾는 게 실력
→ 회사마다 조합 기술이 달라 성능 차이가 남 (Security-Solutions.md "시그니처 성능 차이")
```

> 그래서 벤더 기본 시그니처만으론 부족 → 우리 서비스 특화 공격은 **커스텀 시그니처**로 직접 잡아야 함.

## AD(Active Directory) 공격 — DC가 최종 목표인 이유 (6/25)

AD는 계정·권한·정책을 **중앙(DC)** 에서 관리한다(개념은 Security-Solutions.md "AD"). 중앙집중의 대가는 명확하다 — **DC 하나가 뚫리면 도메인 전체가 넘어간다.** 그래서 AD 공격은 "어떻게 DC까지 가서, 어떻게 전체 권한을 먹느냐"의 이야기.

### Pass-the-Hash — 비번 없이 해시로 횡이동

```
한 PC에서 계정 해시(NTLM) 탈취
 → 평문 비번을 몰라도 그 해시 자체를 인증에 재사용
 → 도메인 내 다른 서버로 옆걸음(lateral movement)
```
- Kerberos/NTLM이 "비번이 아니라 자격증명(해시·티켓)으로 인증"하는 구조를 역이용. (Security-Solutions.md "Kerberos = 티켓으로 인증"의 그림자)

### Golden Ticket — krbtgt 해시로 위조 티켓 발급

```
DC의 krbtgt 계정 해시 탈취  (← KDC가 TGT를 서명하는 마스터 키)
 → 공격자가 임의의 위조 TGT(Golden Ticket)를 직접 만듦
 → 도메인 내 모든 서비스에 무제한·장기 접근 (사실상 도메인 장악)
```
- TGT를 서명하는 키가 털리면 **인증 체계 자체를 위조**할 수 있다 → 그래서 **DC가 공격자의 최종 목표**.

> 🔗 **방어 흐름 연결:** ① DC로 가는 길목을 좁히고(NAC·세그먼트·관리자 권한 최소화), ② 명령어 감사(**snoopy/auditd**)로 횡이동·권한상승 흔적을 남기고(uid 실제/유효 불일치 = sudo·권한상승), ③ 인증 이상(비정상 시간·위치의 티켓 요청)을 **ESM/SIEM 상관분석**으로 잡는다. 단일 로그로는 안 보이고 여러 로그를 엮어야 보이는 전형 — "브루트포스 = 상관관계"와 같은 결.
