# 🛡️ 보안 솔루션 정리

## 방화벽 (Firewall)

내부와 외부 네트워크를 구분하고 불법 침입을 탐지·차단하는 시스템.
OSI 기준 L3/L4에서 동작. MAC 주소를 보유해 L2 인식은 가능하지만 L2 기능으로 동작하진 않음.

### 방화벽 종류

| 구분 | 설명 |
|---|---|
| 어플라이언스 방화벽 | 전용 하드웨어 장비 (Palo Alto, Fortinet 등) |
| 어플리케이션 방화벽 | 서버 내 소프트웨어 형태 (ufw, firewalld 등) |

### 세대별 방화벽

| 세대 | 레이어 | 하는 일 | 예시 |
|---|---|---|---|
| 1세대 패킷필터링 | L3~L4 | IP + 포트 차단 | iptables, UFW |
| 2세대 상태기반 | L4 | TCP 세션 상태 추적 | 대부분의 기본 F/W |
| 3세대 NGFW | L7 | 페이로드 내용까지 검사 | FortiGate, Palo Alto |

### 세대별 방화벽이 이렇게 발전한 이유 (흐름으로)

방화벽은 "이 문을 통과시킬 거냐 말 거냐"를 판단하는 **경계보안의 관문**이다. 시대에 맞춰 1 → 2 → 3세대로 발전했고, 현업 주력은 3세대(NGFW), 그 위에 차세대(NGFW+)까지 나와있다.

#### 1세대 (패킷 필터링) — IP + 포트만 본다

출발지/목적지 IP와 포트만 보고 통과 여부를 결정한다. 아직도 L3 장비로 방화벽 흉내를 내는 곳이 있는데, 여기서 핵심 질문이 하나 나온다.

> **"IP·포트로 거르는 순수 기능만 보면 L3 + ACL로도 되는 거 아냐?"**

순수 차단 기능만 보면 L3로도 가능은 하다. 그런데 **방화벽을 대체하진 못한다.** 이유는 **망 제어(망분리)를 못 하기 때문**.

```
L3 스위치 = 자기가 L3로서 망을 "분리·라우팅하는 주체"
            → 트렁크로 들어온 여러 IP 대역을 ACL로 "거를" 뿐
            → 분리된 망을 찾아 연결해주는 핵심 역할은 없음

ACL = 접근제어(통과/차단)지 망분리가 아니다   ← 핵심
```

> ACL은 검문소 앞에서 "너 통과 / 너 차단"만 외치는 경비.
> 방화벽은 망 자체를 나누고 어느 망으로 보낼지까지 책임지는 관문. 역할의 급이 다르다.

**1세대의 진짜 한계 — 응답을 받으려면 문을 양쪽으로 다 열어야 한다**

내가 `1.1.1.1:80`에 요청을 보내면 응답도 돌아와야 한다. 그런데 1세대는 세션 개념이 없어서 **역방향 ACL을 사람이 직접 한 줄 더 넣어야** 응답이 들어온다.

```
표 1.5  1세대 방화벽 ACL (양방향을 수동 등록해야 함)
src IP    dst IP    src Port  dst Port  control
any       1.1.1.1   any       80        permit   ← 나가는 요청
1.1.1.1   any       80        any       permit   ← 돌아오는 응답 (이걸 또 열어줘야 함)
```

여기서 문제: 응답을 받겠다고 **출발지 포트를 any로 열어두면**, 그 열린 문으로 무분별한 요청까지 같이 들어올 수 있다.

```
netstat 결과 (그림 1.11)
TCP  10.2.1.209:53598 → 203.104.160.12:443   ESTABLISHED
TCP  10.2.1.209:54069 → 52.1.176.134:443      CLOSE_WAIT
                 ↑                       ↑
        출발지 포트는 매번 동적(임의)     목적지 포트는 443 고정
```

- 목적지 포트는 정해져 있는데(443) **출발지 포트는 연결마다 임의로 정해진다.**
- 그 임의 포트가 정해지는 순간 = 세션이 연결된 것. 세션이 닫히면 그 임의 포트도 닫힌다.
- 네트워크의 원칙: **내가 따고 들어간 문으로 다시 들어와야 한다.** 1세대는 그 "내가 딴 문"을 자동으로 기억 못 하니까 어쩔 수 없이 출발지 포트를 any로 다 열어두는 것 → 위험.

#### 2세대 (Stateful Inspection) — 세션을 기억한다

1세대의 그 문제(역방향을 수동으로 열어야 함)를 자동화한 게 2세대다.

```
세션이 연결된 상태면
→ 그 세션으로 돌아오는 패킷(임의 출발지 포트로 정해진 값)은
→ 인바운드 정책을 따로 안 짜도 자동으로 허용
→ 세션이 닫히면 그 허용도 같이 사라짐 (휘발성)
```

> 가제트 형사 팔 비유: 내가 팔(세션)을 쭉 뻗어 문을 따고 나간다.
> 응답은 **내가 뻗은 그 팔을 타고만** 돌아온다. 팔을 거두면(세션 종료) 통로도 사라진다.
> → 내가 시작한 연결의 응답만 들어오고, 생판 남이 그 문으로 못 들어온다.

```
표 1.6 → 표 1.7  (Stateful이 자동 생성하는 ACL)
사람이 넣는 룰:   any → 1.1.1.1 : 80              permit
자동 생성되는 룰: 1.1.1.1 : 80 → [요청자] : 44512  permit
                                            ↑ 임의(ephemeral) 포트를 알아서 열어줌
```

이게 **Stateful Inspection**이고, "세션이 중간에 안 바뀐다"는 가정 하에 설계된 개념이다.

⚠️ 강사님 개인 홈 방화벽 설정에서 강조한 것 — **기본은 전부 Deny, 그리고 "양방향 체크"를 함부로 누르면 큰일난다.** 양방향을 통으로 열면 1세대의 무분별 유입 문제로 되돌아가는 셈. (아래 "방화벽 정책 철칙"의 단방향 원칙과 직결)

#### 3세대 (NGFW) — 페이로드 안의 내용까지 본다

L7까지 커버한다. IP·포트(봉투)만이 아니라 **페이로드(편지 내용) 안의 데이터 성질**까지 들여다본다.

- 하드웨어가 급발전하면서 "방화벽에 보안 솔루션들을 다 얹어도 되겠다"는 판단이 가능해짐 → 통신 내용·상태까지 보고 싶었던 것.
- 덕분에 **애플리케이션 단위 차단**이 가능해짐.

```
예) N드라이브만 차단하고 싶다
→ IP로 막으면? N드라이브 IP가 수시로 바뀌어서 정책이 꼬여버림
→ L7은 페이로드 안의 데이터 성질("이건 N드라이브 트래픽이다")을 보고 차단 → 깔끔
```

- 원천 데이터가 상단(방화벽)에서 대기하다가, 사용자가 잘못된 애플리케이션과 통신하려 하면 **선제 차단** 가능.
- 실제 NGFW(Palo Alto 등)는 `daum-cloud` / `daum-mail`처럼 **애플리케이션을 카테고리 단위로 식별**해 정책을 건다. (그림 1.12)
- 실습 때 **Squid 프록시** 같은 걸 올려서 이 L7 제어를 직접 해볼 예정.

> ⚠️ 음차 교정: 수업 중 "웹스키드/스키드 프록시" = **Squid 프록시**(대표적인 포워드·캐싱 프록시). 아웃바운드 화이트리스트 제어에 쓴다. → Architecture.md "방화벽 디자인 C/D코스" 참고.

> 한 줄 요약: 1세대 = 봉투의 주소·포트만 본다 → 2세대 = 누가 먼저 말 걸었는지(세션) 기억한다 → 3세대 = 봉투를 뜯어 내용까지 읽는다. **점점 더 깊이 본다.**

### 방화벽 레이어별 정리

방화벽은 어느 레이어에서 동작하느냐에 따라 "볼 수 있는 깊이"가 달라진다.

```
L2 방화벽 (TP모드)  → 브릿지로 끼어들어 투명하게 제어 (IP 없음)
L3 방화벽           → IP 기준 제어
L4 방화벽           → IP + 포트만 봄 = 접근제어만 가능
                     ⚠️ IP 변조해서 들어오면 못 잡음 (내용을 안 보니까)
L7 방화벽           → 패킷 안의 문자열(페이로드)까지 필터링 가능
                     pfSense = L7 UTM의 대표적인 예
```

> L4까지는 "누가 어느 문으로 들어오냐"만 본다. L7부터 "들고 온 짐 안에 뭐가 있냐"를 검사.
> 그래서 L4 방화벽은 출발지 IP를 위조한 공격을 걸러내지 못한다.

### 블랙리스트 vs 화이트리스트

| 방식 | 기본 상태 | 동작 방식 |
|---|---|---|
| 블랙리스트 | 열려있음 (Default Allow) | 차단할 IP/포트만 명시 |
| 화이트리스트 | 닫혀있음 (Default Deny) | 허용할 IP/포트만 명시 |

> 실무에서는 인바운드는 화이트리스트, 보안이 중요한 서버망은 양방향 모두 화이트리스트.

### 방화벽 정책 철칙

```
잘못된 방식: web01 ↔ SMB  양방향 허용 한 번에 체크
올바른 방식: web01 → SMB  출발지/목적지/포트 명시 (단방향)
            SMB → web01  별도 정책으로 명시 (단방향)
```

### 방화벽 정책 운영 실무 (금융권 기준)

방화벽 정책은 "그냥 열어주는 것"이 아니라 신청 → 검토 → 반영 프로세스를 거친다.

- 접속 대상이 많고(사람·시스템), 접속 방식도 제각각(원격 / HTTP / 특정 TCP 포트)이라 정책 수가 기하급수로 늘어난다.
- 그래서 보안팀은 신청 받는 시간대를 고정해서 운영한다. (예: 오전 10시 / 점심 후 13시 / 퇴근 전 17시)
- 금융회사는 보통 **2주 전 신청 ~ 3일 전 마감**으로 룰을 박아둔다.

> 검문소 운영시간 같은 것. 아무 때나 와서 문 열어달라고 하면 검문소가 마비됨.
> 신청 창구 시간을 정해두고, 마감일을 박아둬야 보안팀이 업무를 끌고 갈 수 있다.

> ⚠️ 정책을 "이럴 것 같으니 미리 열어두자"는 식으로 **예측해서 넣으면 안 됨**.
> 정확히 출발지 / 목적지 / 포트를 확인하고 문을 열어야 한다 (위 "방화벽 정책 철칙"과 직결).

징징대는 유관부서에 끌려다니며 "우리는 맨날 방화벽만 열어준다" 신세가 되지 않으려면,
신청 프로세스 자체를 잘 설계하는 게 보안팀 실력이다.

### 방화벽 룰 처리 순서 (Top-Down)

방화벽은 **무조건 맨 위 룰부터 아래로 내려오면서 읽고, 처음 매칭되는 룰에서 멈춘다(first match).** 그래서 룰을 어떤 순서로 쌓느냐가 곧 정책 설계다.

```
[최상단]  특정 차단룰(블랙) — 무조건 막아야 할 IP·패턴은 맨 위 (아래서 허용되기 전에 먼저 끊음)
   │
   │      외부 서비스 룰 그룹 — 외부→DMZ 인바운드처럼 트래픽 많은 룰
   │
   ▼
[하단]    내부끼리 통신 룰 — 내부→내부 등
```

- **트래픽 많은 룰일수록 위로** — 위부터 읽으니, 자주 매칭되는 룰이 위에 있어야 매번 끝까지 안 훑는다.
  → 룰이 수천 개면 순서가 곧 **성능(처리 속도)**. 트래픽량을 계산해서 배치 순서를 디자인해야 함.
- **특정 차단룰은 최상단** — 허용룰보다 아래 있으면 위에서 먼저 허용돼버려 차단이 무력화된다.
- UFW도 같은 원리: 좁은 대역 ALLOW를 넓은 대역 DENY보다 위에 둬야 함 (위 UFW 노트와 동일).

> 체크리스트를 위에서부터 읽다가 "예" 나오면 멈추는 것과 같다. 가장 흔한 항목·가장 중요한 금지 항목을 위에 둬야 효율도, 안전도 잡힌다.

### 방화벽 Deny 로그

방화벽은 L3에서 패킷을 직접 제어하는 장비라서, 차단된 트래픽은 **방화벽에서만** 보인다.

- L3에 묶여있는 게 아니라, L3에서 제어를 받아 **차단됐냐 / 안 됐냐**가 핵심
- 접근 통제에 걸리면 패킷이 못 넘어가고 방화벽에 **Deny 로그**가 찍힘
- 이 Deny 로그는 다른 장비에서 안 보이고 방화벽에서만 확인 가능

> DB 접근 권한 없는 사람이 DB에 붙으려 한 시도 = 통제 권한 위배 = 방화벽 Deny 로그로 추적.
> 방화벽은 패킷 제어 장비 → 접근 제어가 방화벽에서 일어남 → 그 결과(차단)도 방화벽 로그에 남는 것.

### UFW 기본 명령어

```bash
ufw status                              # 상태 확인
ufw status verbose                      # 상세 상태 확인
ufw status numbered                     # 번호 포함 룰 확인
ufw enable / disable                    # 방화벽 활성화/비활성화

ufw allow 포트/프로토콜                  # 허용
ufw deny 포트/프로토콜                   # 차단
ufw delete 번호                         # 룰 삭제

ufw allow from IP대역 to any port 번호  # 특정 IP 대역만 허용
ufw default deny incoming               # 인바운드 전체 차단
ufw default deny outgoing               # 아웃바운드 전체 차단
```

> ⚠️ 방화벽 룰은 위에서 아래 순서로 적용됨
> 좁은 대역 ALLOW를 넓은 대역 DENY보다 위에 둬야 함

## pfSense 실전 — NAT 포트포워딩(DNAT)과 Alias

랩에서 직접 박아본 두 가지. 방화벽 정책을 "이론"이 아니라 "운영"으로 만드는 핵심 도구다.

### NAT 포트포워딩 (DNAT) — 외부 포트를 내부 서버로 꽂는다

외부(또는 관리 PC)에서 방화벽의 **WAN 포트 하나**를 치면, pfSense가 내부 서버의 실제 IP·포트로 **목적지를 바꿔(Destination NAT)** 전달한다.

> 비유: pfSense WAN(192.168.1.82)이 건물 **안내데스크**. "82번 건물 80번 창구요" 하면 안내데스크가 안쪽 운영팀(10.10.10.10:80)으로 데려다준다. 방문객은 내부 주소를 몰라도 된다.

```
Firewall → NAT → Port Forward → Add
  Interface:     WAN
  Dest. Address: WAN address (예: 192.168.1.82)   ← "들어오는 입구" 기준
  Dest. Port:    외부에서 칠 포트 (80, 1010 …)
  Redirect IP:   실제 서버 IP (10.10.10.10 …)
  Redirect Port: 실제 서버 포트
```

| 외부 WAN 포트 | → 실제 목적지 | 용도 |
|---|---|---|
| 80 | 10.10.10.10:80 | 웹 서비스 |
| 1010 | 10.10.10.10:1010 | web SSH |
| 2020 | 10.10.20.10:2020 | webdev SSH |

> ⚠️ **Dest. Address는 "보낸 사람" IP가 아니라 방화벽의 WAN IP**(트래픽이 들어오는 입구)다. 관리 PC IP를 여기 넣으면 안 됨 → 출처를 거르려면 Source 쪽에 넣는다.
> NAT 룰을 만들면 그에 맞는 WAN **방화벽 룰이 자동 생성**(체크박스)된다 — NAT(주소 변환)와 Filter(통과 허용)는 별개 단계라, 포워딩만 해두고 허용룰을 빠뜨리면 막힌다.

> 🔗 SNAT(출발지 변환, 여러 내부를 공인 IP 하나로)와 짝 개념. DNAT=밖→안 들어오는 목적지를 바꿈, SNAT=안→밖 나가는 출발지를 바꿈. (SNAT가 출처를 뭉개 추적을 어렵게 하는 건 위 "센서 분산·음영지역" 참고)

### Alias — 반복되는 IP·포트를 별칭 하나로 묶는다

같은 관리 IP를 룰 여러 개에 박아두면, IP가 바뀔 때 **룰을 전부 손봐야** 한다. Alias로 묶으면 **한 곳만 고쳐 모든 룰에 반영**.

```
Firewall → Aliases → Add
  Name:        ADMIN_IP
  Type:        Host  (또는 Network / Port)
  Network/IP:  192.168.1.83
→ 이후 모든 룰의 Source/Dest에 IP 대신 ADMIN_IP 지정
```

> 🧠 **관리 포인트를 하나로 줄이는 게 방화벽 운영의 기본.** 프로그래밍에서 매직넘버를 상수 하나로 빼는 것과 같은 발상. 룰이 수천 개가 되는 실무에선 이게 곧 "유지보수 가능성".

```
실무 Alias 예
  ADMIN_HOSTS   : 관리자 IP 묶음 (여러 개)
  WEB_PORTS     : 80, 443
  INTERNAL_NETS : 내부 대역 전체
```

> ⚠️ 비유: 출입증 명단을 검문소 6곳에 따로 붙여놓으면 한 명 바뀔 때 6곳을 다 뛰어야 한다. Alias = 명단을 **중앙 한 장**으로 두고 검문소들이 그걸 참조하게 하는 것.

## L2 브릿지 방화벽 (TP모드 / Transparent Mode)

NAT 기능 없이 두 장비 사이에 투명하게 자리잡아 모든 트래픽을 제어.
IP 없이 배치하기 때문에 기존 네트워크 구조를 바꾸지 않고 끼워 넣을 수 있음.

```
일반 방화벽 (L4)          TP모드 방화벽 (L2 브릿지)
IP 있음                   IP 없음 (관리용만 있으면 됨)
네트워크 구조 바뀜          네트워크 구조 그대로
기존 서버 설정 변경 필요     기존 설정 하나도 안 바꿔도 됨
```

### 언제 쓰냐

- 1차 방화벽으로 외부 진입은 통제했는데 내부망 사이에 이중 보안을 추가하고 싶을 때
- 기존 IP 설정 건드리지 않고 DMZ → 내부망 길목에 투명하게 끼워 넣는 방식
- 실무에서는 FortiGate 같은 하드웨어 장비를 TP모드로 설정해서 사용

### 배치 위치

```
인터넷 → 1차 F/W (UFW, L4) → L3 라우터 → [TP F/W] → LB → L2 스위치 → 서버
```

> TP모드는 "배치 방식"이고, 안에서 하는 검사는 여전히 L4/L7.
> 배치 방식(어떻게 끼어있냐)과 검사 기능(뭘 보냐)은 별개 개념.

### TP모드(L2 브릿지)의 한계 — 1차 방화벽을 대체하진 못한다

투명하게 끼우는 게 장점이지만, 그만큼 못 하는 것도 명확하다. (그림 1.26 "L2 방화벽 구조")

```
구조가 간단함             → 브릿지로 끼우기만 하면 됨
IP가 없음 (관리용 mgt만)   → L3 라우팅·NAT를 못 함
장애 시 바이패스 필요      → 라인 위에 투명하게 끼어있어 죽으면 통신 단절 → 하드웨어 바이패스 필수
망 확장이 안 됨           → IP가 없으니 새 망을 만들어 붙이는 라우팅 확장 불가
내부 대역 망제어 안 됨     → 망을 분리·라우팅하는 주체가 아니라 내부망 사이 제어 불가
```

> 그래서 TP모드는 "1차 방화벽(L3/L4) 뒤에서 길목에 덧대는 이중 보안"이지, 1차 방화벽 자리를 대신하지 못한다.
> L3-only 방화벽이 망분리를 못 하는 것과 같은 결 — **망을 나누고 길을 정해주는 주체가 아니다.** (위 "1세대 ACL≠망분리"와 직결)

### 심층 방어 (Defense in Depth)

한 겹이 뚫려도 다음 겹이 막는 구조.

```
1차 F/W (UFW)    → 외부 진입 통제
TP 브릿지 F/W    → DMZ → 내부 길목 통제
VLAN + ACL       → VLAN 간 이동 통제
```

## IDS / IPS

### IDS (Intrusion Detection System) — 탐지만

- 패킷 감시 + 탐지 + 로그 (차단 없음)
- 특정 레이어 하나에 고정되지 않음 (L3~L7 동시 감시, 페이로드까지 읽음)
- 대표: Suricata, Snort

### IPS (Intrusion Prevention System) — 탐지 + 차단

- IDS에 차단 기능 추가 (인라인)
- 알려진 악성코드 / 공격 패턴을 시그니처로 차단
- DDos를 막는 기능도 포함 (L7)
- IPS 제조사들은 새로운 시그니처를 지속적으로 업데이트해서 엔진 강화

### 인라인(Inline) vs 미러링(Mirroring) — IDS를 라인에서 빼는 이유

```
IPS = 인라인 (라인 위에 걸쳐 있음, 트래픽이 반드시 통과)
      → 차단해야 하니까 길목에 직접 서 있어야 함

IDS = 미러링 (옆에서 복제된 데이터만 받아봄)
      → 로깅 장비라 라인 위에 둘 이유가 없음
      → 인라인에 넣으면 장애 시 라인 절체 = 다운타임 발생
      → 그래서 별도로 빼서 복제(미러링) 데이터로 분석
      → 넣었다 뺐다 자유로운 구조
```

> IPS는 검문소(차 세워서 검사), IDS는 길가의 CCTV(지나가는 거 찍어서 나중에 본다).
> CCTV가 고장 났다고 도로를 막진 않듯, IDS는 죽어도 서비스에 영향이 없어야 한다.

### 다 인라인에 욱여넣으면 안 되는 이유 (오탐 = 재앙)

```
모든 하드웨어 보안 장비를 인라인에 다 끼워넣으면?
→ 차단·제어를 전부 라인 위에서 직접 하는 구조
→ 오탐(false positive)이 한 번 나면 정상 트래픽까지 끊김 = 재앙
```

그래서 차단까지 하는 장비(방화벽·IPS)만 인라인에 두고, 탐지·로깅만 하는 장비(IDS)는 미러링으로 옆에서 복제만 받는다.

### TAP (Test Access Point) — 미러링 구성의 복제 장비

미러링 구성에서 트래픽을 복제해 IDS로 흘려주는 장비.

```
TAP = 바이패스처럼 트래픽 경로를 능동적으로 절체·개입하는 장치가 아님
      (본선은 그대로 흘려보내고 복제만 하는 수동 장비)

구성: 업링크(up) ─ 다운링크(down) 본선
      + 사이드에 복제(mirror) 인터페이스
      → 본선 트래픽은 그대로 흐르고, 사이드로 똑같이 한 부 더 떠서 IDS에 줌
```

> TAP은 도로 본선은 안 건드리고 옆에서 사진만 한 장 더 찍어 CCTV실(IDS)로 보내는 장치.
> 본선에 전기적으로 개입하지 않으니 TAP이 죽어도 통신은 멀쩡하다 → IDS를 라인 밖에 두는 미러링 구성과 짝이 되는 장비.

### IPS와 IDS를 같이 쓰는 이유

```
회사마다 시그니처 DB가 다름
→ IPS가 못 잡은 걸 IDS가 잡을 수 있음
→ 알약 이중 엔진과 같은 발상 (미탐 방지, 보강)
→ 비용 대비 가치를 따져서 도입 결정
```

### IPS vs IDS 운용 전략

| 구분 | 운용 방식 | 이유 |
|---|---|---|
| IPS | **보수적** | 걸리면 바로 차단 → 오탐 최소화된 시그니처만 사용 (정상 차단하면 사고) |
| IDS | **러프하게** | 탐지·로그만 → 놓치는 것 없게 넓게 탐지, 예외는 나중에 수동 처리 |

> IPS는 차단하니까 신중하게(오탐=장애), IDS는 차단 안 하니까 일단 다 잡아두고 본다.

### IPS 멀티 세그먼트

- IPS 3대를 따로 사는 게 아님 → **1대로 멀티 세그먼트 구성**
- 한 대의 IPS가 여러 구간을 동시에 감시
- 실제 현업에서도 이렇게 구성
- 방화벽이 망별로 분리해 둔 세그먼트 라인을 IPS에서도 세그먼트로 나눠 받으면 → **한 대로 서로 다른 망의 트래픽을 각각 정책 적용**
  - (포트 미러링으로 여러 망을 한 포트에 모아 IDS 센서로 받는 것과 같은 발상 → 아래 "트래픽 복제 — SPAN/포트 미러링" 참고)

### 시그니처란

- 악성코드의 패턴 정보 = 몽타주
- 보안 솔루션은 이 시그니처를 기반으로 패턴 매칭
- AI 발달로 시그니처를 우회하는 변종 악성코드 증가 → AI 기반 탐지로 전환 추세

### 시그니처 성능 차이 (외산 장비가 좋다는 이유)

```
회사마다 시그니처 "조합" 기술이 다름
→ 똑같은 악성코드도 어떻게 시그니처를 짰냐에 따라 성능 차이
→ 오탐이 많을 수도, 적을 수도 있음
→ 패킷 1개당 시그니처 수만 줄을 다 태워야 함 → 처리 속도(스루풋)에 영향

외산(이스라엘·미국 등 보안 강국) 장비가 좋다는 이유
→ 악성코드 분석가가 시그니처를 직접 만들고, 조합 기술이 앞서기 때문
```

### 커스텀 시그니처

```
벤더사 제공 시그니처 → 표준 공격류는 커버, 우리 서비스 특화 공격은 못 잡음
커스텀 시그니처      → 우리 서비스에 특화된 공격 패턴을 직접 만들어 장비에 투입
                     Suricata / Snort 룰로 직접 작성
```

**IPS GUI에서 커스텀 시그니처 넣는 모습 (그림 2.5)** — 룰이라고 거창한 게 아니라, 폼 몇 칸 채우는 일이다.

```
Custom Signature
 Name*              : Facebook                       ← 룰 이름(식별용)
 Protocol*          : TCP                            ← 어느 프로토콜에 적용
 Custom Rule*       : srcport:443; content:"www.facebook.com";   ← 핵심: 매칭 조건
                      (Format  →  Keyword:"value"; )             (페이로드에서 이 문자열 찾기)
 Severity*          : Major                          ← 위험도 등급
 Recommended Action : Drop Packet                    ← 걸리면 뭘 할지(여기선 패킷 폐기)
```

> 즉 "443 포트로 나가는 트래픽 페이로드에 `www.facebook.com`이 있으면 → Drop"이라는 한 줄 정책.
> L7 페이로드 안의 **문자열(content)**을 보고 잡는 게 핵심 — IP가 아니라 내용으로 식별한다(NGFW 애플리케이션 식별과 같은 결).

> 시그니처 우회(난독화) 공격과 한 쌍으로 봐야 함 → Attack-Defense.md "시그니처 우회" 참고.

## IDS 위치 선정과 센서 디자인

앞에서 IDS는 "탐지·로그만 하고 차단은 안 한다", 그래서 미러링으로 라인 밖에 뺀다고 정리했다. 6-8일엔 그 IDS를 **실제로 어디에 어떻게 꽂느냐(센서 디자인)**까지 내려갔다.

### IDS는 IPS와 "태생"이 다르다

강사님이 못 박은 것 — IPS가 놓여야 할 자리와 IDS가 놓여야 할 자리는 **본질적으로 다르다.**

```
IPS = 차단 장비 → 트래픽이 반드시 통과하는 길목(인라인)에 박혀야 함
IDS = 가시성·로깅 시스템 → "네트워크 흐름을 봐야 하는 곳"이면 어디든 적시적소에 꽂음
```

- IDS는 **네트워크 가시성을 확보하기 위한 로깅 시스템**이다. 차단이 목적이 아니라 "지금 무슨 트래픽이 흐르나"를 보는 게 목적.
- 그래서 필요하면 **정상 트래픽도 본다.** (악성만 보는 IPS와 다른 점)
- 네트워크 흐름의 패킷을 **저장**할 수 있다 → 사후 분석·포렌식 재료가 됨.
- 언제든 트래픽을 봐야 하는 구간에 넣었다 뺐다 할 수 있어야 한다 (미러링이라 가능 — 위 "인라인 vs 미러링" 참고).

> ⚠️ 강사님 지론: **IPS와 IDS는 서로 다른 벤더 걸 써야 한다.** 같은 회사 장비면 시그니처 베이스가 같아서, IPS가 못 잡은 걸 IDS도 똑같이 못 잡는다. 다른 시그니처 엔진을 겹쳐야 미탐이 줄어든다. (위 "IPS와 IDS를 같이 쓰는 이유"의 알약 이중 엔진 발상과 같은 결)

### 정탐/오탐 구분이 없다 = 그게 가시성이다

IDS는 "이건 악성, 저건 정상"을 칼같이 가르는 장비가 아니다. **탐지 시스템이라 정탐이든 오탐이든 일단 다 올라온다.**

```
ping을 친다 → 악성은 아니지만 IDS에는 탐지로 뜬다
ssh 접속    → 악성은 아니지만 탐지로 뜬다
근데 새벽 3시에, 평소 안 쓰던 곳에서 ssh가 들어왔다? → 이 "맥락"이 보이는 게 가시성
```

- 즉 IDS의 가치는 "악성만 골라준다"가 아니라 **흐름을 다 보여줘서, 보고 싶은 이상 징후를 입맛대로 뽑아낼 수 있다**는 데 있다.
- "ping이 악성이냐? 아니다. ssh가 악성이냐? 아니다. 근데 비정상적인 시간·상황의 ssh는?" — 이 판단을 할 수 있게 **재료(가시성)를 깔아주는 게 IDS**.

> CCTV 비유의 연장: IDS는 "범인만 찍는 카메라"가 아니라 "지나가는 걸 다 찍는 카메라"다. 다 찍혀 있으니까 나중에 "어, 쟤 왜 이 시간에 여기 있지?"를 잡아낼 수 있다. 다 안 찍혀 있으면 그 판단 자체가 불가능하다.

⚠️ **설치 직후엔 오탐이 쏟아진다** — Snorby(IDS 대시보드) 예시에서, 막 설치한 IDS가 HIGH 등급만 4천여 건, MEDIUM은 61만 건 넘게 잡혔다. 정탐/오탐 구분 없이 다 올라오니 당연한 결과. → 그래서 **룰 튜닝(필요한 시그니처만 남기고 오탐 대상 제거)**이 IDS 운영의 핵심 업무가 된다.

### 싱글 운용 vs 이중화

```
기본 입장(강사님): IDS는 싱글로 운용해도 괜찮다
→ 차단을 안 하니까, 다운돼도 서비스엔 영향 없음
→ 다운타임 동안엔 "로그 수집만 잠깐 멈추는" 것뿐

이중화가 필요해지는 경우
→ IDS 의존도가 높아져 관제(모니터링)의 중심 역할이 되면 이중화 고려
→ 관제 시스템과 연동될 땐 2중화 필요
⚠️ IDS를 클러스터(이중화)로 묶으면 → 두 노드가 같은 탐지 상태를 공유해야 하니
   공유 볼륨(shared volume)이 형성돼 있어야 함
```

> 가용성 관점에서 방화벽(1순위 이중화)과 정반대다. 방화벽은 죽으면 서비스가 끊기니 무조건 HA, IDS는 죽어도 서비스는 멀쩡하니 싱글로 시작 → 관제의 핵심이 되면 그때 이중화. (Architecture.md "왜 방화벽을 1순위로 이중화하나"와 대비)

## 패턴 매칭 기반 IDS — Snort / Suricata 룰 문법

6-5일 "다음에 깊게 볼 것"에 적어둔 **"커스텀 시그니처 직접 작성 — Snort/Suricata 룰 문법"이 바로 이 날 나왔다.** 시그니처가 "몽타주"라는 추상적 얘기에서, 실제 룰 한 줄이 어떻게 생겼나로 내려간 것.

### 룰 한 줄의 구조 분해

testmyids.com이라는 문자열을 탐지하는 시그니처:

```
alert  tcp  any any  ->  any any  (msg:"LOCAL-RULE Test for TestMyIDS.com"; content:"testmyids.com"; classtype:misc-activity; sid:100001; rev:1;)
  │     │    │  │      │   │  │     └──────────────────────── 옵션(괄호 안) ────────────────────────┘
  │     │    │  │      │   │  └ 목적지 포트
  │     │    │  │      │   └ 목적지 IP
  │     │    │  │      └ 방향(트래픽 흐름)
  │     │    │  └ 출발지 포트
  │     │    └ 출발지 IP
  │     └ 프로토콜
  └ 액션 (alert=탐지/로그, drop=차단)
```

| 자리 | 값 예시 | 의미 |
|---|---|---|
| 액션 | `alert` | 매칭되면 뭘 할지 (alert=탐지/로그, IPS면 drop=차단) |
| 프로토콜 | `tcp` | 어느 프로토콜을 볼지 |
| 출발지 | `any any` | 출발지 IP / 포트 (any=전부) |
| 방향 | `->` | 트래픽 방향 |
| 목적지 | `any any` | 목적지 IP / 포트 |
| `msg` | "..." | 탐지 시 로그에 찍을 설명 |
| `content` | "testmyids.com" | **핵심 — 페이로드에서 찾을 문자열/바이너리** |
| `sid` | 100001 | 시그니처 고유 ID |
| `rev` | 1 | 룰 버전(개정 번호) |

- 결국 **출발지/목적지를 정의하고, content에 특정 값(문자열·바이너리)을 넣어 그게 발견되면 탐지**되는 구조. (수업 한 줄 요약: "source와 destination 정의 후 특정 값 넣고 발견되면 탐지")
- 위 "커스텀 시그니처" GUI 폼(Name / Protocol / Custom Rule / Severity / Action)이 결국 이 룰 한 줄을 칸 나눠 채우는 것이었다 — 같은 얘기.

### content는 문자열만이 아니라 바이너리도 본다

```
content:"testmyids.com"        ← 문자열로 매칭
content:"|00 0C 23 45 53|"      ← 바이너리(헥스)로 매칭  (| | 사이가 헥스값)
```

⚠️ **바이너리 매칭의 함정 = 오탐** — 짧은 바이너리 조합(예: 00 0C 23 45 53)은 **전혀 다른 정상 패킷에도 우연히 똑같이 들어 있을 수 있다.** 그래서 같은 바이너리를 탐지하면 오탐 가능성이 높아진다. → 룰을 짤 때 content를 너무 짧고 흔하게 잡으면 정상 트래픽까지 긁어온다.

**방향(reverse connection)으로 악성을 잡는 룰도 있다:**

```
정방향(웹 공격류):  alert tcp $EXTERNAL_NET any -> $HOME_NET $HTTP_PORTS   (밖 → 안)
                     예) REvil Kaseya 랜섬웨어가 로그 지우는 http upload 탐지

역방향(트로이목마):  alert tcp $HOME_NET any -> $EXTERNAL_NET any            (안 → 밖)
                     예) Win.Trojan.Crysis 변종이 밖으로 나가는(outbound) C2 연결
```

> `$HOME_NET -> $EXTERNAL_NET`처럼 **방향이 거꾸로(내부에서 밖으로)** 나가는 게 핵심. 감염된 PC가 C2로 거꾸로 말 거는 게 트로이목마의 특징이라, "안에서 밖으로 나가는 이 패턴"을 시그니처로 잡는다. → Architecture.md "D코스(아웃바운드 차단)", Attack-Defense의 RAT/리버스 셸과 한 쌍.

### 패턴 매칭 방식의 장단점

| | 내용 |
|---|---|
| 장점 | 원하는 타깃을 **정확히** 탐지 (아는 공격엔 강함) |
| 단점 | 같은 문자열·바이너리에 대해 **오탐 가능성 높음** → 정밀한 룰 튜닝 필요 |
| 운영 | 시스템에 필요한 시그니처만 등록해 오탐 대상을 쳐냄 |

- **트래픽 학습 모드** — IDS를 학습 모드로 돌려 내부 네트워크 흐름을 파악할 수 있다. 이 학습 데이터로 보안정책을 세울 때 유용. (← 6-5일 정리한 애노멀리 디텍션의 "학습 모드"가 IDS에서 이렇게 쓰인다)
- **좀비(2차 검증) 탐지** — IDS 기반 기능에 가상화 시스템을 붙여 2차 검증을 하는 좀비 탐지 시스템 = **FireEye(파이어아이)**. (← 6-5일 Anti-APT 샌드박스 = 동적분석 VM과 같은 발상)
- **DB 감사 시스템** — IDS 계열로 DB 접근·쿼리를 감사하는 시스템도 있다.

> 결국 패턴 매칭은 **시그니처 기반의 실체**다. 6-5일 "시그니처 vs 애노멀리" 패러다임에서 시그니처 쪽이 현장에선 이렇게 룰 한 줄로 구현되고, 그 한계(오탐·변종 못 잡음)를 학습 모드와 FireEye류 2차 검증으로 메우는 흐름.

### Snort 룰셋 소스와 ET Open의 한계 (6-15일 실습)

pfSense Snort에서 고를 수 있는 룰 소스:

```
Snort Subscriber(VRT / Cisco Talos)  : 최신, 유료(Oinkcode 필요)
Snort GPLv2 Community Rules          : 무료
Emerging Threats Open (ET Open)      : 무료  ← 랩에서 사용. SID가 2xxxxxx대
ET Pro                               : 유료
```

⚠️ **ET Open엔 `' OR '1'='1` 같은 범용 SQLi 패턴이 없다** — 오탐이 너무 많아 **의도적으로 빠져 있고**, 알려진 CVE·특정 제품 기반 시그니처 위주다. 그래서 범용 SQLi를 잡으려면 `content:"pw="; ... content:"OR"; distance:0; within:50;`처럼 **커스텀 룰을 직접 짜야** 한다.

> 💡 **IDS ≠ WAF (역할 분리).** "왜 IDS만으론 SQLi가 안 잡히지?" → 범용 웹 공격(SQLi/XSS)은 **WAF(ModSecurity + OWASP CRS)** 영역이기 때문. IDS는 네트워크 전반의 알려진 위협을 넓게 탐지, WAF는 HTTP 웹 공격을 정밀 차단. (아래 "WAF는 IPS에서 갈라져 나온 것"과 한 쌍)

> ⚠️ **threshold가 재현을 막는다** — 룰에 `track by_src, count 1, seconds 3600`이면 한 소스에서 1시간에 1번만 alert, `count 5, seconds 60`이면 60초 안에 5번 이상이어야 alert. "한 번 뜨고 다시 안 뜬다 = 룰이 죽은 게 아니라 threshold." 위 애노멀리 "Threshold" 개념이 IDS 룰에 그대로 박혀 있다.

## Snort를 IDS → IPS로 전환 (pfSense 인라인 차단) — 6-16일

6-15일엔 Snort로 **탐지(alert)**만 했다. 6-16일은 그 Snort를 **차단(drop)**까지 하는 IPS로 끌어올린 날. pfSense Snort에서 설정 두 개를 만지면 IDS가 IPS가 된다.

### ① "Block Offenders" — alert를 일으킨 호스트를 자동 차단

Snort 인터페이스 설정(WAN)에 **Block Offenders** 체크박스가 있다. 설명문 그대로:

> *"Checking this option will automatically block hosts that generate a Snort alert."*
> = 이 옵션을 켜면 **alert를 발생시킨 호스트를 자동으로 차단**한다. (기본은 꺼짐 / Default Not Checked)

이거 하나로 Snort의 성격이 바뀐다 — 어제까지 "보고만 있던(IDS)" 장비가 "걸리면 막는(IPS)" 장비가 된다. (= 위 "IDS=탐지만 / IPS=탐지+차단"이 GUI 체크박스 하나로 갈리는 지점)

차단 방식은 두 가지 모드로 갈린다 — 이게 핵심:

```
Legacy Mode (구식, libpcap 복제 기반)
  → Snort는 패킷 "복제본"을 보고 탐지 (라인 밖에서 봄)
  → alert가 뜨면 그 IP를 pf 차단테이블(snort2c)에 등록 → 방화벽(pf)이 그 IP의 "이후" 트래픽을 끊음
  ⚠️ 방아쇠가 된 그 패킷 자체는 이미 지나간 뒤다 (사후 차단)
  → "Which IP to Block: SRC / DST / BOTH" 로 차단 대상 방향 선택

Inline Mode (netmap 기반, 진짜 IPS)
  → Snort가 패킷 경로 위(인라인)에 직접 끼어 실시간으로 본다
  → drop 룰에 걸리면 그 패킷 자체를 그 자리에서 폐기 (실시간 차단)
  → 트래픽이 반드시 Snort를 통과 → 통과 못 하면 못 들어옴
```

> 비유: Legacy = CCTV로 보다가 수상한 차 번호판을 정문 경비한테 무전("저 차 앞으론 막아") — 그 차는 이미 들어왔고 다음부터 막힘. Inline = 검문소 바리케이드를 직접 들고 서서 그 차를 그 자리에서 세움.

> 🔗 = 위 "인라인 vs 미러링"의 실제 구현. Legacy=미러링(복제본 보고 사후 차단), Inline=인라인(라인 위 실시간 차단). 어제 IDS로 쓰던 걸 오늘 인라인 IPS로 바꾼 것.

### ② Hardware Checksum Offloading을 "꺼야" 인라인이 돈다 (System → Advanced → Networking)

인라인 모드를 켜기 전에 **Hardware Checksum Offloading을 비활성화**(체크박스 ON = "Disable hardware checksum offload")해야 한다. 왜 이걸 끄라는지가 노트의 물음표였는데, 두 겹으로 이해하면 된다.

**무엇이냐 — 체크섬 계산을 누가 하느냐**
```
체크섬(checksum) = 패킷이 운반 중 깨졌는지 확인하는 오류검출 값
오프로딩(offloading) = 그 계산을 CPU 대신 "NIC(랜카드) 하드웨어"가 대신 해주는 것
  → 평소엔 CPU 부하를 덜어주는 좋은 기능
```

**왜 인라인 IPS·VM에선 꺼야 하나**
```
오프로딩이 켜져 있으면
 → OS는 체크섬을 비워둔 채(0 또는 placeholder) NIC에 넘김 ("계산은 네가 해")
 → NIC이 "랜선으로 나가기 직전"에 체크섬을 채움

근데 인라인 Snort(netmap)는 그 "나가기 직전"보다 앞에서 패킷을 가로챈다
 → Snort가 보는 패킷의 체크섬이 아직 비어있음/틀림
 → "체크섬 깨진 패킷"으로 오인 → 검사가 꼬이거나 패킷 재주입이 깨짐

오프로딩을 끄면
 → CPU가 패킷을 NIC에 넘기기 전에 체크섬을 정확히 계산해 채움
 → Snort가 "완성된 정상 패킷"을 보게 됨 → 인라인 검사·차단이 안정적으로 동작
```

⚠️ **특히 VirtualBox/VMware 같은 가상 NIC에서 더 문제** — 가상 환경은 실제 랜선을 안 타고 호스트가 체크섬을 대신 처리하는 구조라, 게스트(pfSense) 안에서 보면 체크섬이 비어있는 패킷이 더 흔하다. 그래서 **VM 위 pfSense + 인라인 IPS는 체크섬 오프로딩 끄기가 거의 필수.** (우리 랩이 VirtualBox라 정확히 이 케이스)

> 비유: 택배 무게(체크섬)를 "출고 직전 문 앞에서" 재서 송장에 찍는 시스템인데, 검표원(Snort)이 그 문 앞보다 안쪽에 서 있으면 무게칸이 빈 송장을 보고 "불량!" 하며 반려해버린다. 그래서 무게를 "포장 단계(CPU)"에서 미리 찍게 바꾸는 것.

> ⚠️ 같은 이유로 인라인에선 TSO/LRO(세그먼트 분할·병합 오프로딩)도 끈다 — netmap이 원본 패킷을 그대로 봐야 하니까. 체크섬만의 문제가 아니라 "오프로딩 = NIC이 패킷을 뒤늦게 손대는 것 전반"이 인라인 검사와 충돌.

## IDS/IPS 룰 튜닝 — Suppress · Pass List · SID 관리 (6-16일)

6-8일 노트에 "설치 직후 오탐이 쏟아진다 → **룰 튜닝이 IDS 운영의 핵심**"이라 적어뒀다. 그 튜닝의 실제 도구 3종이 이 날 나왔다. **차단(IPS)을 켜는 순간 오탐은 곧 장애**(정상 호스트가 끊김)라, 튜닝은 선택이 아니라 필수가 된다.

### SID 네임스페이스 — 왜 커스텀은 3000000번부터냐

```
시그니처 ID(SID)는 겹치면 안 된다 (중복이면 에러 → 룰 로드 실패)
그래서 진영별로 번호대가 나뉘어 있다:

  ~999,999            : Snort VRT/Talos (공식 룰)
  2,000,000~2,999,999 : ET(Emerging Threats) 룰   ← 어제 쓴 ET Open이 2xxxxxx대
  3,000,000~          : 우리가 직접 만드는 커스텀(local) 룰은 이 위로
```

- 즉 **"남이 쓰는 번호대를 피해서" 안 겹치는 빈 대역(3000000+)에 커스텀을 박는다.** (어제 커스텀 SQLi가 sid:9000001이었던 것도 같은 발상 — 안 겹치는 높은 번호면 됨)
- Snort 내부엔 이 SID들이 정렬돼 들어있고, 우리는 그 뒤 빈 번호를 쓰는 것.

### Suppress(억제) — "이 알람만 조용히 시켜라"

오탐이거나 너무 시끄러운 alert를 **로그에서만 안 뜨게** 막는다(룰 자체는 살아있음). pfSense는 alert 옆 ➕ 버튼으로 자동 등록해준다.

```
두 방향 (노트 한 줄 요약: "ip가 계속 들어온다→ip를 끈다 / sid로 너무 들어온다→sid를 끈다")

① 특정 IP만 억제 (track by_src / by_dst + ip)
   suppress gen_id 1, sig_id 2404300, track by_dst   ← 이 SID인데 이 방향/IP면 알람 X
   → "이 IP는 정상인데 자꾸 이 룰에 걸린다" → 그 IP에 한해 알람만 끔

② SID 통째로 억제
   suppress gen_id 1, sig_id 2404300                 ← 이 룰은 IP 불문 알람 X
   → "이 룰이 오탐을 너무 뱉는다" → 룰 단위로 알람 끔
```

> ⚠️ Suppress는 **"차단을 푸는 게 아니라 로그를 끄는 것"** 에 가깝다 — gen_id 1은 Snort 기본 룰엔진을 뜻하고, sig_id가 그 룰 번호. "알람 노이즈"를 줄여 진짜 봐야 할 걸 보이게 하는 작업이다.

### Pass List(통과 목록) — "이 IP는 절대 차단하지 마라"

Block Offenders를 켜면 오탐 한 번에 정상 서버·관리 PC가 차단당할 수 있다. 그걸 막는 안전장치가 Pass List — **여기 등재된 IP는 alert가 떠도 절대 차단(block) 안 한다.**

```
많이 쓰는 PC·서버·게이트웨이·DNS·관리자 IP → Pass List에 등재
 → "이 핵심 노드는 무슨 일이 있어도 끊기면 안 된다"를 보장
```

⚠️ **여기서도 IP를 직접 박는 건 하수 — 전부 Alias로.** Pass List 대상이 바뀔 때마다 룰을 손대는 대신, `ADMIN_HOSTS` 같은 Alias 하나만 고치면 끝. (위 "Alias — 반복되는 IP를 별칭으로" / 어제 "ip로 치는 건 하수"가 여기서 또 반복)

> 🔗 정리: **Suppress = 알람을 끈다(로그 노이즈 제거), Pass List = 차단을 면제한다(가용성 보호).** 목적이 다르다. 시끄러운 룰은 Suppress, 절대 끊기면 안 되는 호스트는 Pass List.

> 비유: Suppress = 특정 감지기의 알람음을 음소거. Pass List = VIP 출입증(검색대에서 걸려도 통과). 음소거(알람만 끔)와 면제(차단 안 함)는 다른 일.

### 커스텀 drop 시그니처 — 문자열을 왜 hex/hash로 넣나 (SKT 침해 공문 룰 사례)

웹서버에 로그인하는 admin 계정 패턴으로 drop 룰을 만들어봤다. 그 과정에서 강사님이 짚은 핵심 — **content를 그냥 문자열로 넣으면 잘 안 맞는다.**

```
문제: 문자열은 "문자 그대로" 해석되지 않는 경우가 많다
  → 인코딩(URL/Base64)·이스케이프·대소문자·비출력(non-printable) 바이트 등으로
    같은 "글자"라도 실제 바이트가 달라짐 → 평문 content 매칭이 빗나감
```

그래서 실무 룰은 두 가지로 넣는다:

| 넣는 방식 | 형태 | 무엇을 잡나 | 강점 / 약점 |
|---|---|---|---|
| **헥스(hex)** | `content:"\|68 70 61 73 6D 6D 6C 64\|"` | 바이트를 **있는 그대로** 매칭 | 인코딩·비출력 바이트 무관하게 정확 / 짧으면 오탐 |
| **해시(hash)** | `file_sha256; content:"\|c7f6…b5d4\|"` | **파일 자체**를 지문(해시)으로 매칭 | 이름 바뀌어도 잡음 / 1바이트만 바뀌어도 못 잡음(변종 회피) |

- 위 hex `68 70 61 73 6D 6D 6C 64` 를 ASCII로 풀면 `hpasmmld` — **문자열을 바이트로 박아** 인코딩에 안 흔들리게 한 것.
- SHA256·MD5 룰은 **알려진 악성파일의 해시**를 통째로 박아 그 파일이 흐르면 drop = 파일 단위 탐지.
- 이게 **SKT USIM 침해사고 후 공유된 공식 탐지룰(공문)** 형태였다 — `drop TCP $HOME_NET any -> $EXTERNAL_NET $HTTP_PORTS` 로 **안→밖(outbound)** 방향. 감염 단말이 밖으로 데이터를 빼거나 C2로 나가는 걸 잡는 구조.

> 🔗 = 위 "content는 바이너리(헥스)도 본다"의 실전. 그리고 방향이 `$HOME_NET → $EXTERNAL_NET`인 건 "역방향(트로이목마) outbound 룰"의 실제 사례(감염 PC가 밖으로 거꾸로 통신). 짧은 hex는 오탐 난다는 경고도 그대로 유효 — 그래서 해시(긴 지문)를 같이 쓴다.

> ⚠️ **해시 기반의 한계 = 변종에 무력.** 공격자가 파일을 1바이트만 바꿔도 해시가 통째로 달라져 안 잡힌다(폴리모픽). 그래서 해시 룰은 "알려진 그 검체"에만 강하고, 변종엔 행위기반(EDR·애노멀리)이 필요. (시그니처 vs 애노멀리 한 쌍)

## NIDS vs HIDS — IDS를 어디에 설치하나

IDS는 **어디에 설치하느냐**에 따라 두 갈래로 나뉜다.

| | NIDS (Network IDS) | HIDS (Host-based IDS) |
|---|---|---|
| 위치 | 네트워크에 설치 (미러링으로 트래픽 수집) | 각 호스트(서버)에 에이전트로 설치 |
| 장점 | 모든 호스트에 개별 설치 안 해도 됨, **네트워크 전체를 한 군데서 분석** | 호스트별 상세 분석 + **사용자 단위 분석**까지 가능 |
| 단점 | IDS를 **경유한 공격만** 보임 → 서버 로컬에서 침해당한 건은 모름 | 개별 설치·관리 부담 + 네트워크 패킷은 못 봄 + ⚠️ **호스트가 뚫리면 해커가 IDS를 꺼버릴 수 있음** |

> NIDS = 길목에 세운 검문소 카메라(전체 흐름을 한눈에, 대신 골목 안에서 벌어진 일은 모름).
> HIDS = 집집마다 단 개인 CCTV(집 안까지 상세히 보지만, 도둑이 들어와 카메라를 꺼버리면 끝).

⚠️ **규모가 발목을 잡는다** — SKT처럼 서버가 4만 대 넘는 곳에서 HIDS를 깐다? 4만 대에 전부 에이전트를 설치·관리·업데이트해야 한다는 뜻. 이게 현실적이냐를 따져야 하는 영역. → 대규모는 NIDS로 네트워크를 통으로 보고, 정말 중요한 핵심 서버에만 HIDS를 선별 적용하는 식으로 절충한다.

> FIMS 연결: Wazuh는 각 서버에 에이전트를 깔아 로그를 수집했으니 **HIDS 성격**(호스트 기반)에 가깝다. 거기에 네트워크 트래픽을 보는 Suricata(NIDS 성격)가 더해져 둘이 상호 보완했던 구조.

## 트래픽 복제 — SPAN / 포트 미러링

IDS(미러링 구성)에 트래픽을 어떻게 떠다 주느냐의 실제 방법. 앞의 TAP이 "별도 물리 복제 장비"라면, 이건 **스위치 자체 기능으로 복제**하는 방식이다.

### 복제 방식 3가지 이름

```
시스코            → SPAN (Switched Port Analyzer)
델 / 주니퍼        → 포트 미러링 (Port Mirroring)
크롬캐스트 / 애플TV → 화면 미러링  (← 같은 "미러링"이라도 이건 화면 복제, 개념만 같음)
```

- 이름만 다르지 **"한 포트로 흐르는 트래픽을 다른 포트로 똑같이 한 부 더 떠준다"**는 개념은 같다.
- 스위치가 기본 제공하는 기능이라, **구성 디자인만 정해지면 작업 난이도는 낮다** — 관리자 모드에서 설정, 다운타임 없이 무중단으로 켤 수 있다.

### 포트 미러링 설정이 실제로 어떻게 생겼나

매니지먼트(관리형) 스위치의 포트 미러링 설정 표:

```
Source Port   Destination Port   Session Mode   Direction
   g3              g6               Enable        Tx and Rx
   g4              g6               Enable        Tx and Rx
   g5              g6               Enable        Tx and Rx
   g7              g6               Enable        Tx and Rx
```

- **출발지 포트(g3·g4·g5·g7)로 흐르는 트래픽을 목적지 포트 g6로 미러를 뜬다.** 그 g6에 IDS가 물려 있다.
- 즉 여러 망의 트래픽을 g6 하나로 모아 IDS에 던지는 구조 → IDS 입장에선 여러 구간을 한 센서로 본다.

**Direction의 Tx / Rx**
```
TX = Transmit → 데이터(신호)를 보내는 역할
RX = Receive  → 데이터(신호)를 받는 역할
Tx and Rx     → 양방향(보내고 받는 것) 다 복제
```

### 센서(Sensor)

```
IDS 모듈이 트래픽을 수집하는 포트 = 센서(Sensor)
→ 센서는 계속 늘릴 수 있음 (포트를 더 붙여 더 많은 구간을 감시)
```

> 포트 미러링으로 g6에 트래픽을 모아주면, IDS는 그 g6 포트를 "센서"로 삼아 빨아들인다. 감시할 구간이 늘면 센서(수집 포트)를 추가하면 된다 → IPS 멀티 세그먼트(1대로 여러 구간)와 같은 결의 확장 방식.

### TAP vs 스위치 미러링 — 트레이드오프

```
스위치 포트 미러링 → 스위치가 복제 일을 더 하니 네트워크에 영향이 "있긴" 하다
TAP 방식           → 본선에 전기적으로 개입 안 함 → 네트워크 영향이 거의 없음
                    ⚠️ 대신 TAP 장비를 라인에 끼우는 순간(설치)엔 단선이 한 번 발생
```

> "영향 거의 없는 TAP" vs "스위치 기능으로 간편한 미러링" 두 카드를 요건에 맞게 골라 배치하는 게 핵심. 어느 쪽이든 IDS는 본선 밖에서 복제본만 받는다는 큰 그림은 동일. (위 "TAP" 참고)

## 센서 분산 · TAP 심화 — 미러링의 대가와 음영지역 (6-9일)

6-8일에 "TAP vs 스위치 미러링" 트레이드오프를 짚었다면, 6-9일엔 **왜 결국 TAP으로 가게 되는지**, 그리고 **미러링·TAP으로도 못 보는 음영지역**까지 내려갔다.

### 포트 미러링의 대가 — 발열·성능 저하

```
매니지먼트(관리형) 스위치에 포트 미러를 떠놓으면
→ 복제 일을 더 하니 열이 발생 → 성능 저하
   (모든 전자기기가 동일 — 일을 많이 시키면 열나고 느려진다)

많은 서버·자원이 한 포트(센서) 하나로만 들어오면 그 포트가 병목
→ 인터페이스를 그룹으로 묶어 센서를 늘려 트래픽을 분산시킬 수 있음
```

> 이게 6-8일 "센서는 계속 늘릴 수 있다"의 실제 이유다. 한 센서에 다 몰면 발열·병목 → 여러 인터페이스를 그룹으로 묶어 나눠 받는다.

### 그래도 기존 트래픽에 영향이 간다 → TAP의 존재 이유

```
센서를 늘려 분산해도, 결국 스위치 자체가 복제하는 구조라
기존(본선) 트래픽에 영향을 줄 수밖에 없다
→ rx / tx에 에러를 유발할 수 있음

그래서 나온 게 TAP 장비 (본선에 전기적으로 개입 안 함)
⚠️ 단, TAP을 라인에 끼우는 순간엔 단선(다운타임) 발생 — 선을 끊고 사이에 넣어야 하니까
```

- 실무엔 소형 테스트형 TAP 장비부터 있다 (라이트한 구간 검증용).
- 정리: **스팬(스위치 기능)은 간편하지만 본선에 영향 / TAP은 본선 무영향이지만 설치 시 단선** — 이 둘의 차이를 기억해야 한다. (6-8일 트레이드오프의 "왜"가 이것)

### TAP은 "길목(chokepoint)"이 있어야 한다 — 무선 음영지역

TAP이든 스팬이든 **트래픽이 반드시 지나가는 길목**이 있어야 복제를 뜬다. 길목이 없으면 못 본다.

```
유선: 포트 1·2번에 미러 맵핑 → IDS가 잘 감시함
무선: 공유기에 무선으로 직접 붙은 장치 → 그 트래픽은 길목을 안 거쳐 → IDS가 감지 못 함 (음영지역)
```

> CCTV(IDS) 비유 연장: 골목(길목)에 카메라를 달아야 찍힌다. 무선은 공중으로 바로 날아가버려 그 골목을 안 지나간다 → 사각지대.

### 무선을 억지로 보려면 ISP→TAP→WAN → 근데 NAT가 출처를 지운다

```
한 방법: ISP ─ TAP ─ WAN 형태로, WAN 들어가기 직전에서 복제해 IDS에 넣음
→ 무선까지 포함된 전체 트래픽을 일단 볼 수는 있음
⚠️ 그러나 WAN으로 나갈 때 NAT(SNAT) 처리가 되므로
   "어느 내부 서버·단말에서 발생한 패킷인지" 100% 추적은 안 됨
```

여기서 SNAT 개념이 박힌다 — Outside Interface가 SNAT로 설정돼 있으면, 탭 센서에 복제된 패킷의 출발지 IP는 전부 그 **공인 IP 하나로 보인다.** 내부 출처가 뭉개진다. (Architecture.md "SNAT — 하나로 묶으면 추적이 안 된다"와 정확히 같은 문제)

**강사님 해법 (교재는 무선 결함을 언급만 하고 못 풀었음):**
```
라우터에서 무선 기능을 떼어내고
→ 무선장치를 별도 L3로 뺀 뒤 거기에 연결
→ 라우터 ↔ L3 사이(이제 유선 길목이 생김)에 TAP을 둬서 무선 트래픽까지 보게 함
```

> 핵심: **음영지역을 없애려면 "길목을 인위적으로 만들어준다."** 무선을 L3 뒤로 빼서 유선 구간을 만들고, 거기서 복제를 뜨는 것.

### 결론 — 적절한 위치에 잘 놓는 게 전부

복제 방식 선택은 정답이 정해진 게 아니라 **구간 상황으로 판단하는 의사결정**이다.

```
스위치 캐파(용량)를 본다
 └ 트래픽이 많아지면 → TAP 장비를 고려 (스위치 부담 회피)
     └ 근데 그 구간이 라이트하면 → 굳이 TAP 말고 스팬(스위치 미러링)으로
다만 어느 쪽이든 "길목에 있어야 한다"는 전제는 동일
```

### 음영지역 커버 — 센서를 더 박는다

```
네트워크엔 반드시 음영지역(보이지 않는 구간)이 생긴다
→ 그 구간에 센서를 추가해서 TAP 또는 스팬으로 커버
```

> 6-8일 "센서는 늘릴 수 있다"가 여기서 음영지역 커버 수단으로 다시 쓰인다. 한 번에 다 못 보면, 안 보이는 곳에 센서를 더 박아 시야를 넓힌다.

⚠️ **보안담당자에게 진짜 필요한 능력** — 이 모든 판단(어디에 음영이 있고, 길목이 어디고, 스팬이냐 탭이냐)은 **회사 내부의 서비스 흐름과 네트워크 구조를 정확히 알아야** 나온다. 보안 영역만 잘해선 안 되고, 실무에서 일어나는 문제에 대해 **정확한 요청(스펙)을 줄 수 있어야** 한다.

## UTM (Unified Threat Management)

방화벽 + IDS/IPS + 안티바이러스 + DLP를 장비 하나에 통합한 것.

```
스타트업    → pfSense (무료 소프트웨어 UTM)
중소기업    → FortiGate 중저가 모델 (100~300만원)
중견기업    → FortiGate 고급 / Palo Alto
대기업/금융  → Palo Alto + Cisco 풀스택 + 물리 망분리
```

> FIMS에서 어플라이언스 VM에 Suricata + Wazuh 올린 것 = 소프트웨어 UTM

### 오픈소스 보안 솔루션 — pfSense만 있는 게 아니다

```
pfSense     → BSD 기반 오픈소스 방화벽/UTM (랩에서 사용 중)
OPNsense    → pfSense에서 fork된 오픈소스 방화벽 — UI·업데이트가 더 활발
PacketFence → 오픈소스 NAC. 인증 안 된 단말 격리·차단 (위 NAC 항목의 오픈소스 구현)
ClamAV      → 오픈소스 백신 엔진(안티바이러스)
```

> 💡 "방화벽=상용(FortiGate/Palo Alto)"만 떠올리기 쉽지만, **오픈소스로도 방화벽(pfSense/OPNsense)+NAC(PacketFence)+백신(ClamAV)** 스택을 다 꾸릴 수 있다. 돈 없는 스타트업이 UTM을 세울 때 쓰는 조합.

### 다운로드 무결성 검증 — 해시 대조 습관

> 비유: 택배 받을 때 **송장 번호(공식 사이트 게시 해시)** 와 **박스에 붙은 번호(받은 파일 해시)** 가 같은지 맞춰보는 것. 다르면 운송 중 바꿔치기된 것.

- 보안장비 ISO·설치 파일은 **공식/검증된 사이트**에서 받고, 받은 뒤 **해시값을 게시값과 대조**한다 = **무결성(Integrity) 검증**. 중간자 변조·바꿔치기를 거른다.

```bash
sha256sum pfSense.iso     # Linux
md5sum pfSense.iso        # 공식이 MD5만 주면 MD5로 대조
```
```powershell
Get-FileHash .\pfSense.iso -Algorithm SHA256   # Windows
```

> ⚠️ **MD5는 충돌(collision) 위험**이 알려져 보안 용도로는 부적합. 선택지가 있으면 **SHA-256 이상**을 쓴다.

## WAF (Web Application Firewall)

- 웹 공격 특화 악성코드 방지 장치 (HTTP 트래픽 전용)
- SQL Injection, XSS 등 웹 레벨 공격 탐지·차단
- 위험한 URL 접근 시 차단

### WAF는 IPS에서 갈라져 나온 것

```
IPS = 모든 프로토콜 대상 (HTTP 포함 전체)
WAF = HTTP 전용 전문 장비

→ IPS가 먼저 나왔고, HTTP 트래픽이 폭증하면서 WAF가 따로 등장
→ 웹 서버만 있으면 WAF 하나로 충분
→ 소켓 서버(메신저 등)가 같이 있으면 IPS도 필요
```

> IPS와 WAF가 같이 있어도 이상한 게 아님. 태생이 다르고 담당 트래픽이 다르다.
> "전체 트래픽 검문(IPS) + 웹 전용 정밀 검문(WAF)" 이중 구조.

### 차단 페이지를 안 띄우는 추세

- 예전엔 차단 시 "접근 차단되었습니다" 페이지 노출
- 요즘은 차단 페이지를 안 보여줌 → **WAF의 존재와 벤더사가 노출되기 때문**
- 그냥 빙글빙글 돌다가 일정 시간 후 풀리게 함 (공격자에게 정보를 안 줌)

## 웹 프록시 (포워드 프록시) — Squid / SquidGuard

WAF·IPS가 **밖에서 들어오는(인바운드)** 웹 공격을 본다면, **포워드 프록시는 안에서 밖으로 나가는(아웃바운드)** 트래픽을 URL 단위로 통제한다. 위 NGFW 항목에서 "실습 때 Squid 올려 L7 제어 해볼 예정"이라 적어둔 게 이것. (랩 실습: `../Lab/2026-06-15-웹프록시·Snort.md`)

> 비유: 방화벽이 "나가도 되냐"만 보는 정문이면, 포워드 프록시는 정문 앞 **외출 신고대** — "어느 가게(URL)에 가는데?"를 묻고 블랙리스트면 막는다. 목적지 명단을 신고대 한 곳에서만 관리하니 정문 정책을 매번 안 고쳐도 된다.

### 왜 쓰나 (보안 측면 3가지)

```
1. 악성 사이트 사전 차단        — 유해 URL로 나가는 걸 미리 막음
2. 서버의 무분별한 인터넷 접근 차단 — 감염 예방 + 보안정책 유지
                                 (감염 서버가 외부 C2로 나가려 해도 프록시에서 막힘)
3. 침해 대응 시 가시성 확보      — "누가 어디로 나갔나" 프록시 로그에 다 남아 사고 분석이 됨
```

- **현실적 타협안인 이유**: 이상적으로는 서버마다 아웃바운드를 "필요한 것만 1:1"로 여는 게 가장 안전하다. 하지만 소규모 조직은 보안 인력이 적어 서버별 정책을 일일이 못 짜고 outbound=any로 열려 있는 경우가 흔하다 → 그 앞에 프록시 한 대를 태워 한 곳에서 차단정책을 걸고, 부족한 부분을 중·장·단기로 보완한다.

> ⚠️ 프록시는 **만능이 아니라 차선책.** 가장 좋은 건 여전히 "필요한 목적지만 1:1로 여는 것". 프록시는 그게 현실적으로 안 될 때 아웃바운드를 한 곳에 모아 통제하는 절충이다.

### Squid vs SquidGuard

| | Squid | SquidGuard |
|---|---|---|
| 역할 | 포워드/**캐싱** 프록시 엔진 | Squid 위에 얹는 **URL 블랙리스트 필터** |
| 비유 | 외출 신고대(엔진) | 신고대가 든 "출입금지 가게 명단 책자"(카테고리별) |
| 하는 일 | 트래픽 중계·캐시·포트/접근 ACL | adult/games/malware/warez … 카테고리 단위 deny/allow |

- **Squid 캐시** — 자주 받는 콘텐츠를 디스크에 캐싱(권고 2000MB+) → 트래픽 절감 + 응답 가속. (프록시의 또 다른 본업)
- **SquidGuard 블랙리스트** — `blacklists_for_pfsense.tar.gz` 같은 카테고리 묶음을 받아 Groups ACL에서 "어느 대역(Source)에서 어떤 카테고리를 막을지" 지정.
- **`Do not allow IP-Addresses in URL`** — 도메인 차단을 우회하려 `http://1.2.3.4`로 IP 직접 접속하는 걸 막는 옵션(블랙리스트는 도메인 기준이라 IP로 치면 새는 구멍을 메움).

### 투명 프록시(intercept) vs 명시적 프록시(explicit)

```
투명(Transparent)  → 클라이언트 설정 없이 80 트래픽을 자동 가로챔 (편하지만 HTTPS가 까다로움)
명시적(Explicit)    → 클라이언트가 프록시 주소를 알고 CONNECT로 도메인을 알려줌 (HTTPS 필터링이 깔끔)
```

> ⚠️ **"TP모드" 용어 혼동 주의** — Squid의 **Transparent Proxy(L7, 80 가로채기)** 와 위 "L2 브릿지 방화벽 TP모드"는 **다른 것**이다. 둘 다 "투명(클라이언트가 눈치 못 챔)"이라는 발상만 같고 레이어도 하는 일도 다르다.

### HTTPS를 프록시로 태우려면 — SSL Bump

HTTP(평문)는 URL이 보여 그냥 거른다. HTTPS는 암호화돼 그냥은 못 본다 → **SSL Bump**(방화벽이 CA 인증서를 내주고 중간에서 복호화)로 풀어 본다. = 아래 `SSL Inspection`을 프록시에서 구현한 것.

```
⚠️ 투명 intercept 방식의 SSL Bump는 SNI(접속 도메인) 소실 시 ERR_DNS_FAIL로 깨지기 쉽다.
   → HTTPS 필터링은 명시적 프록시(PAC/WPAD)로 가는 게 정석.
⚠️ 모든 HTTPS를 다 복호화하는 게 정답이 아니다 — 민감하지 않은 사이트까지 복호화하면
   자원 낭비 + 프라이버시 침해. "프록시로 태울지"도 목적·자원으로 유동적 판단.
```

### 실무에서 어디에 쓰나 (규모별)

```
스타트업/소규모  → pfSense + Squid/SquidGuard (무료). 서버 앞단 아웃바운드 통제
중견           → 상용 시큐어 웹 게이트웨이(SWG) 도입 검토
대기업/금융     → Zscaler(클라우드 SWG) / Broadcom-Symantec Blue Coat ProxySG /
                Forcepoint / Skyhigh Web Gateway, 국내 URL필터 플랜티넷 WeBKeeper 등
```

> 🔗 프록시(나가는 걸 거름) + 안 쓰는 아웃바운드 포트 닫기(Default Deny) + PMS 경유 업데이트(서버 직접 인터넷 차단)는 **"서버의 아웃바운드를 최소화"하는 한 묶음**이다. → `IT-Infrastructure.md` PMS·망분리, 위 "방화벽 정책 철칙(단방향·최소허용)".

## Anti-APT (샌드박스 기반)

- IPS가 **알려진** 공격을 막는다면, Anti-APT는 **알려지지 않은** APT 코드를 잡는다.
- 의심 파일을 **샌드박스(격리된 가상 환경)**에서 실제로 실행해보고 악성 행위를 탐지
- 시그니처가 아직 없는 신규 악성코드(제로데이)에 대응

> 시그니처(몽타주)가 없는 범인은 IPS가 못 잡는다.
> Anti-APT는 "수상한 놈을 격리된 방에 넣고 행동을 지켜보는" 방식 → EDR의 행위 기반 탐지와 같은 발상.

### 정적 분석 vs 동적 분석 — 샌드박스가 "동적"인 이유

악성코드를 분석하는 두 방법. 샌드박스는 이 중 **동적 분석**에 해당한다.

| 구분 | 정적 분석 (Static) | 동적 분석 (Dynamic) |
|---|---|---|
| 방식 | 파일을 **실행하지 않고** 코드·문자열·해시·구조를 들여다봄 | 파일을 **실제로 실행**시켜서 행동을 관찰 |
| 잡는 것 | 알려진 패턴·시그니처 | 실행해야 드러나는 악성 행위 (파일 생성, 레지스트리 변조, 외부 통신 등) |
| 한계 | 난독화·패킹하면 안 보임 | 분석에 시간·자원이 듦, 샌드박스를 눈치채고 잠잠해지는 회피도 있음 |

**동적 분석기 = VM(가상머신)으로 이루어진다** ← 강사님이 말한 "동적분석기 → vm"의 정체.

```
해커의 악성코드(의심 파일)가 내부 보안 서버로 유입됨
        ↓
동적 분석기(샌드박스)가 그 파일을 격리된 VM 안에서 "실제로 실행"
        ↓
VM 안에서 뭘 하는지 관찰  →  파일 만드나? 레지스트리 건드리나? 외부 C2로 통신하나?
        ↓
악성 행위가 보이면 → 차단 + 그 특징을 시그니처로 추출 → 다른 장비·백신에 배포
```

왜 하필 VM이냐 — **진짜로 실행해봐야** 행동이 드러나는데, 진짜 운영 서버에서 실행하면 그게 곧 감염이다.
그래서 언제든 갈아엎을 수 있는 **격리된 가상 환경(VM)**에서 일부러 터뜨려보고, 분석이 끝나면 VM을 초기화(스냅샷 복원)한다.

> 🔗 이게 6-5 아키텍처 그림(DDoS → F/W → IPS / WAF / IDS → **분석** → L3 → Server)에서 라인 옆에 따로 빠져있던 "**분석**" 박스의 정체다.
> 라인에서 의심 파일을 떠다가 분석 장비(샌드박스)로 보내, 격리된 VM에서 실행해보는 것.

> 폭발물 처리반이 의심 가방을 방폭 컨테이너(VM) 안에서 열어보는 것과 같다.
> 진짜면 터져도 컨테이너 안이라 피해 없음, 끝나면 컨테이너를 비운다. → EDR 행위 기반 탐지와 발상이 같다(실행·행동을 보고 판단).

## 탐지의 두 가지 패러다임 — 시그니처 기반 vs 애노멀리 디텍션

IPS·WAF·EDR 같은 탐지 장비가 "이게 공격이냐"를 판단하는 방식은 크게 두 갈래다. **이 둘의 차이를 잡으면 왜 보안이 AI 기반으로 넘어가는지가 한 번에 보인다.** (2장 4강 "트래픽 학습" 슬라이드 핵심)

### ① 시그니처(Signature) 기반 — "몽타주랑 대조한다"

알려진 공격의 흔적·패턴을 미리 DB로 들고 있다가, 들어온 트래픽·파일을 그 패턴과 대조한다.

```
무엇을 패턴으로 보나
- 컴퓨터 네트워크·시스템에 대한 악의적 공격과 관련된 흔적·패턴
- 악성코드·해킹 파일 내부 코드에 있는 특정 문자열·바이트 패턴
- 공격의 순서 (예: 특정 요청 → 특정 응답 → 특정 명령으로 이어지는 흐름) 같은 특징
```

- 장점: **알려진 공격은 빠르고 정확하게(오탐 적게)** 잡는다.
- 한계: **DB에 없는(= 처음 보는) 공격은 못 잡는다.** 변종·난독화로 시그니처를 살짝만 비틀어도 빠져나간다.

> 지명수배 몽타주 방식. 얼굴이 등록된 범인은 잘 잡지만, 처음 보는 얼굴·변장한 범인은 못 잡는다.
> → Attack-Defense.md "시그니처 우회(난독화)"와 정확히 한 쌍.

### ② 애노멀리 디텍션(Anomaly Detection) — "평소랑 다르면 의심한다"

정상 상태가 어떤 모습인지를 **먼저 학습**해두고, 거기서 벗어나는 걸 이상으로 본다. 슬라이드의 4개 키워드가 곧 이 흐름이다.

```
학습 모드 (Learning)        → 평소 트래픽·행위의 "정상 기준선(baseline)"을 만든다
        ↓
통계 (Statistics)           → 그 기준선을 통계로 모델링 (평균·분포·임계치)
        ↓
오탐지 (False Positive) 관리 → 정상인데 비정상으로 잘못 잡는 걸 줄여가는 과정
        ↓
패턴의 새로운 정의           → 학습이 쌓이며 "정상/비정상의 경계"를 계속 갱신
```

- 장점: **시그니처가 없는 신규 공격(제로데이)도** "평소와 다르다"는 이유만으로 잡을 수 있다.
- 한계: 정상의 폭이 넓거나 학습이 부족하면 **오탐(false positive)**이 쏟아진다 → 그래서 오탐 관리가 평생 숙제.

> 동네 사람을 다 외운 경비. 몽타주(시그니처)가 없어도 "저 사람 처음 보는데 새벽 3시에 옆집 담을 넘네?"를 잡는다.
> 대신 처음 보는 택배기사도 의심하면 그게 오탐.

### Threshold(임계치) — 애노멀리의 방아쇠

"얼마나 벗어나면 이상으로 칠 거냐"의 기준선.

```
임계치 낮으면 → 조금만 달라도 잡음 → 오탐 폭발
임계치 높으면 → 어지간하면 안 잡음 → 미탐(놓침)
```

> 위 "IPS DDoS 설정"의 SYN Proxy 500/s, CPS Flood 100/s, Connection Flood 25 가 바로 이 Threshold다.
> "정상은 이 정도다"를 숫자로 박아두고, 그 선을 넘으면 공격으로 보는 것.

> 한 줄 요약: **시그니처 = 아는 공격을 정확히 / 애노멀리 = 모르는 공격을 평소와의 차이로.** 그래서 둘은 보완 관계지 대체 관계가 아니다.

## Darktrace — AI 자가학습 기반 이상탐지 (NDR)

애노멀리 디텍션을 AI로 본격화한 대표 솔루션. 강사님이 "**이게 앞으로 보안이 가야 할 방향**"이라고 짚은 게 이것.

### 핵심: 시그니처를 안 쓴다 — 내부를 통째로 학습한다

```
내부의 모든 네트워크·사용자·장비의 평소 행동을 AI가 통째로 학습
        ↓
각 주체의 "정상 패턴(self)"을 스스로 모델링 (사람이 룰·시그니처를 안 짬)
        ↓
거기서 벗어나는 미묘한 이상을 감지 (평소 안 가던 곳 접속, 평소와 다른 시간·전송량)
        ↓
알려지지 않은 공격·내부자 위협·제로데이도 "평소와 다름"만으로 포착
```

- 시그니처 DB에 의존하는 IPS의 한계(모르는 공격 못 잡음)를 정면으로 보완.
- 사람이 일일이 룰·시그니처를 안 짜도 됨 → 운영 부담 ↓, 신종 위협 대응 ↑.
- 분류상 **NDR(Network Detection & Response)** 계열. 인체 면역체계(immune system)에 비유해 마케팅한다.

> 시그니처 = 수배 전단 일일이 외우는 경비 / Darktrace = 우리 회사 사람들 평소 습관을 전부 학습해 "쟤 오늘 이상한데?"를 잡아내는 AI 경비.

### AI 탐지 모드 vs 분석 모드

강사님이 같이 언급한 부분 — AI 보안 장비는 보통 두 모드로 일한다.

```
탐지 모드 (Detection) → 실시간으로 "지금 이게 이상하다"를 잡아 알람/차단
분석 모드 (Analysis)  → 쌓인 데이터를 사후에 파고들어 "무슨 일이 있었나" 추적·조사 (위협 헌팅·포렌식)
```

- 탐지는 **실시간 방아쇠**, 분석은 **사후 수사**. 둘이 한 쌍으로 돈다.
- 큰 그림: 관제(SIEM) 연관분석 + EDR 행위탐지 + 애노멀리 디텍션이 **AI로 합쳐지는 흐름**이 Darktrace 같은 방향.

> FIMS 연결: Wazuh가 여러 로그를 모아 연관분석(관제)한 게 "분석"의 초기형. 여기에 자가학습 AI가 붙으면 사람이 룰을 안 짜도 평소와 다른 걸 알아서 잡는 Darktrace 방향이 된다.

## 관제와 연관 분석 (SIEM 개념)

각 보안 장비 로그를 **따로따로** 보면 의미가 없다. 한 곳으로 모아서 **연관 분석**해야 공격이 보인다.

```
IPS에서 IP 하나 탐지        → 그것만 보면 별 의미 없음
근데 WAF에서도 같은 IP 탐지
내부 IDS에서도 같은 IP 탐지
→ 세 군데서 보이면 "이상하다" → 추적 시작

그래서 한 곳(관제 서버)으로 모은다
→ IPS + WAF + IDS + Anti-APT 로그 전부 수집
→ 연관 분석으로 공격 패턴 파악
```

> 집 앞에서 누가 서성이는 건 OK. 근데 옆집 대문 두들기고, 또 다른 집도 두들기면 이상한 것.
> 따로따로 보면 모르고, 합쳐야 보인다. = SIEM의 상관분석(Correlation).

> FIMS 연결: **Wazuh가 바로 이 관제 서버 역할**. 여러 로그를 수집해 연관 분석한 것.

## Security Onion — 오픈소스 NSM 플랫폼 (분산 IDS + 풀패킷 + 관제 콘솔)

위 SIEM이 "여러 로그를 한 곳에 모아 연관 분석한다"는 **개념**이라면, Security Onion은 그 관제실을 **통째로 깔아주는 조립형 키트**다. 센서(눈) + 수집기(두뇌) + 상황판(콘솔)을 한 묶음으로 준다.

> 비유: SIEM이 "관제실을 이렇게 꾸며라"는 설계도라면, Security Onion은 **턴키(turnkey) 관제실 세트** — 카메라(센서), 녹화서버(Elastic), 모니터 월(SOC 콘솔)을 박스 하나로 받아 전원만 켜는 것.

> 🔗 FIMS 연결: 과정1에서 **Suricata(탐지) + Wazuh(관제)를 따로 깔아 손으로 엮은** 그 NSM을, Security Onion은 **Suricata + Zeek + Elastic + 콘솔을 한 박스로 제품화**한 버전이다. "FIMS에서 수작업으로 조립한 걸 완제품으로 받은 셈."

### 무엇으로 구성되나 (3.1 기준)

```
Suricata   → 시그니처 기반 NIDS + 풀 패킷 캡처(PCAP)   ← 우리 랩의 Snort와 같은 계열
Zeek       → 연결·DNS·HTTP·TLS 등 "프로토콜 메타 로그" 생성 (탐지가 아니라 기록·가시성)
Elastic    → Elasticsearch(저장/검색) + Logstash(수집) + Kibana(시각화)
SOC        → Security Onion Console = 분석가가 보는 웹 관제 UI (Alerts·Hunt·PCAP·Cases…)
부가도구    → osquery, CyberChef, ATT&CK Navigator
```

- **버전**: 2026-06 현재 최신은 **3.1.0**(Elastic 9.3.3 / Suricata 8.0.5 / Zeek 8.0.8, OS는 Oracle Linux 9).
- ⚠️ **3.0부터 바뀐 두 가지**: ① **UI가 새로 갈아엎어졌다**(과거 2.4 화면 자료와 다름) ② **Stenographer가 빠지고 풀 패킷 캡처를 Suricata가 담당**한다. → 옛 자료가 "stenographer로 PCAP 받는다"고 하면 구버전 설명.

### 노드 구조 — 센서(눈) / 매니저(두뇌) / 서치(기억)

```
sensor(forward) node : 모니터링 NIC을 SPAN/TAP에 물려 복제 트래픽 감시 (Suricata·Zeek)  ← 눈
manager node         : Logstash·Kibana·SOC 콘솔. 정책·룰·콘솔의 중심                     ← 두뇌
search node          : Elasticsearch 저장/검색 분산                                       ← 기억
standalone           : 소규모는 위 셋을 한 박스에 (우리 랩처럼 PoC·학습용)
```

### 센서는 어떻게 보나 — SPAN/TAP로 "복제본"을 받는다

```
스위치 SPAN(미러) / TAP ──복제 트래픽──> 센서의 모니터링 NIC (promiscuous, IP 없음)
                                          ├ Suricata : 시그니처 탐지 + 풀 PCAP
                                          └ Zeek     : 프로토콜 메타 로그
                                                      ↓
                                          Elasticsearch 적재 → SOC/Kibana에서 조회
* 관리 NIC은 따로 (SOC 웹 접속용, IP 있음)
```

- 🔗 6-9 "센서는 길목(chokepoint)에 둬야 한다" + 위 **SPAN/포트 미러링·TAP** 섹션이 정확히 이 그림이다. Security Onion 센서 = 그 미러 포트에 물리는 **실제 수집기**.
- ⚠️ **설치 직후 점검 2종**(놓치기 쉬움):
  - `so-status` → 모든 서비스(도커 컨테이너)가 **running**인지 확인. 하나라도 죽어 있으면 수집/탐지에 구멍.
  - `ifconfig` → **모니터링 NIC의 RX 패킷이 늘어나는지** 확인 = "미러가 실제로 들어오고 있나". **RX가 0이면 룰·콘솔이 멀쩡해도 탐지 0**(장님 센서) → SPAN 설정/케이블/포트그룹부터 의심.
- 모니터링 NIC에 **IP를 안 주는 이유**: 듣기만(promiscuous) 하면 되지 응답할 일이 없다. IP가 없어야 센서 자신이 탐지 대상이 되지 않고, 미러 트래픽만 조용히 받는다. (인라인 IPS가 라인에 끼는 것과 정반대 — 미러는 옆에서 본다)

### SOC(Security Onion Console) — 관제 UI 구성과 로그 보는 법

좌측 메뉴 기준, 분석가가 실제로 쓰는 흐름:

```
Grid       : 배포된 노드(매니저/센서/서치) 상태판 — 어느 센서가 살아있나
Alerts     : Suricata 등에서 뜬 경보 모음. 여기서 시작해 드릴다운
Dashboards : 데이터 유형별 미리 만든 대시보드 (NIDS 경보 / Zeek 프로토콜 메타 / Elastic Agent / 방화벽 로그)
Hunt       : 위협 헌팅 — 쿼리로 능동적으로 뒤진다 (경보가 안 떠도 직접 사냥)
PCAP       : 선택한 스트림의 전체 패킷 끌어와 보기 (3.x는 Suricata가 캡처)
Cases      : 의심 건을 "사건"으로 만들어 분석가끼리 협업·종결
Detections : 탐지 룰(Suricata/Sigma/YARA) 관리·활성화
Kibana / CyberChef / ATT&CK Navigator : Elastic 원본 탐색 / 데이터 디코딩 / MITRE 매핑
```

**로그 보는 표준 파이프라인:**
```
Alerts(경보 발생) → 수상한 것 클릭해 드릴다운 → Hunt로 관련 Zeek 메타·로그 더 파기
 → 필요하면 PCAP로 실제 패킷(증거) 확인 → 진짜면 Cases로 승격해 추적·종결
= "경보 → 헌팅 → 증거(PCAP) → 사건화" 한 줄 흐름
```

> 💡 강의 못 들은 부분 보충: Security Onion에서 "로그 본다"는 건 그냥 텍스트 grep이 아니라 **Alerts에서 시작해 Hunt/PCAP/Cases로 피벗하는 분석 동선**을 말한다. Kibana는 그 밑의 원본 데이터(Elasticsearch)를 직접 뒤질 때 쓰는 하위 도구.

### 우리 랩의 Snort와의 관계 — 경쟁이 아니라 역할 분담

```
pfSense Snort(인라인 IPS, 6-16) → 라인 위에서 "막는다" (차단·가용성 리스크)
Security Onion(미러 NSM)        → 옆에서 "다 본다" (가시성·증거·헌팅, 차단은 안 함)
```

- 🔗 6-8 "인라인 vs 미러링" 그대로다. IPS=차단(끊으면 장애), NSM 센서=가시성(끊을 일 없음). 둘을 같이 둬야 "막으면서 + 놓친 것도 본다".
- ⚠️ 면접 함정: **"Security Onion은 IPS다" → 틀림.** 기본은 **미러 기반 NSM/IDS(탐지·가시성)** 플랫폼이지 인라인 차단 장비가 아니다.

### 실무에서 어디에 쓰나 (규모별)

```
스타트업/중소  → standalone 한 대로 무료 NSM 가시성 확보 (상용 SIEM 사기 전 단계)
중견          → 센서 노드 분리 배치(지점·세그먼트별) + 매니저/서치 분리
대기업/금융    → 상용 SIEM(Splunk·QRadar) + 상용 NDR이 주력. Security Onion은 교육·PoC·
                특정 세그먼트 관제·블루팀 훈련(침해대응 연습)에 강함
```

> ⚠️ 무료지만 **운영 손이 많이 간다** — 룰 튜닝(Snort 때와 동일한 오탐 문제), 디스크/Elastic 용량 관리, 센서 성능. "공짜 = 공짜로 굴러간다"가 아니다.

### 면접에서 이렇게 말해라

✅ "Security Onion은 **Suricata·Zeek·Elastic·SOC 콘솔을 묶은 오픈소스 NSM 플랫폼**입니다. 센서가 SPAN/TAP로 **복제 트래픽**을 받아 Suricata(시그니처 탐지+풀 PCAP)·Zeek(프로토콜 메타)로 분석하고 Elastic에 적재한 뒤, SOC에서 **Alerts→Hunt→PCAP→Cases**로 분석합니다."
❌ "Security Onion으로 공격을 차단합니다" — 기본은 **미러 기반 탐지/가시성**이지 인라인 차단이 아님. (차단은 pfSense Snort 같은 인라인 IPS의 몫)

## DDos 방어 장비 (Anti-DDoS)

- 디도스(DDoS) 공격을 방어하기 위한 전용 장비
- 대용량 트래픽을 견딜 수 있는 크기의 서버로 구축 (CDN도 DDos 완화 역할)
- 페이로드까지 읽을 수 있어 **L7까지 커버** → HTTP Flood 같은 응용 자원 고갈 공격도 방어

### 왜 맨 앞에 두냐

```
인터넷
   ↓
Anti-DDoS   ← 대량 트래픽(플러딩)을 여기서 먼저 거른다
   ↓
방화벽 / IPS / ...
```

- 대량 트래픽을 먼저 걸러야 **뒤에 있는 장비들이 안 죽음**
- 그래서 DDoS 장비는 보호 대상보다 훨씬 높은 스루풋(고성능)이 필요

### DDoS 장비가 없을 때 (꿩 대신 닭)

```
IPS로 대체 가능
→ IPS 임계치(Threshold) 설정으로 플러딩 방어
→ 완벽하진 않지만 없는 것보단 나음
→ 단점: IPS에 부하 걸리면 뻗어버림 → 뻗었을 때 대응 체계도 함께 마련해야 함
```

**IPS의 DDoS 방어 설정 화면 (그림 2.7)** — 결국 "초당 몇 개 넘으면 막아라"라는 **임계치(Threshold)** 설정이다.

```
SYN Proxy        : Notification Threshold 500   Syns per second (1~10000)
                   → 초당 SYN이 500개 넘으면 알림/차단 (SYN Flood 대응)
CPS Flood        : Threshold 100.0   Max average connections per second
                   → 초당 평균 연결 수 임계치 (Connection Per Second)
Connection Flood : Threshold 25      Max simultaneous open connections
                   → 동시에 열려 있는 연결 수 임계치
```

> 세 가지 다 "정상은 이 정도인데, 그 선을 넘으면 공격으로 본다"는 애노멀리 디텍션의 Threshold 개념(아래 참고).
> 전문 DDoS 장비만큼 정교하진 않지만, IPS에 이 임계치를 걸어두는 것만으로도 1차 방어가 된다.

## 인라인 모드와 하드웨어 바이패스

### 인라인 모드 (Inline Mode)

```
인라인 = 라인 위에 걸쳐 있는 것
→ 트래픽이 반드시 이 장비를 거쳐야 함
→ 장비가 빠지면 통신 단절

ISP
   ↓
방화벽 (업링크 인터페이스)
   ↓
IPS (업링크 ─ 다운링크)
   ↓
L3 스위치
   ↓
서버들
```

방화벽, IPS처럼 **차단을 해야 하는 장비**는 인라인으로 들어간다. (IDS는 미러링이라 인라인 아님)

### 하드웨어 바이패스 (Hardware Bypass) — 장비 내장

```
문제: 인라인 장비가 전원 꺼지거나 장애나면 → 트래픽 단절 → 서비스 중단

해결: 장비 안에 하드웨어 바이패스 회로 내장
→ 전원 없어도 물리적으로 입출력 포트를 직접 연결
→ 장비 꺼져도 트래픽은 그냥 통과 (보안 검사는 못 하지만 서비스는 유지)
```

> 자동문이 정전되면 수동으로 밀고 들어갈 수 있게 만들어 둔 것과 같은 발상.
> "검사는 못 해도 사람은 통과시킨다" → 가용성 우선.

### 외부 바이패스 장비 (별도 장비)

```
내장 바이패스 → 장비 안 회로. 전원 꺼지면 자동 직결
외부 바이패스 → 인라인 장비 옆에 따로 두는 독립 장비
```

왜 따로 두냐 — 인라인 장비를 **계획적으로** 빼야 하는 상황 때문:

```
1. 장비 단선 (케이블 문제)
2. 장비 교체 (업그레이드, 고장)
3. 장비 점검 (유지보수)
4. 펌웨어 업데이트
```

- 이때 인라인 장비를 그냥 빼면 트래픽 단절 = 서비스 중단
- 외부 바이패스가 있으면 → 빼기 전에 바이패스로 전환 → 트래픽 우회 → **서비스 유지하며 장비 교체·점검 가능**

> 내장 바이패스는 "사고 났을 때 자동 대비", 외부 바이패스는 "계획된 작업 때 수동 전환".

## 방화벽 실물 구조

코앞에 보이던 구성도의 "선 3개"가 실물에서는 인터페이스 3개에 케이블을 각각 꽂는 것이다.

```
인터페이스 (Interface)
→ 방화벽에 케이블 꽂는 구멍 하나하나
→ 각 인터페이스 = 각각의 망
→ 예) 인터페이스 4개 = 업링크 + DMZ + WAS + DB 4개 망 분리

광 모듈 (Optical Transceiver)
→ 광케이블 꽂는 모듈, 슬롯 추가하면 포트 확장
→ 처음 도입 시 필요한 포트만 라이센스 활성화 (나머진 비활성)

스루풋 (Throughput)
→ 장비가 동시에 처리할 수 있는 트래픽 양
→ 수백 Gbps = 고성능, 스루풋 높을수록 비쌈
→ 이게 고급 장비(4천만~1억)와 일반 장비의 차이
```

### 인터페이스 구성 (3-Tier 연결)

```
방화벽
├── 인터페이스 1 (업링크) → ISP (인터넷)
├── 인터페이스 2          → DMZ (Web Server)
├── 인터페이스 3          → WAS
└── 인터페이스 4          → DB
```

> 48포트 장비라도 필요한 포트만 라이센스 켜고 쓴다. 안 쓰는 포트는 비활성 상태.

## VPN (Virtual Private Network) — 통신 구간을 암호화로 묶는다

서로 떨어진 네트워크(혹은 단말)를 **공용 인터넷 위에 암호화 터널을 깔아** 사설망처럼 연결하는 기술. 핵심은 "인터넷이 통하는 서로의 통신 공간을 암호화한다"는 것 — 중간 구간(인터넷)은 그대로인데, 그 위에 우리만 읽을 수 있는 관을 하나 더 까는 셈.

> 비유: 인터넷은 누구나 다니는 **공용 도로**다. VPN은 그 도로 위에 **우리 회사 전용 지하 터널**을 뚫는 것. 도로(인터넷)는 그대로지만, 터널 안에서 오가는 내용은 밖에서 안 보인다.

> 🔗 FIMS 연결: 과정1에서 **WireGuard로 깐 터널**(`FIMS-Project.md` "WireGuard VPN")이 정확히 이 발상이었다. WAS는 VPN이 없어 AWS에서 직접 못 갔다 — 즉 "터널이 깔린 구간만 사설망처럼 통한다"는 걸 그때 몸으로 겪은 것.

### 암호화가 일어나는 레이어 — IPsec(L3) vs SSL-VPN(L7)

VPN이 "어느 구간을 암호화하느냐"는 동작 레이어에 따라 갈린다.

```
L2~L4 구간 암호화  → IPsec VPN 계열
  - 패킷(L3) 자체를 통째로 암호화해 게이트웨이↔게이트웨이로 터널링
  - 망과 망을 통으로 묶는 데 적합 (site-to-site)

L7(애플리케이션) 암호화 → SSL-VPN 계열
  - TLS(HTTPS와 같은 방식)로 응용 구간을 암호화
  - 개인 단말이 브라우저/클라이언트로 접속하는 데 적합 (remote access)
```

> 한 줄: **IPsec = 망째로 묶는 터널(L3), SSL-VPN = 개인이 접속하는 터널(L7).** 둘 다 "공용 인터넷 위 암호화 관"이라는 큰 그림은 같다.

### 두 가지 사용 형태 — Site-to-Site vs Remote Access

| 구분 | Site-to-Site VPN | Remote Access VPN |
|---|---|---|
| 묶는 대상 | **지역 ↔ 지역**(망 ↔ 망) | **개인 단말 ↔ 본사망** |
| 터널 주체 | 양쪽 게이트웨이(방화벽/VPN장비) | 개인 PC의 VPN 클라이언트 |
| 주 프로토콜 | IPsec | SSL-VPN / WireGuard |
| 예 | 본사 ↔ 지사 상시 연결 | 재택근무자가 사내망 접속 |

**Site-to-Site = 지역과 지역을 묶는 방법**이 이 날의 핵심 단어다.

```
VPN 본점(본사)  ──── 암호화 터널 ────  VPN 지점(지사 A)
       │
       └────────── 암호화 터널 ────────  VPN 지점(지사 B)
```

- **VPN 본점** = 말 그대로 본사(허브). 모든 지사 터널이 모이는 중심.
- **VPN 지점** = 본점에 연결돼 있는 각각의 지사. 본점을 거쳐 서로/본사와 통신.

> 비유: 본사가 **중앙 우체국(허브)**, 각 지사가 **동네 우체국(스포크)**. 동네끼리 직접 길을 안 내고, 일단 중앙 우체국으로 보냈다가 다시 뿌리는 허브-스포크 구조.

### 실제로 어떻게 연결되나 — 키 교환 · OTP · SSL VPN 인증서 (6-16 심화)

스타벅스에서 노트북 하나로 회사 서버에 붙는 상황으로 흐름을 따라가 보면:

```
[스벅의 나] ── 인터넷 ──> [회사 VPN 게이트웨이] ──> [서버망: PC/우분투/NAS]
   내 VPN 클라이언트               서버쪽 VPN에 걸린 "정책"에 따라 접근 허용
```

- 외부의 누군가가 내 서버로 접근하려면 **VPN으로 묶인 뒤**, 서버쪽 VPN에 적용된 **정책**을 통과해야만 서버에 닿는다. 터널만 뚫는다고 다 들어오는 게 아니라 **터널 + 정책 2단**이다.
- **서로 키를 나눠 갖는다** — IPsec은 양쪽이 **사전 공유키(PSK) 또는 인증서**를 갖고 그걸로 터널을 암호화·인증한다. "각 키를 가지고 있다"는 게 이 말.

**Site-to-Site의 현실적 제약 — 양끝 장비가 짝이 맞아야**
```
서버쪽이 pfSense의 IPsec을 쓴다면 → 반대쪽 사이트도 IPsec을 지원하는 장비가 필요
(IPsec은 표준이라 이론상 타벤더끼리도 되지만, Phase1/2 파라미터·인증방식을 양끝이 똑같이
 맞춰야 해서 실무에선 "같은 장비/같은 설정"으로 맞추는 게 압도적으로 편하다)
```

**Client-to-Site(원격접속)의 인증 — OTP가 핵심**
```
클라이언트가 아무 네트워크(스벅·집·LTE)를 타고 와서
 → 로그인 + 인증을 거친 뒤 접속
 → 보통 휴대폰 OTP를 추가(MFA) — 그 OTP는 "유일한 한 개"라 양도·복제가 안 됨
⚠️ OTP 기기(폰)를 잃어버리면 그대로 막힘 → 분실/재발급 절차가 운영 포인트
```

**SSL-VPN = 인증서 기반 + 브라우저 접속**
```
SSL-VPN은 TLS(=HTTPS)와 같은 인증서 통신
 → 크롬 브라우저로 포털에 접속해 로그인 (전용 클라이언트 없이도 가능)
 → 여기에도 OTP(MFA) 적용
 ⚠️ 인증서/OTP 수단을 잃어버리면 접속 불가 — 분실 시 타격이 크다
```

**들어온 다음이 진짜 — SSL Inspection으로 본다 (아키텍처 개념)**
```
SSL-VPN으로 직원 패킷이 들어오면, 그 안에서 무슨 짓을 하는지 어떻게 아나?
 → 들어오자마자 SSL Inspection(복호화)으로 평문화해서 본다
 → 관제 상황판에서 "누가 어디로 무슨 트래픽을 보내는지" 다 보임
```

> 🔗 = 위 "SSL Inspection"이 VPN 입구에서 쓰이는 자리. VPN은 "터널을 깔아 들여보내는" 것까지고, 들어온 뒤의 행위 가시성은 **SSL 복호화 + 관제**가 책임진다. 터널만 믿지 않고 안에서 또 본다 = ZTNA로 가는 발상(터널 통째로 신뢰 X, 요청마다 검증).

> 비유: VPN 터널 = 회사 정문까지 오는 전용 통근버스. 근데 버스에서 내린 뒤 사옥 안에서 뭘 하는지는 **로비 보안검색(SSL Inspection) + CCTV 상황판(관제)**이 따로 본다. 버스를 태워줬다고 사옥 프리패스를 준 게 아니다.

### ⚠️ VPN은 그 자체가 공격 표면이다 — "위험 수용(Risk Acceptance)"

VPN 게이트웨이는 **인터넷에 얼굴을 내놓고 있어야** 외부에서 터널을 걸 수 있다. 그 말은 곧 **장비 자체가 인터넷에 노출된 표적**이 된다는 뜻.

```
VPN 결함(취약점)이 노출되면
→ 그 익스플로잇으로 게이트웨이가 뚫림
→ 터널을 타고 내부망까지 들어옴 (암호화 관이 오히려 침투 통로가 됨)
→ 침해 가능성이 급격히 높아짐
```

- 그래서 VPN 운영은 **"인터넷에 장비를 노출하는 위험을 감수(수용)하고 쓰는 것"** — 위험 처리 4전략(회피/전가/완화/수용) 중 **수용(Risk Acceptance)**에 해당. 편의(원격 접속)를 얻는 대신 노출 위험을 안고 가는 것.
- ⚠️ **음차 불명확** — 수업 노트의 "DOA 위험 수용"은 원본 표기가 흐릿하다. 문맥상 **"VPN 노출에 따른 잔여 위험을 수용한다"**는 위험 처리의 '수용' 전략으로 이해. (정확한 약어는 수업 자료에서 재확인 필요)

> 실무 사례: Fortinet SSL-VPN, Ivanti/Pulse Secure, Cisco ASA 등 **VPN 장비의 CVE는 해마다 초기 침투(Initial Access) 단골 경로**다. "VPN 켜놨으니 안전"이 아니라, **VPN 장비 펌웨어 패치(→ `PMS`)와 인증 강화(MFA)**가 같이 가야 한다.

### SSL-VPN은 왜 정책을 any로 여나 + VPN ≠ 포트포워딩 (6-17)

**① SSL-VPN 입구를 any로 여는 이유 — 폐쇄형 → 공개형**
```
원래 VPN = 폐쇄형(정해진 상대만 붙는 구조)
 → 근데 재택·외근 직원은 스벅·집·LTE로 "잦게 이동"한다
 → 사용자마다 1:1로 출발지를 지정해 정책을 짜면 사람이 일일이 대응 불가(수가 많고 IP가 계속 바뀜)
 → 그래서 입구(출발지)는 any로 열 수밖에 없다 = 공개형으로 전환
```
- ⚠️ any로 열면 **해커도 문 앞까지 온다**. 그래서 "문은 누구나 두드리게(any) 열되, **들어오는 건 SSL 인증으로 막는다**" — VPN 로그인=인증 절차를 통과해야 서버에 닿게 한 것. 폐쇄형의 한계(이동성↓)를 풀고, 보안 책임을 **출발지 제한 → 인증**으로 옮긴 셈.

**② VPN ≠ 포트포워딩 (헷갈리기 쉬움)**
```
포트포워딩(DNAT, 6-12) : 외부 포트를 내부 IP:포트로 "꽂는" 것 (특정 서비스 하나를 밖에 노출)
VPN                    : 접속자에게 "사설 IP"를 발급해 내부망의 일원처럼 만드는 것
```
- ⚠️ 핵심: **사설 IP를 줬다고 다 들어오는 게 아니다.** 그 IP에 대한 **정책이 생성돼야** 한다 — 특정 목적지로만 갈 수 있는 정책만 부여한다. 부여자(관리자) 입장에선 "클라이언트에게 권한을 주고 + 어떤 경로를 허가할지 정책을 짜는" 2단 작업. (= 위 "터널 + 정책 2단"의 출발지 쪽 설명)
- 비유: 포트포워딩 = 건물 외벽에 **특정 호실 직통 우편투입구** 하나 뚫기. VPN = 방문자에게 **사원증(사설 IP)** 발급 — 근데 사원증이 있다고 전 층 프리패스가 아니라, 그 카드로 열리는 문(정책)만 열린다.

### 재택 이중망의 함정 — VPN 계정 2개·"닌자게이트"·ISMS (6-17)

```
재택에서 두 망(개발망 + 인터넷망)을 쓰려면 → VPN 계정이 2개 필요
 이유: 터널은 한 계정당 하나만 뚫린다 (한 계정 = 한 터널)
```

- 개발망에서 일하다 인터넷이 필요하면? **두 라우팅을 동시에** 쓸 수 있다 — 근데 이게 위험하다.
- ⚠️ **"닌자게이트"**(강사님 표현) = 눈에 안 띄게 두 망을 잇는 **우회 게이트웨이 / 피벗 경로**.
```
재택 노트북이 RAT로 원격당하면
 → 두 망에 동시에 붙은 그 노트북이 "다리"가 된다
 → 해커가 인터넷 쪽 → 노트북 경유 → 개발망까지 피벗(침투)
 → 망분리가 노트북 한 대로 무력화됨
```
- 🔗 6-17 OpenVPN의 **redirect-gateway(게이트웨이로 WAN 쓰겠다) 체크박스**에서 본 위험과 동형이다: "게이트웨이는 하나"라, 다른 대역 사용자가 내 게이트웨이를 타고 트래픽을 오가게 되면 그 자체가 우회 경로가 된다.
- **바람직한 운영**: VPN 끊고 → 인터넷 쓰고 → 다시 VPN 연결 (**동시 연결 금지**). "번거롭다"고 불평한 직원에게 강사님 왈 *"그럼 그만둬라"* (실제 자회사 사례) — 망 동시연결은 그만큼 치명적이라는 것.
- 🔗 ISMS 연결: **코로나 이후 VPN이 활성화되면서, ISMS 심사 때 바로 이 "재택 두 망 동시연결" 취약 환경부터 확인**한다. 보안 인증이 실제로 들여다보는 포인트.
- 🔗 `Attack-Defense.md` / EDR: RAT 원격 → 피벗은 결국 **단말 탐지(EDR) + 망분리 원칙**으로 막는 영역. 닌자게이트는 "단말 하나가 망분리를 무너뜨리는" 시나리오의 이름.

> 🔗 실습: pfSense **OpenVPN 실제 구축**(CA·서버 인증서·터널 대역·redirect gateway·client-export)은 `Lab/2026-06-17-OpenVPN·SecurityOnion.md` 참고.

### 실무에서 어디에 쓰나 (규모별)

```
스타트업/중소  → pfSense·OpenVPN·WireGuard (무료) 또는 클라우드 VPN
                재택·외주 인력 remote access 위주
중견          → FortiGate·Palo Alto의 IPsec site-to-site로 본사-지사 상시 연결
대기업/금융    → 전용 VPN 장비 + MFA + (인터넷 노출 최소화). 점차 VPN을
                ZTNA(Zero Trust Network Access)로 대체하는 추세
```

> VPN의 한계(터널만 뚫리면 내부망이 통째로 노출)를 보완하려고 나온 게 **ZTNA** — "터널을 깔아 망을 통째로 믿는" 대신 **요청마다 검증**하는 방향. VPN 익스플로잇이 잦아질수록 이 흐름이 빨라진다.

## WIPS (Wireless Intrusion Prevention System)

- 인가된 Wi-Fi만 Access 허용
- Jamming 기술로 강한 전파를 보내 전자기기 신호 방해 → SSID 식별 불가

## EDR (Endpoint Detection and Response)

기존 백신의 한계를 보완한 엔드포인트 보안 솔루션.

### 기존 백신의 한계

- 시그니처 기반이라 처음 발견된 악성코드는 탐지 불가
- 탐지 후 격리는 가능하지만 리버싱 분석까지 어려움
- 숙련된 보안 인력 부족

### EDR의 해결 방식

- **행위 기반 탐지** 사용 → 시그니처 없는 신규 악성코드도 이상 행위로 선제 차단
- Threat Intelligence(TI) 결합 → 의심 행위를 점수화해 AI가 자동 분류·조치

```
악성 행위 탐지 (EDR 행위 기반)
    ↓
시그니처 추출
    ↓
백신에 전달 → 이후 동일 패턴 자동 차단
```

> EDR + 백신 = EDR이 행위 탐지, 백신이 시그니처 차단하는 이중 구조

## DLP (Data Loss Prevention) — 정보 유출 방지

### NDLP (Network DLP)
- 네트워크 안 데이터를 탐지
- 네트워크 길목에 배치

```
F/W → Backbone → Mail Server
               ↑
        여기서 NDLP가 탐지
```

### HDLP (Host DLP)
- 사용자 하드웨어에 Agent로 하여금 중요 정보 탐지
- 취급 담당자가 무단으로 고객 데이터를 하드웨어에서 옮기는 행위 방지

> 엔드포인트만으론 불충분(에이전트 삭제 가능) → 네트워크 DLP 필수, 둘 결합해야 완결

### NDLP는 보통 "관찰용" — 인라인 차단이 아니라 TAP 복제로 본다 (6-9일 메일서버 사례)

메일 서버 구성도에서 DLP는 **코어 상단에 TAP을 달아, 악성 분석이 끝나기 전 단계의 트래픽을 복제한 라인**으로 받는다. 즉 본선(인라인)에 끼어 차단하는 게 아니라 **복제본을 받아 관찰**한다.

```
왜 인라인 차단이 아니라 관찰용이냐?
→ 이 메일서버는 애초에 "발신할 때 담당자 승인이 필요한" 흐름
→ 사람이 승인 단계에서 거르니, DLP는 차단보다 "유출 정황을 보는" 역할이면 충분
→ DLP가 본선에 인라인으로 끼면 오탐 시 정상 메일까지 끊김 (IDS를 라인 밖에 빼는 것과 같은 논리)
```

> IDS를 미러링으로 빼는 발상이 DLP에도 그대로 적용된다 — **차단이 목적이 아닌 가시성 장비는 TAP 복제로 옆에서 본다.** (위 "인라인 vs 미러링", "TAP" 참고. 상세 메일 흐름은 Architecture.md "메일 서버 구성도".)

## DRM (Digital Rights Management)

- 무단 디지털 콘텐츠 공급/제공 막는 기술
- Agent를 통해 하드웨어 파일을 암호화
- 각 라이센스 키가 존재 → 수신자는 라이센스 키 있어야 열람 가능

> 같은 DRM 솔루션이라도 라이선스 서버가 다르면 키가 달라서 타 기업과 문서 공유 불가

---

## ECM (Enterprise Contents Management) — 문서 중앙화

- 기업 정보 중앙 집중화
- 업무 발생 모든 문서 강제 저장
- 정보 유출 원천 차단
- 문서 접근 권한 관리
- DRM과 같이 쓰면 보안 효율 증가

---

## PMS (Patch Management System)

보안 패치 및 솔루션 등의 원격 설치 유도 솔루션.

### 도입 배경
- 제로데이 취약점: 패치가 나오기 전 공격이 먼저 발생
- APT 공격: 취약점 미패치 환경이 주요 침투 경로
- 수동 관리 한계: 수백~수천 대 PC를 담당자가 직접 패치 불가

### 폐쇄망 환경에서의 PMS

```
[인터넷] → [패치 수집 서버 (DMZ)] → 망연계 → [내부 패치 배포 서버] → 엔드포인트
```

---

## 허니팟 (Honeypot)

- 외부 공격을 유도하는 보안 장치로 수동적으로 공격을 기다리는 서버
- 공격자의 패턴 및 특성 파악이 핵심
- 보안 회사뿐 아니라 일반 회사도 아키텍처 상 허니팟 배치해 연구 가능
