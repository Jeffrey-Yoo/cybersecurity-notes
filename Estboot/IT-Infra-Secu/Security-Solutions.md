## 방화벽 (Firewall)

내부와 외부 네트워크를 구분하고 불법 침입을 탐지·차단하는 시스템.
OSI 기준 L3/L4에서 동작. MAC 주소를 보유해 L2 인식은 가능하지만 L2 기능으로 동작하진 않음.

### 방화벽 종류

| 구분 | 설명 |
|---|---|
| 어플라이언스 방화벽 | 전용 하드웨어 장비 (Palo Alto, Fortinet 등) |
| 어플리케이션 방화벽 | 서버 내 소프트웨어 형태 (ufw, firewalld 등) |

### 세대별 방화벽

| 세대 | 레이어 | 하는 일 | 예시 |
|---|---|---|---|
| 1세대 패킷필터링 | L3~L4 | IP + 포트 차단 | iptables, UFW |
| 2세대 상태기반 | L4 | TCP 세션 상태 추적 | 대부분의 기본 F/W |
| 3세대 NGFW | L7 | 페이로드 내용까지 검사 | FortiGate, Palo Alto |

### 방화벽 레이어별 정리

방화벽은 어느 레이어에서 동작하느냐에 따라 "볼 수 있는 깊이"가 달라진다.

```
L2 방화벽 (TP모드)  → 브릿지로 끼어들어 투명하게 제어 (IP 없음)
L3 방화벽           → IP 기준 제어
L4 방화벽           → IP + 포트만 봄 = 접근제어만 가능
                     ⚠️ IP 변조해서 들어오면 못 잡음 (내용을 안 보니까)
L7 방화벽           → 패킷 안의 문자열(페이로드)까지 필터링 가능
                     pfSense = L7 UTM의 대표적인 예
```

> L4까지는 "누가 어느 문으로 들어오냐"만 본다. L7부터 "들고 온 짐 안에 뭐가 있냐"를 검사.
> 그래서 L4 방화벽은 출발지 IP를 위조한 공격을 걸러내지 못한다.

### 블랙리스트 vs 화이트리스트

| 방식 | 기본 상태 | 동작 방식 |
|---|---|---|
| 블랙리스트 | 열려있음 (Default Allow) | 차단할 IP/포트만 명시 |
| 화이트리스트 | 닫혀있음 (Default Deny) | 허용할 IP/포트만 명시 |

> 실무에서는 인바운드는 화이트리스트, 보안이 중요한 서버망은 양방향 모두 화이트리스트.

### 방화벽 정책 철칙

```
잘못된 방식: web01 ↔ SMB  양방향 허용 한 번에 체크
올바른 방식: web01 → SMB  출발지/목적지/포트 명시 (단방향)
            SMB → web01  별도 정책으로 명시 (단방향)
```

### 방화벽 정책 운영 실무 (금융권 기준)

방화벽 정책은 "그냥 열어주는 것"이 아니라 신청 → 검토 → 반영 프로세스를 거친다.

- 접속 대상이 많고(사람·시스템), 접속 방식도 제각각(원격 / HTTP / 특정 TCP 포트)이라 정책 수가 기하급수로 늘어난다.
- 그래서 보안팀은 신청 받는 시간대를 고정해서 운영한다. (예: 오전 10시 / 점심 후 13시 / 퇴근 전 17시)
- 금융회사는 보통 **2주 전 신청 ~ 3일 전 마감**으로 룰을 박아둔다.

> 검문소 운영시간 같은 것. 아무 때나 와서 문 열어달라고 하면 검문소가 마비됨.
> 신청 창구 시간을 정해두고, 마감일을 박아둬야 보안팀이 업무를 끌고 갈 수 있다.

> ⚠️ 정책을 "이럴 것 같으니 미리 열어두자"는 식으로 **예측해서 넣으면 안 됨**.
> 정확히 출발지 / 목적지 / 포트를 확인하고 문을 열어야 한다 (위 "방화벽 정책 철칙"과 직결).

징징대는 유관부서에 끌려다니며 "우리는 맨날 방화벽만 열어준다" 신세가 되지 않으려면,
신청 프로세스 자체를 잘 설계하는 게 보안팀 실력이다.

### 방화벽 Deny 로그

방화벽은 L3에서 패킷을 직접 제어하는 장비라서, 차단된 트래픽은 **방화벽에서만** 보인다.

- L3에 묶여있는 게 아니라, L3에서 제어를 받아 **차단됐냐 / 안 됐냐**가 핵심
- 접근 통제에 걸리면 패킷이 못 넘어가고 방화벽에 **Deny 로그**가 찍힘
- 이 Deny 로그는 다른 장비에서 안 보이고 방화벽에서만 확인 가능

> DB 접근 권한 없는 사람이 DB에 붙으려 한 시도 = 통제 권한 위배 = 방화벽 Deny 로그로 추적.
> 방화벽은 패킷 제어 장비 → 접근 제어가 방화벽에서 일어남 → 그 결과(차단)도 방화벽 로그에 남는 것.

### UFW 기본 명령어

```bash
ufw status                              # 상태 확인
ufw status verbose                      # 상세 상태 확인
ufw status numbered                     # 번호 포함 룰 확인
ufw enable / disable                    # 방화벽 활성화/비활성화

ufw allow 포트/프로토콜                  # 허용
ufw deny 포트/프로토콜                   # 차단
ufw delete 번호                         # 룰 삭제

ufw allow from IP대역 to any port 번호  # 특정 IP 대역만 허용
ufw default deny incoming               # 인바운드 전체 차단
ufw default deny outgoing               # 아웃바운드 전체 차단
```

> ⚠️ 방화벽 룰은 위에서 아래 순서로 적용됨
> 좁은 대역 ALLOW를 넓은 대역 DENY보다 위에 둬야 함

## L2 브릿지 방화벽 (TP모드 / Transparent Mode)

NAT 기능 없이 두 장비 사이에 투명하게 자리잡아 모든 트래픽을 제어.
IP 없이 배치하기 때문에 기존 네트워크 구조를 바꾸지 않고 끼워 넣을 수 있음.

```
일반 방화벽 (L4)          TP모드 방화벽 (L2 브릿지)
IP 있음                   IP 없음 (관리용만 있으면 됨)
네트워크 구조 바뀜          네트워크 구조 그대로
기존 서버 설정 변경 필요     기존 설정 하나도 안 바꿔도 됨
```

### 언제 쓰냐

- 1차 방화벽으로 외부 진입은 통제했는데 내부망 사이에 이중 보안을 추가하고 싶을 때
- 기존 IP 설정 건드리지 않고 DMZ → 내부망 길목에 투명하게 끼워 넣는 방식
- 실무에서는 FortiGate 같은 하드웨어 장비를 TP모드로 설정해서 사용

### 배치 위치

```
인터넷 → 1차 F/W (UFW, L4) → L3 라우터 → [TP F/W] → LB → L2 스위치 → 서버
```

> TP모드는 "배치 방식"이고, 안에서 하는 검사는 여전히 L4/L7.
> 배치 방식(어떻게 끼어있냐)과 검사 기능(뭘 보냐)은 별개 개념.

### 심층 방어 (Defense in Depth)

한 겹이 뚫려도 다음 겹이 막는 구조.

```
1차 F/W (UFW)    → 외부 진입 통제
TP 브릿지 F/W    → DMZ → 내부 길목 통제
VLAN + ACL       → VLAN 간 이동 통제
```

## IDS / IPS

### IDS (Intrusion Detection System) — 탐지만

- 패킷 감시 + 탐지 + 로그 (차단 없음)
- 특정 레이어 하나에 고정되지 않음 (L3~L7 동시 감시, 페이로드까지 읽음)
- 대표: Suricata, Snort

### IPS (Intrusion Prevention System) — 탐지 + 차단

- IDS에 차단 기능 추가 (인라인)
- 알려진 악성코드 / 공격 패턴을 시그니처로 차단
- DDos를 막는 기능도 포함 (L7)
- IPS 제조사들은 새로운 시그니처를 지속적으로 업데이트해서 엔진 강화

### 인라인(Inline) vs 미러링(Mirroring) — IDS를 라인에서 빼는 이유

```
IPS = 인라인 (라인 위에 걸쳐 있음, 트래픽이 반드시 통과)
      → 차단해야 하니까 길목에 직접 서 있어야 함

IDS = 미러링 (옆에서 복제된 데이터만 받아봄)
      → 로깅 장비라 라인 위에 둘 이유가 없음
      → 인라인에 넣으면 장애 시 라인 절체 = 다운타임 발생
      → 그래서 별도로 빼서 복제(미러링) 데이터로 분석
      → 넣었다 뺐다 자유로운 구조
```

> IPS는 검문소(차 세워서 검사), IDS는 길가의 CCTV(지나가는 거 찍어서 나중에 본다).
> CCTV가 고장 났다고 도로를 막진 않듯, IDS는 죽어도 서비스에 영향이 없어야 한다.

### IPS와 IDS를 같이 쓰는 이유

```
회사마다 시그니처 DB가 다름
→ IPS가 못 잡은 걸 IDS가 잡을 수 있음
→ 알약 이중 엔진과 같은 발상 (미탐 방지, 보강)
→ 비용 대비 가치를 따져서 도입 결정
```

### IPS vs IDS 운용 전략

| 구분 | 운용 방식 | 이유 |
|---|---|---|
| IPS | **보수적** | 걸리면 바로 차단 → 오탐 최소화된 시그니처만 사용 (정상 차단하면 사고) |
| IDS | **러프하게** | 탐지·로그만 → 놓치는 것 없게 넓게 탐지, 예외는 나중에 수동 처리 |

> IPS는 차단하니까 신중하게(오탐=장애), IDS는 차단 안 하니까 일단 다 잡아두고 본다.

### IPS 멀티 세그먼트

- IPS 3대를 따로 사는 게 아님 → **1대로 멀티 세그먼트 구성**
- 한 대의 IPS가 여러 구간을 동시에 감시
- 실제 현업에서도 이렇게 구성

### 시그니처란

- 악성코드의 패턴 정보 = 몽타주
- 보안 솔루션은 이 시그니처를 기반으로 패턴 매칭
- AI 발달로 시그니처를 우회하는 변종 악성코드 증가 → AI 기반 탐지로 전환 추세

### 시그니처 성능 차이 (외산 장비가 좋다는 이유)

```
회사마다 시그니처 "조합" 기술이 다름
→ 똑같은 악성코드도 어떻게 시그니처를 짰냐에 따라 성능 차이
→ 오탐이 많을 수도, 적을 수도 있음
→ 패킷 1개당 시그니처 수만 줄을 다 태워야 함 → 처리 속도(스루풋)에 영향

외산(이스라엘·미국 등 보안 강국) 장비가 좋다는 이유
→ 악성코드 분석가가 시그니처를 직접 만들고, 조합 기술이 앞서기 때문
```

### 커스텀 시그니처

```
벤더사 제공 시그니처 → 표준 공격류는 커버, 우리 서비스 특화 공격은 못 잡음
커스텀 시그니처      → 우리 서비스에 특화된 공격 패턴을 직접 만들어 장비에 투입
                     Suricata / Snort 룰로 직접 작성
```

> 시그니처 우회(난독화) 공격과 한 쌍으로 봐야 함 → Attack-Defense.md "시그니처 우회" 참고.

## UTM (Unified Threat Management)

방화벽 + IDS/IPS + 안티바이러스 + DLP를 장비 하나에 통합한 것.

```
스타트업    → pfSense (무료 소프트웨어 UTM)
중소기업    → FortiGate 중저가 모델 (100~300만원)
중견기업    → FortiGate 고급 / Palo Alto
대기업/금융  → Palo Alto + Cisco 풀스택 + 물리 망분리
```

> FIMS에서 어플라이언스 VM에 Suricata + Wazuh 올린 것 = 소프트웨어 UTM

## WAF (Web Application Firewall)

- 웹 공격 특화 악성코드 방지 장치 (HTTP 트래픽 전용)
- SQL Injection, XSS 등 웹 레벨 공격 탐지·차단
- 위험한 URL 접근 시 차단

### WAF는 IPS에서 갈라져 나온 것

```
IPS = 모든 프로토콜 대상 (HTTP 포함 전체)
WAF = HTTP 전용 전문 장비

→ IPS가 먼저 나왔고, HTTP 트래픽이 폭증하면서 WAF가 따로 등장
→ 웹 서버만 있으면 WAF 하나로 충분
→ 소켓 서버(메신저 등)가 같이 있으면 IPS도 필요
```

> IPS와 WAF가 같이 있어도 이상한 게 아님. 태생이 다르고 담당 트래픽이 다르다.
> "전체 트래픽 검문(IPS) + 웹 전용 정밀 검문(WAF)" 이중 구조.

### 차단 페이지를 안 띄우는 추세

- 예전엔 차단 시 "접근 차단되었습니다" 페이지 노출
- 요즘은 차단 페이지를 안 보여줌 → **WAF의 존재와 벤더사가 노출되기 때문**
- 그냥 빙글빙글 돌다가 일정 시간 후 풀리게 함 (공격자에게 정보를 안 줌)

## Anti-APT (샌드박스 기반)

- IPS가 **알려진** 공격을 막는다면, Anti-APT는 **알려지지 않은** APT 코드를 잡는다.
- 의심 파일을 **샌드박스(격리된 가상 환경)**에서 실제로 실행해보고 악성 행위를 탐지
- 시그니처가 아직 없는 신규 악성코드(제로데이)에 대응

> 시그니처(몽타주)가 없는 범인은 IPS가 못 잡는다.
> Anti-APT는 "수상한 놈을 격리된 방에 넣고 행동을 지켜보는" 방식 → EDR의 행위 기반 탐지와 같은 발상.

## 관제와 연관 분석 (SIEM 개념)

각 보안 장비 로그를 **따로따로** 보면 의미가 없다. 한 곳으로 모아서 **연관 분석**해야 공격이 보인다.

```
IPS에서 IP 하나 탐지        → 그것만 보면 별 의미 없음
근데 WAF에서도 같은 IP 탐지
내부 IDS에서도 같은 IP 탐지
→ 세 군데서 보이면 "이상하다" → 추적 시작

그래서 한 곳(관제 서버)으로 모은다
→ IPS + WAF + IDS + Anti-APT 로그 전부 수집
→ 연관 분석으로 공격 패턴 파악
```

> 집 앞에서 누가 서성이는 건 OK. 근데 옆집 대문 두들기고, 또 다른 집도 두들기면 이상한 것.
> 따로따로 보면 모르고, 합쳐야 보인다. = SIEM의 상관분석(Correlation).

> FIMS 연결: **Wazuh가 바로 이 관제 서버 역할**. 여러 로그를 수집해 연관 분석한 것.

## DDos 방어 장비 (Anti-DDoS)

- 디도스(DDoS) 공격을 방어하기 위한 전용 장비
- 대용량 트래픽을 견딜 수 있는 크기의 서버로 구축 (CDN도 DDos 완화 역할)
- 페이로드까지 읽을 수 있어 **L7까지 커버** → HTTP Flood 같은 응용 자원 고갈 공격도 방어

### 왜 맨 앞에 두냐

```
인터넷
   ↓
Anti-DDoS   ← 대량 트래픽(플러딩)을 여기서 먼저 거른다
   ↓
방화벽 / IPS / ...
```

- 대량 트래픽을 먼저 걸러야 **뒤에 있는 장비들이 안 죽음**
- 그래서 DDoS 장비는 보호 대상보다 훨씬 높은 스루풋(고성능)이 필요

### DDoS 장비가 없을 때 (꿩 대신 닭)

```
IPS로 대체 가능
→ IPS 임계치(Threshold) 설정으로 플러딩 방어
→ 완벽하진 않지만 없는 것보단 나음
→ 단점: IPS에 부하 걸리면 뻗어버림 → 뻗었을 때 대응 체계도 함께 마련해야 함
```

## 인라인 모드와 하드웨어 바이패스

### 인라인 모드 (Inline Mode)

```
인라인 = 라인 위에 걸쳐 있는 것
→ 트래픽이 반드시 이 장비를 거쳐야 함
→ 장비가 빠지면 통신 단절

ISP
   ↓
방화벽 (업링크 인터페이스)
   ↓
IPS (업링크 ─ 다운링크)
   ↓
L3 스위치
   ↓
서버들
```

방화벽, IPS처럼 **차단을 해야 하는 장비**는 인라인으로 들어간다. (IDS는 미러링이라 인라인 아님)

### 하드웨어 바이패스 (Hardware Bypass) — 장비 내장

```
문제: 인라인 장비가 전원 꺼지거나 장애나면 → 트래픽 단절 → 서비스 중단

해결: 장비 안에 하드웨어 바이패스 회로 내장
→ 전원 없어도 물리적으로 입출력 포트를 직접 연결
→ 장비 꺼져도 트래픽은 그냥 통과 (보안 검사는 못 하지만 서비스는 유지)
```

> 자동문이 정전되면 수동으로 밀고 들어갈 수 있게 만들어 둔 것과 같은 발상.
> "검사는 못 해도 사람은 통과시킨다" → 가용성 우선.

### 외부 바이패스 장비 (별도 장비)

```
내장 바이패스 → 장비 안 회로. 전원 꺼지면 자동 직결
외부 바이패스 → 인라인 장비 옆에 따로 두는 독립 장비
```

왜 따로 두냐 — 인라인 장비를 **계획적으로** 빼야 하는 상황 때문:

```
1. 장비 단선 (케이블 문제)
2. 장비 교체 (업그레이드, 고장)
3. 장비 점검 (유지보수)
4. 펌웨어 업데이트
```

- 이때 인라인 장비를 그냥 빼면 트래픽 단절 = 서비스 중단
- 외부 바이패스가 있으면 → 빼기 전에 바이패스로 전환 → 트래픽 우회 → **서비스 유지하며 장비 교체·점검 가능**

> 내장 바이패스는 "사고 났을 때 자동 대비", 외부 바이패스는 "계획된 작업 때 수동 전환".

## 방화벽 실물 구조

코앞에 보이던 구성도의 "선 3개"가 실물에서는 인터페이스 3개에 케이블을 각각 꽂는 것이다.

```
인터페이스 (Interface)
→ 방화벽에 케이블 꽂는 구멍 하나하나
→ 각 인터페이스 = 각각의 망
→ 예) 인터페이스 4개 = 업링크 + DMZ + WAS + DB 4개 망 분리

광 모듈 (Optical Transceiver)
→ 광케이블 꽂는 모듈, 슬롯 추가하면 포트 확장
→ 처음 도입 시 필요한 포트만 라이센스 활성화 (나머진 비활성)

스루풋 (Throughput)
→ 장비가 동시에 처리할 수 있는 트래픽 양
→ 수백 Gbps = 고성능, 스루풋 높을수록 비쌈
→ 이게 고급 장비(4천만~1억)와 일반 장비의 차이
```

### 인터페이스 구성 (3-Tier 연결)

```
방화벽
├── 인터페이스 1 (업링크) → ISP (인터넷)
├── 인터페이스 2          → DMZ (Web Server)
├── 인터페이스 3          → WAS
└── 인터페이스 4          → DB
```

> 48포트 장비라도 필요한 포트만 라이센스 켜고 쓴다. 안 쓰는 포트는 비활성 상태.

## SSL Inspection

- 전송 계층에서 암호화된 정보를 복호화해 평문화한 뒤 검사
- NGFW에 포함된 기능
- 기밀성만 유지된다면 악성코드 유입 차단에 유리

## WIPS (Wireless Intrusion Prevention System)

- 비인가 장치를 찾아 네트워크를 차단
- 인가된 Wi-Fi만 Access 허용
- Jamming 기술로 강한 전파를 보내 전자기기 신호 방해 → SSID 식별 불가

---

## EDR (Endpoint Detection and Response)

기존 백신의 한계를 보완한 엔드포인트 보안 솔루션.

### 기존 백신의 한계

- 시그니처 기반이라 처음 발견된 악성코드는 탐지 불가
- 탐지 후 격리는 가능하지만 리버싱 분석까지 어려움
- 숙련된 보안 인력 부족

### EDR의 해결 방식

- **행위 기반 탐지** 사용 → 시그니처 없는 신규 악성코드도 이상 행위로 선제 차단
- Threat Intelligence(TI) 결합 → 의심 행위를 점수화해 AI가 자동 분류·조치

```
악성 행위 탐지 (EDR 행위 기반)
    ↓
시그니처 추출
    ↓
백신에 전달 → 이후 동일 패턴 자동 차단
```

> EDR + 백신 = EDR이 행위 탐지, 백신이 시그니처 차단하는 이중 구조

---

## DLP (Data Loss Prevention) — 정보 유출 방지

### NDLP (Network DLP)
- 네트워크 안 데이터를 탐지
- 네트워크 길목에 배치

```
F/W → Backbone → Mail Server
               ↑
        여기서 NDLP가 탐지
```

### HDLP (Host DLP)
- 사용자 하드웨어에 Agent로 하여금 중요 정보 탐지
- 취급 담당자가 무단으로 고객 데이터를 하드웨어에서 옮기는 행위 방지

> 엔드포인트만으론 불충분(에이전트 삭제 가능) → 네트워크 DLP 필수, 둘 결합해야 완결

---

## DRM (Digital Rights Management)

- 무단 디지털 콘텐츠 공급/제공 막는 기술
- Agent를 통해 하드웨어 파일을 암호화
- 각 라이센스 키가 존재 → 수신자는 라이센스 키 있어야 열람 가능

> 같은 DRM 솔루션이라도 라이선스 서버가 다르면 키가 달라서 타 기업과 문서 공유 불가

---

## ECM (Enterprise Contents Management) — 문서 중앙화

- 기업 정보 중앙 집중화
- 업무 발생 모든 문서 강제 저장
- 정보 유출 원천 차단
- 문서 접근 권한 관리
- DRM과 같이 쓰면 보안 효율 증가

---

## PMS (Patch Management System)

보안 패치 및 솔루션 등의 원격 설치 유도 솔루션.

### 도입 배경
- 제로데이 취약점: 패치가 나오기 전 공격이 먼저 발생
- APT 공격: 취약점 미패치 환경이 주요 침투 경로
- 수동 관리 한계: 수백~수천 대 PC를 담당자가 직접 패치 불가

### 폐쇄망 환경에서의 PMS

```
[인터넷] → [패치 수집 서버 (DMZ)] → 망연계 → [내부 패치 배포 서버] → 엔드포인트
```

---

## 허니팟 (Honeypot)

- 외부 공격을 유도하는 보안 장치로 수동적으로 공격을 기다리는 서버
- 공격자의 패턴 및 특성 파악이 핵심
- 보안 회사뿐 아니라 일반 회사도 아키텍처 상 허니팟 배치해 연구 가능
