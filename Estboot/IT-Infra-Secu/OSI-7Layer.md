# 📡 OSI 7계층과 네트워크 프로토콜

---

## TCP/IP 4계층 (실무 기준)

OSI 7계층을 실무에서는 4계층으로 압축해서 사용한다.

```
L7 응용계층      → HTTP, DNS, FTP, SMTP
                 OSI 5,6,7계층을 묶어서 응용계층으로 표현
                 모든 앱에서 만들어진 데이터 = 페이로드

L4 전송계층      → TCP, UDP
                 포트로 앱 구분, 신뢰성 제어

L3 네트워크계층  → IP, ICMP, ARP
                 IP 주소로 경로 결정

L2 데이터링크    → Ethernet, Wi-Fi
                 MAC 주소로 같은 망 안 전달
```

> 실무에서 L5·L6는 거의 따로 말하지 않는다. 전부 L7으로 뭉뚱그림 → **실무 기준 L2 / L3 / L4 / L7 4개**.

## 레이어 포함 관계와 비용

상위 레이어 장비는 하위 레이어 기능을 **전부 포함**한다.

```
L4 ⊃ L3 ⊃ L2
→ L4 장비 = L3 + L2 기능 포함
→ L3 장비 = L2 기능 포함
→ L2 장비(깡통 허브) = L2 기능만, 상위 기능 없음 → VLAN / ACL 불가

비용도 같은 구조
L2 = 100원 (깡통)
L3 = 1,000원
L4 = 10,000원
→ 숫자 올라갈수록 기능 많고 비쌈
```

그래서 **NAT / VLAN 분리는 L3 레벨 이상부터 가능**하다.

```
L2 깡통(허브)      → 매니지먼트 없음 → VLAN·ACL 불가
L2 매니지먼트 스위치 → VLAN 분리 가능 + 상단 L3에서 ACL 걸면 제어 가능
L3 스위치          → NAT·VLAN 직접 가능
L4 방화벽          → L3 포함하니 당연히 NAT·VLAN 가능
```

### 브릿지(L2) vs L3 라우팅

```
브릿지 방식 (L2)   → 네트워크 대역이 같아서 다 접속 가능, 제어 안 됨
L3 라우팅 방식     → 네트워크가 구분되어 제어 가능
```

> "엔지니어는 웹서버에 터미널(SSH) 접속 O, 인사팀은 브라우징만" 같은 통제를 구현하려면?
> → L3에서 NAT 만들고 VLAN 분리(엔지니어용 / 인사팀용) → L3 이상에서만 가능한 작업.

## 와이어샤크로 본 패킷 구조

패킷 하나 안에 OSI 레이어가 전부 내장되어 있다.

```
1번줄 → Frame        : 패킷 전체 설명 (몇 바이트인지 등)
2번줄 → Ethernet II  : MAC 통신 = L2
3번줄 → IP           : IP 통신 = L3
4번줄 → TCP / UDP    : 전송 프로토콜 = L4
5번줄 → Data         : 페이로드 = L7
우측 코드             : 기계가 읽는 16진수 HEX 코드
```

- 송신: L7에서 만들어진 데이터가 아래로 내려오면서 각 레이어 헤더가 붙음 → **캡슐화**
- 수신: L1부터 올라가며 헤더를 벗김 → **역캡슐화**
- IPS는 이 패킷 안의 페이로드에서 악성 패턴을 탐지하고 차단함

## L7 — 응용 계층

- 사용자가 직접 쓰는 앱 레이어
- HTTP, DNS, FTP, SMTP 등
- FIMS에서 FastAPI 라우터(/api/login 등)가 전부 L7에서 동작한 것

---

## L6 — 표현 계층

- 데이터 형식 변환 + 암호화 담당
- TLS/SSL, JPEG, ASCII
- HTTPS의 암호화가 여기서 일어남
- FIMS에서 Apache에 Let's Encrypt 인증서 달아서 HTTPS 붙인 것 = L6 작업

---

## L5 — 세션 계층

- 연결 시작 · 유지 · 종료 관리
- 화상통화를 1시간 하면 그 1시간 동안 연결이 끊기지 않도록 세션을 유지하는 역할
- "세션 만료됐습니다" = L5 세션이 끊긴 것
- FIMS에서 JWT 8시간 설정 = 세션 유지 개념

> 핸드셰이크(L4)가 연결을 맺어주면, L5가 그 연결 상태를 유지·관리·종료한다.
> L4가 전화를 받는 순간이라면, L5는 통화가 유지되는 시간 전체다.

---

## L4 — 전송 계층 (TCP vs UDP)

### TCP (신뢰성 프로토콜)
- 재전송이 가능해서 신뢰할 수 있는 프로토콜
- 분할된 데이터에 시퀀스 번호가 있어서 합산이 안 맞으면 재전송 요청 후 무한대기
- 금융, 주요 거래, 메시지처럼 데이터 유실이 치명적인 곳에 사용
- 3-way handshake로 연결, 4-way handshake로 종료

### UDP (속도 우선 프로토콜)
- 재전송 규약 없음, 헤더가 심플하고 속도 빠름
- 게임, 스트리밍처럼 유실보다 실시간성이 중요한 곳에 사용

### L4의 정체성 — IP + 포트 둘 다 있어야 함

```
포트만 있어도 안 됨, IP만 있어도 안 됨
→ "어느 장비(IP)의 어느 서비스(포트)냐"가 L4의 핵심
```

### 세션 테이블 (Stateful Inspection)

한번 연결된 세션 정보를 L4가 기억하는 것.

```
이유: 서버가 2천만 명 연결을 계속 물고 있으면 터짐
→ 잠깐 연결을 끊어도 세션 값을 기억해 둠
→ 재연결 시 "그 서버"로 다시 붙여줌
→ 세션 유지 시간은 L4에서 설정 (10분, 20분 등 서비스 특성에 맞게)
```

> 단골 식당이 내 자리를 잠깐 비워둬도 기억해두는 것. 다시 오면 그 자리로 안내.

### Sticky 세션

```
라운드로빈인데도 기존에 붙었던 서버로 계속 보내는 것
→ 다른 서버로 가면 로그인 세션이 날아가니까
```

### DNS 라운드로빈 vs L4 로드밸런서 (핵심 차이)

```
DNS 라운드로빈 → 서버가 죽어도 그냥 보냄 (헬스체크 없음)
L4 로드밸런서  → 서버가 죽으면 안 보냄 (헬스체크 있음)
                서버를 "자원"으로 등록 → 죽으면 자원에서 자동 제거
```

> DNS RR은 명단 보고 순서대로 돌리는 것뿐(생사 확인 안 함),
> L4 LB는 "이 서버 살아있나?" 계속 확인하고 죽은 서버는 건너뛴다.

### VIP (Virtual IP)

```
사용자는 VIP(가상 IP)로 접속
→ 방화벽에서 DNAT(1:1 NAT)으로 실제 서버 IP에 매핑
→ L4가 VIP 하위 자원(실제 서버들)에 로드밸런싱
```

> `cmd → nslookup → 도메인` 쳤을 때 나오는 IP = 보통 이 VIP다 (실제 서버 IP가 아님).

### L4 / L7 로드밸런싱 기준

- L4 LB: IP + 포트 기준 분배 (순수 로드밸런싱에 집중, 빠름)
- L7 LB: URL, 쿠키, 헤더 기준 분배 (nginx, AWS ALB)

> 로드밸런서 설계 디테일(L3→LB 순서, HA)은 Architecture.md "로드밸런서" 참고.

## TCP 플래그 정리

| 플래그 | 이름 | 역할 |
|---|---|---|
| SYN | Synchronization | 연결 요청, 시퀀스 번호 동기화 |
| ACK | Acknowledgement | 패킷 수신 응답, 받은 시퀀스 +1 |
| FIN | Finish | 세션 연결 종료 |
| RST | Reset | 비정상적인 세션 강제 종료 |
| PSH | Push | 버퍼 안 기다리고 L7으로 바로 올려보냄 (SSH, 텔넷에서 자주 사용) |
| URG | Urgent | 긴급 데이터, 다른 데이터보다 우선순위 높음 |

> Suricata 룰에서 `flags:S` = SYN만 탐지
> SYN만 계속 보내고 ACK 안 보내면 half-open connection = **SYN Flood 공격**

## 3-way Handshake (TCP 연결)

```
클라이언트                         서버
    |                               |
    |──── SYN (연결요청) ───────────▶|  시퀀스 번호 임의 결정
    |                               |
    |◀─── SYN + ACK ───────────────|  받은 시퀀스 +1 해서 응답
    |                               |
    |──── ACK (확인) ───────────────▶|  Established 완료
    |                               |
```

- `telnet [ip] [port]` = L4에서 해당 IP의 포트가 열려있는지 확인
- `ping [ip]` = L3 ICMP 구간에서 해당 IP가 살아있는지 확인
- 처음엔 서버가 디폴트로 Listening 상태 유지

### TCP 연결 상태 (Connection States)

```
LISTEN       → 서버가 연결 대기 중 (기본 상태, 포트 오픈된 상태)
SYN_SENT     → 클라이언트가 SYN 보내고 응답 기다리는 중
SYN_RECEIVED → 서버가 SYN 받고 SYN+ACK 보낸 상태
ESTABLISHED  → 3-way handshake 완료, 실제 통신 중
FIN_WAIT     → FIN 보내고 종료 대기 중
CLOSE_WAIT   → 상대방 FIN 받고 내 FIN 준비 중
TIME_WAIT    → 종료 후 잠시 대기 (지연 패킷 처리용)
CLOSED       → 연결 완전 종료
```

- `netstat -an` 명령어로 현재 연결 상태 확인 가능
- 포트가 열려 있으면 LISTEN 상태 → `telnet [ip] [port]`로 ESTABLISHED 되는지 테스트

> SYN_SENT가 쌓이고 ESTABLISHED가 안 되면? → 방화벽이 막고 있거나 서버가 응답 안 주는 것
> SYN_SENT만 무한히 보내고 ACK 없이 반복 → **SYN Flood 공격**

## 4-way Handshake (TCP 종료)

```
클라이언트                         서버
    |                               |
    |──── FIN (종료요청) ───────────▶|
    |                               |
    |◀─── ACK ─────────────────────|
    |                               |
    |◀─── FIN ─────────────────────|
    |                               |
    |──── ACK ─────────────────────▶|  연결 종료 완료
```

- 종료는 **FIN(정상 종료)** 또는 **RST(강제 종료)** 로 시작된다 — "FIN이 날아갔냐, RST가 날아갔냐"로 끊는 방식이 갈린다.
- 클라이언트가 마지막에 **TIME_WAIT(보통 1~2분)** 로 잠시 머무는 이유: 뒤늦게 도착하는 지연 패킷을 흡수하고, 마지막 ACK가 유실됐을 때 재전송에 응답하기 위함.

> ⚠️ **세션이 유지되는 동안 포트가 바뀌면 안 된다.** 통신이 끝날 때까지 (클라이언트) 출발 포트가 고정돼야 같은 세션으로 인식된다 — 위 "세션 테이블(Stateful)"과 직결. 포트가 바뀌면 방화벽/서버가 "새 연결"로 보고 기존 세션이 끊긴다.

## TCP 시퀀스·ACK 넘버 — "서로의 시작번호를 맞추는 작업"

3-way handshake의 진짜 목적은 연결뿐 아니라 **양쪽의 시퀀스 시작번호를 동기화**하는 것이다. (와이어샤크는 보기 편하게 상대값 0, 1…로 보여주지만 실제론 임의의 난수에서 시작)

```
클라이언트 SYN     → 내 시퀀스 시작번호 제공 (Seq=0)
서버 SYN+ACK       → 받은 시퀀스 시작번호 확인 → ACK=1(받았다, +1) / 내 시퀀스도 제공(Seq=0)
클라이언트 ACK     → 서버 시퀀스 확인 → ACK=1(+1) / 내 Seq=1
 = 이제 양쪽이 서로의 Seq·ACK 기준점을 알게 됨 → 이후 데이터는 이 번호로 순서·유실 판단
```

> ACK 넘버 = "여기까지 잘 받았으니 **다음엔 이 번호부터** 보내라"는 영수증. 받은 만큼 +1(또는 +데이터길이) 해서 돌려준다. 이 번호가 안 맞으면 유실로 보고 재전송 → TCP 신뢰성의 핵심 장치.

### 윈도우 사이즈 (Window Size)

```
Window Size = "내가 한 번에 받아둘 수 있는 버퍼 크기" → 상대에게 알려줌
 → 3-way 과정에서 서로의 수신 능력(패킷 크기)을 맞추는 표시
 → 받는 쪽이 벅차면 윈도우를 줄여 "천천히 보내" 신호 (흐름 제어)
```

> 🔗 IP 헤더의 ECN(혼잡 표시)이 "망이 막힌다"는 신호라면, TCP 윈도우는 "내 수신 버퍼가 벅차다"는 신호 — 둘 다 **밀려서 터지는 걸 막는** 흐름/혼잡 제어 장치다.

### 와이어샤크 필터링 — 수많은 패킷에서 원하는 것만

```
ip.addr == 172.30.1.32 && tcp.port == 80
 → 특정 IP + 특정 포트 트래픽만 추려서 봄
```

- 캡처는 순식간에 수천 줄이 쌓이므로 **필터를 거는 게 분석의 기본기**다. `Statistics → Flow Graph`로 3-way·데이터·4-way 흐름을 한눈에 본다.

## UDP 상세 — 단순함이 곧 속도

TCP가 신뢰성을 위해 시퀀스·ACK·세션을 다 달았다면, UDP는 **"빨리 보내는 것"** 이 유일한 역할이라 헤더가 몇 줄밖에 없다.

```
UDP 헤더 = 출발 포트 / 목적 포트 / 길이(Length) / 체크섬(Checksum)  — 끝.
 → 시퀀스·ACK·연결상태 전부 없음 → 재전송도 없음
 → 중간에 유실되면? 그냥 손실. 끝. (스트리밍·게임·DNS·VoIP)
```

### 닫힌 포트로 UDP를 보내면 → ICMP Port Unreachable

```
UDP는 핸드셰이크가 없어 "포트 열림" 확인을 못 한다
 → 하지만 닫힌 포트로 보내면 받는 쪽이 ICMP "Destination Unreachable (Port Unreachable)" 를 돌려줌
 → 즉 응답이 없으면 "열렸거나 필터됨", ICMP가 오면 "닫힘"
```

> 🔗 이게 nmap의 **UDP 스캔(-sU)** 이 느리고 부정확한 이유 — 열린 포트는 응답이 없어 구분이 모호하고, 닫힌 포트만 ICMP로 확실히 잡힌다. (포트스캔: Attack-Defense.md "포트 스캔(nmap)")

### UDP 위에 신뢰성을 얹은 신종 프로토콜

```
QUIC (Quick UDP Internet Connections)
 → 구글이 미는 프로토콜. UDP 위에서 TLS 암호화 + 연결관리를 직접 구현
 → TCP의 느린 핸드셰이크를 걷어내고, UDP의 속도에 신뢰성·암호화를 얹음 → HTTP/3의 기반

DCCP (Datagram Congestion Control Protocol)
 → UDP의 속도를 살리되 망 혼잡도를 계산해 전송률(QoS)을 조절하는 혼잡제어형 프로토콜
```

> 정리: "TCP는 안전하지만 느리고, UDP는 빠르지만 막 던진다"의 **중간을 메우려는** 시도들. QUIC는 특히 모바일·웹에서 TCP를 빠르게 대체하는 중. (이 둘은 L4 영역의 새 흐름 정도로만 기억)

## L3 — 네트워크 계층

```
IP    → 데이터를 목적지까지 배달하는 규칙 (항상 동작)
ICMP  → 배달 경로가 살아있는지 확인 (ping, traceroute)
        IP보다 통신 우선순위가 낮음 (확인용이라 늦어도 무방)
ARP   → IP 주소를 MAC 주소로 변환
        "이 IP가 누구 MAC이야?" 물어보면 해당 장비가 응답
```

> IP주소(내가 가진 고유번호)와 IP프로토콜(배달 규칙)은 이름은 같지만 다른 개념이다.

### IP / ICMP / ARP 흐름

```
데이터 보내려면
  1. 목적지 IP 주소 확인        ← IP
  2. 그 IP의 MAC 주소 알아내기  ← ARP
  3. 경로 살아있는지 확인       ← ICMP
  4. 실제 배달                  ← IP
```

### ARP 동작 원리

ARP = IP 주소를 MAC 주소로 변환하는 프로토콜.

```
1. ARP 테이블 먼저 검사 (이미 알면 끝)
2. 없으면 브로드캐스트 → "이 IP 가진 사람 누구야?" (모두에게 외침)
3. 해당 장비가 유니캐스트로 응답 → "나야" (1:1로 대답)
4. MAC 주소 획득 후 통신
```

**왜 굳이 MAC으로 통신하냐**

```
IP  = 논리적 주소 → 미국까지도 찾아감 (어느 동네인지)
MAC = 물리적 주소 → 실제 그 장비 (그 집 문패)

L2 레이어에서는 프레임 통신 → 결국 MAC으로 바꿔야 실제 장비에 도달
→ ARP가 L3에서 L2 MAC 정보를 관리하는 다리 역할
```

> 택배로 치면 IP는 "서울시 강남구 OO동"(주소), MAC은 "302호 김철수"(받는 사람).
> 동네까지는 IP로 찾아가지만, 마지막에 실제 문 앞에 꽂으려면 MAC이 필요하다.

> ⚠️ ARP는 응답(Reply) 패킷의 MAC 정보를 **검증하지 않는다** → ARP 스푸핑의 근본 원인.
> 공격 시나리오는 Attack-Defense.md "ARP 스푸핑" 참고.

### ICMP 우선순위가 낮은 이유

- ping이 느리다고 서비스가 느린 게 아님 (ICMP는 후순위 처리)
- ping이 막혀도 실제 서비스는 동작할 수 있음 (방화벽에서 ICMP만 drop하는 경우)
- ping 안 된다고 서버가 죽은 게 아님

### 💡 단편화(MF)를 눈으로 보려면 ICMP를 써라 (실습 트릭)

위 "MTU와 단편화"에서 정리한 MF 플래그를 **와이어샤크에서 실제로 보려면 TCP는 불리하다.** TCP는 연결 시점에 MSS를 협상해 MTU에 맞춰 미리 쪼개 보내므로, MF가 켜진 큰 조각을 잡기 어렵다. 반면 **ICMP는 페이로드를 협상 없이 그대로 실어 보내기 때문에**, MTU(1500)보다 큰 패킷을 쏘면 그 자리에서 단편화가 일어난다.

```bash
ping 192.168.1.1 -l 3500     # -l = 페이로드 크기 3500바이트 (MTU 1500 초과)
```

```
관찰: 한 ping이 1480바이트짜리 조각 여러 개로 쪼개짐
 → 앞 조각들 MF=1 (뒤에 더 있음) / 마지막 조각 MF=0
 → Fragment Offset이 0 → 1480 → 2960 … 으로 늘어남 (조각 위치)
 → 마지막 조각은 남은 데이터만 실려 Total Length가 뚝 떨어짐
```

> 🔗 TCP에서 MF가 안 보였던 이유 = MSS 협상으로 **미리** MTU에 맞춰 보냈기 때문. "단편화는 보낼 쪽이 MTU를 모를 때(또는 ICMP처럼 협상 없이 보낼 때) 일어난다"가 핵심. 위 "MTU와 단편화", Attack-Defense.md "단편화 기반 DoS"와 직결.

### 라우팅 — 최적 경로 · Next-hop · 블랙홀 라우팅

```
Next-hop(다음 홉)  → 이 패킷을 넘길 "바로 다음 라우터"를 지정
경로 탐색           → 라우팅은 항상 최적 경로를 찾으려 한다
경로 꼬임           → 드물지만 라우팅 정보가 엉키면 최적이 아닌 길로 돌아감
                     → 사람은 못 느껴도 기계 입장에선 지연/손해가 숫자로 잡힘
```

통신 중간엔 라우터가 여럿 있고, **라우터끼리 각자의 MTU가 다르다.** 그래서 경로 위에서 가장 작은 MTU에 맞춰 서로 합의(Path MTU)하고 보낸다. → 위 단편화가 일어나는 또 다른 지점.

⚠️ **블랙홀 라우팅 (Blackhole Routing) — ISP의 DDoS 흡수책**

```
ISP에 DDoS가 들어오는데 라우터 한 대가 감당 못 할 때
 → 공격 대상 IP로 가는 트래픽을 "아무 데도 안 가는 더미 경로"로 보내버림
 → 갈 곳 없는 패킷은 홉을 돌다 TTL이 0이 되어 소멸
 → 정상 서비스를 살리기 위해 그 IP 트래픽을 통째로 버리는(=구멍에 빠뜨리는) 것
```

> 비유: 식당에 폭도 200명이 몰려오면, 그 무리가 향하는 통로 자체를 막아 **빈 방으로 흘려보내** 매장(서비스)을 지키는 것. 대상 IP는 잠깐 죽지만 ISP망 전체는 산다 → 손절 방어. (Anti-DDoS 장비가 정밀 필터라면, 블랙홀 라우팅은 ISP 차원의 거친 차단)

> 🔗 TTL(위 "TTL — 무한 루프 방지")을 **방어에 역이용**하는 사례 — 원래는 루프 방지용인 TTL 소멸을, 공격 트래픽을 말려 죽이는 데 쓴다.

## IPv4 / IPv6 주소 체계

위 "TCP/IP 4계층"에서 정리한 그대로, **OSI 7계층은 네트워크가 만들어질 때의 이론적 청사진**이고, 실제로 우리가 쓰는 주력은 **TCP/IP**다. 그 TCP/IP의 핵심 주소가 IP다.

```
IPv4 = 32비트 = 8비트 × 4옥텟 = 2진수 8자리짜리 4칸
       → 2^32 ≈ 약 43억 개의 주소
IPv6 = 128비트 → 사실상 무한대
```

- IoT 기기가 폭증하면서 43억 개로는 부족 → 그래서 나온 게 **IPv6(128비트)**.
- 테스트는 거의 끝나가고, **내부망 실무부터 IPv6로 바꾸는 추세**다.

> 비유: IPv4가 "전화번호 8자리 체계"라 번호가 동나는 상황. IPv6는 "전화번호 자릿수를 통째로 늘려버린" 것 — 자릿수가 많아지면 가능한 번호 수가 폭발적으로 늘어난다.

## 캐스트 종류 — 유니/브로드/멀티

패킷을 "몇 명에게 보내느냐"로 갈린다. ARP를 보면 두 개가 한 번에 보인다.

```
유니캐스트(Unicast)    → 1 : 1     특정 한 대에게만
브로드캐스트(Broadcast)→ 1 : 전체   같은 네트워크 대역 전원에게
멀티캐스트(Multicast)  → 1 : 그룹   가입한 그룹에게만 (IPTV 셋탑박스)
```

- **ARP가 캐스트의 교과서**: "이 IP 가진 사람 누구야?"라고 **브로드캐스트**로 외치면, 해당 장비가 "나야"라고 **유니캐스트**로 대답한다. (위 "ARP 동작 원리"와 직결)
- **멀티캐스트** = IPTV·셋탑박스 영역. 채널을 본 사람(그룹 가입자)에게만 같은 스트림을 한 번에 뿌린다.

⚠️ **브로드캐스트 경계 = 그 네트워크 대역**이다. C클래스(/24)일 수도 있고 아닐 수도 있다 — 서브넷을 어떻게 잘랐느냐에 따라 다르다.

> 그래서 **"내 네트워크 현황(어디까지가 같은 대역인지)을 읽을 줄 아는 것"** 이 기본기다. 브로드캐스트가 어디까지 퍼지는지를 모르면 ARP 스푸핑 범위도, 세그먼트 경계도 가늠 못 한다.

## 사설 IP 대역 (Private Address)

```
A class : 10.0.0.0     /8   (10.x.x.x)
B class : 172.16.0.0   /12  (172.16 ~ 172.31.x.x)
C class : 192.168.0.0  /16  (192.168.x.x)
```

- 회사 내부망은 보통 **B·C 대역**으로 구성한다. (우리 랩도 192.168.x = C, 10.10.x = 내부)
- ⚠️ **내부의 DNS 정보가 외부로 노출되면 절대 안 된다** → 내부용 IP는 내부 DNS에서만 풀이 = **Split DNS** (위 "DNS" 참고). 내부 대역·호스트명이 새면 공격자에게 내부 지도를 그냥 주는 셈.

## IPv4 헤더 분석 (와이어샤크로 뜯어보기)

`ipip 했던 게 사실 이 프로토콜이었구나` — 패킷 하나의 L3 헤더 안에 이 정보들이 다 들어있다. 와이어샤크에서 `Internet Protocol Version 4` 줄을 펼치면 보이는 필드들.

| 필드 | 값 예시 | 의미 |
|---|---|---|
| Version | 4 | IPv4 / IPv6 구분 |
| Header Length | 20 bytes (5) | IP 헤더 크기 |
| **DSCP (CS0~)** | CS0 | **우선순위** — 숫자 높을수록 우선 처리 |
| **ECN** | Not-ECT / 11 | **혼잡 표시** — 11이면 "패킷 밀리는 중" 경보 |
| **Total Length** | 40 ~ 65,535 | 헤더 + 데이터 전체 길이 (최대 65,535바이트) |
| **Identification** | 0xd86d | **단편 묶음 식별자** (같은 원본에서 쪼개진 조각은 같은 ID) |
| **Flags / Fragment Offset** | MF, offset | 단편화 제어 (아래 상세) |
| **TTL** | 128 | 살아남을 수 있는 홉(라우터) 수 |
| **Protocol** | TCP(6) | 상위 프로토콜 번호 |
| **Header Checksum** | 0x81ef | 헤더 무결성 검증 값 |
| **Source / Destination** | 192.168.1.83 → 170.114.52.5 | 출발지 / 목적지 IP |

### DSCP(우선순위)와 ECN(혼잡도)

```
DSCP (CS0, CS1 …) = 우선순위 등급. 숫자가 높을수록 먼저 처리
ECN               = 네트워크 혼잡 상태 표시. 11로 뜨면 "삐용삐용, 패킷 밀리는 중"
```

> 💡 실사례: **산사태 현장에서 소방관 무전이 다른 트래픽에 밀려 안 나갔던** 사고 → 그 뒤로 긴급 통신은 우선순위를 최상으로 올린다. DSCP가 바로 "이 트래픽 먼저 통과시켜라"를 정하는 자리. (위 TCP 플래그의 URG와 발상이 같다 — 급한 건 새치기)

### Total Length & Identification — 헷갈리지 말 것

```
Total Length   = 이 패킷이 실제로 쓴 용량 (헤더 + 데이터), 최대 65,535바이트
Identification = 한 데이터를 여러 조각으로 쪼갰을 때, "이 조각들은 한 묶음"이라고 표시하는 식별자
```

> 💡 **수업노트 정정**: "Identification = 네트워크가 보낼 수 있는 최대전송 단위"는 **틀린 정리**다. 최대전송단위는 **MTU**(아래)고, Identification은 **단편들을 같은 원본으로 다시 합치기 위한 ID**다. 둘은 완전히 다른 필드. (조각이 뒤섞여 도착해도 같은 Identification끼리 모아 offset 순서대로 재조립한다)

### MTU와 단편화(Fragmentation)

```
MTU(Maximum Transmission Unit) = 한 번에 보낼 수 있는 최대 크기
 → 이더넷 환경에서 보통 1500바이트로 통용

데이터가 MTU보다 크면? → 여러 조각으로 쪼개서 보냄 = 단편화(Fragmentation)
 → 조각이 순서 뒤죽박죽 도착할 수 있으니, 다시 합칠 장치가 필요
```

> 💡 **인터페이스별 MTU 확인 (Windows, 6/24 번외):** `netsh interface ipv4 show subinterfaces`
> → 인터페이스마다 MTU 값과 In/Out 바이트가 같이 나온다. VPN 터널은 MTU가 1500이 아니라 1350처럼 작은데(헤더 캡슐화 오버헤드만큼 깎임), 이게 위 Path MTU·단편화가 실제로 갈리는 지점이다. (Linux는 `ip link` / `ifconfig`)

조각을 합치는 3개 장치 = **Identification(같은 묶음 표시) + Fragment Offset(조각 위치) + MF 플래그(More Fragments)**.

```
표 1.05  단편화된 패킷 (MTU 1500 기준, 헤더 20 빼면 데이터 1480씩)

첫 번째 패킷  Total Length 1500 │ data 1480 │ MF=1 │ Fragment Offset 0
두 번째 패킷  Total Length 1500 │ data 1480 │ MF=1 │ Fragment Offset 1480
   …
마지막 패킷   나머지 데이터        │ MF=0 │  ← MF=0 = "내가 마지막 조각이다"
```

- **MF=1** = "뒤에 조각이 더 있다, 기다려라" / **MF=0** = "이게 마지막, 이제 합쳐라"
- 받는 쪽은 MF=0을 받을 때까지 조각들을 버퍼에 모아두고 기다린다.

⚠️ **여기가 공격 포인트다 (단편화 기반 DoS — Teardrop류)**

```
공격자가 MF=1짜리 조각만 계속 보내고 → 마지막(MF=0)을 영원히 안 보냄
 → 받는 쪽은 "다음 조각 오겠지" 하고 버퍼에 쌓아둔 채 무한 대기
 → 미완성 조각들이 메모리를 점유 → 자원 고갈로 말라 죽음
```

> 🔗 위 "SYN Flood"(SYN만 보내고 ACK 안 줘서 half-open으로 자원 고갈)와 **완전히 같은 발상**이다 — "끝맺지 않는 미완성 연결/조각으로 상대 자원을 잠가버린다." (Attack-Defense.md "DoS — 미완성 상태로 자원 고갈" 참고)

### TTL (Time To Live) — 무한 루프 방지

```
TTL = 이 패킷이 살아남을 수 있는 홉(라우터) 수
 → 라우터를 하나 지날 때마다 1씩 감소
 → 0이 되면 그 패킷은 폐기됨
```

> 💡 **수업노트 보강**: TTL의 본질은 "재전송 횟수 제한"이 아니라 **라우팅 루프 방지**다. 만약 경로가 꼬여서 A→B→A→B로 패킷이 뱅뱅 돌면(라우팅 루프), TTL이 없으면 그 패킷이 네트워크에 영원히 남아 회선을 좀먹는다. TTL이 매 홉마다 깎여 0이 되면 강제 폐기 → 죽은 패킷이 망을 떠도는 걸 막는다.

> 비유: 택배에 "최대 10개 물류센터까지만 거쳐라" 도장을 찍어두는 것. 주소가 꼬여서 센터끼리 무한 핑퐁해도, 10번 거치면 자동 폐기되어 시스템이 안 막힌다. `traceroute`가 바로 이 TTL을 1, 2, 3…으로 늘려가며 경로를 그린다.

### Protocol 번호 & 무결성 체크섬

```
Protocol 필드 = 이 IP 패킷 안에 든 상위 프로토콜이 뭔지
  0x01(1)  ICMP   |  0x06(6)  TCP
  0x11(17) UDP    |  0x2F(47) GRE
  (전체 목록: 위키피디아 "List of IP protocol numbers")
```

```
Header Checksum = 패킷이 운반 중 깨졌는지 확인하는 무결성 값
  보낼 때 체크섬 계산 → 받을 때 다시 계산해 비교 → 다르면 깨진 패킷
```

> 🔗 이 체크섬이 **인라인 IPS에서 문제를 일으키는 그 체크섬**이다 — NIC이 "나가기 직전"에 체크섬을 채우는 오프로딩이 켜져 있으면, 인라인 Snort가 그 전에 패킷을 가로채 "체크섬 비어있는 깨진 패킷"으로 오인한다. 그래서 인라인 IPS는 체크섬 오프로딩을 끈다. (Security-Solutions.md "Hardware Checksum Offloading을 꺼야 인라인이 돈다" 참고)

> **한 줄 정리**: IP 헤더 안에 ① 우선순위(DSCP)·혼잡(ECN) ② 크기(Total Length)·단편 식별(Identification) ③ 단편 합치기(MF/Offset) ④ 루프 방지(TTL) ⑤ 상위 프로토콜(Protocol) ⑥ 무결성(Checksum) ⑦ 출발/목적지가 다 들어있다. **"ipip"로 뭉뚱그려 쓰던 게 사실 이 정교한 규약이었던 것.**

## L2 — 데이터링크 계층

- MAC = 네트워크 장비끼리 서로를 구분하는 고유 주소
- 하드웨어에 고정된 값이지만 소프트웨어로 임의 변경 가능 = **MAC 스푸핑**
- 기업에서 퇴사자 PC의 IP + MAC 주소를 폐기하는 이유

### L2 스위치 vs 허브

```
허브      → 깡통. 들어온 패킷을 모든 포트에 그냥 뿌림
L2 스위치  → 매니지먼트 시스템 있음
            MAC 정보를 포트에 매핑해서 해당 MAC 아니면 DOWN 가능 = PORT SECURITY
```

### NAC (Network Access Control)

MAC 스푸핑을 막기 위해 등장한 시스템.
네트워크에 접속하는 기기의 보안 상태를 검사해서 접근을 허용/차단.

```
[사용자 PC 접속 시도]
    ↓
[NAC] → 백신 설치 여부 확인
      → OS 패치 최신 여부 확인
      → 인가된 기기인지 확인
      → MAC 충돌 탐지 (같은 MAC이 두 포트에서 잡히면 스푸핑으로 판단)
    ↓
허용 → 내부망 접근
차단 → 격리망으로 이동
```

> MAC 필터링만으로는 약함. 802.1X 인증(인증서 기반)과 함께 써야 완결.
> NAC 재정이 부족하다면 L2 스위치 PORT SECURITY로 대체 가능.

## L1 — 물리 계층

- 0과 1을 전기신호(랜선), 빛(광섬유), 전파(와이파이)로 바꿔서 물리적으로 전송
- 랜선 뽑으면 여기서 끊기는 것
- 허브는 L1 장비 (깡통)

### Full Duplex vs Half Duplex (전이중/반이중)

```
Full Duplex (전이중) → 송신·수신을 동시에. "1Gbps 양방향" = 보내면서 받는다
Half Duplex (반이중) → 한 번에 한 방향만 (무전기처럼 번갈아)
```

> 비유: Full = **왕복 2차선**(오는 차·가는 차 동시), Half = **외길 1차선**(한쪽 지날 동안 반대쪽 대기).
> 옛날 허브는 충돌(collision) 때문에 Half, 요즘 스위치 포트는 Full. "1기가 통신"은 보통 Full Duplex 1Gbps.

## ThroughPut (처리량)

```
ThroughPut = 단위 시간당 실제로 처리된 데이터량
```

- IPS, 방화벽이 패킷을 검사하면 속도가 느려질 수 있음
- 고성능 보안 장비일수록 ThroughPut 수치가 중요한 스펙
- 이론상 대역폭과 실제 처리량은 차이가 있음

## SSL 암호화와 보안의 딜레마

```
해커      → 악성 데이터를 SSL로 암호화해서 숨기려 함
화이트햇   → SSL 복호화해서 내용 분석
NGFW      → SSL Inspection으로 복호화 후 검사 후 재암호화
```

- HTTPS 트래픽은 암호화되어 있어서 일반 방화벽으로는 내용 볼 수 없음
- 그래서 차세대 방화벽(NGFW)은 SSL을 한번 풀어서 검사하고 다시 암호화해서 보냄

## DNS (Domain Name System)

IP를 외울 수 없어서 도메인 이름을 IP로 변환해주는 역할.

### DNS 서버 동작 방식

```
1. 로컬 DNS 서버에 조회 요청
2. 없으면 Root DNS 서버에 조회
3. Root DNS가 .org/.com 등 TLD DNS 정보 제공
4. TLD DNS에서 해당 도메인 DNS 서버 정보 제공
5. 해당 도메인 DNS에서 IP 반환
6. 로컬 DNS가 TTL 동안 캐시 유지 후 기기로 전달
```

- 보안상 내부에서만 공개되어야 하는 IP는 내부 DNS에서 운영
- 외부에 공개되어야 하는 IP는 외부 DNS에서 운영 = **Split DNS**
- DNS는 보통 UDP 53, 응답이 512바이트 초과하거나 Zone Transfer 시 TCP 53

### DNS 스푸핑

- 공격자의 사이트로 유도해 사용자 데이터 탈취
- hosts 파일 변조 (`C:\Windows\System32\drivers\etc\hosts`)로 다른 사이트 유도 가능

### 라운드 로빈

- 같은 도메인에 A 레코드를 여러 개 등록하면 요청마다 순서대로 다른 IP로 분산
- 서버 장애 감지 기능 없음 → 실무에서는 로드밸런서와 함께 사용

## VLAN (Virtual LAN)

### 세그멘테이션 vs VLAN

```
세그멘테이션 (FIMS 방식)
→ VirtualBox 내부 네트워크 이름 기반
→ 소프트웨어 격리
→ 하이퍼바이저 취약점 뚫리면 격리 붕괴 가능

VLAN (802.1Q)
→ 물리 스위치 하드웨어가 태그로 나눔
→ 소프트웨어 취약점과 무관하게 격리 유지
→ 다른 VLAN끼리 통신하려면 반드시 L3 라우터 경유 필요
```

> 면접에서 FIMS를 "VLAN 망분리"라고 하면 틀림.
> "VirtualBox 내부 네트워크로 세그멘테이션 했다"가 정확한 표현.

### VLAN 실습 방법 (물리 장비 없이)

```bash
# VirtualBox + ip link 명령어로 802.1Q 태그 직접 생성
ip link add link enp0s8 name enp0s8.10 type vlan id 10
ip link add link enp0s8 name enp0s8.20 type vlan id 20
ip link set enp0s8.10 up
ip link set enp0s8.20 up
ip addr add 192.168.10.1/24 dev enp0s8.10
ip addr add 192.168.20.1/24 dev enp0s8.20
```

- GNS3: Cisco IOS 이미지 올려서 실제 VLAN 명령어 실습 가능 (CCNA 준비에 직결)

### 액세스 포트 vs 트렁크 포트 — 태그를 누가 다루나

```
액세스(Access) 포트 → 한 VLAN만. 단말이 붙는 끝단. 태그 없이 단일 VLAN 트래픽만.
트렁크(Trunk) 포트   → 여러 VLAN을 태그를 단 채로 한 선에 실어 나름 (스위치↔스위치, 스위치↔라우터/방화벽)
```

> 비유: 액세스 = **한 동네만 도는 마을버스**, 트렁크 = **여러 동네 우편을 태그 붙여 한 트럭에 싣는 간선버스**. 다른 VLAN끼리 통신하려면 이 트렁크를 타고 L3(라우터/방화벽)까지 가서 라우팅돼야 한다.

### ESXi에서의 VLAN — 포트그룹 VLAN ID로 모드가 갈린다 (실습 6/11)

가상화(ESXi)에서 VLAN은 **포트그룹의 VLAN ID 값**으로 동작 방식이 정해진다.

| VLAN ID | 모드 | 누가 태그를 다루나 |
|---|---|---|
| 0 | EST | 외부 물리 스위치가 태깅 |
| 1~4094 | **VST** | ESXi vSwitch가 태깅/언태깅 (포트그룹마다 고정 VLAN) |
| **4095** | **VGT (트렁크)** | **게스트 VM이 직접** — 모든 태그를 안 지우고 게스트로 넘김 |

> 🧠 **4095 = VGT = 트렁크.** pfSense 같은 게스트를 VLAN 라우터로 쓰려면, 그 게스트가 붙은 포트그룹을 **4095로 열어 모든 VLAN 태그를 통과**시켜야 한다. 그래야 pfSense가 10번·20번 태그를 직접 보고 VLAN 10/20으로 라우팅한다. 특정 ID로 고정(VST)하면 vSwitch가 태그를 벗겨버려 게스트가 VLAN을 구분 못 한다. → 실습 상세: `../Lab/2026-06-11-VLAN구축.md`

### ACL (Access Control List)

> 누가 어디로 가는 걸 허용/차단할지 정한 목록

- VLAN으로 망을 나눴다면 ACL로 그 망끼리의 이동을 통제
- 표준 ACL: 출발지 IP만 보고 차단
- 확장 ACL: 출발지 IP + 목적지 IP + 포트까지 보고 차단 (실무에서 주로 사용)

```
VLAN = 망을 나누는 것
ACL  = 나눠진 망끼리 이동을 통제하는 것
```
