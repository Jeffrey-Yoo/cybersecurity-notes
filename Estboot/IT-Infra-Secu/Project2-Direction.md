## 0. 한 장 요약 — 이번 프로젝트가 1차(FIMS)와 다른 점

```
FIMS(과정1)          : 단일 IDC 6대 + AWS, "돌아가게 만든다"가 목표
과정2 프로젝트        : 두 사이트(서비스 IDC ↔ R&D/Dev)를 site-to-site로 묶고,
                       배운 솔루션 전부를 "왜 여기 두는지" 정당화하며 배치한다
```

- **키포인트(강사님 원문)**: *"수많은 배운 것들을 어디에, 어떤 목적을 가지고, 어떻게 배치할 것인가."*
- **진짜 평가 포인트**: 장비를 많이 그리는 게 아니라 — **보안 모듈이 살아있고, 그 정책이 얼마나 고도화돼 있는가.** *"정책도 아키텍처의 일부다."*
- 그래서 이 문서의 절반은 "토폴로지", 절반은 "정책·배치 정당화 논리"다.

## 1. 큰 그림 — 두 개의 "나눔"을 헷갈리지 마라 ⭐ (내가 막혔던 지점)

내가 "IDC가 두 사이트로 나뉘고 한쪽은 데이터·한쪽은 채널"이라고 이해했는데, **실은 서로 다른 두 단계의 나눔이 겹쳐 있어서** 헷갈린 거다. 분리해서 보면 풀린다.

### 나눔 A — 사이트 레벨 (지역 ↔ 지역, site-to-site VPN으로 연결)

```
[서비스 IDC]  ◄──── IPsec Site-to-Site VPN ────►  [R&D / Dev 사이트]
 대외 서비스가                                       개발·연구·사내운영·관제가
 실제로 도는 곳                                       모여 있는 곳
```

- 강사님 원문: *"하나는 서비스를 하는 곳, 하나는 R&D→dev. 데이터센터만 있다? 안 되고, R&D만 있다? 안 된다."* → **둘 다 있어야** 현실적인 구성이다.
- 둘을 잇는 건 **Site-to-Site VPN(IPsec, pfSense↔pfSense)**. 그래서 *"NMS가 양쪽에 다 있어야 하나? R&D 한쪽에 두고 site-to-site로 IDC를 봐도 되지"* 같은 판단이 가능해진다.

### 나눔 B — IDC 내부 레벨 (한 사이트 안에서 데이터 ↔ 채널)

```
서비스 IDC 내부
 ├── 채널 영역(처리/프론트) : 외부 인입을 받아 처리 — DMZ·WAF·웹/WAS·API·외부DNS·망연계
 └── 데이터 영역(메인/저장)  : 라이브 DB·파일서버·스토리지 (뒤쪽, 폐쇄망에 가깝게)
```

- 강사님 원문: *"IDC도 데이터를 뿌려주는 메인이 있을 것이고, 처리하는 채널로 나뉜다."*
- **이 "데이터/채널"이 바로 내가 R&D랑 헷갈린 부분.** 데이터/채널은 **IDC 안쪽의 앞단/뒷단 나눔(나눔 B)**이지, IDC↔R&D 나눔(나눔 A)이 아니다.

> 💡 비유: **나눔 A = 본사 건물 vs 연구소 건물**(서로 다른 주소, 전용선으로 연결). **나눔 B = 본사 건물 안의 민원창구(채널) vs 금고(데이터)**. 내가 "민원창구 vs 금고"를 "본사 vs 연구소"랑 같은 층위로 본 게 혼선의 원인.

🔗 FIMS 연결: FIMS의 `DMZ(로비) ─ WAS(사무실) ─ DB(금고앞) ─ Storage(금고)`가 정확히 **나눔 B(채널→데이터)**였다. 이번엔 그 위에 **나눔 A(R&D 사이트)를 한 겹 더** 얹는 것.

## 2. R&D 사이트에는 어떤 망(網)이 존재해야 하나 ⭐⭐ (방향성)

> "틀은 이해되는데 R&D쪽 망 설계가 막힌다" — 그래서 **요건 하나당 망 하나**로 역산했다.
> 망을 먼저 그리지 말고, **"이 사이트에 누가 있고 무슨 일을 하나 → 그래서 어떤 위험이 생기나 → 그걸 끊으려면 어떤 망이 필요한가"** 순서로 가면 자연스럽게 나온다.

R&D 사이트에 사는 사람·일(노트 기준): **개발자, 데이터 관리자/정보취급자, 고객센터(CS) 상담원, 보안관제 인력.** 여기서 망을 역산하면:

| # | 망(VLAN) | 왜 존재해야 하나 (요건) | 핵심 통제 |
|---|---|---|---|
| 1 | **개발망 (Dev)** | 개발 사고(설정 실수·악성코드 테스트)가 운영으로 번지면 안 됨 | 인터넷 직접 X(프록시 경유), 형상관리/빌드서버, 운영과 단방향 |
| 2 | **배포 게이트웨이/스테이징** | 코드가 함부로 운영에 못 들어가게 — 승인·검증 거쳐 IDC로 | 개발→스테이징→운영 **단방향 배포 파이프라인** |
| 3 | **검역/격리망 (Quarantine·Guest)** | NAC 자세점검 실패 단말을 던질 곳 | 패치 인프라(아래 #4)에만 접근 허용, 정상망 차단 |
| 4 | **패치/배포 인프라망** | 격리된 단말을 치료해서 다시 들여보내야 함 | PMS·OS패치 설치서버·SSL복호화 에이전트 배포·백신 업데이트 서버 |
| 5 | **DB 접근 프록시망** | 정보취급자/CS가 운영 라이브DB를 직접 못 건드리게 | **레플리카 DB + DB Safer/ProxySQL** 게이트웨이 경유만, 위험쿼리 차단 |
| 6 | **고객센터(CS) 업무망** | 상담원이 개인정보를 열람함 → 유출·캡처 위험 | DLP·DRM·화면통제, DB Safer 경유 조회만 |
| 7 | **보안관제망 (SOC)** | 두 사이트 로그를 한 곳에서 봐야 함 | ESM·NMS(Zabbix)·EMS·ELK, 접근 극도로 엄격, site-to-site로 IDC 로그 수집 |
| 8 | **관리망 (OOB Mgmt)** | ESXi·pfSense·스위치 관리 인터페이스 노출 금지 | 호스트 전용/전용선으로만 접근, 업무망과 물리/논리 분리 |
| 9 | **사용자 업무망 (OA)** | 일반 임직원 PC (개발자와도 분리) | NAC 자세점검 대상, 프록시 경유 인터넷 |
| 10 | **원격접속(VPN) 인입망** | 재택 개발자 Remote Access 종단 | 닌자게이트 방지(개발망+인터넷망 동시연결 금지), MFA |

### 이 표를 만들 때 쓴 강사님 단서들 (근거)

- **NAC(PacketFence)는 대외계에 두지 않는다 → R&D로 간다.** NAC가 보는 것 = 단말에 *백신 있나, SSL 복호화 에이전트 있나, 패치 됐나*. 미달이면 **격리 게스트망**으로 보냄(→ 망 #3). → "격리됐는데 패치를 어디서 받지?"의 답이 **패치 인프라망(#4)**. *PMS·패치 설치서버·SSL에이전트 서버가 격리 단말을 치료 후 입장시킨다.*
- **무분별한 데이터 추출 방지 → DB Safer / ProxySQL(#5).** 입력 쿼리가 `DELETE`면 DB가 통째로 날아간다(라이브는 실시간으로 쌓이는데 계정정보를 지웠다면?). → **위험 명령어를 블록하고 전달.** 원래 운영DB는 원격세션을 닫아둔다(왜 DBA가 운영에 직접 쿼리를 날려?). 그래서 정보취급자/CS는 **라이브가 아니라 레플리카(복사본)**를 본다 — 라이브에 직접 패킷을 유발하면 트래픽 문제도 생기니까. 그 레플리카를 **프록시 게이트웨이(DB Safer)**로만 조회.
  - 🔗 이건 Attack-Defense의 *snoopy로 rm·ssh 명령을 추출해 감사하던 것*과 같은 발상 — "위험 행위를 길목에서 잡아 차단·기록".
- **NMS는 양쪽 다 둘 필요 없다(#7).** R&D에 두고 site-to-site VPN으로 IDC를 모니터링하면 됨. 그때 **VPN 방화벽 정책**으로 어느 망이 어느 망을 보는지 통제.
- **관리망은 전용선/호스트 전용으로(#8).** 아래 4절(랩 구성)에서 상세.

> 💡 R&D쪽 핵심 한 줄: **"만드는 곳(개발)·고치는 곳(패치)·보는 곳(관제)·다루는 곳(데이터취급)"** 네 덩어리. 서비스 IDC가 "외부에 내주는 곳"이라면, R&D는 **"내부에서 굴리고 지키는 곳"**이다.

🔗 FIMS 연결: FIMS엔 R&D 사이트라는 개념이 없었다(개발=운영이 한 박스). 이번에 **개발망을 운영에서 떼어내고 그 사이에 배포 게이트웨이·NAC·패치 인프라를 끼우는 것**이 1차 대비 한 단계 성숙해지는 지점.

## 3. 솔루션 어디에 둘 것인가 — 강사님 솔루션 브레인덤프 배치

강사님이 칠판에 배운 솔루션을 쭉 늘어놓고 *"어디에 어떤 목적으로 배치할래?"*라고 던졌다(`Pasted image 20260629132950`). 그 리스트를 **사이트별·목적별로 정렬**하면:

```
[서비스 IDC — 외부에 내주는 쪽]
 채널(프론트)  : DMZ · WAF · 외부DNS · 망연계 · 웹/WAS
 탐지/가시성   : TAP(미러 분기) · IDS · 지능형(APT 샌드박스) · ADS(이상행위)
 데이터(백)    : 라이브 DB · 파일서버(분산·암호화 저장) · NAS/스토리지 · AV(악성코드 검사)
 권한/유출     : DRM(문서암호화) · 취약점진단

[R&D / Dev — 내부에서 굴리는 쪽]
 단말통제      : NAC(PacketFence) · PMS · SSL 복호화 에이전트
 데이터취급    : DB Safer/ProxySQL · 레플리카 DB · DLP
 관제          : NMS(Zabbix) · ESM · EMS · ELK
 연결          : VPN(site-to-site + remote access)
```

⚠️ 이건 **확정 배치가 아니라 출발점**이다. 강사님 말대로 *"맞는 배치는 없다 — 요건이 배치를 결정한다"*(`Architecture.md` "틀린 아키텍처는 없다"). DRM/DLP가 어느 쪽이냐도 요건 나름. **중요한 건 위치가 아니라 "왜 거기 뒀는지 설명할 수 있느냐."**

> 🔗 배치할 때 거꾸로 쓰는 법: `Solution-Map.md`의 **탐지공백 맵**("이건 X가 못 잡으니 Y")을 뒤집으면 그대로 **장비 도입 정당화 논리**가 된다. 예) "방화벽은 정상 포트로 들어오는 악성 페이로드를 못 본다 → 그래서 WAF/IPS" → 구성도 옆 주석으로 박으면 그게 곧 정책 고도화의 흔적.

## 4. 랩 구성 — 집에서 멀티사이트를 어떻게 만들까 (현실 제약)

내일부터의 실제 구축 관점. 노트 + `Pasted image 20260629145324 / 151010` 기준.

```
집 통신사 모뎀 ── NIC 직통 연결 ── 내 PC가 공인 IP를 그대로 물어옴
        │
        └── pfSense를 Site-to-Site로 팀원의 pfSense와 묶음
            → 서로 통신하며 역할 분담(한쪽 IDC·한쪽 R&D 식)
```

⚠️ **함정 — ESXi 관리 페이지를 공인 IP로 노출하면 끝장.** 위 그림대로 하면 ESXi 관리자 페이지 주소가 공인 IP를 그대로 쓰게 되어 보안에 취약. → **VMware 호스트 전용(Host-Only) 네트워크로 관리 페이지에 들어가는 그림**으로 가야 한다.
- 제공되는 **호스트 전용선(VMnet1: 192.168.230.x)**으로만 ESXi에 접속(`...151010`의 ESXi 방화벽 규칙 화면 — vSphere Web Client 443/902 등).
- 그쪽으로만 접속 가능하게 하고 **ESXi 방화벽 규칙으로 제어**.
- 🔗 이게 곧 **나눔 B/관리망(#8)**의 실현 — 관리 인터페이스는 절대 서비스망/공인망과 같은 줄에 두지 않는다. FIMS에서 관리망(192.168.56.x)을 호스트 전용으로 뺀 것과 동일한 발상.

> ⚠️ **"OS는 무조건 섞어라"** (강사님 마지막 한마디). 전부 같은 OS·같은 버전이면 취약점 하나에 전부 쓸려나간다(monoculture). Windows/Linux/Unix 섞어서 단일 취약점의 폭발 반경을 줄여라. 🔗 `IT-Infrastructure.md` OS 항목과 연결.

## 5. 이미 정리해 둔 "프로젝트 유념사항" 총집합 (흩어진 것 모음)

그동안 날짜별로 흘려 적어둔 프로젝트 관련 경고들. **제출 전 체크리스트로 쓸 것.**

### 도식화 실수 (Architecture.md "과정2 프로젝트 주의사항")
- ⚠️ **클라우드 서버를 내부망 안쪽에 그리지 마라.** 클라우드는 명백히 외부(인터넷 너머).
- ⚠️ **내부→내부도 방화벽 타면 트래픽이다** → "내부 스루풋"으로 장비 캐파 산정.
- ⚠️ **S3 업로드 회선은 Stateful** → 나가는 트래픽만이 아니라 **돌아오는 응답까지** 캐파에 포함.

### 설계 철학 (Architecture.md "고민의 흔적" / 메일서버 구성도)
- ⚠️ **장비를 무자비하게 인라인으로 욱여넣지 마라.** 장비마다 *"이게 죽으면 무슨 문제가 오나"*를 시뮬레이션하고, 그 고민의 흔적(이중화·바이패스·미러링 선택 근거)이 구성도에 드러나야 함. **배치는 정답이 아니라 장애 시나리오를 돌려본 의사결정.**
- ⚠️ **모든 곳에 L7을 다 깔면 오히려 이상하게 본다.** 역할에 맞는 레이어 배치(L4는 LB 순수기능에 집중). 트래픽량 안 맞는 장비 끼우면 동맥경화(병목).
- **메일서버 TAP 복제라인·DLP 흐름**(6-9일)을 도식에 반영 — 미러는 별도 라인으로 뺀다.
- **방화벽을 1순위로 이중화**(가장 많은 일 = 가장 잘 죽음). 단, 디도스로 스루풋 초과 시엔 failover 금지(넘겨도 스탠바이도 뻗음).

### 망분리·VPN (Lab 6-16·6-17 / Security-Solutions)
- ⚠️ **닌자게이트** — 재택 단말이 개발망+인터넷망을 동시에 물면, RAT 감염 시 그 노트북이 다리가 되어 망분리가 한 대로 무력화. **동시연결 금지**(끊고→인터넷→다시 연결).
- **Site-to-Site = IPsec(망↔망)**, 양끝 장비 짝 맞추는 게 편함. **Remote Access = SSL-VPN + MFA(OTP)**.
- **운영/개발 VLAN 분리** = 장애·침해 전파 차단. 운영망=엄격(화이트리스트), 개발망=생산성 고려(블랙리스트).
- VPN 자체가 공격 표면(CVE 익스플로잇 → 터널 타고 침투) → **펌웨어 패치(PMS)+MFA 필수.**

### 데이터 보호 (오늘 오전 — 금융권 파일 저장)
- 업무 시스템은 **DB쪽 / 파일쪽 양분**. 미들웨어 인입 시 텍스트·정보값은 DB로, 첨부파일은 **실행 안 되는 형태로 암호화해 파일서버로**.
- 금융권은 **파일 위치가 식별되면 안 됨** → 수많은 랜덤 폴더에 분산. 100MB면 25MB씩 쪼개 `a·b·c·d` 공간에 분할·암호화 저장 → 침입자가 들고 가도 **조각 하나만** 가져간 꼴. (클라우드가 쓰는 방식)
