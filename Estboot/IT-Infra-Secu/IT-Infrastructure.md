## IT 인프라 유형

### 컨버지드 (On-Premises)
- 전통적인 인프라. 필요 요소를 전부 구비하고 있는 형태
- 비용, 전력, 물리적 공간 필요

### 클라우드 (Cloud)
- 공유 개념. 해당 클라우드에 접속해 서비스를 빌림

### 하이퍼 컨버지드 (HCI)
- 단일 인터페이스에서 컴퓨터, 네트워크, 데이터 리소스를 관리
- 4대의 컴퓨터 자원을 1개처럼 통합해 사용하는 기술 = 슈퍼컴퓨터 개념

## 데이터 센터 (IDC)

인터넷과 연결된 데이터를 모아두는 시설.

```
Public       → 누구에게나 열림 (Hosting)
Public+Owned → 고객 및 내부 사용 (Hybrid)
Owned        → 내부 사용 Only
```

### 전형적인 컨버지드 삼각 구조

```
인터넷 회선 → 보안 시스템(방화벽) → 네트워크(백본) → 스위치 → 사용자
```

### 데이터 센터 보안 통제

- 관리자 출입 통제, 방문자 출입 통제
- 서버 반입/반출 시 무결성 확보 (새 장비인지, 패치가 이전에 이루어지진 않았는지)
- Disk 폐기 시 Degaussing (반자기화) → 자성을 잃게 해 데이터 영구 폐기
- USB 등 반출입 통제 (CD 사용 = 데이터 입력 방지)

## 인터넷 회선

### ISP (Internet Service Provider)
- 인터넷을 개인이나 기업에게 제공하는 회사
- IT 업계에선 "회선"으로 읽힘
- 보안 측면에서 회선을 두 개 써 가용성 확보 가능 (데이터센터 운영)

### HCX (Hyper Connect Experience)
- 클라우드(CSP) ↔ 데이터센터(IDC) ↔ 내 회사 데이터센터를 물리적 케이블로 직접 연결
- 기밀성, 비용, 속도 측면 이점
- 스니핑 불가

### 국제망 (Submarine Cable)
- 실제 해저 케이블로 나라끼리 연결된 인터넷 인프라

### DWDM
- 광케이블 하나로 여러 파장을 동시 전송 → 속도 매우 유리

## OS 종류와 특성

### Windows
- 사용자 환경 중심 설계
- Close Source (유료), 라이센스 유통기한 있음 (EOL)
- 개발 환경에서 비공개 영역이 많아 불리

### Linux
- GNU 철학. 소스 코드 공개된 Open Source
- 대부분의 개발/서버 환경은 Linux
- 개발 및 관리를 사용자가 직접 해야 함

### Unix
- 안정성, 신뢰성이 높음. 주로 금융/은행에서 사용
- 특정 CPU(IA-64) 필요
- 최근 U to L 추세 (Linux로 전환)

## 웹 서버 구조 (3-Tier)

```
Web Server (정적 영역)  → HTML, CSS, JS 저장 및 제공 (Apache, Nginx)
WAS Server (동적 영역)  → 비즈니스 로직 처리, DB 연동 (FastAPI, Tomcat)
DB Server              → 사용자 데이터 저장. 외부 직접 접근 불가
```

```
사용자 → 프로그램 입력 ↔ WS(정적) ↔ WAS(동적 처리) ↔ DB(필요시 저장)
```

## 가상화 (VM)

### Hypervisor
- 한 개의 머신을 여러 운영체제가 실행할 수 있도록 나누는 기술

## 망분리

내부 정보 유출을 차단하기 위한 내부/외부망 분리.
법적 의무 (전자금융감독 규정, 정보통신망법 등)로 일부 강제 시행.

### 업무망 / 인터넷망 / 폐쇄망 3개로 구성

### 물리적 망분리
- 업무망과 인터넷망을 물리 하드웨어로 분리
- 망 연계를 통해 데이터 송수신
- KVM (물리 스위치): 모니터·키보드를 두 PC가 공유 (데이터는 안 넘어감)

### 논리적 망분리 (서버 기반, SBC)
- 가상화 기술로 서버를 분리 (VLAN)
- HCI로 설계된 서버에 가상 데스크탑 화면을 송출

### 논리적 망분리 (클라이언트 기반, CBC)
- 사용자 PC에 가상화 프로그램을 써서 구분
- 이식성 좋음, 보안은 서버 기반보다 약함

### ⚠️ VLAN ≠ 세그먼트 분리 ≠ 망분리 (6-18 정정)

우리 랩 도식을 "망분리 했다"고 말하면 틀린다. 강사님이 못 박은 구분 — **VLAN은 망분리가 아니다.**

```
VLAN           : 하나의 물리 LAN 안을 "논리적으로" 쪼갠 것
                 → 우리 랩(VLAN 10~40)이 정확히 이것. 물리적으론 한 LAN, 가상으로만 분리
세그먼트(여러 인터페이스/LAN) : IPS·방화벽에 인터페이스를 여러 개 꽂아
                 "물리적으로 별개인 LAN을 여러 개" 구성 → 망 자체가 나뉨
망분리          : 업무망/인터넷망을 물리적으로(또는 강한 논리적 격리로) 끊어
                 데이터가 함부로 못 넘게 한 것 (위 물리적/논리적 망분리)
```

- **우리 랩 구조는 VLAN(논리 분리)일 뿐 망분리가 아니다.** 한 LAN 위에서 가상으로 칸막이만 친 것 — 실제 격리(LAN 세그먼트 분리)와는 다르다.
- **"세그먼트를 나눈다"**(IPS가 여러 인터페이스를 받는 것 = 여러 LAN)와 **"VLAN으로 나눈다"**(한 LAN 아래 논리 분할)는 층위가 다르다. VLAN은 LAN **아래** 네트워크를 논리적으로 나누는 개념이지, 망을 물리적으로 쪼개는 게 아니다.
- 🔗 Security-Solutions "IPS 멀티 세그먼트" — IPS 한 대가 **여러 인터페이스(세그먼트=여러 LAN)**를 받아 망별로 정책을 거는 것. 데이터 성격에 맞게 망을 나눠 거기에 솔루션을 배치하는 게 제대로 된 분리 설계. 우리 랩은 아직 그 전 단계(VLAN 논리분리)다.

> 비유: VLAN = 사무실 하나를 파티션으로 칸 나눈 것(벽은 한 건물, 칸막이만 침). 세그먼트/망분리 = 아예 다른 층·다른 건물로 부서를 떼어놓는 것. 파티션 친 걸 "건물 분리했다"고 하면 안 된다.

## 망연계

보안 구역과 비보안 구역 간의 데이터 이동 (연계 구간).

```
자료 전송 로그인 → 하드웨어 정보 일치 확인 → 데이터 파일 전송 시도
→ 내부 중계 서버 업로드 → 승인 요청 → 검토 및 승인 → 전송
```

### 방식

- **데이터 다이오드**: 반대 통로가 물리적으로 없는 단방향 (군·원자력 초고보안)
- **CDG 연계서버**: 백신+콘텐츠검사+승인+로그. 방향별 정책 (가장 흔함)
  - inbound(받기) = 악성코드 검사
  - outbound(보내기) = 개인정보 유출 검사 (DLP 역할)
- **스토리지 중계**: 중간 창고 경유

> 한국 벤더: 한싹, 휴네시온

## 백업 및 복구

### 재해 등급 판정

- **RTO (Recovery Time Objective)**: 얼마나 빨리 복구 완료해야 하나 → 복구 시간 목표
- **RPO (Recovery Point Objective)**: 얼마나 과거 시점으로 복구할 수 있나 → 데이터 손실 허용 범위

> 복구 → 문제 발생 → 정책 수립 ❌
> **정책 수립 → 문제 발생 → 빠른 대처** ✅

### 효율적인 백업 방법

- 소스와 타겟 중복 제거
- Tape-Out 기능으로 2차 백업 소산 (AIR Gap: 물리적으로 테이프 분리 보관 → 네트워크 원천 차단)
- SAN 방식으로 백업 서버와 데이터 송수신

### LTO 테이프 + PTL (Physical Tape Library)

```
LTO (Linear Tape-Open) → 장기 보관용 테이프 매체 (대용량·저렴·오프라인)
PTL (Physical Tape Library) → 그 LTO 테이프를 로봇 암으로 자동 적재·교체·관리하는 물리 테이프 라이브러리
```

- 로그·아카이브처럼 **양이 계속 쌓이는데 서버 내장(인터널 아카이브) 용량은 한정**될 때, 테이프로 빼서 소산 보관.
- 위 Tape-Out / AIR Gap이 결국 이 LTO·PTL로 구현된다. (DLP 로그서버 → PTL→LTO 소산 사례: Architecture.md "메일 서버 구성도")

## CDN (Contents Delivery Network)

정적인 콘텐츠를 가지고 있는 네트워크.

```
사용자 ↔ 서버(Edge Pop) ← CDN 연결 → Origin 서버
```

- 자신에게 가장 가까운 서버에 접근해 콘텐츠 전달받음 → 낮은 레이턴시
- DDos 완화 역할 (매우 큰 용량의 트래픽을 견딜 수 있는 서버로 구축)
- AWS CloudFront = CDN 서비스
