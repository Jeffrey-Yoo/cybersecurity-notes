# 🏛️ 아키텍처 (Architecture)

---

## 아키텍처란

정보 시스템 구축 종합 서비스.
전산자원을 통합해 최적 해결점 제시.
목적에 맞는 구조를 치밀하게 설계하는 개념.

- **아키텍처**: 시스템 전체의 설계 방식
- **아키텍트**: 다양한 IT 솔루션을 기획·설계하는 사람

---

## 아키텍처 목적

- **신뢰성**: 안전한 이미지 구축 및 침해 공격에 강한 설계
- **신속성**: 개발/테스트/배포 업데이트에 대한 신속한 설계
- **공용성**: 전국민 PC, 모바일을 대상으로 대용량 Transaction 처리

---

## 아키텍처 종류

### EA (Enterprise Architecture)
- 비즈니스 이해를 바탕으로 한 장기적 IT 전략
- 기업 IT의 구조 전체를 담당
- ISP (Information Strategy Planning) 정보전략계획 수립 = 설계도
- AS-IS 분석 → TO-BE 기획 → 실행 담당

### AA (Application Architecture)
- ERP 등 솔루션을 설계
- 실제 앱들을 어떻게 구성하고 배치할 것인가
- 예) 웹 서버는 어디, 내부망 안 데이터 관리는 어디

### SA (Solution Architecture)
- 개발 환경과 같은 솔루션 설계
- AA가 방화벽을 배치했다면 어떤 방화벽 솔루션을 도입할 것인가

### DA (Data Architecture)
- 데이터 흐름 제어, 최적의 경로 탐지
- 데이터가 어디에 저장되고 어떤 권한을 가지며 어디에만 존재할 것인가

### TA (Technical Architecture)
- 전반적인 기술적 문제 해결
- 위 모든 설계들을 어떻게 구현할 것인가
- 회선, 이중화, 메일, 웹서버 구동 방법 등

---

## 아키텍처 설계 6단계 접근법

```
1단계. 서비스 정의     → 어떤 서비스인가, 누가 쓰는가
2단계. 요건 정의       → 외부 요건 + 내부 요건
3단계. 구성 요소 도출  → 필요한 서버 / 솔루션 도출
4단계. 데이터 흐름     → 요청이 어디서 시작해서 어디서 끝나는가
5단계. 보안 레이어     → 방화벽 / DMZ / 망분리 / 솔루션 배치
6단계. 고가용성        → 이중화 / 로드밸런서 / 백업
```

### 요건 정의 구조

| 구분 | 질문 |
|---|---|
| 외부 요건 | 동시 접속자 수는? 외부 연동이 필요한가? 응답 속도는? |
| 내부 요건 | 데이터 보관 기간은? 장애 복구 시간은? 자동화가 필요한가? |

> 서비스의 특성이 인프라를 결정한다.
> 게임 → 실시간 처리 → CacheDB 필수
> 금융 → 외부 연동 많음 → EAI 필수
> 검색 → 읽기 요청 압도적 → 캐시 + 검색엔진 필수

## 실제 3-Tier 아키텍처 구성

```
인터넷
    ↓
보안 장비 (IPS / WAF / DDoS / WIPS)
    ↓
DMZ 구간
Web Server — 외부와의 접점. 내부망과 분리
    ↓
내부망
WAS — 비즈니스 로직 처리
    ↓
DB — 사용자 데이터 저장. 직접 외부 접근 불가
```

- Web은 DMZ 구간으로 내부와 인터넷의 중간 지점
- WAS, DB는 내부망에 위치
- SSO, DLP, NDLP 등 비즈니스 목적에 맞는 솔루션 선택

## 네트워크 흐름에서 장비 순서

### 기본 원칙

```
외부 트래픽
    ↓
F/W (1차, L4) — 누가 어느 포트로 들어오냐 검사
    ↓
L3 라우터    — 어느 방향으로 가냐 길 결정
    ↓
[TP F/W]     — IP 없이 투명하게 끼어서 내부 트래픽 검사 (선택)
    ↓
LB           — 서버 분배
    ↓
L2 스위치    — MAC으로 최종 전달
    ↓
서버
```

### L3 → LB 순서인 이유

```
L3 = 어느 네트워크로 갈지 결정 (서울시 어느 구)
LB = 그 네트워크 안에서 어느 서버로 갈지 결정 (그 구 안에서 어느 집)
```

LB가 먼저 오면 경로도 모르는 상태에서 분배하는 구조가 됨.
클라우드(AWS ALB)에서 L3가 안 보이는 건 VPC 라우팅이 추상화된 것이지 없는 게 아님.

## 보안 장비 배치 전체 흐름

보안 장비는 **큰 것부터 거르고 → 점점 정밀하게** 검사하는 순서로 배치한다.

```
인터넷
    ↓
Anti-DDoS    → 플러딩 / 대량 트래픽 차단 (맨 앞, 고성능)
    ↓
방화벽 (L4)  → IP + 포트 기준 차단
    ↓
IPS          → 악성 패킷, 시그니처 기반 차단 (인라인)
    ↓
Anti-APT     → 샌드박스로 신규 악성코드 탐지
    ↓
WAF          → HTTP 웹 공격 차단
    ↓
L3 라우터    → 경로 결정
    ↓
[IDS]        → 미러링으로 탐지 + 로그 (라인 밖 별도 배치)
    ↓
서버들
```

> 체로 거르는 순서 비유:
> ```
> Anti-DDoS = 굵은 체 (큰 것 먼저 거름)
> 방화벽    = 중간 체
> IPS       = 미세한 체
> WAF       = 거름종이 (HTTP 전용)
> IDS       = 옆에서 복제해서 관찰 (라인에 안 끼움)
> ```

- 왜 DDoS가 맨 앞 → 대량 트래픽을 먼저 걸러야 뒤 장비들이 안 죽음
- 왜 IDS만 옆으로 빠짐 → 차단 안 하는 로깅 장비라 인라인에 둘 이유가 없고, 장애 시 라인 절체 위험 (Security-Solutions.md "인라인 vs 미러링" 참고)

## 요건 기반 보안 장비 배치 — "위협을 먼저 정의하라"

위의 "큰 것부터 거르는" 표준 순서는 출발점일 뿐이다. 6-8일 핵심은 **"정답 배치는 없다. 서비스 요건이 솔루션과 배치를 결정한다"**였다. 슬라이드의 세 질문이 시작점이다.

```
① 회사에서 가장 필요한 보안 시스템은?
② 가장 위협이 되는 대상은?
③ 보호해야 할 자산은?
```

이 세 개를 정의해야 "어디에 무슨 장비를 둘지"가 나온다. 무작정 다 깔면 비효율(동맥경화·비용)만 생긴다.

### 서비스마다 받는 공격이 다르다 → 장비도 달라진다

같은 망에 웹서버·개발팀·그룹웨어·게임서버가 매달려 있어도, 각자 받는 위협이 다르니 거기 맞는 장비를 골라야 한다.

```
웹서버      → 주 공격은 웹 공격 → WAF
  ⚠️ 단, 웹서버 있다고 WAF 무조건? 아니다.
     - 게시판/업로드 같은 "입력받는 서비스"가 없는 정적 페이지면? 웹 공격 표면이 작음
     - CloudFlare(CF) 같은 CDN에 올렸다면? 오리진을 직접 때리기 어려움
     → "WAF가 필요한 요건이 생겼는가"로 접근해야지, 장비부터 깔면 안 됨

게임서버    → 게임 서비스 대상 공격 → 그에 맞는 탐지

개발팀      → 해커의 진짜 주 표적일 수 있음 (소스코드·내부 접근 권한)
            → "해커는 웹서버만 노린다"는 선입견을 버려야
```

> 핵심 사고법: **"장비가 있으니 둔다"가 아니라 "이런 요건(공격 표면)이 생겼으니 이 장비가 필요하다"** — 요건에서 장비로 가는 방향. 정적 페이지 웹서버에 WAF를 욱여넣는 건 과설계.

### 교집합으로 묶는다 — 같은 성격은 한 장비로

개발팀과 그룹웨어는 **둘 다 인트라넷(내부망)**이다. 성격이 같으니 IPS 한 대로 두 구간을 함께 커버하는 식으로 **교집합을 만들어** 묶는다. (Security-Solutions.md "IPS 멀티 세그먼트" — 1대로 여러 구간 감시)

### 같이 둘까 분리할까 — 스루풋이 결정한다

웹서버와 게임서버를 한 장비/구간에 같이 둘지 분리할지는 **처리량(스루풋)**으로 판단한다.

```
게임 2G + 웹 100G 가 한 구간에?
→ 처리량 격차가 크고 합이 인터페이스 용량을 넘김 → 분리해야 함 (안 그러면 병목)

근데 인터페이스가 1G이고, 둘 다 1G 안에서 처리 가능하다면?
→ 그 정도는 같이 둬도 괜찮다
```

- 처리량이 안 맞거나 인터페이스 한계를 넘기면 분리(비효율 회피).
- 같은 라인에 굳이 둬야 하는데 요건이 다르면 → **바이패스를 두고 필터링으로 IPS·WAF 기능을 요건/특성에 맞게 갈라 적용**할 수도 있다.

> 위 "트래픽량도 고려 — 동맥경화"와 같은 원리. 6-8일엔 게임/웹 구체 수치로 한 번 더 짚었다.

### 직원에게서 들어오는 위협 → 네트워크냐 엔드포인트냐

위협의 상당수는 외부 직접 공격이 아니라 **직원의 단말을 거쳐** 들어온다. 정리해보면 "이 위협은 네트워크 보안에서 처리할 건가, 엔드포인트에서 처리할 건가"가 갈린다.

| 유입 경로(직원발) | 처리 레이어 | 솔루션 |
|---|---|---|
| 피싱(악성 URL) | 엔드포인트 | EDR (+ 메일/URL 필터) |
| 메일 첨부파일 | 네트워크 | 안티스팸 / 메일 보안 |
| 악성코드(키젠·영화파일·MS오피스 매크로 사칭) | 엔드포인트 | EDR / 백신 (행위·시그니처) |
| 주식 MTS 등 외부 프로그램 | 엔드포인트 | 단말 통제 |
| RAT(원격 쉘) 공격 | 네트워크 | IPS / Darktrace(이상 탐지) |

```
네트워크 보안에서 처리할 요건  ↔  엔드포인트에서 처리할 요건
→ RAT으로 원격 쉘? → 네트워크단(IPS·Darktrace)이 잡을 수 있음
→ 단말에 악성코드가 떨어짐? → 엔드포인트단(EDR·백신)이 행위/시그니처로 잡음
→ 같은 "악성코드"라도 어느 단계에서 막을지를 요건으로 계속 따져야 함
```

> 위협을 나열만 하면 의미 없다. **각 위협을 "어느 보안 레이어가 책임지나"로 매핑**하는 게 요건 정의의 실체. (피싱→EDR은 Security-Solutions "EDR", RAT→Darktrace는 "Darktrace", 메일→안티스팸 흐름 참고)

### 장비 스루풋은 데이터시트로 계산한다

배치를 정했으면 "이 구간 트래픽을 이 장비가 감당하나"를 **데이터시트의 스루풋 수치로 계산**한다. 예) SECUI MFI 라인업:

| 모델 | MFI 1500 | MFI 2200 | MFI 4200 | MFI 21000 |
|---|---|---|---|---|
| 64Byte 성능 | 1G | 2G | 40G | 20G |
| 최대 성능 | 4G | 10G | 40G | 100G |
| Memory | 16GB | 16GB | 32GB | 64GB |
| Power | Dual | Dual | Dual | Dual |

- **64Byte 성능 vs 최대 성능을 같이 본다** — 작은 패킷(64Byte)이 쏟아질 때의 성능이 진짜 부하 상황(플러딩 등)에 더 가깝다. "최대 성능" 수치만 믿으면 안 됨.
- 구간 트래픽 합 < 장비 스루풋이어야 병목이 안 생긴다. (Security-Solutions.md "방화벽 실물 구조 — 스루풋" 참고)

> ⚠️ 카탈로그 "최대 성능"은 대개 큰 패킷 기준 이상치다. 실제로는 작은 패킷 처리 성능(64Byte)과 시그니처를 몇 줄 태우느냐(Security-Solutions "시그니처 성능 차이")에 따라 실효 스루풋이 확 떨어진다.

### 아키텍처는 요건 따라 계속 움직인다

```
맞는 답은 없다. 정확한 요건에 의해 솔루션과 배치가 바뀌어야 한다.
→ 프로젝트 때 가장 고민 많이 해야 하는 영역
→ (아래 "아키텍처 최적화 원칙 — 틀린 아키텍처는 없다"와 직결)
```

## 방화벽 디자인 (구간별 정책 설계)

방화벽 한 대가 인터넷 / DMZ(서비스 대역) / 직원 내부망을 인터페이스로 가르고 있을 때, **각 구간이 어느 방향으로 통신하느냐(코스)에 따라 deny·allow 정책을 따로 설계**한다. "일단 다 열고 막힌 것만 푼다"가 아니라, 구간별로 들어오고 나가는 길을 하나씩 정의하는 작업.

```
                인터넷 / 클라우드
                  ▲ D     ▲ C
                  │
                Router
                  │
    ┌──────── Firewall ────────┐
    │ ▷ A (서비스 인바운드)      │── Switch ── 서비스 대역(DMZ): Web 서버들
    │ ▷ B                        │
  Switch
    │
  직원 내부망(Intranet)
```

| 코스 | 방향 | 한 줄 설명 |
|---|---|---|
| A | 인터넷 → DMZ | 외부 사용자가 서비스(웹)로 들어오는 길 |
| C | DMZ 서버 → 인터넷 | 서버가 밖으로 나가는 길 (업데이트 등) |
| D | 내부망 → 인터넷 | 직원이 밖으로 나가는 길 |

### DMZ(서비스 대역)는 공인 IP와 1:1 매칭 → 인바운드를 막아야 함

```
DMZ = 서비스 대역 = SNAT/DNAT으로 공인 IP와 1:1 매칭돼 있는 상태
→ 외부에서 공인 IP로 직접 때릴 수 있음
→ 그래서 22(SSH)처럼 "들어올 수 있는 경로"를 deny하는 정책이 반드시 필요

직원 내부망 = DNAT 1:1 매칭이 안 돼 있음 (밖에서 직접 못 들어옴)
→ 인바운드 차단 정책을 따로 안 박아도 됨
```

> DMZ는 대문에 주소(공인 IP)가 붙어있는 집 → 아무나 문 두드릴 수 있으니 SSH 같은 문은 잠가야 함.
> 내부망은 주소가 안 붙은 집(DNAT 없음) → 애초에 밖에서 못 찾아오니 덜 신경 써도 됨.

### C코스 — DMZ 서버가 인터넷으로 나가야 할 때

원칙은 "서비스 대역에서 인터넷으로 못 나가게" 막는 것. 그런데 목적에 따라 나가야 할 때가 있다 — `apt update`, 리눅스 레포지토리/패치 서버 업데이트 등.

```
all deny (기본 전부 차단)
→ 업데이트에 필요한 목적지만 L7 레이어로 콕 집어 허용
→ Squid(포워드 프록시)로 "리눅스 repo·패치 서버는 permit, google·naver는 deny"
```

> ⚠️ 음차 교정: 수업 중 "스키드/웹스키드 프록시" = **Squid 프록시**(대표적인 포워드·캐싱 프록시). 아웃바운드 화이트리스트 제어에 쓴다.

💡 **"퍼지(purge)를 날린다" 보충** — 노트에 애매하게 적힌 부분.
CloudFlare(CF) 같은 CDN을 예로, DMZ가 **오리진(origin, 원본 서버)**이 되고 CF **엣지(edge) 서버**가 그 콘텐츠를 복제(캐싱)해 전 세계에 뿌린다. 오리진 내용이 바뀌면 엣지에 남은 옛 캐시를 비워야 하는데, 이때 "캐시 **퍼지(purge)**를 날린다"고 한다 = 엣지 캐시 무효화 → 다음 요청 때 오리진에서 최신 내용을 다시 받아오게 함. "서비스 대역(오리진)에서 CDN 엣지로 동기화한다"는 게 이 얘기.

### D코스 — 직원 내부망이 인터넷으로 나갈 때

```
여기도 Squid 프록시로 제어 가능하지만,
→ 가장 좋은 건 화이트리스트 방식 (허용할 목적지만 열기)
→ 경우에 따라 아웃바운드에도 IPS를 같이 태우거나 SSL 복호화(SSL Inspection)를 걸 수도 있음
```

악성 사이트 접속(access to malicious site)을 막는 게 핵심 목적.

> 인바운드만 막으면 끝이 아니다. **나가는 길(아웃바운드)도 설계 대상**이다.
> 감염된 내부 PC가 C2 서버로 나가는 것도 여기서 막는다 (RAT/악성코드 → Attack-Defense.md 연결).

## NAT 설계 — DNAT / SNAT

NAT는 사설↔공인 IP를 바꿔주는 것. 방향에 따라 둘로 갈린다.

```
DNAT (Destination NAT) = 들어오는 방향. 공인 IP → 내부 사설 서버로 꽂아줌 (외부 진입용)
SNAT (Source NAT)      = 나가는 방향. 내부 사설 IP → 공인 IP로 바꿔 내보냄 (외부 egress용)
```

### DNAT — 공인 IP는 사설 서버 수만큼 필요 → 그래서 VIP + LB

```
DNAT으로 외부에 노출할 서버가 N대면, 매칭되는 공인 IP도 그만큼 필요
→ 사용자가 공인 IP로 들어오면 일단 VIP(가상 IP)로 떨궈주고
→ VIP 밑에 묶인 실제 서버들로 로드밸런싱해서 내려보냄
```

> VIP·DNAT 기본 개념은 OSI-7Layer.md "VIP" 참고. 여기서 포인트는 **그 LB가 물리적으로 어디 붙어있냐**.

**물리 L4 스위치 LB vs LVS / AWS ALB (논리 LB)**

```
물리 L4 스위치 LB → 방화벽 "밑에 물린" 라인들끼리 분배 (장비가 경로 위에 실제로 있음)

LVS / AWS ALB    → 방화벽 "옆 선상"에 논리적으로 존재
                  ① 사용자가 방화벽을 타고 들어옴
                  ② LVS/ALB에서 VIP를 받고 패킷을 던짐
                  ③ ALB가 등록(바인딩)된 서버 목록을 보고 해당 서버로 전달
```

> 물리 LB는 길목에 실제로 서 있는 분배기, ALB/LVS는 옆에서 "명단(바인딩 목록)" 보고 던져주는 분배기.
> 경로 자체에 끼어있는 게 아니라 논리적으로 옆에 있다는 게 핵심 차이.

### SNAT — 하나로 묶으면 편하지만 추적이 안 된다

```
모든 서버·망이 SNAT 하나의 공인 IP로 나가면 → 관리는 편함
하지만 사고가 터지면 "누가 나간 거냐" 추적이 불가능/어려움
   (전부 같은 공인 IP로 찍히니까)

→ 사고 위험이 높은 구간은 SNAT를 쪼개거나, 분류별로 정책에 맞게 분리
```

> 단체버스 한 대로 다 같이 나가면 명단 관리는 편한데, 사고 나면 누가 뭘 했는지 모름.
> 위험한 그룹은 따로 차를 태워서 꼬리표(공인 IP)를 구분해 둔다.

## 과정2 프로젝트 주의사항 — 도식화 & 회선 산정

> 번외로 강사님이 짚어준 LB·클라우드 쓸 때 실수 포인트. (1차 FIMS 프로젝트 땐 잘했던 부분)

### 클라우드 서버를 "내부"에 그리지 마라

```
온프레미스 vs 클라우드를 도식화할 때
→ 클라우드 안에 있는 서버를 우리 내부망 안쪽에 그려넣는 표현은 없어야 함
→ 클라우드는 명백히 외부(인터넷 너머)에 있는 자원이다
```

### 내부 → 내부도 방화벽을 타면 트래픽이다 (캐파 산정)

```
인터페이스 3 → 4번으로 가는 통신도 방화벽을 거침 = 트래픽 발생
→ 내부에서 내부를 볼 땐 "내부 쓰루풋(throughput)"으로 장비 캐파를 산정

인터페이스 3번이 방화벽에서 ISP를 찍고 클라우드/외부로 나간다면
→ 인터넷 회선이 필요한 구간

3번에서 클라우드 S3에 저장한다면?
→ Stateful이니까 나가는(아웃바운드) 트래픽뿐 아니라
   응답으로 다시 들어오는(인바운드) 트래픽까지 회선 산정에 넣어야 함
```

> Stateful 세션은 나간 만큼 돌아온다(가제트 팔 — Security-Solutions.md "2세대" 참고).
> 그래서 S3 업로드 회선을 잡을 때 올리는 트래픽만 보면 안 되고, 돌아오는 응답 트래픽도 캐파에 포함해야 한다.

## 아키텍처 최적화 원칙

```
틀린 아키텍처는 없다
→ "개선할 게 있나, 최적화할 게 있나"로 봐야 함
→ 요건에 맞게 설계하는 것이 핵심
```

### 서비스 특성에 따라 분리

```
웹 서버만 있으면        → WAF 하나로 충분
소켓 서버(메신저)가 있으면 → IPS 추가
```

### 트래픽량도 고려 (동맥경화 주의)

```
메신저 서비스 100Mbps
웹 서버      1Gbps
→ 100Mbps짜리 장비를 1Gbps 구간에 넣으면 동맥경화 (병목)
→ 분리해서 각자 맞는 처리량의 장비를 배치
```

> 면접 포인트: **모든 곳에 L7을 다 갖다 놓으면 오히려 이상하게 본다.**
> 역할에 맞는 레이어 장비를 배치하는 게 현실 감각. L4는 로드밸런싱 순수 기능에 집중시키는 식.

> FIMS 연결: 과정1 FIMS 아키텍처도 틀린 게 아니다. 요건이 더 정확해지면 개선점이 보일 뿐.
> 웹 서비스만 있었으니 WAF 하나로 충분했을 수 있고, Suricata = IPS(인라인) 역할, Wazuh = 관제 서버 역할이었다.

## 고가용성 (HA, High Availability)

서버나 장비가 장애가 나도 서비스가 끊기지 않도록 이중화하는 개념.

> HA는 L3 구간 이름이 아니라 "고가용성(High Availability)"이라는 설계 개념이다. (면접 단골 함정)

- 회선을 두 개 써서 가용성 확보
- 서버 이중화 (미러링, RAID)
- 로드밸런서로 서버 분산

### 왜 방화벽을 1순위로 이중화하나

실무에서 가장 잘 고장나는 하드웨어가 보통 **방화벽**이다.

```
모든 트래픽을 받고 → 패킷을 제어하고 → 로깅·기록하고 → 다음 장비로 넘겨주는
= 한 장비가 가장 많은 일을 함 → 가장 잘 죽음 → HA 구성 1순위
```

### Active-Standby (액티브-스탠바이)

```
주(Active) 장치  ──세션 동기화──  보조(Standby) 장치
   살아서 일함                     옆에서 대기 (동기화 값만 계속 받음)
```

- 주 장치가 죽으면 → 주 장치가 들고 있던 **VIP를 스탠바이 장치로 넘긴다(failover)**
  - 사용자는 같은 VIP로 계속 접속하니 장비가 바뀐 줄 모름 (무중단)
- 주/보조는 **반드시 동일 성능 장비**로 맞춰야 한다
  - 세션을 동기화(= 클러스터 기능)해서 주가 들고 있던 세션 상태를 그대로 이어받아야 하기 때문
  - 성능이 다르면 넘겨받는 순간 스루풋을 못 버텨 보조도 같이 뻗음

> 부조종사 비유: 기장(Active)이 쓰러지면 부기장(Standby)이 같은 조종간(VIP)을 그대로 잡는다.
> 단, 부기장도 같은 기종을 몰 줄 알아야(동일 성능) 비행이 이어진다.

⚠️ **언제는 안 넘겨야 하나** — 디도스처럼 비정상 트랜잭션으로 장비 스루풋이 100%를 초과한 경우.
이건 장비 고장이 아니라 **공격 상황**이라, 스탠바이로 넘겨봤자 스탠바이도 똑같이 뻗는다 → 넘길 필요 없음(넘기면 안 됨).
그 외 정상적인 장애는 전부 넘기는 게 맞다 (failover 정상 동작).

### Active-Active (액티브-액티브)

```
주 장치 2대가 둘 다 일하면서 세션을 서로 동기화
→ 두 장치 밑으로 똑같은 트래픽 라인이 양쪽에 다 구현돼 있어야 함
→ 상하로 L4까지 다 묶으면 풀 메쉬(Full Mesh) 구조
```

- 장점: 평소에도 두 대가 부하를 나눠 처리 (스탠바이처럼 놀지 않음)
- 단점: 양쪽 라인을 전부 깔아야 해서 **돈이 많이 든다**

### 본딩 / 티밍 (Bonding / Teaming)

```
한 서버의 여러 인터페이스(NIC)를 논리적으로 하나로 묶는 기술
→ 한 회선(포트)이 끊겨도 다른 회선으로 통신 지속 (가용성)
→ 대역폭을 합쳐 처리량을 늘리기도 함 (성능)

리눅스 = 본딩(bonding) / 윈도우·일부 벤더 = 티밍(teaming) — 사실상 같은 개념
```

> Active-Active에서 "두 장치 밑 라인을 양쪽 다 구현"하는 게 결국 이 본딩/티밍으로 인터페이스를 묶는 작업이다.
> 회선 한 가닥이 끊겨도 묶인 나머지로 버티게 하는 것.

## 로드밸런서 (Load Balancer)

트래픽이 몰릴 때 여러 서버로 분산 처리.

| 종류 | 레이어 | 기준 |
|---|---|---|
| L4 LB | L4 | IP + 포트 기준으로 분배 |
| L7 LB | L7 | URL, 쿠키, 헤더 기준으로 분배 |

- 대표 방식: 라운드로빈 (순서대로 분배)
- 관련 기술: VIP (가상 IP), Session Table
- DNS 라운드로빈은 세션 유지 문제 있음 → 로드밸런서 함께 사용 필요

## 파일 공유 서버 아키텍처

```
글(메타데이터) → DB 저장
파일 자체      → 별도 스토리지 (NAS)
```

- NAS 형태: SMB/CIFS (Windows/Linux 혼용), NFS (Linux 전용)
- 소규모: Samba. 보안 취약
- 금융/엔터프라이즈: EMC VNX 같은 전문 장비

### NAS 보안 리스크

```
주요 포트: 139 (NetBIOS), 445 (SMB Direct)
불필요한 포트는 반드시 닫아야 함
SMB는 최신 버전으로 지속 업데이트 필수
```

### 보안 장비 배치 원칙

- 모든 구간에 배치하는 건 비효율적
- **아키텍처 구조 먼저 파악 → 목적과 리스크에 맞는 포인트에만 배치**
- 보안 업무자라면 아키텍처를 보고 리스크를 스스로 유추할 수 있어야 함

## 메일 서버 구성도 — 패킷 흐름 & DLP 복제라인 (6-9일) ⭐ 2차 프로젝트 참고

실제 메일 시스템 구성도를 따라가며 **패킷이 어디서 시작해 어디로 흐르는지**, 그리고 **DLP를 위한 복제라인을 어떻게 따로 빼는지**를 봤다. 강사님이 "2차 프로젝트 때 참고할 중요한 흐름(TAP 장비·DLP 흐름)"이라고 못 박은 부분.

### 좌측 — 메일 본선 (수신·처리·보관)

```
외부 메일
   ↓
스팸 서버 (Anti-Spam) ── 1차로 스팸/악성 거름
   ↓
Mail Proxy (Active-Standby) ── 메일 진입점, 이중화
   ↓
L4 스위치 → 로드밸런싱
   ↓
Mail Main & Anti Spam/Virus (Active-Standby) ── 본처리 + 백신
   ↓
Mail-DB 서버
   ↓
SAN Switch → 스토리지에 메일 보관
```

- 좌측 구간 = 메일이 각 단계를 거쳐 **로드밸런싱되며 처리되고 최종 스토리지에 보관**되는 본선.
- 핵심 장비들은 Active-Standby로 이중화 (Mail Proxy, Mail Main 등). (HA → 위 "고가용성" 참고)

### 우측 — DLP 복제라인 (관찰용, 본선과 분리)

```
코어 상단에 TAP 설치
   ↓ (악성 분석이 되기 전 단계의 트래픽을 복제)
복제 라인 ──→ 메일 모니터링 수집서버 ──→ 메일 모니터링 로그서버 (= DLP)
```

- 우측 = **DLP 역할**을 위해, 코어 상단에 TAP을 달아 **악성분석 전 트래픽을 복제한 별도 라인**을 만든 것.
- 본선(좌측)에 끼어 차단하는 게 아니라 **복제본을 받아 관찰**한다. → 이 메일서버는 애초에 **발신에 담당자 승인이 필요한 흐름**이라 DLP는 차단(인라인)이 아니라 관찰 용도로 충분. (Security-Solutions.md "NDLP는 보통 관찰용", "인라인 vs 미러링"과 한 쌍)

> IDS를 라인 밖으로 빼는 그 논리 그대로다 — **차단이 목적이 아닌 가시성·로깅 장비(여기선 DLP)는 TAP 복제로 옆에서 본다.**

### 로그서버 데이터는 PTL → LTO로 소산

```
DLP(로그서버)에서 발생한 데이터 → PTL 방식으로 LTO(테이프)에 보관
이유: 로그서버의 인터널 아카이브(내장 저장) 용량이 적기 때문
```

- **PTL(Physical Tape Library)** = LTO 테이프를 로봇 암으로 자동 적재·관리하는 물리 테이프 라이브러리. (백업 상세 → IT-Infrastructure.md "백업 및 복구")
- 로그는 양이 계속 쌓이는데 서버 내장 용량은 한정 → 테이프로 빼서 장기 보관.

### 보안 강화 아이디어 — 엣지 메일서버 (이미지·코드만 분석 후 승인)

```
"스팸서버에서 안 걸러진 악성 패킷을 더 막으려면?"
→ 메일서버 앞단에 엣지(edge) 메일서버를 둠
→ 메일의 이미지·코드를 프로그램이 먼저 분석해서 (안전하게 변환·렌더링한 결과만) 담당자에게 보여줌
→ 담당자가 확인 후 승인하는 형태
```

> 사용자가 원본 첨부·코드를 직접 받기 전에, 엣지에서 "안전하게 처리한 버전"만 보여주고 사람이 한 번 더 승인 → 악성 실행 표면을 줄이는 발상. (CDR / 콘텐츠 무해화와 같은 결)

### 바이패스 기반 Inline / Out-of-Band 이중화 구성도

같은 날 본 또 다른 구성도 — **코어라우터·코어스위치 사이에 장비들을 어떻게 배치하느냐**를 인라인이냐 아니냐로 가른 그림.

```
코어라우터 ─┬─ [Inline Tools]  Firewall1·2, IPS, WAF        ← 차단하니 본선(인라인)에
            │     (바이패스로 묶어 장애 시 우회)
코어스위치 ─┴─ [Out of Band Tools]  Network Forensics, APT   ← 분석·탐지라 본선 밖(미러링)
```

- **Inline Tools** = 차단하는 장비(FW/IPS/WAF)는 본선에 인라인 + **바이패스로 묶어 장애 시 자동 우회**. (Security-Solutions.md "하드웨어 바이패스")
- **Out of Band Tools** = Network Forensics·APT처럼 분석/탐지만 하는 장비는 본선 밖에 두고 복제본만 받음. (IDS·DLP를 라인 밖에 빼는 것과 같은 원리)

> 한 그림에 "차단=인라인(+바이패스)", "분석=아웃오브밴드(미러링)"가 같이 그려져 있어, 지금까지 배운 배치 원칙이 한눈에 들어온다.
