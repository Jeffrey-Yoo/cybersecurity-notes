# 🗺️ 보안 솔루션 배치·로그·탐지공백 통합 맵

> 내가 쓰고 있는(또는 쓸) 솔루션·프로그램이 **어디에·어떻게 배치되어야 하고**, 거기서 **어떤 로그가 나오며**, 그 로그를 **어떻게 읽어야 하는지**, 그리고 **이 장비가 못 잡는 건 뭘로 메우는지**를 한 장에 모은 지도.
>
> 다른 파일들이 "개념"이라면 이 파일은 **"내 프로젝트에서 실제로 어떻게 굴러가야 하나"의 조립 설명서**다.
> (개념 깊이는 `Security-Solutions.md`, 배치 철학은 `Architecture.md`, 실제 구축 기록은 `../Lab/`)
>
> ⚠️ **이 문서는 살아있는 문서다.** 새 솔루션을 도입하거나, 배치 위치·로그·탐지공백에 변화가 생기면 여기를 갱신한다. (Lab README의 "현재 환경 상태"와 짝)
>
> 🎯 이 지도를 읽고 나면 이런 생각이 자동으로 나와야 한다:
> *"아, 이 구간에선 이런 로그가 뜨고, 이 로그는 이렇게 읽어야 하고, 이건 IPS가 못 잡으니까 이 솔루션을 붙여야겠다."*

## 🧭 한눈에 보는 배치 (현재 랩 기준)

```
                          [인터넷 / 공유기 192.168.1.1]
                                   │
                  ┌────────────────┴───────────────────┐
                  │  Kali (공격기) 192.168.1.250        │  ← WAN쪽 외부 공격자 포지션 (검증용)
                  └────────────────┬───────────────────┘
                                   │ WAN(em0) 192.168.1.82
   ┌───────────────────────────────────────────────────────────────────┐
   │  pfSense (UTM / 방화벽)  ── 막는 쪽 (인라인) ──                     │
   │   ├ 방화벽 룰 (L3/L4)      : 통과/차단 → Filter/Deny 로그           │
   │   ├ Squid + SquidGuard     : 아웃바운드 URL 통제 → access.log       │
   │   ├ Snort (IPS·인라인)     : 시그니처 차단(drop) → alert/block 로그 │
   │   └ OpenVPN                : 원격접속 터널 → VPN 연결 로그          │
   └──────────┬──────────────────────────────────────┬─────────────────┘
              │ LAN(em1) 트렁크 VLAN 4095              │ SPAN(미러)
              │                                        ▼
   ┌──────────┴──────────┐               ┌──────────────────────────────┐
   │ VLAN10 Web 10.10.10 │               │ Security Onion (NSM 센서)     │
   │ VLAN20 WebDev .20   │               │  ── 보는 쪽 (미러) ──         │
   │ VLAN30 Samba  .30   │               │  Suricata(탐지)+Zeek(메타)    │
   │ VLAN40 DNS    .40   │               │  +Elastic+SOC 콘솔 / full PCAP│
   │ MGT    Zabbix .60   │◄──SNMP/agent──┤  → 사후분석·상관분석·포렌식   │
   └─────────────────────┘  자원·가용성  └──────────────────────────────┘
              ▲
        Zabbix (NMS) ── 모니터링 ── baseline·골든타임 감지
```

> 큰 그림 한 줄: **pfSense = 막는 쪽(인라인) / Security Onion = 보는 쪽(미러) / Zabbix = 상태 보는 쪽(모니터링) / Kali = 때리는 쪽(검증).**
> 막는 쪽은 라인 위에 박혀 죽으면 통신이 끊기고, 보는 쪽은 옆에서 복제만 받아 죽어도 서비스는 멀쩡하다. 이 둘을 한 라인에 몰면 같이 뻗는다 → 역할별로 분리. (`Security-Solutions.md` "Anti-DDoS와 IPS 분리 배치")
>
> 🧭 위 배치도는 **단일 랩 기준**. 과정2 프로젝트의 **멀티사이트(서비스 IDC ↔ R&D/Dev, site-to-site VPN) 배치와 R&D 망 설계**는 `Project2-Direction.md`로 분리. 이 솔루션들을 두 사이트에 "왜 여기 두는지" 정당화하는 게 프로젝트 과제.

## 📊 표 1 — 목적별 분류 (어디에·어떻게·왜)

역할로 4덩어리: **막는 쪽 / 보는 쪽 / 모니터링 / 통제·검증.**

### 🛡️ 막는 쪽 (인라인 — 라인 위에 박혀 직접 차단)

| 솔루션 | 계층 | 우리 랩 배치 | 방식 | 핵심 목적 |
|---|---|---|---|---|
| **pfSense 방화벽 룰** | L3/L4 | WAN·VLAN 게이트웨이 | 인라인 | IP·포트 단위 통과/차단(접근통제), 망분리·라우팅의 주체 |
| **Squid + SquidGuard** | L7(프록시) | pfSense 내장, 아웃바운드 길목 | 인라인(투명/명시) | 나가는 URL 통제(화이트/블랙리스트) — 내부→밖 통제 |
| **Snort (IPS)** | L3~L7 | pfSense 내장, WAN·VLAN10 | 인라인(netmap) | 알려진 공격 시그니처를 **실시간 drop** |
| **OpenVPN** | L3(터널) | pfSense 내장 | 인라인 | 원격접속 구간 암호화 + 인증된 사용자만 진입 |

### 👁️ 보는 쪽 (미러 — 옆에서 복제만 받아 탐지·기록)

| 솔루션 | 계층 | 우리 랩 배치 | 방식 | 핵심 목적 |
|---|---|---|---|---|
| **Security Onion** | L3~L7 | SPAN 미러 NIC(IP 없음) + 관리 NIC | 미러(passive) | 네트워크 가시성·탐지(Suricata)·메타로그(Zeek)·풀패킷 저장 → 사후분석/관제 |
| **Wireshark** | L2~L7 | 분석 PC에서 그때그때 | 수동 캡처 | 패킷 1개 단위 정밀 해부(헤더·플래그·페이로드) — 포렌식/검증용 |

### 📈 모니터링 (상태·자원·가용성을 본다)

| 솔루션 | 계층 | 우리 랩 배치 | 방식 | 핵심 목적 |
|---|---|---|---|---|
| **Zabbix (NMS)** | 관리(SNMP) | MGT 10.10.60.10, 에이전트 10050 | 폴링/에이전트 | CPU·메모리·트래픽 baseline 수집 → 임계치 이탈(골든타임) 감지 |

### 🧾 호스트 내부·계정 (서버 안 / 신원·권한 — 네트워크 장비가 못 보는 영역)

| 솔루션 | 계층 | 우리 랩 배치 | 방식 | 핵심 목적 |
|---|---|---|---|---|
| **Zabbix FIM** | 호스트(파일) | 각 서버 에이전트 | 해시 폴링 | 핵심 파일(authorized_keys·passwd·sshd_config) **변조 감지**(상태) |
| **snoopy** | 호스트(명령어) | 각 서버 `/etc/ld.so.preload` | execve 래핑 | 누가·무슨 **명령어**를 쳤나 감사(내부자·침투 후 행위·sudo 남용) |
| **journald + rsyslog** | 호스트(로그) | 각 서버 | 수집→분류 | 로그 수집(journald 바이너리)·분류저장(rsyslog 텍스트) → **SIEM 전송 입구** |
| **AD (개념/백로그)** | 신원·권한 | (윈도우 DC, 미구축) | 중앙관리 | 계정·인증(Kerberos)·정책(GPO) 중앙통제 — 내부망 한정 |

### 🎯 통제·검증 (관리·공격 시뮬레이션)

| 솔루션 | 계층 | 우리 랩 배치 | 방식 | 핵심 목적 |
|---|---|---|---|---|
| **Kali Linux** | 공격기 | WAN쪽 192.168.1.250 | 능동(필요시) | nmap·Hydra·hping3로 방어 장비가 **실제로 막나/잡나** 검증 |

> ⚠️ 같은 "L7을 본다"라도 목적이 다르다: **Squid는 나가는 걸 통제(막는 쪽), Snort는 들어오는 공격을 차단(막는 쪽), Security Onion은 양쪽을 다 보지만 차단은 안 함(보는 쪽).** "본다 = 막는다"가 아니다.

## 📜 표 2 — 솔루션별 로그 & 분석 포인트

"이 구간에선 이런 로그가 발생하고, 이 로그는 이렇게 읽는다"의 본체.

| 솔루션 | 발생 로그 | 어디서 보나 | 어떻게 읽나 (분석 포인트) |
|---|---|---|---|
| **pfSense 방화벽** | Filter 로그 / **Deny 로그** | pfSense Status→System Logs→Firewall | 차단된 트래픽은 **방화벽에서만** 보인다. Deny 로그 = "권한 없는 접근 시도"의 추적 단서(예: DB 권한 없는 호스트가 붙으려 함). 누가·어디로·왜 막혔나 |
| **Squid/SquidGuard** | access.log (`TCP_DENIED`/`TCP_MISS` 등) | pfSense Squid Proxy 리포트 / `/var/squid/logs/access.log` | "누가 어디로 나갔나/막혔나". 화이트리스트면 허용 외 전부 DENIED, 블랙리스트면 차단 대상만 DENIED. 비정상 도메인(C2 의심) 아웃바운드 포착 |
| **Snort (IPS)** | alert(탐지) / block·snort2c(차단) | pfSense Snort→Alerts / Blocked | **sid별·src IP별**로 본다. ⚠️ 한 번 뜨고 안 뜨면 룰이 죽은 게 아니라 **threshold**다(count/seconds). 알람 폭주 시 Suppress/Pass로 노이즈 정리 후 진짜를 본다 |
| **Security Onion** | Suricata Alerts + **Zeek** conn/dns/http.log + **full PCAP** | SOC 콘솔(Alerts·Hunt·PCAP·Cases) | 정탐/오탐 안 가리고 다 올라옴 = 그게 가시성. Alert로 "뭔가 떴다" → Zeek 메타로 "맥락"(새벽 3시·낯선 IP의 ssh) → PCAP로 "실제 패킷" 확인. 사후 포렌식의 재료 |
| **Zabbix (NMS)** | 메트릭 시계열(CPU·net·세션) / 트리거 이벤트 | Zabbix 웹 대시보드 | **baseline(평소값) 대비 스파이크**를 본다. 평소 곡선을 알아야 "지금 비정상"을 판단(DDoS 골든타임). 절대값이 아니라 "평소 대비 얼마나 튀었나" |
| **서버 자체 로그** | `auth.log`(SSH) / 웹 access·error 로그 | 각 서버 `/var/log/` | SSH 실패 로그 무더기 = Hydra 무차별 대입. **실패 로그의 주체는 비번 틀린 관리자 아니면 공격자 둘뿐.** 웹 로그는 SQLi/스캔 흔적 |
| **Wireshark** | (실시간 캡처, 저장은 pcap) | 분석 PC | 헤더·플래그·시퀀스/ACK·윈도우·페이로드를 눈으로. 장비가 "왜 그렇게 판단했나"를 패킷으로 역검증 |
| **Zabbix FIM** | 파일 해시 변경 이벤트(트리거) | Zabbix Problems | `#1<>#2`(직전 해시와 다름)면 변조. ⚠️ **보안 알람은 자동해제 금지**(OK None + manual close)로 누락 방지. 폴링이라 "변경 사실"만, 누가/언제는 모름 |
| **snoopy** | 명령어 실행 로그(→ auth.log) | `journalctl`/`/var/log/auth.log` | `login`(누가)·`uid:실제/유효`(다르면 sudo·권한상승)·`cwd`·**명령어(무엇을)**. `rm -rf`·`wget`·`nc`·`nmap` 등 의심 명령 grep |
| **서버 로그(journald/rsyslog)** | auth.log/syslog/kern.log | `journalctl -u`/`/var/log/` | facility로 분류된다(auth=인증, syslog=나머지). 24.04는 `journalctl`로 봐야 함. rsyslog가 외부 SIEM 전송 입구 |
| **윈도우 이벤트 로그** | 보안/시스템 이벤트 | 이벤트 뷰어 | **이벤트 ID로 필터**(같은 오류=같은 ID). 무슨 로그를 남길지는 **로컬 보안 정책→감사 정책**으로 정의 |

> 🔗 **로그는 흩어져 있다 → 한곳에 모아야 의미가 생긴다.** 방화벽 Deny + Snort alert + 서버 auth.log + Zabbix 스파이크를 따로 보면 점이지만, 시간축으로 겹쳐 보면 "스캔 → 침투시도 → 자원 급증"이라는 선이 보인다. 이 상관분석을 사람이 손으로 하다 한계 → **ESM/SIEM**으로 수렴(아래 "로그가 흐르는 길").

## 🕳️ 탐지 공백 맵 — "이건 못 잡으니까 이걸 써야 한다"

이 표가 이 문서의 핵심. **"왜 장비를 더 붙이는가"의 근거.** 한 장비의 한계가 다음 장비를 부르는 흐름.

| 못 잡는 것 (한계) | 왜 못 잡나 | 메우는 솔루션 | 비고 |
|---|---|---|---|
| 범용 SQLi/XSS (`' OR '1'='1`) | ET Open 룰엔 오탐 많아 **의도적으로 빠짐**. IDS/IPS는 웹공격 전담이 아님 | **WAF (ModSecurity + OWASP CRS)** | IDS≠WAF. 백로그. 커스텀 룰로 일부만 임시 커버 |
| 암호화된(HTTPS) 페이로드 속 공격 | IPS가 봉투는 봐도 **내용물(L7)은 암호화돼 못 봄** (SSL Bump 보류 중) | **SSL Inspection** / 엔드포인트단 검사 | 복호화는 성능·프라이버시 트레이드오프 |
| 대역폭 고갈형 DDoS (bps) | IPS는 **스루풋 한계** — 자기가 먼저 뻗음 | **Anti-DDoS(syncookies)** 앞단 + 클라우드(AWS Shield/Anycast) | IPS는 1차, 본격 방어는 앞단. 한 라인에 몰면 같이 다운 |
| 알람 폭주 속 진짜 공격 (상관관계) | IPS/IDS는 **점(이벤트)만 뱉지 선을 못 그림** | **ESM/SIEM** (Security Onion의 상용 확장판) | nmap 한 방에 알람 폭주 → 중앙 수집·상관분석 필요성 체감(6/19) |
| 느린 무차별 대입 (Hydra slow) | 임계치(threshold) 밑으로 깔면 IPS가 안 걸림 | **fail2ban / Rate Limit** (서버단) | FIMS의 slowapi를 인프라 레벨로 옮긴 것 |
| 차단 자체 (Security Onion) | 미러라서 **보기만 하고 못 막음** | **Snort 인라인(IPS)** | 보는 쪽↔막는 쪽 역할 분리가 정상 |
| 무선 구간 트래픽 | 유선 SPAN/TAP에 안 걸리는 **음영지역** | **WIPS** | 무선 단말은 IDS 사각 |
| 엔드포인트 내부 행위(파일·프로세스) | 네트워크 장비는 **단말 안을 못 봄** | **EDR** | 네트워크 가시성 ≠ 엔드포인트 가시성 |
| 인증된 내부자의 정상 유출 | 권한 통과한 트래픽이라 방화벽·IPS가 정상으로 봄 | **DLP / DRM** | 막는 게 아니라 "나가는 콘텐츠"를 봄 |
| 위험 쿼리·무분별 DB 추출 (`DELETE`·대량 SELECT) | 정상 인증 세션이 친 SQL이라 FW/IPS는 못 봄. 라이브DB 직접접속이면 트래픽·사고 위험 | **DB Safer / ProxySQL(쿼리 프록시) + 레플리카 분리** | 위험 명령어 블록·감사. 운영DB 원격세션은 닫고 복사본만 본다 |
| 정상 SaaS로 위장한 C2 비콘/반출 | Dropbox·GDrive·HTTPS는 정상 도메인이라 시그니처로 못 거름 | **아웃바운드 통제 + DNS 터널 탐지(allowlist) + SSL 복호화 + ADS** | 주기적 콜백은 행위/주기 이상으로 잡음 (Attack-Defense C2 심화) |
| baseline 없는 상태의 이상 판단 | "평소"를 모르면 "비정상"을 못 정함 | **NMS(Zabbix) baseline + 트리거** | 골든타임 감지의 전제조건 |
| 알려지지 않은 변종(시그니처 우회) | 패턴 매칭은 아는 것만 잡음 | **애노멀리/AI 탐지(Darktrace류)·Anti-APT 샌드박스** | 시그니처의 근본 한계 |
| 쉘에서 친 명령어(무엇을 했나) | 네트워크 로그는 "나갔다"만 앎, `bash_history`는 지워짐·실시간 X | **snoopy(명령어 감사)** | execve 래핑, 못 지움. 내부자·침투 후 행위 추적 |
| snoopy 자체 우회 / 파일·시스템콜 전반 | LD_PRELOAD는 무력화 가능, 명령어만 봄 | **auditd / Wazuh** | 커널 레벨, 우회 어려움 |
| 변조 "행위·주체"(누가·언제 바꿨나) | Zabbix FIM은 해시 폴링 = 변경 "사실"만 | **auditd/inotify(실시간) / Wazuh FIM** | FIM=상태 / auditd=이벤트 |
| AD 도메인 장악(DC 단일 급소) | 중앙집중이라 DC 뚫리면 전체(Pass-the-Hash·Golden Ticket) | **DC 권한최소화+명령어감사+ESM 상관분석** | DC가 최종 표적, 단일 로그로 안 보임 |
| 흩어진 서버 로그를 SIEM으로 | journald 바이너리는 외부 전송 어려움 | **rsyslog forwarding / Filebeat** | 텍스트 전송이 호환성 최고. SIEM 입구 |

> 💡 읽는 법: 왼쪽이 "현장에서 부딪히는 구멍", 오른쪽이 "그래서 뭘 붙이나". 프로젝트 설계할 때 이 표를 거꾸로 돌리면 곧 **장비 도입 정당화 논리**가 된다.

## 🌊 로그가 흐르는 길 — 개별 로그 → 상관분석 → ESM 수렴

```
[개별 장비 로그]                 [한데 모아 보기]               [사람의 판단]
pfSense Deny ───┐
Squid access ───┤
Snort alert ────┼──► ESM/SIEM(상관분석) ──► "스캔→침투시도→자원급증"의 선 ──► 대응
Zeek/PCAP ──────┤      (Security Onion이                                  (차단·격리·신고)
auth.log ───────┤       오픈소스 축소판)
Zabbix 스파이크 ┘
```

- **1단계 — 각자 본다:** 각 장비 콘솔에서 자기 로그를 본다(위 표 2). 여기까진 점.
- **2단계 — 겹쳐 본다:** 같은 시간대에 여러 장비 로그를 포개면 공격의 줄거리가 보인다. Security Onion이 이 역할의 오픈소스 입문판.
- **3단계 — 자동 상관분석:** 규모가 커지면 사람이 못 따라감 → ESM/SIEM이 룰로 자동 상관·알림. (Zabbix는 자원 상태(NMS)지 보안 상관분석(ESM)이 아님 — **NMS ≠ ESM** 혼동 주의)

> 🔗 FIMS 연결: FIMS에서 로그를 한 군데로 모아 본 발상이 인프라 레벨로 커진 게 ESM. socat이 통로를 뚫었듯, ESM은 흩어진 로그의 통로를 중앙으로 모은다.

## 📦 현재 보유 vs 백로그 (탐지공백 메우기 우선순위)

| 상태 | 솔루션 | 메우는 공백 |
|---|---|---|
| ✅ 보유 | pfSense(FW), Squid, Snort(IPS), OpenVPN, Security Onion(NSM), Zabbix(NMS+FIM), **snoopy(명령어 감사)**, **journald/rsyslog**, Kali | 접근통제·아웃바운드통제·시그니처차단·원격접속·가시성·자원/무결성 모니터링·명령어 감사·로그 파이프라인·검증 |
| ⏳ 백로그 | **WAF(ModSecurity+CRS)** | 범용 SQLi/XSS (IDS가 못 잡는 웹공격) |
| ⏳ 백로그 | **fail2ban / Rate Limit** | 느린 무차별 대입 (threshold 밑) |
| ⏳ 백로그 | **SSL Inspection(SSL Bump 재시도)** | 암호화 트래픽 속 공격 |
| ⏳ 백로그 | **ESM/SIEM (rsyslog forwarding으로 SO에 합류)** | 흩어진 로그 상관분석 |
| ⏳ 백로그 | **auditd/Wazuh** | FIM/snoopy 한계(실시간·주체·우회) 보완 |
| ⏳ 백로그 | **AD(Windows DC) 랩** | 계정·권한·정책 중앙관리 직접 실습 |
| ⏳ 백로그 | **DB Safer / ProxySQL + 레플리카** | 위험 쿼리·무분별 DB 추출 (과정2 프로젝트 R&D 데이터취급망) |
| ⏳ 백로그 | **NAC(PacketFence) + PMS·패치 인프라** | 단말 자세점검·격리·치료 (과정2 R&D 단말통제망) |
| 💭 개념만 | EDR / DLP·DRM / WIPS / Anti-DDoS / 허니팟 | 엔드포인트·정보유출·무선·대역폭DDoS·유인 |

## 🔧 이 문서 유지 규칙 (메타)

1. **새 솔루션/프로그램 도입** → 표 1(목적별 분류)·표 2(로그)·"보유 vs 백로그"에 추가.
