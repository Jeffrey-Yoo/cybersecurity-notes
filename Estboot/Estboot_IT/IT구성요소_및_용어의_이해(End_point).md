# IT 구성요소 및 용어의 이해 (End point)

## 📝 DLP (Data Loss Prevention)

내부 정보 유출 방지 솔루션

- 데이터 반/입출 탐지 → 중간 관제 역할 (지속적인 탐지)
- 탐지뿐만 아니라 **차단까지** 수행

---

### 1. NDLP (Network DLP)

네트워크 안 데이터를 탐지

> ex) Mail-i → 개인 정보가 포함된 데이터를 탐지하는 패턴 탐지 시스템

**네트워크 길목 배치 예시:**

```
F/W ──> Backbone ──> Mail Server
                          |
                          |
                     이 구간에서 NDLP 탐지
                     (네트워크 길목에 배치)
```

---

### 2. HDLP (Host DLP)

사용자 하드웨어에 Agent를 설치해 중요 정보를 탐지 → 내 컴퓨터 안에서 작동하는 탐지 시스템

> **강사님 예시)** 고객의 개인 정보를 취급할 수 있는 담당자가 무단으로 고객 데이터를 하드웨어 상에서 옮기는 행위를 방지

---

## 📝 PMS (Patch Management System)

보안 패치 & 솔루션 등의 원격 설치 유도 솔루션

- 각 큰 틀의 망에 설치되어야 함
- 아웃바운드를 열어 업데이트 진행
  - 폐쇄망 같은 물리적으로 접근해야 하는 고위 보안 서버는 물리적으로 접근
- 패치 후 망연계를 통해 각 내부망끼리 통신

**패치 흐름:**

```
패치 파일 업데이트 필요
      ↓
아웃바운드 정책 통해 패치 파일 수신
      ↓
PMS 중계 서버 (DMZ) → 보안 검사/다운로드
      ↓
PMS 서버 (내부망) → 각 하드웨어 전달
      ↓
망연계 통해 각 망으로 배포
```

> **WSUS** (Windows Server Update Services) — Microsoft에서 제공하는 Windows 전용 패치 관리 도구. 기업 환경에서 WSUS를 PMS와 연동해 Windows 패치를 중앙에서 관리

---

## 📝 ECM (Enterprise Contents Management)

문서중앙화 솔루션

1. 기업 정보 중앙 집중화
2. 업무 발생 모든 문서 강제 저장
3. 정보 유출 원천 차단
4. 문서 접근 권한 관리
5. DRM과 같이 쓰는 경우 보안 효율 증가 (문서 저장 + 암호화 동시 적용)

> 기업에서 생성되는 모든 파일을 **문서중앙화**를 통해 데이터 유실을 방지하는 솔루션

---

## 📝 GitLab

1. 사내 시스템 구축 형태 운영 → 소스코드가 외부로 유출되지 않음
2. 개발 소스코드 관리
3. CI/CD 구조를 활용해 빌드 배포 자동화

```
CI : Build → Test → Merge
              ↓
CD : Deploy
```

> GitHub는 외부 클라우드 서비스라 소스코드 유출 가능성 존재. 기업이 GitLab을 선호하는 이유

---

## 📝 CDN (Contents Delivery Network)

정적인 콘텐츠를 가지고 있는 네트워크

```
개발자 웹사이트 이미지 파일
      ↓
CDN 서버 수신 및 저장
      ↓
이용자에게 가장 가까운 서버에서 콘텐츠 전달 (낮은 레이턴시)
```

> **강사님 예시)** 사용자는 Amazon CloudFront (CDN)를 통해 방대히 분포된 정적 콘텐츠 서버의 데이터를 출력받음

**보안 관점:**
- **DDoS 완화** → 대용량 트래픽을 여러 서버로 분산해 공격 흡수
- **WAF와 함께 구성** (CloudFront + WAF) → 웹 공격 방어
- CDN 보안을 위해 개발자의 Deploy 자동화 필수

---

## 📝 GitHub (Microsoft 인수)

Git에 프로젝트 관리 지원을 확장해 제공하는 웹 호스팅 서비스

**보안 관점:**
1. 소스 코드 유출 가능성
2. 권한 제어 어려움
3. 외부 소스코드 사용 → 악성코드 침입 가능성

> GitLab(사내 구축)과 달리 외부 서버에 코드가 올라가므로 기업 기밀 코드 관리 시 주의 필요

---

## 📝 Docker Hub

Docker 관리 하 Docker 레지스트리 → 여러 개발자가 소스코드를 이용한 이미지를 공개해 놓은 Hub

**보안 관점:**
- 도커 이미지를 사내 네트워크망에 반입 → 보안 노출 위험
- **프라이빗 레지스트리** 따로 구축 후 인증된 이미지만 사용
- Image Pull 전 **취약점 스캔** 필수 → 전용 Tool 사용 (ex. Trivy)

---

## 💭 오늘의 회고

### 배운 점
- 기업 보안 아키텍처 안에서 이루어져야 할 여러 솔루션들을 알게 됨
- 단순히 각 솔루션의 기능을 배우는 것을 넘어 DLP/PMS/ECM/DRM이 어떤 구간에 위치하고 서로 어떻게 연계되는지 흐름으로 이해함
- GitLab, GitHub, Docker Hub 등 개발 인프라 솔루션도 보안 관점에서 바라봐야 한다는 시각을 갖게 됨

## 📚 알게 된 IT 용어

| 용어 | 설명 |
|---|---|
| **FDS (Fraud Detection System)** | 이상 거래 감지 시스템. 카드 결제 패턴/로그인 위치 등을 분석해 비정상 거래 실시간 탐지/차단. 머신러닝 기반으로 발전 중 |
| **SOAR (Security Orchestration, Automation and Response)** | 보안 작업 자동화 솔루션 |
| **F/W Policy - Outbound** | 내부 영역 데이터 → 외부 (방화벽 기준 내부에서 나가는 트래픽) |
| **F/W Policy - Inbound** | 외부 → 내부 영역 데이터 (방화벽 기준 외부에서 들어오는 트래픽) |
| **DMZ** | 내부망과 외부망 사이의 완충 구간. 외부에서 접근 가능하지만 내부망과는 분리된 구역 |
| **WSUS** | Windows Server Update Services — Microsoft 제공 Windows 전용 패치 관리 도구 |
