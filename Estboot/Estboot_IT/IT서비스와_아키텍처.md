# What Is Architecture? — 아키텍처의 이해

## 📝 What Is Architecture?

- 정보 시스템 구축 종합 서비스
- 전산자원을 통합해 최적 해결점 제시
- 정보시스템 개발/유지/보수 과정을 포함하는 광범위한 개념

> IT 인프라 기술을 제공해 산업 등에 제공하는 서비스 → 목적에 맞는 아키텍처 구조를 생각하자

---

## 📝 인프라 아키텍처 구성과 특징

### 목적

신뢰성, 안정성, 신속성, 공용성을 겸비한 아키텍처 구성 필요

| 목적 | 설명 |
|------|------|
| **신뢰성** | 안전한 이미지 구축 및 침해 공격에 강한 설계 |
| **신속성** | 개발/테스트/배포를 업데이트에 대한 신속 설계 |
| **공용성** | 전국민 PC/모바일을 대상으로 대용량 Transaction 처리 |

## 📝 CDN (Contents Delivery Network)

정적 콘텐츠 저장 네트워크

```
사용자 ↔ Edge POP ← (CDN 연결) → Edge POP ↔ Origin 서버
```

- **Origin 서버** → 원본 콘텐츠 보관
- **Edge POP** → 각 지역에 분산된 캐시 서버. 정적 콘텐츠를 저장해 가장 가까운 POP에서 사용자에게 전달
- **Akamai** → 세계 최대 CDN 기업. Edge POP 개념의 대표 사례

```
한국 서울 사용자 → 서울 POP에서 수신
한국 부산 사용자 → 부산 POP에서 수신
미국 사용자      → 미국 POP에서 수신
```

## 📝 아키텍처 종류

### 1. EA (Enterprise Architecture)

- 비즈니스 이해를 바탕으로 한 장기적 IT 전략
- 기업 IT 구조 전체를 담당
- AS-IS 분석 → TO-BE 기획 → 실행 담당

**산출물:**
- ISP 문서 기획
- AS-Is / To-Be 아키텍처 다이어그램
- 기술 로드맵

#### ISP (Information Strategy Planning) — 정보전략계획

| 종류 | 설명 |
|------|------|
| **Typical-ISP** | 전형적 ISP. 정기적으로 반복 수행 (3~5년 주기) |
| **Macro-ISP** | 근본적 변화를 위한 대규모 장기 계획 |
| **Micro-ISP** | 단일 정보 시스템 개선 목적의 정보화 계획 |
| **ISMP** | 정보시스템 마스터플랜. RFP 작성이 목적인 총괄 계획 |

#### ISP 수립 4단계

```
1. 기업 모형 분석
   → 설계도의 바탕 작업 (현재 상태 파악)

2. 정보 기술 환경 분석
   → 설계도 안에 필요한 기술 및 바탕 분석

3. 정보 시스템 정의
   → 필요한 기술의 프로세스를 분석/설계해 전체 구조 정의

4. 정보 기술 정의
   → 각 설계도를 기반해 필요한 기술의 구성도 정의
```

---

### 2. AA (Application Architecture)

- 앱들을 어디에 배치하고 어떻게 연결할 것인가
- ERP 등 업무 관련 모듈 및 기능 설계
- 설계를 도식화하는 역할

**산출물:**
- 애플리케이션 배치도
- API 명세서
- 시스템 연동 다이어그램

```
예시:
웹서버 → DMZ 배치
WAS   → 내부망 배치
DB    → 내부망 깊숙이 배치
각 앱 → REST API로 연결
```

---

### 3. SA (Solution Architecture)

- 비즈니스 문제를 해결하기 위해 어떤 솔루션을 조합할 것인가
- 해당 상황에 맞는 기술적 설계
- AA가 방화벽을 배치했다면 어떤 방화벽 솔루션을 도입할 것인가

**산출물:**
- RFP (Request for Proposal) — 제안요청서
- 솔루션 비교 분석서
- 구현 로드맵

---

### 4. DA (Data Architecture)

- 데이터가 어디에 저장되고, 어떻게 흐르고, 누가 접근할 수 있는지 설계
- 사용자 기반 데이터를 어떻게 산출해 분석/통합해서 유의미한 데이터로 만들 것인가
- 흐르고 있는 데이터는 어떤 권한을 가지고 어디에만 존재할 것인가

**산출물:**
- ERD (데이터 모델)
- 데이터 흐름도 (DFD)
- 데이터 접근 권한 정책서

---

### 5. TA (Technical Architecture)

- 위 모든 설계들을 어떻게 구현해낼 것인가
- 각 계층 물리적인 환경에 대한 구조 설계
- 전반적인 기술적 문제 해결

```
설계된 데이터가 이중화가 필요하다면 어떻게 할 것인가
메일은 어떻게 주고받을 것인가
웹 서버는 어떻게 구동될 것인가
회선은 어떤 걸 써야 가용성/기밀성/무결성 모두 확보할까
효율적이며 확실한 방법을 목적에 맞게 구성했는가
```

|------|------|------|

| **DAS (Direct Attached Storage)** | 서버에 직접 연결된 저장장치 | 내 PC에 붙은 외장하드 |
| **NAS (Network Attached Storage)** | 네트워크로 연결된 파일 공유 저장소 | 공유 드라이브 |

| 구분 | 설명 |
|------|------|
| **HA (High Availability)** | 고가용성. Active 서버 장애 시 Standby 서버가 즉시 인계해 서비스 중단 없이 유지 |
| **WAS Cluster** | WAS 서버 여러 대를 묶어 부하분산 및 이중화 |
| **L4 Switch** | 로드밸런서. 트래픽을 여러 서버로 분산 |
| **MS Cluster** | Microsoft 서버 클러스터링 솔루션 |
| **HP Serviceguard** | HP 서버 고가용성 클러스터 솔루션 |

## 💭 오늘의 회고

### 배운 점
- 각 목적에 맞는 인프라 구성을 보고 아키텍처 구조를 생각하는 시각 획득
- 우리가 배우고 있는, 배워야 하는 시작점 파악
- 보안 회사 입장에서 어떤 관점으로 아키텍처를 바라봐야 하는지
- 아키텍처의 종류와 각 역할에 맞는 기능

### 어려운 점 / 개선할 점
- 종류의 이해는 어느 정도 됐지만 실제로 일어나는 흐름과 기능 이해 부족
- 배웠던 보안 시스템이나 네트워크 용어 숙지 부족
- 복습을 통해 척하면 척 나오는 정도까지 숙지

## 📚 알게 된 IT 용어

| 용어 | 설명 |
|------|------|
| **ERP (Enterprise Resource Planning)** | 회계, 재고, 인사, 세무 등 기업 자원을 통합 관리하는 시스템 |
| **Akamai** | 세계적 CDN/보안 기업. Edge POP 기반 콘텐츠 전송 서비스 제공 |
| **Edge POP** | 정적 콘텐츠를 저장한 네트워크 캐시 서버. 사용자와 가장 가까운 위치에서 콘텐츠 전달 |
| **Region** | 데이터센터 2곳 이상을 묶어둔 클라우드 인프라 단위 |
| **DW (Data Warehouse)** | 대용량 데이터 저장/분석을 위한 데이터 저장소 |
| **RFP (Request for Proposal)** | 제안요청서. 발주사가 벤더에게 솔루션 제안을 요청하는 문서 |
| **NAC (Network Access Control)** | 네트워크 접근 통제. PC의 보안 정책 준수 여부 확인 후 접근 허용/차단. PMS와 연동해 패치 미적용 PC는 네트워크 접근 차단 |
| **Load Balancer (L4)** | 트래픽을 여러 서버로 분산해 부하를 줄이고 가용성 확보 |
| **SSO (Single Sign-On)** | 통합 로그인 솔루션. 한 번 로그인으로 여러 시스템 접근 가능. 보안 관점에서 SSO 탈취 시 연결된 모든 시스템이 위험해져 MFA와 함께 사용 필수 |
| **HA (High Availability)** | 고가용성. 장애 발생 시 Standby 서버가 즉시 인계해 서비스 중단 방지 |
| **DAS** | Direct Attached Storage — 서버 직접 연결 저장장치 |
| **NAS** | Network Attached Storage — 네트워크 기반 파일 공유 저장소 |
| **SAN** | Storage Area Network — 고속 전용 네트워크 스토리지 |
