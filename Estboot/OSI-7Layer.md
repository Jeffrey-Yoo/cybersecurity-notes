# 📡 OSI 7계층과 네트워크 프로토콜

---

## TCP/IP 4계층 (실무 기준)

OSI 7계층을 실무에서는 4계층으로 압축해서 사용한다.

```
L7 응용계층      → HTTP, DNS, FTP, SMTP
                 OSI 5,6,7계층을 묶어서 응용계층으로 표현
                 모든 앱에서 만들어진 데이터 = 페이로드

L4 전송계층      → TCP, UDP
                 포트로 앱 구분, 신뢰성 제어

L3 네트워크계층  → IP, ICMP, ARP
                 IP 주소로 경로 결정

L2 데이터링크    → Ethernet, Wi-Fi
                 MAC 주소로 같은 망 안 전달
```

---

## 와이어샤크로 본 패킷 구조

패킷 하나 안에 OSI 레이어가 전부 내장되어 있다.

```
1번줄 → Frame        : 패킷 전체 설명 (몇 바이트인지 등)
2번줄 → Ethernet II  : MAC 통신 = L2
3번줄 → IP           : IP 통신 = L3
4번줄 → TCP / UDP    : 전송 프로토콜 = L4
5번줄 → Data         : 페이로드 = L7
우측 코드             : 기계가 읽는 16진수 HEX 코드
```

- 송신: L7에서 만들어진 데이터가 아래로 내려오면서 각 레이어 헤더가 붙음 → **캡슐화**
- 수신: L1부터 올라가며 헤더를 벗김 → **역캡슐화**
- IPS는 이 패킷 안의 페이로드에서 악성 패턴을 탐지하고 차단함

---

## L7 — 응용 계층

- 사용자가 직접 쓰는 앱 레이어
- HTTP, DNS, FTP, SMTP 등
- FIMS에서 FastAPI 라우터(/api/login 등)가 전부 L7에서 동작한 것

---

## L6 — 표현 계층

- 데이터 형식 변환 + 암호화 담당
- TLS/SSL, JPEG, ASCII
- HTTPS의 암호화가 여기서 일어남
- FIMS에서 Apache에 Let's Encrypt 인증서 달아서 HTTPS 붙인 것 = L6 작업

---

## L5 — 세션 계층

- 연결 시작 · 유지 · 종료 관리
- 화상통화를 1시간 하면 그 1시간 동안 연결이 끊기지 않도록 세션을 유지하는 역할
- "세션 만료됐습니다" = L5 세션이 끊긴 것
- FIMS에서 JWT 8시간 설정 = 세션 유지 개념

> 핸드셰이크(L4)가 연결을 맺어주면, L5가 그 연결 상태를 유지·관리·종료한다.
> L4가 전화를 받는 순간이라면, L5는 통화가 유지되는 시간 전체다.

---

## L4 — 전송 계층 (TCP vs UDP)

### TCP (신뢰성 프로토콜)
- 재전송이 가능해서 신뢰할 수 있는 프로토콜
- 분할된 데이터에 시퀀스 번호가 있어서 합산이 안 맞으면 재전송 요청 후 무한대기
- 금융, 주요 거래, 메시지처럼 데이터 유실이 치명적인 곳에 사용
- 3-way handshake로 연결, 4-way handshake로 종료

### UDP (속도 우선 프로토콜)
- 재전송 규약 없음, 헤더가 심플하고 속도 빠름
- 게임, 스트리밍처럼 유실보다 실시간성이 중요한 곳에 사용

### L4에서 눈여겨볼 것 — 로드밸런싱(LB)
- 라운드로빈 방식 등으로 트래픽을 여러 서버에 분배
- L4 LB: IP + 포트 기준 분배
- L7 LB: URL, 쿠키, 헤더 기준 분배 (nginx, AWS ALB)

---

## TCP 플래그 정리

| 플래그 | 이름 | 역할 |
|---|---|---|
| SYN | Synchronization | 연결 요청, 시퀀스 번호 동기화 |
| ACK | Acknowledgement | 패킷 수신 응답, 받은 시퀀스 +1 |
| FIN | Finish | 세션 연결 종료 |
| RST | Reset | 비정상적인 세션 강제 종료 |
| PSH | Push | 버퍼 안 기다리고 L7으로 바로 올려보냄 (SSH, 텔넷에서 자주 사용) |
| URG | Urgent | 긴급 데이터, 다른 데이터보다 우선순위 높음 |

> Suricata 룰에서 `flags:S` = SYN만 탐지
> SYN만 계속 보내고 ACK 안 보내면 half-open connection = **SYN Flood 공격**

---

## 3-way Handshake (TCP 연결)

```
클라이언트                         서버
    |                               |
    |──── SYN (연결요청) ───────────▶|  시퀀스 번호 임의 결정
    |                               |
    |◀─── SYN + ACK ───────────────|  받은 시퀀스 +1 해서 응답
    |                               |
    |──── ACK (확인) ───────────────▶|  Established 완료
    |                               |
```

- `telnet [ip] [port]` = L4에서 해당 IP의 포트가 열려있는지 확인
- `ping [ip]` = L3 ICMP 구간에서 해당 IP가 살아있는지 확인
- 처음엔 서버가 디폴트로 Listening 상태 유지

---

## 4-way Handshake (TCP 종료)

```
클라이언트                         서버
    |                               |
    |──── FIN (종료요청) ───────────▶|
    |                               |
    |◀─── ACK ─────────────────────|
    |                               |
    |◀─── FIN ─────────────────────|
    |                               |
    |──── ACK ─────────────────────▶|  연결 종료 완료
```

## L3 — 네트워크 계층

```
IP    → 데이터를 목적지까지 배달하는 규칙 (항상 동작)
ICMP  → 배달 경로가 살아있는지 확인 (ping, traceroute)
        IP보다 통신 우선순위가 낮음 (확인용이라 늦어도 무방)
ARP   → IP 주소를 MAC 주소로 변환
        "이 IP가 누구 MAC이야?" 물어보면 해당 장비가 응답
```

> IP주소(내가 가진 고유번호)와 IP프로토콜(배달 규칙)은 이름은 같지만 다른 개념이다.

### IP / ICMP / ARP 흐름

```
데이터 보내려면
  1. 목적지 IP 주소 확인        ← IP
  2. 그 IP의 MAC 주소 알아내기  ← ARP
  3. 경로 살아있는지 확인       ← ICMP
  4. 실제 배달                  ← IP
```

### ICMP 우선순위가 낮은 이유

- ping이 느리다고 서비스가 느린 게 아님 (ICMP는 후순위 처리)
- ping이 막혀도 실제 서비스는 동작할 수 있음 (방화벽에서 ICMP만 drop하는 경우)
- ping 안 된다고 서버가 죽은 게 아님

---

## L2 — 데이터링크 계층

- MAC = 네트워크 장비끼리 서로를 구분하는 고유 주소
- 하드웨어에 고정된 값이지만 소프트웨어로 임의 변경 가능 = **MAC 스푸핑**
- 기업에서 퇴사자 PC의 IP + MAC 주소를 폐기하는 이유

### L2 스위치 vs 허브

```
허브      → 깡통. 들어온 패킷을 모든 포트에 그냥 뿌림
L2 스위치  → 매니지먼트 시스템 있음
            MAC 정보를 포트에 매핑해서 해당 MAC 아니면 DOWN 가능 = PORT SECURITY
```

### NAC (Network Access Control)

MAC 스푸핑을 막기 위해 등장한 시스템.
네트워크에 접속하는 기기의 보안 상태를 검사해서 접근을 허용/차단.

```
[사용자 PC 접속 시도]
    ↓
[NAC] → 백신 설치 여부 확인
      → OS 패치 최신 여부 확인
      → 인가된 기기인지 확인
      → MAC 충돌 탐지 (같은 MAC이 두 포트에서 잡히면 스푸핑으로 판단)
    ↓
허용 → 내부망 접근
차단 → 격리망으로 이동
```

> MAC 필터링만으로는 약함. 802.1X 인증(인증서 기반)과 함께 써야 완결.
> NAC 재정이 부족하다면 L2 스위치 PORT SECURITY로 대체 가능.

## L1 — 물리 계층

- 0과 1을 전기신호(랜선), 빛(광섬유), 전파(와이파이)로 바꿔서 물리적으로 전송
- 랜선 뽑으면 여기서 끊기는 것
- 허브는 L1 장비 (깡통)

## ThroughPut (처리량)

```
ThroughPut = 단위 시간당 실제로 처리된 데이터량
```

- IPS, 방화벽이 패킷을 검사하면 속도가 느려질 수 있음
- 고성능 보안 장비일수록 ThroughPut 수치가 중요한 스펙
- 이론상 대역폭과 실제 처리량은 차이가 있음

## SSL 암호화와 보안의 딜레마

```
해커      → 악성 데이터를 SSL로 암호화해서 숨기려 함
화이트햇   → SSL 복호화해서 내용 분석
NGFW      → SSL Inspection으로 복호화 후 검사 후 재암호화
```

- HTTPS 트래픽은 암호화되어 있어서 일반 방화벽으로는 내용 볼 수 없음
- 그래서 차세대 방화벽(NGFW)은 SSL을 한번 풀어서 검사하고 다시 암호화해서 보냄

## DNS (Domain Name System)

IP를 외울 수 없어서 도메인 이름을 IP로 변환해주는 역할.

### DNS 서버 동작 방식

```
1. 로컬 DNS 서버에 조회 요청
2. 없으면 Root DNS 서버에 조회
3. Root DNS가 .org/.com 등 TLD DNS 정보 제공
4. TLD DNS에서 해당 도메인 DNS 서버 정보 제공
5. 해당 도메인 DNS에서 IP 반환
6. 로컬 DNS가 TTL 동안 캐시 유지 후 기기로 전달
```

- 보안상 내부에서만 공개되어야 하는 IP는 내부 DNS에서 운영
- 외부에 공개되어야 하는 IP는 외부 DNS에서 운영 = **Split DNS**
- DNS는 보통 UDP 53, 응답이 512바이트 초과하거나 Zone Transfer 시 TCP 53

### DNS 스푸핑

- 공격자의 사이트로 유도해 사용자 데이터 탈취
- hosts 파일 변조 (`C:\Windows\System32\drivers\etc\hosts`)로 다른 사이트 유도 가능

### 라운드 로빈

- 같은 도메인에 A 레코드를 여러 개 등록하면 요청마다 순서대로 다른 IP로 분산
- 서버 장애 감지 기능 없음 → 실무에서는 로드밸런서와 함께 사용

## VLAN (Virtual LAN)

### 세그멘테이션 vs VLAN

```
세그멘테이션 (FIMS 방식)
→ VirtualBox 내부 네트워크 이름 기반
→ 소프트웨어 격리
→ 하이퍼바이저 취약점 뚫리면 격리 붕괴 가능

VLAN (802.1Q)
→ 물리 스위치 하드웨어가 태그로 나눔
→ 소프트웨어 취약점과 무관하게 격리 유지
→ 다른 VLAN끼리 통신하려면 반드시 L3 라우터 경유 필요
```

> 면접에서 FIMS를 "VLAN 망분리"라고 하면 틀림.
> "VirtualBox 내부 네트워크로 세그멘테이션 했다"가 정확한 표현.

### VLAN 실습 방법 (물리 장비 없이)

```bash
# VirtualBox + ip link 명령어로 802.1Q 태그 직접 생성
ip link add link enp0s8 name enp0s8.10 type vlan id 10
ip link add link enp0s8 name enp0s8.20 type vlan id 20
ip link set enp0s8.10 up
ip link set enp0s8.20 up
ip addr add 192.168.10.1/24 dev enp0s8.10
ip addr add 192.168.20.1/24 dev enp0s8.20
```

- GNS3: Cisco IOS 이미지 올려서 실제 VLAN 명령어 실습 가능 (CCNA 준비에 직결)

### ACL (Access Control List)

> 누가 어디로 가는 걸 허용/차단할지 정한 목록

- VLAN으로 망을 나눴다면 ACL로 그 망끼리의 이동을 통제
- 표준 ACL: 출발지 IP만 보고 차단
- 확장 ACL: 출발지 IP + 목적지 IP + 포트까지 보고 차단 (실무에서 주로 사용)

```
VLAN = 망을 나누는 것
ACL  = 나눠진 망끼리 이동을 통제하는 것
```
