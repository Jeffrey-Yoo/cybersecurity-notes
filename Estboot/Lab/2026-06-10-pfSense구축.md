## 1️⃣ "물리 NIC"는 어디서 추가하나 — ESXi vs Workstation

막판에 헷갈렸던 지점. **ESXi 호스트 관리자 페이지(브라우저)에서는 물리 하드웨어(서버·NIC 같은 물리 어댑터)를 추가할 수 없다.** 진짜 물리(처럼 보이는) 장치를 늘리려면 한 층 아래인 **VMware Workstation**으로 내려가서 작업해야 한다.

```
Workstation에서  → 기존 물리 NIC 1장이던 것에 LAN Segment로 1장 더 추가
                  → 이제 pfSense가 쓸 물리 스위치가 2개:
                     · WAN (기존 포트, 공유기 브릿지)
                     · LAN (방금 추가한 LAN Segment)
```

> 💡 **왜 이렇게 되나** — ESXi 입장에선 Workstation이 만들어 준 가상 NIC을 **"물리 어댑터"로 인식**한다(6/9 중첩 가상화에서 정리한 "물리는 그 OS 입장에서의 표현"). 그래서 ESXi가 인식하는 '물리' 장치를 늘리고 줄이는 건 Workstation에서 해야 하고, ESXi는 내가 Workstation에서 더한/뺀 걸 **물리적 변화로 받아들인다**(= 내 Workstation을 진짜 물리 서버로 가정).

- **vSwitch** = ESXi가 그 '물리 어댑터' 위에 만드는 **가상 스위치**. ESXi가 자체적으로 만들 수 있는 내부 스위치.

> 비유: Workstation = 건물에 **랜포트 구멍(NIC)을 뚫는 시공**, ESXi의 vSwitch = 그 구멍에 연결된 **건물 내부 배선반(스위치)**. 구멍 뚫기는 시공(Workstation)에서, 내부 배선은 관리실(ESXi)에서.

⚠️ **가상화 숨기기/되살리기** — 띄워둔 게스트를 **"등록 취소(unregister)"** 하면 목록에서 사라진다(삭제가 아님). 다시 **"기존 가상화 불러오기(register existing VM)"**로 vmx를 지목하면 그대로 올라온다. 디스크는 그대로 두고 인벤토리에서만 내렸다 올리는 것.

## 2️⃣ pfSense 설치 — WAN/LAN 인터페이스 할당

### WAN 할당 (수동 · MAC 대조)

```
설치 중 인터페이스 할당 화면에 어댑터별 MAC이 같이 표시됨
→ ESXi에서 확인해 둔 "WAN용 어댑터 MAC"과 눈으로 대조
→ 일치하는 em0을 WAN으로 지정
```

> 💡 어댑터가 2장이면 화면엔 em0/em1로 뜨는데, 둘 중 뭐가 WAN인지는 **MAC으로 식별**하는 게 확실하다. 포트그룹 이름만 믿지 말고 MAC을 맞춰본다.

### LAN 할당 (스킵해도 자동)

```
LAN 수동 지정을 스킵
→ 그런데 pfSense가 어댑터 2장을 감지하면
→ 남은 em1을 LAN으로 자동 지정 (기본 규칙: em0=WAN, em1=LAN)
```

> 🧠 **핵심** — "LAN 스킵"은 *수동 지정을 안 한 것*이지 *LAN이 없는 것*이 아니다. 설치 후 콘솔에 `LAN(lan) → em1`이 떴던 게 자동 할당된 증거. (어댑터 = 가상 랜포트 / 포트그룹 = 그 포트를 어느 망에 꽂을지 — 6/9 개념의 실전판)

## 3️⃣ 대역 충돌 발견 & 해결 ⭐ (오늘의 핵심 삽질)

설치 직후 상태가 이랬다:

```
WAN → em0 → 192.168.1.82/24   (공유기 DHCP에서 받음)
LAN → em1 → 192.168.1.1/24    (pfSense LAN 기본값)
```

> ⚠️ **문제: WAN과 LAN이 같은 192.168.1.x 대역** — 게다가 pfSense LAN(192.168.1.1)이 **공유기 게이트웨이(192.168.1.1)와 IP까지 똑같다.** WAN/LAN이 같은 대역이면 pfSense가 어디로 라우팅할지 갈피를 못 잡는다. **방화벽 양쪽 인터페이스는 반드시 다른 대역**이어야 한다.

### 해결 — 콘솔 메뉴 2번으로 LAN 대역 변경

```
2) Set interface(s) IP address → LAN 선택

DHCP로 설정?      → n              (LAN은 게이트웨이라 IP를 직접 박는다)
LAN IPv4          → 192.168.2.1    (충돌 없는 새 대역)
서브넷 비트       → 24             (255.255.255.0)
upstream gateway  → Enter(공란)    (LAN쪽은 상위 게이트웨이 없음)
IPv6 DHCP6?       → n
IPv6 주소         → Enter
DHCP 서버 켤까?   → y   ★          (내부 기기에 IP 자동 부여)
DHCP 시작         → 192.168.2.100
DHCP 끝           → 192.168.2.200
HTTP로 되돌릴까?  → n              (HTTPS 관리 유지)
```

결과:

```
WAN → 192.168.1.82/24   (공유기 대역, 그대로)
LAN → 192.168.2.1/24    (내부 전용 — 충돌 해소)
```

> 💡 **개념 분리** — "대역 변경(2.1로)"은 *주소 정하기*, "DHCP 켜기"는 *IP 나눠줄 준비*. 두 개는 다른 작업인데 메뉴 2번 안에서 연달아 물어봐서 헷갈리기 쉽다.

## 4️⃣ DHCP는 pfSense가 찾아다니는 게 아니다 — DORA

> 🧠 pfSense가 능동적으로 "누구 있나" 스캔하는 게 아니다. **기기가 먼저 "IP 줘"라고 외치고**, pfSense가 응답한다.

```
1. Discover     : 기기가 "DHCP 서버 있나요?" 브로드캐스트
2. Offer        : pfSense가 "192.168.2.100 어때요?" 응답
3. Request      : 기기가 "그거 쓸게요"
4. Acknowledge  : pfSense가 "OK 등록" → 기기 사용 시작
```

> 💡 비유: LAN 포트그룹 = 같은 **회의실(201호)**, pfSense = 201호 관리자, 새 기기 = 방금 들어온 사람. "제 자리번호요?"(같은 방이라 들림) → "100번 앉으세요." **같은 포트그룹(=같은 브로드캐스트 도메인)이어야 이 요청이 닿는다.**

> ⚠️ 그래서 기기가 엉뚱한 포트그룹(예: WAN)에 꽂히면 → 브로드캐스트가 LAN쪽 pfSense에 안 닿아 **DHCP를 못 받는다.** 안 받아지면 제일 먼저 "얘가 LAN 포트그룹 맞나"부터 본다.

## 5️⃣ xubuntu 올리기 — LAN 전용 관리 데스크탑

| 항목 | 값 | 비고 |
|---|---|---|
| CPU | 4 | 데스크탑용 적당 |
| 메모리 | 8GB | RAM 빠듯하면 4GB로 줄여도 충분 |
| 디스크 | 30GB | 데스크탑+도구용 |
| 네트워크 어댑터 | **LAN만** | 내부망 식구라 WAN 불필요 |
| CD/DVD | xubuntu ISO | 설치 이미지 |

> 💡 **왜 LAN만 꽂나** — xubuntu는 내부망 식구다. pfSense LAN(192.168.2.1)과 같은 망에 붙고, **인터넷도 pfSense(게이트웨이)를 거쳐 나가므로** WAN을 직접 꽂을 이유가 없다. (FIMS에서 내부 서버가 방화벽을 게이트웨이로만 두고 직접 외부 인터페이스를 안 가졌던 것과 같은 결)

```
설치 중 "Use wired connection" 선택
→ VM 안에선 ESXi 가상 NIC이 "유선"으로 보임
→ "No Wi-Fi devices detected"는 가상머신이라 정상 (무선카드가 없을 뿐)
```

## 7️⃣ 정리 — 192.168.2.x = 앞으로의 "관리 대역"

```
192.168.2.x = 관리 대역으로 고정 사용
→ xubuntu 같은 관리용 기기를 여기에 붙인다
→ 게이트웨이 pfSense(2.1)가 2.100~200 범위로 IP 자동 부여
→ 앞으로 LAN 포트그룹에 묶는 기기는 자동으로 이 대역 IP를 받는다
```

> 망 설계의 출발점이 잡혔다. 이제 이 LAN 대역 위에 서버들을 올리고, pfSense 방화벽 룰로 통제하는 게 다음 단계.

## ⚠️ 오늘 막혔던/짚어둘 것

- **물리 NIC을 ESXi에서 못 더한다** — 한참 ESXi 페이지에서 NIC 추가를 찾았는데, 그건 Workstation 일이다. "ESXi가 보는 물리 = Workstation이 준 가상" 구조를 다시 체감.
- **WAN/LAN 대역 충돌** — pfSense LAN 기본값(192.168.1.1)이 하필 우리 공유기 게이트웨이와 똑같아 정통으로 충돌. WAN/LAN은 **무조건 다른 대역**이라는 걸 손으로 박았다.
- **DHCP를 못 받으면 포트그룹부터 의심** — 대역만 맞으면 되는 게 아니라, **같은 브로드캐스트 도메인(같은 포트그룹)**에 있어야 DORA가 오간다.
