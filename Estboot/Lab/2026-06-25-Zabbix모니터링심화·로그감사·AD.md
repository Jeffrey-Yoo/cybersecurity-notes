## 1. UserParameter — Zabbix가 "내가 원하는 값"을 수집하는 법

Zabbix 기본 템플릿은 CPU·메모리·디스크 같은 **표준 지표**만 준다. "netstat의 SYN_RECV 개수" 같은 임의의 값을 수집하려면 에이전트에 **UserParameter**를 정의해야 한다.

```
/etc/zabbix/zabbix_agentd.conf
UserParameter=키,실행할쉘명령
```

- 서버가 그 **키**를 요청하면 → 에이전트가 **해당 쉘 명령을 실행** → 그 **표준출력 1줄**을 값으로 돌려줌.
- 즉 **"키 ↔ 쉘 명령" 매핑**. 키가 호출되면 명령이 실행되는 구조.

> 비유: 자판기 버튼(키)을 누르면 정해진 음료(쉘 명령 결과)가 나온다. 버튼=key, 안에서 실제로 음료를 만드는 동작=쉘 명령. Zabbix 서버는 버튼만 누르고, 뭘 만들지는 에이전트(자판기)가 안다.

### 수집/저장/표시 분리 (전체 구조)

```
서버: "net.syn_recv 줘"
  → 에이전트: ss -ant | grep -c SYN-RECV 실행
  → 출력 숫자를 서버로 반환
  → 서버가 MariaDB(history 테이블)에 저장
  → 웹UI(Apache+PHP)가 DB를 읽어 대시보드에 표시
```

### 디버깅 명령 2종 — 반드시 구분 (오늘의 핵심 함정)

```bash
zabbix_agentd -t net.syn_recv                       # ① 에이전트 단독 테스트 (root로 즉석 실행)
zabbix_get -s 10.10.10.10 -p 10050 -k "net.syn_recv" # ② 서버에서 실제 통신경로로 가져오기
```

- `-t` 결과 표기: `[t|값]`=text, `[d|값]`=numeric, `[m|ZBX_NOTSUPPORTED]`=키 인식 실패.
- `zabbix_get`은 별도 패키지(`apt install zabbix-get`).
- ⚠️ **중요 차이:** `-t`는 **root**로 즉석 실행이라 되는데, `zabbix_get`은 **데몬(zabbix 유저)** 으로 도는 실제 경로 → **권한 문제가 여기서 드러난다.** (7절 FIM에서 정확히 이 함정에 빠짐)

> 💡 교훈: "`-t`로 됐는데 왜 그래프엔 안 떠?" → 권한은 **데몬(zabbix 유저) 기준**으로 봐야 한다. root로 된 결과에 속지 말 것.

## 2. SYN_RECV 모니터링 — DoS 탐지

### UserParameter + 아이템 + 트리거

```
UserParameter=net.syn_recv,ss -ant | grep -c SYN-RECV
```
- `ss -ant` : 소켓 상태를 (a)전체 (n)숫자그대로 (t)TCP만 출력. `netstat` 대체 최신 도구.
- `grep -c SYN-RECV` : SYN-RECV 상태 줄 **개수만** 셈.
- ⚠️ (삽질) 처음 `wc -1`(숫자 1) 오타 → `wc -l`(엘) 이어야 함. 또 `netstat`은 `SYN_RECV`(언더바), `ss`는 `SYN-RECV`(하이픈) 표기가 달라서 헤맴.

```
아이템:  Key=net.syn_recv / Numeric(unsigned) / Update interval 10s / History 7d
트리거:  Name="SYN_RECV 공격 감지" / Severity High
        Expression: last(/web_server/net.syn_recv)>50
```
- `last(/host/key)` = 그 키의 **가장 최근 수집값**. 최근값이 50 초과면 발동.

### 실험에서 체득한 SYN Flood의 본질

- `--rand-source`로 **출발지 IP 위조** → 서버는 SYN-ACK를 엉뚱한 곳에 보내고 ACK가 안 와서 **half-open(SYN_RECV)** 이 쌓임 = SYN flood.
- ⚠️ **CPU(htop)는 거의 안 오른다.** SYN_RECV는 커널 연결테이블 항목일 뿐 애플리케이션이 처리하는 게 아니라서. **진짜 위협은 backlog 큐가 차서 정상 연결을 못 받는 것.**
  - 확인: `cat /proc/sys/net/ipv4/tcp_max_syn_backlog` (기본 128~512). 이 수 넘으면 신규 연결 거부.
- 실무 대응은 IP 차단이 아니라 **SYN Cookie**(half-open을 메모리에 안 올림): `sysctl -w net.ipv4.tcp_syncookies=1`.

> 🔗 6/24 Snort 임계치 룰(`flags:S` + threshold)이 "네트워크 길목에서 끊는 쪽"이었다면, 오늘 Zabbix는 "서버 안에서 그 결과(SYN_RECV 적체)를 **보는 쪽**". 같은 SYN Flood를 막는 쪽/보는 쪽 두 각도로 본 셈.

## 3. 연결상태 3종 모니터링 (SYN_SENT / ESTABLISHED / TIME_WAIT)

```
UserParameter=net.syn_sent,ss -ant | grep -c SYN-SENT
UserParameter=net.established,ss -ant | grep -c ESTAB
UserParameter=net.time_wait,ss -ant | grep -c TIME-WAIT
```

| 상태 | 뜻 | 비정상 신호 |
|---|---|---|
| **SYN_SENT** | **이 서버가 밖으로** 연결 시도 후 응답 대기 | 급증 = 내부서버가 외부로 대량연결 → **C2 의심** |
| **ESTABLISHED** | 완전히 맺어진 활성 연결 | 평소 대비 폭증 = 부하/공격 |
| **TIME_WAIT** | 정상 종료 후 대기 | 폭증 = 연결을 빠르게 맺고 끊음(스캔/벤치) |

- ESTABLISHED 테스트(칼리에서 동시 연결 유지):
  ```bash
  for i in {1..50}; do curl -s --max-time 30 http://192.168.1.100 & done
  ```
  연결이 너무 빨리 끝나면 안 잡히므로 `--max-time`(또는 `--limit-rate 1`)으로 **연결을 오래 유지**시켜야 그래프에 보인다.
- 모니터링 용도이므로 트리거 없이 **그래프만** 구성.

> 💡 **SYN_SENT가 보안적으로 제일 중요.** 서버가 *밖으로* 거는 연결이 갑자기 늘면 = 감염된 내부 호스트가 C2로 말 거는 그림. (Snort 역방향 룰 `$HOME_NET -> $EXTERNAL_NET`과 같은 발상의 호스트판)

### ⚠️ 헷갈린 점 — 아이템 수집 간격 vs 위젯 새로고침

- **아이템 Update interval(예: 1s)** = 에이전트가 DB에 값을 쌓는 주기 → 그래프 **정밀도**.
- **위젯/대시보드 Refresh(예: 10s)** = 브라우저가 DB를 다시 읽어 **화면을 다시 그리는** 주기.
- 둘은 **별개**. 데이터는 1초로 촘촘히 쌓이고 화면은 10초마다 갱신돼도, 갱신될 때 1초 단위 변화가 다 보인다.

## 4. SSH 무차별 대입 탐지 — Ubuntu 24.04는 journald

### 로그 위치가 바뀌었다

```bash
ls /var/log/ | grep -i auth        # auth.log 없음 (구버전 방식)
journalctl -u ssh --no-pager | tail -5
```
- Ubuntu 24.04는 `/var/log/auth.log` 대신 **systemd-journald**가 로그를 관리 → 파일 grep이 아니라 `journalctl`로 읽어야 함. (원리는 9절)

### UserParameter + 권한 + 트리거

```
UserParameter=ssh.failed,journalctl -u ssh --no-pager | grep -c "Failed password"
```
```bash
usermod -aG systemd-journal zabbix      # 저널은 아무나 못 읽음 → zabbix 유저에 읽기권한
systemctl restart zabbix-agent
```
```
아이템:  Key=ssh.failed / Numeric(unsigned) / 10s
트리거:  last(/web_server/ssh.failed)>20  (Severity High)
```

### 공격 테스트 (Hydra)

```bash
gunzip /usr/share/wordlists/rockyou.txt.gz      # rockyou는 .gz → 풀어야 함
cat > /tmp/pass.txt <<'EOF'
1234
password
admin
secu
secu1234
EOF
hydra -l secu -P /tmp/pass.txt ssh://192.168.1.100:1010 -t 4 -V
```
- `-l` 계정 / `-P` 사전파일 / `-t 4` 동시 4스레드 / `-V` 시도 과정 표시.
- ⚠️ 이 웹서버 SSH 포트는 표준 22가 아니라 **1010**으로 바꿔놨으니 포트 명시 필요.
- 발동 시: Latest data 값↑ → Problems 알람 → **맵에서 호스트가 빨갛게**.

> 🔗 6/19에 "SSH 실패 로그 무더기 = Hydra, 실패 로그의 주체는 관리자 아니면 공격자 둘뿐"이라 적었다. 오늘은 그 실패 로그를 **Zabbix가 자동으로 세서 임계치로 알람**까지 띄운 것 — 사람이 grep하던 걸 시스템이 한다.

## 5. 파일 무결성 모니터링 (FIM) — 오늘 제일 많이 삽질한 곳

### 개념

- 공격자는 침투 후 **특정 파일**을 건드려 백도어/권한상승/방어무력화를 한다.
- 그 파일의 **해시(지문)** 를 주기적으로 떠서, **직전 해시와 달라지면 = 변조**로 판정.

| 파일 | 변조 시 의미 |
|---|---|
| `~/.ssh/authorized_keys` | 공격자가 자기 공개키 추가 = **영구 SSH 백도어** |
| `/etc/passwd` | 계정 추가/변경 |
| `/etc/ssh/sshd_config` | `PermitRootLogin yes` 등 **보안설정 무력화** |

### UserParameter (해시 추출)

```
UserParameter=file.passwd,md5sum /etc/passwd | awk '{print $1}'
UserParameter=file.sshconfig,md5sum /etc/ssh/sshd_config | awk '{print $1}'
UserParameter=file.authkeys,sudo md5sum /home/secu/.ssh/authorized_keys 2>/dev/null | awk '{print $1}'
```
- `md5sum 파일` : 내용의 **고정 길이 지문**. 1바이트만 바뀌어도 해시가 완전히 달라짐.
- `awk '{print $1}'` : md5sum 출력은 "해시 파일명" 두 칸 → **해시만** 추출.
- 빈 파일의 md5는 항상 `d41d8cd98f00b204e9800998ecf8427e` → authorized_keys가 0바이트일 때 이 값이 나옴(정상, 키 추가하면 바뀜).

### ⚠️ 권한 함정과 해결 (sudo 한정 허용)

- `passwd`·`sshd_config`는 **누구나 읽기 가능** → 그냥 됨.
- `authorized_keys`는 **소유자(secu)만 접근**(`drwx------`) → 데몬(zabbix 유저)이 못 읽음.
  - 증상: `zabbix_agentd -t`(root)는 값 나오는데 `zabbix_get`(zabbix 유저)은 **빈 값**. (1절 함정이 여기서 터짐)
- 채택 해결 = **sudo 한정 허용**(파일 권한 자체는 그대로 둠 = 최소권한):
  ```bash
  echo 'zabbix ALL=(ALL) NOPASSWD: /usr/bin/md5sum' > /etc/sudoers.d/zabbix
  chmod 440 /etc/sudoers.d/zabbix          # sudoers 권장 권한, 안 맞으면 sudo가 무시
  systemctl restart zabbix-agent
  ```
  → UserParameter도 `sudo md5sum ...`으로 작성.

### ⚠️ 아이템 설정 함정 (FIM 특이점)

```
Type of information: Character    # 해시는 텍스트라 숫자형 아님
History: Store up to 31d          # ★ 반드시 저장. 안 하면 직전값(#2) 비교 불가
Trends: Do not store              # 문자형은 트렌드 불가
```
- **함정:** History를 "Do not store"로 두면 Latest data가 **빈칸**, 트리거 동작 안 함. (여기서 한참 헤맴)
- Custom intervals(Flexible)가 잘못 끼면 수집이 꼬일 수 있어 제거. 막히면 **정상 동작 아이템을 Clone**해서 Key만 바꾸는 게 가장 확실.

### ⚠️ 트리거 — 변조 감지 + "보안 알람은 자동해제 금지"

기본 변조 감지식:
```
last(/web_server/file.authkeys,#1)<>last(/web_server/file.authkeys,#2)
```
- `#1` 최신 해시 / `#2` 직전 해시 / `<>` 다름 → **직전과 달라지면 발동**.

**문제:** 변조 후 다음 수집 때 `#1=#2`가 되면 Zabbix가 "정상 복귀"로 보고 **알람을 자동 해제**해버림.
**해결 (보안 알람 원칙):** 트리거 편집에서
```
OK event generation: None     # 자동 해제 끔 → 한번 켜지면 계속 유지
Allow manual close: 체크       # 매니저가 직접 확인 후 닫아야 사라짐
```
- 담당자가 확인/조치 후 **수동 Close** 전까지 알람이 남아 **누락 방지**.
- 수동 닫기: `Problems → 알람 → Update → Close problem 체크 → Update`.

### 테스트 & 원복

```bash
echo "ssh-rsa FAKEKEY attacker@kali" >> /home/secu/.ssh/authorized_keys   # 백도어 흉내
useradd hacker_test                                                       # passwd 변조
echo "# malicious" >> /etc/ssh/sshd_config                                # sshd 변조
```
→ 10초 내 Problems에 3종 변조 알람, OK생성 None이라 **계속 유지**.
원복도 파일을 바꾸는 거라 해시가 또 달라져 **변조 트리거가 다시 울릴 수 있음**(정상 동작).

### ⚠️ FIM의 본질적 한계 (해시 폴링)

- 10초 간격 스냅샷이라 **수집~수집 사이에 변조 후 원복**하면 놓침.
- "누가/언제/무엇을" 정보가 없음(해시 비교라 **변경 사실만** 앎).
- 실무 보완: **auditd/inotify(실시간 이벤트)** 또는 **Wazuh FIM**.
  - 정리: **Zabbix FIM = "지금 정상인가"(상태)** / **Wazuh·auditd = "변조 행위가 있었나, 누가 했나"(이벤트)**.

## 6. 서비스 가용성 감시

```
UserParameter=svc.apache,systemctl is-active apache2
UserParameter=svc.ssh,systemctl is-active ssh
UserParameter=svc.zabbix,systemctl is-active zabbix-agent
```
- `systemctl is-active 서비스` : 살아있으면 문자열 `active` 출력. 아이템 Type=Character.
- 트리거(다운 감지): `last(/web_server/svc.apache)<>"active"` (Severity Disaster) → 값이 `active`가 아니면 발동.
- ⚠️ **실수:** FIM 테스트 때 멈춘 Apache를 안 켜서 나중에 80포트 접속이 안 되는 사고. `ss -lnt | grep :80`로 리스닝 확인 → `systemctl restart apache2`로 해결. (테스트로 멈춘 서비스는 반드시 되살리기)

## 7. 시각화 — 맵 + 대시보드 = SOC 콘솔

```
Monitoring → Maps → Create map → Edit → Add(요소) → 더블클릭 Type:Host
→ 두 요소 선택 → Link → Add (연결선) → Update
```
- 요소에 호스트를 연결하면 **그 호스트 트리거 발동 시 자동으로 빨갛게**.
- 대시보드 위젯: Graph(그래프) / **Item value(현재값 숫자 크게)** / Map / Problems 목록.
- 권장 레이아웃:
  ```
  [네트워크 맵]        [SYN_RECV 그래프]
  [SYN_RECV 현재값]    [Problems 목록]
  ```
- 효과: 칼리 공격 시 한 화면에서 **그래프↑ + 숫자↑ + 맵 빨강 + 알람** 동시 관측 = **직접 만든 미니 SOC 콘솔**.

> 🔗 6/17 Security Onion SOC 콘솔이 "네트워크 트래픽을 보는 관제"였다면, 오늘 Zabbix 대시보드는 "**서버 자원·가용성·무결성**을 보는 관제". 둘이 합쳐져야 진짜 관제 화면(NMS+NSM). 단 둘 다 ESM은 아님(상관분석은 별개).

## 8. 트러블슈팅 모음 (증상 → 원인 → 해결)

| 증상 | 원인 | 해결 |
|---|---|---|
| `Unable to locate package zabbix-agent` | 대상 서버에 Zabbix 저장소 미등록 | `.deb` 등록 + `apt update` 후 재설치 |
| 사전점검 `System locale ... Fail` | 로케일 패키지 없음 | `apt install locales` → `locale-gen en_US.UTF-8` |
| 웹UI DB연결 `Access denied (zabbix@localhost)` | **설정파일 DBPassword ≠ 실제 DB 비번** | 비번 일치시키고 `systemctl restart zabbix-server` |
| 대시보드 "Zabbix server is not running" | 위와 동일(서버↔DB 인증 실패) | `tail -f /var/log/zabbix/zabbix_server.log`에 `database is down`/`Access denied` 확인 → 비번 통일 후 재시작 |
| FIM 아이템 Latest data **빈칸** | History "Do not store"(문자형) | History를 **Store up to 31d**로 변경(필요시 Clone) |
| `-t`는 값 나오는데 `zabbix_get`은 빈값 | 데몬(zabbix 유저) **파일 읽기 권한 없음** | `sudoers.d/zabbix`로 `md5sum` 한정 sudo + UserParameter에 `sudo` |
| 변조 알람이 10초 뒤 자동 사라짐 | 트리거 OK 이벤트 자동 생성 | **OK event generation: None** + **Allow manual close** |
| 본컴에서 80 접속 불가(NAT는 정상) | FIM 테스트로 멈춘 Apache 미기동 | `ss -lnt | grep :80`로 리스닝 확인 → `restart` |

## 9. 로그 감사 ① snoopy — 모든 명령어 실행을 가로채 기록

auth.log를 보며 "SSH 실패를 임계치로 잡듯, 내가 원하는 탐지/차단 룰을 커스텀할 수 있다"는 데서 출발했다. 그런데 **쉘 안에서 친 명령 자체가 악성이면?** 방화벽 차단로그·SYN_SENT로는 "나갔다"는 건 알아도 "무슨 명령을 쳤나"는 모른다. 그래서 **명령어 감사 도구 snoopy**.

> **snoopy = `execve()` 시스템콜을 가로채서 모든 명령어 실행을 로깅하는 경량 감사 도구.** 한 줄로: "이 서버에서 누가 어떤 명령어를 언제 쳤는지" 전부 기록.

### 핵심 원리 — LD_PRELOAD 인터셉트

리눅스에서 명령어 실행 흐름: `bash가 fork() → 자식이 execve("/bin/ls") → 커널이 로드·실행`. snoopy는 그 **execve() 앞에 끼어들어 기록하고 원래 걸 실행**한다(대체가 아니라 wrap → 실행 자체엔 영향 없음).

```
일반:           ls → glibc의 execve()
LD_PRELOAD 시:  ls → libsnoopy.so 먼저 로드 → snoopy의 execve() 래퍼가 glibc 것을 덮음
               → ls가 execve() 호출하면 snoopy 것이 먼저 실행(기록) → 원래 execve() 진행
```
- 리눅스는 실행 시 라이브러리를 동적 로드(Dynamic Linking)하는데, **LD_PRELOAD에 지정된 라이브러리를 가장 먼저** 로드하는 점을 이용.
- 환경변수 `LD_PRELOAD`는 그 세션에만 적용 → 시스템 전체엔 **`/etc/ld.so.preload`** 에 `.so` 경로를 박는다(모든 유저/프로세스/SSH·로컬 무관 자동 주입).
- ⚠️ **새 프로세스 생성 시에만 적용** → 이미 떠있는 세션엔 소급 안 됨. 설정 후 **새 SSH 세션**에서 검증해야 함.

### 로그에 담기는 정보

```
snoopy[1376]: [login:secu ssh:((undefined)) sid:1192 tty:/dev/pts/3 (0/root) uid:root(0)/root(0) cwd:/root]: ls --color=auto
```
| 필드 | 의미 |
|---|---|
| `login:secu` | 로그인 계정 (누가) |
| `ssh:(...)` | 출발지 IP (어디서, 설정이슈로 undefined) |
| `tty:/dev/pts/3` | 접속 경로 |
| `uid:secu(1000)/root(0)` | **실제UID/유효UID — 두 개가 다르면 sudo 사용 흔적(권한상승 감지)** |
| `cwd:/root` | 작업 디렉토리 |
| `ls --color=auto` | **실행 명령어 (무엇을)** |

### snoopy vs `~/.bash_history` — 왜 history로는 부족한가

| | history | snoopy |
|---|---|---|
| 삭제 | 사용자가 지움(`history -c`) | **못 지움**(커널 레벨 개입 불가) |
| 시점 | 세션 종료 시에만 저장 | **실시간** |
| 범위 | bash 명령만, 쉘 바꾸면 누락 | **모든 execve()**(쉘 무관, 스크립트 내부도) |
| sudo | 전환 후 다른 history | **sudo 전후 전부** |

### ⚠️ 이번 삽질 (로그가 안 쌓임)

```
1. ld.so.preload 경로가 심볼릭 링크 경로(/lib/... vs 실제 /usr/lib/...) → 내부 경로 불일치로 로깅 실패
2. 설정 변경 후 기존 세션에서 테스트 → 새 프로세스에만 적용되는데 기존 세션이라 안 보임
해결: ld.so.preload에 실제 경로(/usr/lib/...) 기재 + 새 SSH 세션에서 검증
```
- 로그 목적지는 `/etc/snoopy.ini`의 `output = syslog:LOG_AUTHPRIV` → auth.log로 가게 함(facility 분류는 10절).

### snoopy vs auditd

| | snoopy | auditd |
|---|---|---|
| 원리 | LD_PRELOAD execve 래핑 | 커널 audit 서브시스템 |
| 우회 | LD_PRELOAD 무력화 가능 | 커널 레벨, 우회 매우 어려움 |
| 정보량 | 명령어 위주 | 파일접근/네트워크/시스템콜 전부 |
| 성능 | 낮음 | 설정 따라 높을 수 있음 |

> 정리: **snoopy = 가볍고 빠른 명령어 감사 / auditd = 무겁지만 완전한 감사.** 실무는 **둘을 같이** — snoopy로 명령어, auditd로 파일/시스템 이벤트.
> 🔗 Zabbix 연동(랩 확장): `UserParameter=cmd.suspicious,journalctl | grep snoopy | grep -cE "rm -rf|wget|curl|nc |nmap"` → 의심 명령 실행 시 알람 = **침투 후 행위를 실시간 탐지**.

## 10. 로그 감사 ② journald vs rsyslog — 로그가 어디에 쌓이나

snoopy 로그가 auth.log로 가게 하면서 "리눅스 로그가 대체 어디서 어떻게 분류되나"를 정리.

```
전통 방식: syslog 프로토콜 → rsyslog/syslog-ng 외부 데몬이 처리 → 텍스트 파일
현대 방식(Ubuntu 24.04 기본): systemd-journald가 처리 → 바이너리 DB → journalctl로 조회
```

### systemd-journald = systemd 내장 로그 수집 전담 데몬

systemd는 단일 프로그램이 아니라 데몬 묶음(`systemd-journald` 로그 / `-networkd` 네트워크 / `-resolved` DNS / `-timesyncd` NTP / `-logind` 세션…). journald는 그중 **로그 전담**이라 설치 없이 기본 내장.
- 모든 서비스+커널 메시지 수집, `/dev/log` 소켓 수신, **바이너리 DB**(`/run/log/journal/`) 저장, 재부팅 시 사라짐(기본값), **`journalctl`로만 조회**.
- 자주: `journalctl -f`(실시간) / `-u ssh`(서비스) / `--since "1h ago"` / `-p err`(심각도).

### rsyslog = journald 로그를 텍스트 파일로 분류 저장하는 **독립 실행 데몬**

- **라이브러리가 아니라 서비스**(`systemctl status rsyslog`). journald가 `/dev/log`로 공유한 로그를 받아 **facility 기준 분류** → 텍스트 저장.
- **facility(출처) → 파일** 매핑:
  ```
  /var/log/auth.log : auth, authpriv (인증/인가)
  /var/log/syslog   : 나머지 전부 (auth/authpriv 제외)
  /var/log/kern.log : 커널
  ```
  규칙(`/etc/rsyslog.d/50-default.conf`): `auth,authpriv.* → auth.log` / `*.*;auth,authpriv.none → syslog`.
  → **syslog = 전체에서 auth 빼고 / auth.log = auth·authpriv만.** (그래서 snoopy를 `LOG_AUTHPRIV`로 보내면 auth.log로 간다)

### 관계 = 경쟁이 아니라 협력

```
모든 로그 발생 → journald 수신·바이너리 저장 → /dev/log 소켓으로 rsyslog 공유
            → rsyslog가 facility 분류 → 텍스트 파일(auth.log, syslog…)
```
| | journald(바이너리) | rsyslog(텍스트) |
|---|---|---|
| 조회 | journalctl만 | cat/grep/tail |
| 검색 | 빠름(인덱싱) | 느림(전체 스캔) |
| 외부 전송 | 어려움 | **쉬움** |
| 변조 | 어려움 | 쉽게 편집 |

- rsyslog가 없으면: grep/tail 불가, **Zabbix UserParameter로 읽기 불편, 외부 SIEM(Security Onion/Splunk) 전송 어려움.** → 실무 표준은 **둘 다** (바이너리=빠른 조회 + 텍스트=외부 전송).

> 🔗 **SIEM 연동 관점:** 가장 흔한 방식이 **rsyslog forwarding**(텍스트를 TCP/UDP로 SIEM 전송). 그 외 systemd-journal-remote, Filebeat/Logstash→Elasticsearch. 즉 오늘 정리한 로그가 결국 Security Onion/ESM으로 흘러가는 입구가 rsyslog다.

## 11. AD (Active Directory) — 윈도우 진영의 중앙 계정·권한·정책 플랫폼

> **AD = Microsoft가 만든 중앙 계정/권한/정책 통합 관리 플랫폼.** "누가 / 어떤 시스템에 / 어떤 권한으로 접근하고 / 어떤 규칙을 따르나"를 중앙에서 관리.

핵심 역할 3가지: **신원확인(Authentication, Kerberos 티켓)** · **권한부여(Authorization)** · **정책강제(Policy, GPO)**.

### 구성 & 도메인

- **AD = 소프트웨어(AD DS 역할)** / **DC(도메인 컨트롤러) = AD가 설치된 Windows Server.**
  `Windows Server → AD DS 역할 추가 → 도메인 생성(company.local) → 이 서버가 DC`.
- **AD 도메인 ≠ DNS 도메인** (이름만 같고 다른 개념): DNS 도메인=건물 주소(찾아가는 용도) / AD 도메인=회사 이름(어느 조직 소속인지). 단 AD가 내부 이름해석에 DNS를 **도구로** 씀.
- **도메인 가입(Domain Join):** 서버/PC가 DC 관리 아래로 들어옴. 가입 전엔 서버 10대=계정 10개, 가입 후엔 **AD 계정 하나로 전체 접근**(가입 시 DC 관리자 계정 필요 — 아무나 못 함).

### Kerberos 인증 흐름

```
① PC 로그인 → ② DC(KDC)에 인증요청 → ③ DC가 TGT(Ticket Granting Ticket) 발급
→ ④ 파일서버 접근 시 TGT 들고 서비스 티켓 요청 → ⑤ DC가 서비스 티켓 발급 → ⑥ 서버에 제출 → 접근 허용
```
- 핵심: **비밀번호를 서버마다 보내는 게 아니라 티켓으로 인증.** 비번은 DC에만 한 번.

### 관리 오브젝트 & GPO

- Users / Computers / Groups / **OU(조직 단위 — 조직 구조대로 분류, OU별 다른 정책)**.
- **GPO(그룹 정책)** = DC가 도메인 전체에 정책 배포: USB 금지, 프로그램 설치 제한, 비번 복잡도, 소프트웨어 자동배포 등. **GPO 한 번 수정 → 전체 자동 적용.**

### 적용 범위 & 비교

- **내부망 한정** — DC에 네트워크로 도달 가능한 범위 안에서만. 재택은 **VPN으로 내부망 연결해야** 도메인 로그인. 클라우드판 = **Azure AD(Entra ID)**(별도 제품).
- **AD vs DLP:** DLP=데이터 유출 방지 전문 / AD=계정·인증·권한·정책 전체 인프라 플랫폼. **DLP는 AD가 할 수 있는 일(GPO) 중 일부**, AD가 더 큰 개념.
- **AD가 없으면:** 500명·서버 50대에서 입퇴사마다 서버 50대 돌며 계정 생성/삭제(하나 빠뜨리면 보안구멍). AD면 **계정 하나 비활성화로 모든 접근 즉시 차단**.

### ⚠️ 보안 관점 (공격 시나리오) → Attack-Defense.md로

- **Pass-the-Hash:** 해시 탈취 → 도메인 내 다른 서버로 횡이동.
- **Golden Ticket:** DC의 `krbtgt` 계정 해시 탈취 → **위조 Kerberos 티켓**으로 도메인 전체 무제한 접근. → **DC가 공격자의 최종 목표인 이유**(DC 하나 뚫리면 도메인 전체가 넘어감).

> ❓ (아직 애매) **포레스트(Forest)** = 여러 AD 도메인을 묶은 최상위 단위. "naver.com이 메일 도메인을 AD에 등록한다?" 같은 건 DNS 도메인과 AD 도메인을 섞어 생각해서 생긴 혼동 — 둘은 별개라는 것만 분명히. 포레스트/트러스트 관계는 다음에 더 팔 것.

## 12. 윈도우 로그 — 이벤트 뷰어 & 로컬 보안 정책

- **이벤트 뷰어:** 로그마다 **고유 이벤트 ID**가 있어서, 같은 종류(예: 같은 오류)는 같은 ID로 뜬다 → **ID로 필터**해서 본다. (리눅스 facility/journalctl -u 로 거르는 것의 윈도우판)
- `services.msc` : 실행창에 치면 서비스 관리 진입.
- **로컬 보안 정책 → 감사 정책(Audit Policy)** 변경 후 적용 → 그 시점부터 해당 이벤트가 보안 로그에 기록(덮어쓰기되며 파일에 저장).
- 💡 정리: **윈도우도 결국 "어떤 보안 로그를 남길지 정의(감사 정책)"하고 "ID로 걸러 본다(이벤트 뷰어)"** — 리눅스에서 한 로그 정의·분류·필터링과 같은 사고방식.

## 13. 핵심 한 줄 정리

- **UserParameter = 키↔쉘명령 매핑** — 키 호출 시 명령 실행, 표준출력 1줄이 값.
- **권한은 데몬(zabbix 유저) 기준** — `-t`(root) 결과에 속지 말 것.
- **History 저장 없으면 트리거 비교 불가** (특히 문자형 FIM).
- **보안 알람은 자동해제 금지**(OK None + manual close)로 누락 방지.
- **SYN flood의 본질은 CPU가 아니라 backlog 고갈** → SYN Cookie.
- **FIM=상태(지금 정상인가) / auditd·Wazuh=이벤트(누가 변조했나).**
- **snoopy=명령어 감사(LD_PRELOAD execve 래핑), history보다 신뢰성↑, 못 지움.**
- **journald=수집(바이너리) / rsyslog=분류·텍스트 저장 → SIEM 전송 입구.**
- **AD=계정·권한·정책 중앙관리(Kerberos·GPO), 내부망 한정, DC가 최종 표적.**
