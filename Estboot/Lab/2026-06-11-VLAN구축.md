# 2026-06-11 — VLAN으로 웹서버 망 가르기 (ESXi VGT 4095 트렁크 · pfSense VLAN 라우팅)

과정2 8일차. 6/10에 잡은 LAN(192.168.2.0/24) 위에서 **드디어 서버를 올리기 시작**한 날. 단순히 VM 하나 더 띄운 게 아니라, **VLAN으로 "운영 웹(Web)"과 "개발 웹(Web_Dev)"을 다른 대역으로 갈라** 놓은 게 핵심. ESXi 포트그룹의 **VLAN ID 4095(트렁크)** 를 처음 만져봤고, pfSense가 VLAN 게이트웨이(10.10.10.1 / 10.10.20.1)로 직접 라우팅하게 세웠다.

> FIMS 연결: 과정1 FIMS는 VirtualBox **내부 네트워크 이름**으로만 망을 갈랐다(소프트웨어 세그멘테이션). 오늘은 **802.1Q VLAN 태그**로 갈랐다 — 스위치(여기선 ESXi vSwitch)가 태그로 나누고, 다른 VLAN끼리는 **반드시 L3(pfSense)를 경유**해야 통신된다. "이름으로 격리"에서 "태그로 격리 + 라우터 경유"로 한 칸 올라온 셈.

## 0️⃣ 시작 전 — 보안담당자 습관 두 가지

### 오픈소스 보안 솔루션은 pfSense만 있는 게 아니다

```
pfSense   → BSD 기반 오픈소스 방화벽/UTM (지금 쓰는 것)
OPNsense  → pfSense에서 갈라져 나온(fork) 오픈소스 방화벽 — UI·업데이트가 더 활발
PacketFence → 오픈소스 NAC(네트워크 접근통제). 인증 안 된 단말 격리·차단
ClamAV    → 오픈소스 백신 엔진(안티바이러스)
```

> 💡 "방화벽=상용(FortiGate/Palo Alto)" 만 떠올리기 쉬운데, **오픈소스로도 방화벽(pfSense/OPNsense)·NAC(PacketFence)·백신(ClamAV)** 한 스택을 다 꾸릴 수 있다. 스타트업이 돈 없이 UTM을 세울 때 쓰는 조합. (⚠️ 수업 음차 메모의 "ovnsense=OPNsense", "clamvnet"은 문맥상 ClamAV로 정정 — 강사 자료로 한 번 더 확인할 것)

### 다운로드는 검증된 사이트 + 해시(MD5/SHA) 대조

> 비유: 택배 받을 때 **송장 번호(공식 사이트가 게시한 해시)** 와 **실제 박스에 붙은 번호(내가 받은 파일의 해시)** 가 같은지 맞춰보는 것. 다르면 운송 중 누가 바꿔치기한 것.

- 응용프로그램(ISO·설치 파일)은 **공식/검증된 사이트**에서 받고, 받은 뒤 **해시값을 게시된 값과 대조**하는 게 보안담당자의 기본 습관.
- 노린 건 **무결성(Integrity) 검증** — 다운로드 중 변조·중간자 공격으로 바꿔치기된 파일을 거르는 것.

```powershell
# Windows
Get-FileHash .\pfSense.iso -Algorithm SHA256
```
```bash
# Linux
sha256sum pfSense.iso
md5sum pfSense.iso        # 공식이 MD5만 주면 MD5로 대조 (MD5는 충돌 위험 있어 SHA256 권장)
```

> ⚠️ MD5는 충돌(collision)이 발견된 알고리즘이라 보안 용도로는 **SHA-256 이상**을 쓴다. 공식이 MD5만 제공하면 어쩔 수 없이 쓰되, 선택지가 있으면 SHA256.

## 1️⃣ "패킷이 안 가요" 트러블슈팅 순서 — pfSense 쉘 ping

서버를 추가했는데 패킷이 안 오갈 때, 무턱대고 방화벽 룰부터 뒤지지 말고 **아래에서 위로** 좁혀간다.

```
1. 내부 통신은 되나?           (같은 망 안에서는 오가나)
2. ping을 날려 패킷이 가나 확인 (L3 도달성부터)
3. pfSense 콘솔 8번(Shell)에서 직접 ping  ← 방화벽 자신이 그 대상에 닿나
```

> 💡 **왜 8번(Shell)에서 ping을 치나** — 클라이언트에서 안 될 때, *방화벽 자신*이 대상에 닿는지를 봐야 "경로(라우팅) 문제"인지 "방화벽 룰 문제"인지 갈린다. pfSense가 ping은 되는데 클라이언트는 안 되면 → 룰/포트그룹 문제, pfSense조차 안 되면 → 라우팅/대역 문제.

> 비유: 식당에서 음식이 안 나올 때, 손님 자리에서만 따지지 말고 **주방(pfSense)에서 재료가 들어오긴 하나**부터 확인하는 것. 주방에도 재료가 없으면 배송(라우팅) 문제, 주방엔 있는데 손님 테이블엔 안 오면 서빙(룰) 문제.

### SSH 원격 — 콘솔 14번 활성화

```
pfSense 콘솔 메뉴 14) Enable Secure Shell (sshd)
→ 원격에서 SSH 세션으로 pfSense 접속 가능
```

> 💡 콘솔(화면) 앞에 매번 앉을 수 없으니, 14번으로 sshd를 켜서 **원격 세션**으로 관리한다. (FIMS에서 서버마다 콘솔 대신 SSH로 붙던 것과 같은 결)

## 2️⃣ Full Duplex vs Half Duplex — 1기가 "양방향"의 의미

```
Full Duplex (전이중)  → 송신·수신을 동시에. "1Gbps 양방향" = 보내면서 받는다
Half Duplex (반이중)  → 한 번에 한 방향만 (무전기처럼 번갈아)
```

> 비유: Full = **왕복 2차선 도로**(오는 차·가는 차 동시), Half = **외길 1차선**(한쪽이 지나갈 동안 반대쪽은 대기). 옛날 허브는 Half, 요즘 스위치 포트는 Full.

> 💡 "1기가 통신 쓴다"는 보통 **Full Duplex 1Gbps**를 말한다 — 다운로드 1G, 업로드 1G를 동시에 깔 수 있다는 뜻.

## 3️⃣ pfSense 트래픽 그래프 읽기 — WAN 파랑 / LAN 빨강 ⭐

pfSense 대시보드의 트래픽 그래프(`Status > Traffic Graph`)는 **인터페이스별로 색이 갈린다.** 이걸 읽을 줄 알면 "허용됐나/차단됐나"를 그래프만 봐도 안다.

```
WAN으로 100Mbps 인입       → 화면상 WAN(파랑) 그래프 상승
└ 허용 정책에 걸려 내부로 감 → LAN(빨강) 그래프도 같이 상승   (통과 = 두 그래프 동반 상승)
└ 차단 정책에 걸림          → LAN(빨강) 그래프는 안 오름        (막힘 = 파랑만 오르고 빨강 잠잠)
```

> 💡 **그래프로 룰 동작을 검증한다** — "WAN엔 들어왔는데(파랑↑) LAN으로 안 나갔다(빨강 그대로)"면 방화벽이 그 트래픽을 **막은 것**. 룰이 의도대로 동작하는지 패킷 캡처 없이 1차로 확인하는 방법. (TAP/스팬으로 복제떠서 보던 가시성 개념의 가벼운 버전 — `Security-Solutions.md` IDS 미러링과 연결)

> 비유: 건물 정문 검문소(WAN)에 사람이 100명 몰려왔는데(파랑↑), 통과시켜 로비(LAN)로 들어가면 로비 인원도 늘고(빨강↑), 정문에서 막으면 로비는 그대로(빨강 평평).

## 4️⃣ VLAN — 왜 가르나 & 오늘의 목표

```
VLAN(Virtual LAN) = 물리 스위치 하나를 논리적으로 쪼갠 가상 네트워크 (802.1Q 태그 기반)

왜 만드나?
→ 서버들의 대역을 "가상으로" 나눠 대역(망)을 따로 관리하기 위해
→ 운영/개발/관리를 같은 물리 스위치 위에서도 격리

오늘 목표
→ 웹서버 1대를 올리되, VLAN으로 "운영 Web"과 "개발 Web_Dev"를 다른 대역으로 분리
```

> 비유: 한 층(물리 스위치)을 **칸막이(VLAN 태그)** 로 나눠 운영팀 방·개발팀 방을 만드는 것. 벽을 새로 안 쌓아도(케이블 추가 X) 논리적으로 방이 갈린다. 단, **다른 방끼리 가려면 복도(pfSense=L3 라우터)** 를 반드시 거쳐야 한다.

> 🔗 연결: `OSI-7Layer.md`의 "세그멘테이션 vs VLAN" — FIMS는 세그멘테이션(이름 기반 소프트웨어 격리), 오늘은 VLAN(태그 기반). 같은 "망 가르기"지만 격리 레이어가 다르다.

## 5️⃣ ESXi에서 VLAN 세우기 — VGT(4095) 트렁크가 핵심 ⭐⭐

오늘 제일 중요한 개념. ESXi에서 VLAN을 다루는 방식은 포트그룹의 **VLAN ID 값**으로 결정된다.

| VLAN ID | 모드 | 누가 태그를 다루나 | 쓰임 |
|---|---|---|---|
| 0 | EST | 외부 물리 스위치 | (외부 스위치가 태깅) |
| 1~4094 | **VST** | ESXi vSwitch가 태깅/언태깅 | 포트그룹마다 고정 VLAN 부여 |
| **4095** | **VGT (트렁크)** | **게스트(pfSense)가 직접** | 모든 VLAN 태그를 게스트로 통과시킴 |

### 오늘 한 것

```
1. (ESXi) 인터페이스/포트그룹 추가 → 활성화 → 편집
2. VLAN 포트그룹 추가
3. ★ 기존 LAN 포트그룹의 VLAN ID = 4095 로 할당   ← 꼭 기억!
   (LAN으로 들어오는 모든 VLAN 태그를 pfSense가 받아 처리하도록 "트렁크"로 개방)
4. (pfSense) VLAN 10 / VLAN 20 인터페이스 생성 → 게이트웨이 10.10.10.1 / 10.10.20.1
5. 웹서버 Ubuntu에 VLAN 포트에 맞는 정보값(IP/대역) 설정
```

> 🧠 **왜 4095인가** — pfSense가 "VLAN 라우터" 노릇을 하려면, LAN으로 들어오는 트래픽의 **VLAN 태그가 지워지지 않고 그대로** pfSense까지 와야 한다. VLAN ID 4095(=VGT, Virtual Guest Tagging)는 **"태그를 건드리지 말고 게스트한테 다 넘겨"** 라는 뜻. 그래서 pfSense가 10번 태그·20번 태그를 직접 보고 VLAN 10/20으로 라우팅할 수 있다.

> ⚠️ **삽질 포인트** — 만약 LAN 포트그룹을 특정 VLAN ID(예: 10)로 고정(VST)해버리면, ESXi vSwitch가 태그를 벗겨서 pfSense는 VLAN 구분을 못 한다 → VLAN 라우팅 불가. **트렁크가 필요한 인터페이스는 무조건 4095.** "모두를 포함해야 하니 4095"가 이 뜻.

> 비유: 4095 = **우편물 분류를 안 하고 통째로 넘기는 우편 트럭**. 동(VLAN)별로 미리 안 갈라주고, 우체국(pfSense)이 받아서 직접 동별로 분류·배달. 반대로 VST(고정 VLAN)는 **트럭이 한 동 우편만 싣는 것** — 그 동만 다룰 수 있다.

## 6️⃣ 결과 — VLAN 10(Web) / VLAN 20(Web_Dev)

```
VLAN 10 대역 → Web 서버      (G/W 10.10.10.1) — 운영 환경
VLAN 20 대역 → Web_Dev 서버  (G/W 10.10.20.1) — 개발 환경
```

> 💡 **운영/개발을 VLAN으로 가르는 이유** — 개발 중 사고(설정 실수·악성코드 테스트)가 운영 웹에 번지지 않도록 **장애·침해 전파를 끊는다.** 실무에서 운영망/개발망 분리는 기본. 다른 VLAN끼리는 pfSense를 거치므로 그 길목에 **방화벽 룰로 통제**까지 걸 수 있다.

### 편의 세팅 — 호스트 크롬으로 바로 작업

```
1. 웹서버에 HTTP 8080 포트 오픈
2. pfSense WAN 방화벽 룰: source any → destination WAN subnet 허용
3. → xubuntu의 무거운 X 데스크탑은 끄고, 호스트 PC 크롬으로 바로 접속해 작업
```

> ⚠️ `source any` 허용은 **랩이니까** 편의로 연 것. 실무에선 출처를 관리 IP로 좁혀야 한다(any inbound는 곧 공격 표면). 6/10에 정리한 "VPN/인터페이스가 곧 공격 표면" 원칙과 같은 맥락.

## 7️⃣ 웹서버 초기 세팅 — 최소설치 후 필수 유틸

Ubuntu Server를 **최소설치(minimal)** 로 깔아서 도구가 거의 없다. 자주 쓰는 것부터 채웠다.

```bash
apt install net-tools        # ifconfig 등 네트워크 툴 (구식이지만 손에 익음)
apt install vim-tiny         # vi 편집기 (경량판)
apt install iputils-ping     # ping
apt install telnet           # 포트 열렸나 확인 (telnet IP PORT)
apt install dnsutils         # nslookup / dig — DNS 조회
apt install unzip            # 압축 해제
apt install bmon             # 네트워크 트래픽 실시간 현황 (인터페이스별 대역)
apt install nmon             # 서버 자원(CPU/메모리/디스크/네트워크) 종합 현황
apt install htop             # 프로세스·CPU 코어별 사용률 (top 강화판)
apt install openssh-server   # SSH 서버 — 원격 접속 개방
```

> 💡 **이 조합의 의도** — `net-tools/iputils-ping/telnet/dnsutils`는 **"통신이 되나" 진단용**(1️⃣ 트러블슈팅과 직결), `bmon/nmon/htop`은 **"서버가 버티나" 모니터링용**, `openssh-server`는 원격 관리용. 새 리눅스 서버 올리면 거의 항상 깔고 시작하는 세트.

> ⚠️ `telnet`은 원격 접속용이 아니라 여기선 **포트 도달성 확인용**(`telnet 10.10.10.1 8080`처럼). 실제 원격 셸은 SSH. telnet으로 로그인하는 습관은 평문이라 금물.

## ⚠️ 오늘 막혔던/짚어둘 것

- **VLAN ID 4095의 정체** — "왜 LAN 포트그룹만 4095?"에서 한참 멈칫. 정리: 4095=VGT=트렁크. pfSense가 VLAN 라우터가 되려면 태그가 안 지워진 채 들어와야 해서, 그 인터페이스만 "모든 VLAN 통과(4095)"로 연다.
- **트러블슈팅은 아래에서 위로** — 패킷 안 가면 룰부터 의심하지 말고 `내부통신 → ping → pfSense 8번 쉘 ping` 순으로 좁힌다. 방화벽 자신이 닿는지가 라우팅/룰 갈림길.
- **`any` inbound는 랩 한정** — 호스트 크롬 편의로 WAN any 허용을 열었지만, 실무에선 출처를 좁혀야 한다.
- **MD5 < SHA256** — 다운로드 무결성 검증 습관은 좋지만, MD5는 충돌 위험이라 가능하면 SHA256으로.
