## 1️⃣ 단편화(MF)를 ICMP로 강제로 보기 — `ping -l 3500`

TCP 캡처에선 MF 켜진 큰 조각이 잘 안 보였다. TCP는 연결 시 **MSS를 협상**해 MTU(1500)에 맞춰 미리 쪼개 보내기 때문. 그래서 **협상 없이 페이로드를 그대로 싣는 ICMP**로 MTU보다 큰 패킷을 쐈다.

```bash
ping 192.168.1.1 -l 3500     # -l = 보낼 페이로드 크기(바이트). 1500(MTU) 초과 → 강제 단편화
```

```
관찰
 → 한 ping이 1480바이트 조각 여러 개로 쪼개짐
 → 앞 조각 MF=1(뒤에 더 있음) / 마지막 조각 MF=0(끝)
 → Fragment Offset 0 → 1480 → … 으로 증가, 마지막은 남은 길이만 → Total Length 뚝 떨어짐
```

- 통신 중간 라우터들은 각자 MTU가 달라 **경로상 가장 작은 MTU(Path MTU)** 에 맞춰 합의하고 보낸다 → 단편화가 일어나는 또 다른 지점.
- 이론·정정(Identification≠MTU, MF=0이 마지막, 단편화 DoS)은 **OSI-7Layer.md "MTU와 단편화 / 단편화를 눈으로 보려면 ICMP"**.

## 2️⃣ TCP 3-way handshake 캡처 — 상태·시퀀스·플래그·윈도우

서버는 데몬이 떠서 **LISTEN** 상태여야 요청을 받는다. 그 상태에서 3-way가 선행돼야 연결이 선다.

```
클라이언트 SYN 전송      → 클라 SYN_SENT  / 서버는 LISTEN에서 받음
서버 SYN+ACK            → 서버 SYN_RECEIVED
클라이언트 ACK          → 양쪽 ESTABLISHED (연결 완성)
```

와이어샤크 `Statistics → Flow Graph`로 위 흐름이 깔끔히 맺힌 걸 확인.

```
시퀀스/ACK 넘버 (와이어샤크 상대값)
 SYN      Seq=0
 SYN+ACK  Seq=0, ACK=1   ← 받은 시퀀스 +1
 ACK      Seq=1, ACK=1
 = 양쪽이 서로의 시작번호를 맞추는 작업 (이후 데이터 순서·유실 판단의 기준)

플래그   SYN/ACK/FIN/RST/PSH/URG  (TCP 헤더에서 켜짐/꺼짐 확인)
윈도우   Window Size = 한 번에 받아둘 버퍼 크기 → 서로 맞춘 표시 (흐름 제어)
```

- 필터링이 기본기: `ip.addr == 172.30.1.32 && tcp.port == 80` 처럼 걸어 수천 줄 중 원하는 세션만 본다.
- 종료는 **4-way**(FIN→ACK→FIN→ACK), 클라이언트는 마지막에 **TIME_WAIT 1~2분** 대기. 이론은 OSI-7Layer.md "3/4-way, 시퀀스·ACK 넘버, 윈도우".

## 3️⃣ UDP 캡처 — 헤더 4칸 + 닫힌 포트는 ICMP로 답

```
UDP 헤더 = 출발포트 / 목적포트 / Length / Checksum — 끝. (시퀀스·ACK·세션 없음)
 → 빠르지만 유실되면 그냥 끝. 재전송 없음.
```

닫힌 UDP 포트로 보냈더니 **ICMP "Destination Unreachable (Port Unreachable)"** 가 돌아오는 걸 캡처로 확인 — UDP는 핸드셰이크가 없어 "열림"은 모호하고 "닫힘"만 ICMP로 확실히 잡힌다.

- 곁다리 정리: **QUIC**(구글, UDP 위 TLS+연결관리 → HTTP/3), **DCCP**(UDP에 혼잡제어 얹음) 같은 신종 프로토콜도 있음. (OSI-7Layer.md "UDP 상세")

## 4️⃣ nmap 포트스캔 — `-sT`(Connect) vs `-sS`(Stealth) 와이어샤크 대조

6/19엔 "열린 포워딩 포트만 뜬다"만 봤다. 오늘은 **스캔 방식별 패킷 반응**을 갈라봤다.

```bash
nmap -sT 192.168.1.100 -p 1-65535     # TCP Connect: 3-way 끝까지 맺음 → 세션/로그 남음
nmap -sS 192.168.1.100 -p 1-65535     # SYN(Stealth): SYN→SYN+ACK까지만, ACK 대신 RST로 끊음 (root 필요)
```

와이어샤크에서 본 세 가지 반응:

```
열린 포트 → 서버 SYN+ACK 응답   (살아있음)
닫힌 포트 → 서버 RST 응답       ("그런 서비스 없음")
필터됨    → 무응답              (방화벽 drop)
```

**스텔스 스캔의 핵심**: 해커가 ACK까지 보내 ESTABLISHED가 되면 자기 IP가 세션·로그에 박힌다. 그래서 SYN+ACK(열림 확인)를 받는 순간 **ACK 대신 RST로 끊어** "열렸다"는 정보만 챙기고 흔적은 안 남긴다.

> 손잡이만 살짝 돌려 열린 걸 확인하고 바로 손 떼는 것 — 안에 들어가 앉으면(ESTABLISHED) CCTV에 찍히니까. 이론 표/비유는 Attack-Defense.md "포트 스캔(nmap)".
