# 2026-06-19 — Kali 포트스캔(nmap) + Hydra 사전공격 + IPS 인라인에 nmap 투입 → ESM 필요성 체감

과정2 13일차. 6/16에 토폴로지에 박아둔 **공격기 Kali**를 오늘 처음 제대로 휘둘렀다. ① 웹서버 **포트스캔(nmap)**, ② **Hydra 사전공격**으로 SSH 비밀번호 무차별 대입, ③ **IPS를 WAN 인라인 + 전체 룰 ON** 상태에서 nmap을 쏴 알람이 쏟아지는 걸 직접 보고 → **"이래서 ESM이 필요하다"** 를 몸으로 이해했다. 오후엔 와이어샤크로 **IP 헤더를 필드 단위로 뜯어봤다**(이론은 OSI-7Layer.md).

> 🔗 어제까지와 연결: 6/15~18에 **막는 쪽(Snort 인라인 IPS)**과 **보는 쪽(Security Onion)**을 다 세웠다. 오늘은 드디어 **때리는 쪽**을 가동해, 세워둔 방어/관제가 실제로 반응하는지 검증한 날. 6/16에 "static 250으로 박은 공격기"가 오늘 일을 했다.

> FIMS 연결: 과정1 FIMS에선 slowapi로 **로그인 5회 실패 차단(Rate Limiting)**을 구현했었다. 오늘 Hydra로 그 "무차별 대입하는 공격자" 입장을 직접 해본 셈 — 방어로직을 짤 때 상상만 하던 공격을 실제 도구로 재현.

## 🗺️ 오늘의 구도 — 공격기 ↔ 방어/관제

```
   Kali(공격기) 192.168.1.250        ← nmap 포트스캔 · Hydra 사전공격 발사
        │  (WAN/외부 포지션)
        ▼
   pfSense WAN 192.168.1.82 ── Snort(WAN) : 인라인 IPS + 전체 룰 ON
        │
   NAT 포트포워딩으로 열린 포트 ───▶ Web 서버(VLAN10 10.10.10.10) SSH·웹
        │
   [알람 폭주] ──▶ 한곳에서 모아 봐야 한다 = ESM 필요성
```

- 포트스캔이 걸리는 전제: **포트포워딩(DNAT)으로 외부에 연 포트**가 곧 스캔에 잡히는 공격 표면이다. (6/12에 박은 WAN 80/1010/2020… 포워딩이 그대로 스캔 대상)

## 1️⃣ 웹서버 포트스캔 (nmap)

Kali에서 웹서버를 향해 nmap 포트스캔.

```
관찰: 서버의 포트포워딩에서 "열린 포트"만 스캔에 걸려 올라온다
 → 닫힌 포트는 응답이 없거나 reset → 스캔 결과에 안 뜸
 → 열어둔 포트 = 외부에 노출한 공격 표면 그 자체
```

> 💡 그래서 "안 쓰는 아웃바운드/인바운드는 닫는다(최소권한)"가 곧 공격 표면 축소다. 6/15에 samba·dns의 80/443을 닫은 게 정확히 이 맥락 — 안 열면 스캔에도 안 걸린다.

## 2️⃣ Hydra 사전공격(Dictionary Attack)으로 SSH 무차별 대입

Kali에서 **Hydra**에 패스워드 목록(wordlist) 파일을 물려 웹서버 SSH 비밀번호를 무차별 대입.

```
사전공격 = 자주 쓰는 패스워드 목록을 차례로 대입 (순수 브루트포스의 현실적 버전)
 → Hydra가 wordlist를 한 줄씩 SSH 로그인에 시도
```

웹서버 쪽에서 본 결과:

```
서버 auth 로그에 "Failed password ..." 가 무더기로 쌓임
 → SSH 접근(로그인 시도) 자체가 로그로 다 남는다
```

⚠️ **핵심 통찰** — 서버 입장에서 **비밀번호 실패 로그를 만드는 주체는 둘뿐**이다: ① 오타 낸 진짜 관리자, ② 크리덴셜을 노린 공격자. 그래서 **실패 로그가 급증 = 거의 무조건 공격**으로 본다. (단발 오타는 묻혀도, 무더기 실패는 패턴이 명확)

> 💡 음차 정정: 수업노트의 *"디렉터리 공격"* → **딕셔너리(dictionary) 공격**의 오기. 디렉터리를 터는 게 아니라 패스워드 **사전(wordlist)** 을 대입하는 것. (이론: Attack-Defense.md "Hydra 사전공격")

## 3️⃣ IPS 인라인 + 전체 룰 ON 상태에서 nmap → 알람 폭주 → ESM 필요성

오늘의 하이라이트. **IPS를 WAN에 인라인으로 걸고 모든 룰을 켠 뒤** Kali에서 nmap을 한 번 쏘자:

```
→ 수많은 알람이 한꺼번에 올라옴
→ "어디서 / 어떻게 접근했는지"가 룰에 의해 줄줄이 alert로 떴다
```

장비 **한 대(IPS)**에서만도 이 정도다. 방화벽·IDS·DLP·EDR·프록시가 각자 로그를 뱉으면 사람이 도저히 못 따라간다. → **그래서 ESM(Enterprise Security Management)** 이 필요하다.

```
방화벽   IPS   IDS   프록시   EDR   서버 syslog
   └──────┴─────┴──────┴──────┴────────┘
                    ↓
        ESM : 로그를 한곳에 모아 수집·관리·상관분석
```

> 🔗 이게 6/17~18에 올린 **Security Onion**의 발상(여러 센서 로그를 Elastic+SOC 콘솔 한곳에서) — ESM은 그걸 전사 규모로 확장한 상용 관제 플랫폼. (이론: Security-Solutions.md "보안관제와 ESM")

> 💡 단일 장비 로그론 "브루트포스"인지 모른다. **방화벽 세션 급증 + 웹서버 인증실패 급증을 같이 봐야** 공격으로 판단된다 — ESM의 존재 이유가 바로 이 상관분석.

## 4️⃣ 와이어샤크로 IP 헤더 뜯어보기 (오후)

`ipip 하던 게 사실 이 프로토콜이었구나` — 패킷 L3 헤더 안에 DSCP(우선순위)·ECN(혼잡)·Total Length·Identification·MF/Fragment Offset(단편화)·TTL·Protocol·Checksum·출발/목적지가 다 들어있다는 걸 필드 단위로 확인.

- 상세 이론·정정(Identification≠MTU, TTL=홉 카운트, 단편화 DoS)은 **OSI-7Layer.md "IPv4 헤더 분석"** 으로 정리. 여기선 "와이어샤크에서 실제로 펼쳐 봤다"는 실습 사실만 기록.
