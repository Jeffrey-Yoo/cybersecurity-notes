# 2026-06-15 — 웹 프록시(Squid/SquidGuard)로 아웃바운드 통제 + Snort IDS 탐지 실험

과정2 10일차. 6/12까지 **"망을 가르고(VLAN) 그 사이 길목을 방화벽으로 통제"**하는 데까지 왔다. 오늘은 그 길목 위에 **L7 통제 장비 두 개**를 얹은 날 —

- **Squid/SquidGuard (웹 프록시)** = 서버가 인터넷으로 **나갈 때(아웃바운드)** 어디로 가는지 URL 단위로 거른다.
- **Snort (IDS)** = 트래픽을 보며 **공격 패턴이 보이면 탐지·로깅**한다.

즉 "들어오는 것(인바운드 NAT·방화벽)"만 보던 데서, **나가는 것(아웃바운드 프록시)**과 **흐르는 것(IDS 가시성)**까지 손을 댄 날이다.

> FIMS 연결: 과정1 FIMS에선 분석 인프라가 인터넷에 직접 나갈 일을 socat·WireGuard로 좁게 터줬을 뿐, "서버의 아웃바운드를 URL로 거르는 장비"는 없었다. 오늘 Squid가 한 일 = **아웃바운드 화이트리스트 통제**를 전용 장비로 박은 것. 그리고 FIMS의 Suricata가 했던 "트래픽 보고 탐지"를 오늘은 Snort로 다시 했는데, **fast.log가 비어서 헤맸던 그 삽질이 오늘 Snort에서 똑같이 재연**됐다(아래 6번).

> 🔗 연결: `Security-Solutions.md`의 ① 3세대 NGFW(L7 페이로드 검사)에 "실습 때 Squid 올려서 L7 제어 해볼 예정"이라 적어둔 게 바로 오늘, ② IDS/IPS·Snort 룰 문법, ③ WAF. 이론에서 깔아둔 걸 pfSense 위에 실제로 세웠다.

## 0️⃣ 워밍업 — 웹 프록시(포워드 프록시)를 왜 쓰나

웹서버가 인터넷으로 **그냥 직접** 나가면, 나가는 트래픽마다 인터넷망 방화벽에 정책을 박아야 해서 관리가 지옥이 된다. 그래서 서버 → **웹 프록시(악성 URL 차단)** → 방화벽(SNAT) → 인터넷 순으로 **프록시를 한 번 거치게** 만든다.

```
http://naver.com  →  웹 프록시 (악성 URL 차단)  →  방화벽(SNAT)  →  인터넷
                       ↑ 여기서 "어디로 나갈 수 있나"를 URL 단위로 통제
```

> 비유: 회사 정문(방화벽)은 "사람이 나가도 되냐"만 본다. 웹 프록시는 정문 앞에 세운 **외출 신고대** — "어느 가게(URL)에 가는데?"를 묻고, 블랙리스트에 있는 데면 못 나가게 막는다. 가게 목록을 신고대 한 곳에서만 관리하면 되니 정문 정책을 매번 안 고쳐도 된다.

### 보안적 측면에서 프록시가 필요한 이유 (3가지)

1. **악성 사이트 사전 차단** — 유해 URL로 나가는 걸 미리 막는다.
2. **서버의 무분별한 인터넷 접근 차단** → 감염 예방 + 보안정책 유지. (감염된 서버가 외부 C2로 나가려 해도 프록시에서 막힘)
3. **침해 대응 시 시야(가시성) 확보** — 프록시 로그에 "누가 어디로 나갔나"가 다 남아 사고 분석이 된다.

### 실무에서 프록시가 "현실적 타협안"인 이유

```
이상적:  서버마다 아웃바운드를 "필요한 것만 1:1로" 열어두기  ← 가장 안전하지만 손이 많이 감
현실:    소규모 회사는 개발자·서버는 많은데 보안담당이 적음
        → 서버별 아웃바운드 정책을 일일이 세울 여력이 없어 outbound=any로 열려 있는 경우가 흔함
타협안:  그 서버(들) 앞에 프록시를 한 대 태우고 거기서 차단정책을 건다
        → "일단 프록시로 묶어 거르자"고 건의해 부족한 보안을 중·장·단기로 보완
```

> ⚠️ 프록시는 **만능이 아니라 차선책**이다. 가장 좋은 건 여전히 "필요한 목적지만 1:1로 여는 것". 프록시는 그게 현실적으로 안 될 때 "아웃바운드를 한 곳에 모아 통제"하는 절충이다.

### 💡 음차 교정 — Squid의 상용 대안은?

수업 노트의 "웹스퀴드", "웹클라이드"를 교정:

- **Squid** = 대표적인 무료 오픈소스 **포워드/캐싱 프록시**. (pfSense 패키지로 제공)
- "웹클라이드 같은 제품" = 돈 주고 쓰는 **상용 시큐어 웹 게이트웨이(SWG)** 카테고리를 말한 것으로 보임. 원 발음 표기가 불분명해 제품명을 단정하긴 어렵지만, 이 카테고리의 실제 제품은 ↓
  - 해외: **Zscaler(클라우드 SWG)**, **Broadcom/Symantec Blue Coat ProxySG**, **Forcepoint**, **Skyhigh(구 McAfee) Web Gateway**
  - 국내 URL 필터링: **플랜티넷 WeBKeeper(웹키퍼)** 등
- → "무료=Squid / 유료=상용 SWG"의 구도로 이해하면 됨. (정확한 제품명은 수업 자료 재확인)

### 💡 "QA 설계와 OA 설계가 달라야 한다" — 둘이 뭐냐 (노트의 물음표 해소)

수업 중 흘린 말이라 표기가 애매했는데, 가장 자연스러운 해석은 **망(존)의 용도가 다르면 보안·프록시 정책도 달라야 한다**는 것:

```
OA(Office Automation)망  = 임직원 사무용 일반 업무망 (PC·메일·웹서핑 단말이 많음)
   → 사람이 인터넷·메일을 많이 쓰니 악성코드 유입 위험 ↑
   → URL 필터링/프록시·EDR·메일보안이 핵심. 아웃바운드를 넓게 쓰되 프록시로 거른다.

QA(Quality Assurance)망  = 품질검증/테스트 환경 (배포 전 운영과 유사하게 검증)
   → 운영과 비슷하게 격리하되 외부 노출은 최소. 단말 많은 OA와는 위협모델 자체가 다름
```

> 핵심: **"한 가지 보안 설계를 전 구간에 복붙하면 안 된다."** 사무망(OA)에 맞는 프록시·필터 정책과 테스트망(QA)/서버망에 맞는 정책은 위협이 달라서 달라야 한다. 오늘 web(운영)·webdev(개발)을 서로 다른 정책(화이트 vs 블랙리스트)으로 건 것도 같은 발상. (해석이라 수업 맥락 재확인 권장)

## 🦑 Squid — 포워드/캐싱 프록시 세우기

pfSense → **System / Package Manager**에서 `squid` 설치 → Services에 **Squid Proxy Server**가 자동 등록된 걸 확인.

### 설정 포인트

```
Local Cache  → 하드디스크 캐시 사이즈를 권고값 2000 MB로 상향
              (자주 받는 콘텐츠를 디스크에 캐싱 = 트래픽 절감 + 응답 빨라짐)

ACL → Squid Allowed Ports
  ACL SafePorts: 8080 8181 추가   ← 기본(21 70 80 210 280 443 488 563 591 631 …)에 더해 허용할 포트
```

- **왜 포트를 추가로 여나** — 포트는 바뀔 수 있다. 테스트 목적이나 여러 경로의 웹을 다른 포트로 접근하고 싶을 때, 그 포트를 SafePorts에 넣어줘야 프록시가 통과시킨다.
- 안 그러면 서버마다 **1:1 DNAT(포트포워딩)**를 일일이 해줘야 하니, 프록시에서 포트를 열어 한 번에 처리하는 게 편하다.

### 투명 프록시(TP모드)로 동작 확인

```
Transparent HTTP Proxy = Enable + Proxy Server = Enable
 → 클라이언트가 프록시를 몰라도 pfSense가 80 트래픽을 자동으로 Squid로 가로챔(intercept)
 → http://window31.com 접속 → Squid 로그에 통과/차단이 찍히는지 확인
 → ACL 블랙리스트에 window31.com 넣고 재접속 → "ERROR: requested URL could not be retrieved / Access Denied"
```

> ⚠️ **용어 혼동 주의 — "TP모드" 두 개가 다르다.**
> - 여기 Squid의 **Transparent Proxy(투명 프록시)** = 클라이언트 설정 없이 80 트래픽을 가로채는 것 (L7 프록시).
> - `Security-Solutions.md`의 **방화벽 TP모드** = L2 브릿지로 투명하게 끼어드는 방화벽 배치.
> - 둘 다 "투명(클라이언트가 눈치 못 챔)"이라는 발상만 같고 **레이어도 하는 일도 다르다.** 같은 단어라 헷갈리니 분리해서 기억.

### SSL을 프록시로 태울 거냐 — "목적과 자원"으로 판단

HTTP(평문)는 프록시가 URL을 그냥 읽어서 거를 수 있다. 그런데 HTTPS는 암호화돼 있어 **그냥은 URL을 못 본다.**

```
SSL을 프록시로 태우려면 = SSL Bump (방화벽이 인증서를 내주고 중간에서 복호화)
 → pfSense에 CA 인증서를 만들어 클라이언트에 배포 → 그 키로 복호화해서 URL을 보고 차단/허용
```

> ⚠️ 그런데 **모든 HTTPS를 다 복호화하는 게 정답이 아니다.** 평문이어도 상관없는(민감하지 않은) 사이트까지 SSL Bump로 다 복호화하면 그게 곧 **자원 낭비 + 프라이버시 침해**. → "이 트래픽을 프록시로 태울 거냐 말 거냐"도 **목적과 자원 현황에 따라 유동적으로** 판단할 줄 알아야 한다. (방화벽이 "다 열지 말고 필요한 것만"인 것과 같은 절제)

## 🛡️ SquidGuard — URL 블랙리스트 필터

Squid가 "프록시 엔진"이라면, **SquidGuard는 그 위에 얹어 "어떤 카테고리(도박/성인/멀웨어 등)를 막을지"를 거르는 필터 플러그인**이다.

> 비유: Squid가 외출 신고대라면, SquidGuard는 신고대 직원이 들고 있는 **"출입금지 가게 명단 책자"**. 책자를 카테고리별로(도박집·불법사이트…) 들고 있다가 그 가게에 가려 하면 막는다.

### 블랙리스트 다운로드 → 적용

```
SquidGuard → Blacklist URL 에 등록 후 다운로드:
  https://dsi.ut-capitole.fr/blacklists/download/blacklists_for_pfsense.tar.gz
 → 수천 개 도메인이 카테고리(adult/games/malware/warez/webmail …)별로 압축돼 있음
 → Download 후 카테고리가 [blk_blacklists_*] 형태로 떠야 정상
```

### Groups ACL — "어느 대역에서" 무엇을 막을지

```
SquidGuard → Groups ACL → Add (예: PC_dev_access)
  Source:        10.10.20.0/24            ← 이 대역(webdev)에서 나가는 트래픽에
  Target Rules:  카테고리별 access를 deny/allow/whitelist로 지정
                 (games·webmail 등 deny, Default access = allow)
  Do not allow IP addresses in URL: ✅    ← IP로 직접 친 사이트는 차단(FQDN 우회 방지)
  Log: ✅                                  ← 이 ACL 매칭 로그를 남김
 → 맨 아래 Apply 까지 눌러야 반영
```

- **`Do not allow IP-Addresses in URL`** = 도메인 차단을 우회하려고 `http://1.2.3.4` 처럼 **IP로 직접 접속하는 걸 막는** 옵션. (블랙리스트는 도메인 기준이라, IP로 치면 빠져나갈 수 있는 구멍을 메우는 것)
- 맨 아래 **Log** 체크 = 이 ACL이 매칭한 접근을 로그로 남기겠다는 선언.

### 오늘 완성한 정책

```
web 서버(VLAN10) — 화이트리스트 방식
  허용: ubuntu.com, security.ubuntu.com, kr.archive.ubuntu.com   (= 우분투 업데이트만)
  나머지: 전부 deny
  검증: curl로 허용 도메인 200 / 차단 도메인 403 확인

webdev(VLAN20) — 블랙리스트 방식
  차단: SquidGuard games 카테고리(수천 개 게임 도메인)
  나머지: 기본 허용(Default allow)
  검증: 1001games.com → 403(차단),  google.com → 301(허용)
```

> 💡 **같은 회사 안에서도 망마다 정책 방향이 다르다.** 운영(web)은 "딱 필요한 것만(화이트리스트)", 개발(webdev)은 "특정 위험만 막고 나머지 허용(블랙리스트)". 운영망은 엄격, 개발망은 생산성 고려 — 0번 워밍업의 "OA/QA처럼 용도가 다르면 설계가 다르다"가 여기서 실제로 구현됨.

> ⚠️ **SquidGuard는 Save만으론 안 먹는다 — Apply 버튼을 따로 눌러야** 룰이 실제 적용된다. (6/12 SquidGuard 계열에서 흔히 밟는 함정. "분명 저장했는데 안 막힌다"의 범인)

> ⚠️ HTTP(80) 차단은 **NAT 포워딩 룰을 따로 안 만들어도 된다** — 투명 프록시가 80을 자동으로 가로채기 때문. (인바운드 DNAT와 다른 흐름)

> 결과: web에선 우분투 업데이트만 통하고, webdev에선 블랙리스트 카테고리(게임)가 막힌다. **단, SSL을 안 태워서 HTTPS는 못 거른다 → http로 나가는 것 중 deny 카테고리만 차단**되는 상태.

## 🔐 SSL Bump 시도 → 보류

HTTPS도 거르려고 SSL Bump를 시도했다.

```
1. pfSense에 CA 인증서 생성 (System → Cert. Manager → Authorities → 'Squid-SSL-CA')
2. 그 CA를 webdev / web 클라이언트에 배포(신뢰 등록)
3. NAT: 443 → 3129 리다이렉트(투명 SSL intercept 포트)
4. issuer: CN=Squid-SSL-CA 확인 → SSL Bump 자체는 작동
```

> ⚠️ **문제: `ERR_DNS_FAIL`** — 투명 프록시로 가로챈 HTTPS는 **SNI(접속하려는 도메인 정보)가 소실/누락**되면 Squid가 어느 도메인인지 못 찾아 DNS 해석에 실패한다. 투명 intercept 방식은 클라이언트가 `CONNECT 도메인:443`을 안 보내주니, ClientHello의 SNI에 의존하는데 거기서 막힌 것.

```
결론: 복잡도 대비 효용이 낮아 오늘은 보류.
      나중에 "명시적 프록시(explicit proxy)" 방식으로 재시도 예정
      → 명시적 프록시는 클라이언트가 프록시 주소를 알고 CONNECT로 도메인을 알려주므로
        SNI 의존 문제가 없어 HTTPS 차단이 훨씬 깔끔하다.
```

> 💡 교훈: **투명 HTTPS 가로채기는 원래 까다롭다.** 실무에서도 HTTPS 필터링은 PAC/WPAD로 브라우저에 프록시를 명시 지정하는 방식을 더 많이 쓴다. "되는 길로 돌아가는" 판단도 실력. (위 0번 "프록시 태울지 목적·자원으로 판단"과 같은 절제)

> 2차 프로젝트 때 이 SSL Bump(복호화 키 pfSense 등록 → 차단/허용)를 본격적으로 다룰 예정.

## 🚪 방화벽 최소권한 정리 — 안 쓰는 아웃바운드를 닫는다

프록시를 만지면서 "그럼 애초에 인터넷이 필요 없는 서버는 80/443을 왜 열어뒀지?"를 다시 봤다.

```
VLAN30 (samba): 아웃바운드 80/443 제거 — 파일서버는 인터넷 나갈 일이 없다
VLAN40 (dns):   아웃바운드 80/443 제거 — DNS는 53번만 쓰고, 직접 인터넷에 안 나간다
```

> ⚠️ **DNS 서버가 80/443을 쓸 이유가 없다** — 흔한 오해 해소:
> - DNS = **이름 안내만** 한다(질의/응답은 53번). 외부 도메인을 포워딩할 때도 53번(8.8.8.8:53)으로 나간다.
> - 실제 웹 트래픽(80/443)은 **각 서버가 게이트웨이를 통해 직접** 나가는 것이지, DNS를 경유하는 게 아니다.
> - **DNS ≠ 게이트웨이** — DNS는 "주소 안내", 라우팅(실제 트래픽 운반)은 pfSense 게이트웨이가 한다. **완전히 별개 레이어.** (6/12 "이름풀이 OK ≠ 통신 OK"의 연장)

> 🔗 = `Security-Solutions.md` 방화벽 "최소허용·Default Deny"의 실천. 프록시로 "나가는 걸 거르는" 것과, 애초에 "나갈 필요 없는 포트를 닫는" 건 한 쌍. 둘 다 공격 표면 줄이기.

## 👃 Snort IDS — 설치부터 4가지 탐지 실험까지

### 0) Snort란 / 룰셋 음차 교정

- **Snort** = IDS/IPS 진영의 대표 오픈소스. (pfSense 패키지로 설치, 인터페이스별로 가동)
- 💡 **음차 교정 "vgt 무료 오픈소스 코드"** → Snort의 **무료 룰 소스**를 말한 것. 정리하면:

```
Snort 룰 소스 (pfSense Snort 기준)
  Snort Subscriber(VRT/Talos)  : 최신, 유료(Oinkcode 필요) — "vgt"는 이 VRT를 잘못 들은 듯
  Snort GPLv2 Community Rules  : 무료
  Emerging Threats Open (ET Open): 무료  ← 오늘 실제로 쓴 룰셋
  ET Pro                       : 유료
```
- 오늘 탐지에 쓴 SID들(2404300·2002677·2049400 …)이 전부 **2xxxxxx대 = ET Open** 룰. 즉 무료 ET Open을 불러와 인터페이스별로 룰을 업데이트해서 가동했다.

### 0-1) 인터페이스마다 룰을 다르게 — "다 넣으면 안 된다"

```
모든 인터페이스에 모든 룰을 다 켜면? → 비효율 + 목적에 안 맞음
  - 삼바서버 인터페이스에 SQLi 룰을 켠다? → 삼바는 웹이 아닌데 왜? (불필요한 부하·오탐)

목적에 맞게:
  공통/외부 위협 룰  → WAN
  SQLi 같은 웹 공격 → VLAN10(web)
  DNS 관련 룰       → VLAN40
```

> 🔗 = `Security-Solutions.md` "IDS 룰 튜닝(필요한 시그니처만 남기기)"과 "IPS 멀티 세그먼트(한 대로 구간별 다른 정책)"의 실제 적용. 인터페이스(센서)별로 그 망에 맞는 룰만 태운다.

---

### 1) 설치 삽질 — "GUI 초록불인데 프로세스가 안 뜬다"

```
증상: Snort 설치 후 프로세스가 안 뜸. 로그엔 'em0: promiscuous mode enabled'까지만 찍히고 조용히 죽음
     /var/log/snort/ 에는 에러가 없음 → 로그 위치를 잘못 본 것
```

**왜 `/var/log/snort/`가 비었나**
- 거긴 **탐지(alert) 결과**를 쌓는 곳이지, 데몬이 죽은 사유는 **stderr**로 간다.
- → 과정1 **Suricata 삽질과 똑같은 구조** (tcpdump엔 패킷이 잡히는데 fast.log는 비어 있던 그 건).

**진단 (test 모드로 어디서 죽는지 본다)**
```sh
/usr/local/bin/snort -T -c /usr/local/etc/snort/snort_5935_em0/snort.conf -i em0 2>/tmp/snort_err.txt
#  -T = test 모드(설정 파싱만 검증하고 종료). 어디서 깨지는지 stdout/stderr에 나옴
```

**실제 원인**
```
pkg info | grep snort        → 패키지 "등록"은 됨
find /usr/local -name snort -type f → 바이너리 파일이 아예 없음(!)
GUI 경고: "Package is configured but not (fully) installed"
→ 설치 중 파일 추출 실패. 메타데이터만 남고 실제 바이너리 미설치 상태였음
```

**해결**
```
Snort 완전 삭제(Remove) → Available Packages에서 재설치 → /usr/local/bin/snort 정상 확인
```

> ⚠️ **GUI 초록불 ≠ 실제 프로세스 실행.** 패키지 등록(메타데이터)과 바이너리 설치는 별개다. "떴다는데 안 도는" 경우 `find`로 바이너리 존재부터, `-T`로 설정 파싱부터 본다.

---

### 2) 탐지 실험 4종 (전부 alert 확인)

| # | 탐지 대상 | 위치 | 룰(SID) | 재현 방법 | threshold |
|---|---|---|---|---|---|
| 1 | C2(C&C) 서버 통신 | WAN | ET CNC Feodo Tracker Reported CnC Server (2404300) | pfSense 셸 `telnet 162.243.103.246 443` | type limit, by_src, 3600s, count 1 |
| 2 | 외부 스캐너(Nikto) | WAN | ET SCAN Nikto Web App Scan in Progress (2002677) | 본컴 PowerShell에서 Nikto User-Agent 위조 10회 | type both, count 5, 60s |
| 3 | 디렉터리 트래버설 | VLAN10 | ET WEB_SERVER /etc/passwd Detected in URI (2049400) | 브라우저로 `http://192.168.1.82/../../../../etc/passwd` | — |
| 4 | 커스텀 SQLi | VLAN10 | custom.rules (SID 9000001) | `pw=...OR...` 패턴 요청 | — |

**실험 4 커스텀 SQLi 룰** — ET Open엔 범용 SQLi 패턴이 없어서(특정 CVE 기반만 존재) 직접 작성:
```
alert tcp any any -> any any (msg:"Custom SQLi OR 1=1"; content:"pw="; nocase; content:"OR"; nocase; distance:0; within:50; sid:9000001; rev:5;)
```
- `content:"pw="` 찾고, 그 **뒤(distance:0) 50바이트 안(within:50)**에 `OR`이 또 나오면 매칭.
- `' OR '1'='1`을 그대로 매칭하려 했더니 **URL 인코딩 이슈**(`'`·공백이 `%27`·`%20`로 변환)로 안 잡혀서, `pw=` 다음 `OR` 패턴으로 우회.

> 🔗 룰 문법은 `Security-Solutions.md` "Snort/Suricata 룰 한 줄 구조 분해"와 정확히 같다. `content`로 페이로드의 문자열을 잡고, `distance/within`으로 "어디 근처에 있냐"를 좁힌다. 실험1의 C2(안→밖 outbound)는 그 노트의 "역방향(트로이목마) 룰"의 실제 사례.

> ⚠️ **threshold 때문에 같은 IP로 재테스트가 안 된다.** 실험1은 `by_src 3600s count 1` = **한 소스에서 1시간에 딱 1번만 alert**. 다시 보려면 다른 소스 IP로 쳐야 한다. 실험2는 `count 5 / 60s` = 60초 안에 5번 이상 들어와야 alert(그래서 10회 전송). → "한 번 떴는데 다시 안 뜬다 = 룰이 안 먹는 게 아니라 threshold다." 애노멀리의 Threshold 개념(`Security-Solutions.md`)이 IDS 룰에 박혀 있는 것.

### 3) IDS 위치별 탐지 구조

```
본컴(192.168.1.83)
  │
WAN(em0) ─ Snort      ← 외부→내부 인바운드 1차 탐지 (인라인 윗단)
  │  포트포워딩 (WAN:80 → Web_SV:80)
VLAN10(em1.10) ─ Snort ← 세그먼트 내부 트래픽 탐지 (포워딩 이후 트래픽도 봄)
  │
web VM (10.10.10.10)
```

- **WAN Snort** = 외부 공격을 가장 먼저 보는 윗단.
- **VLAN10 Snort** = 세그먼트별 탐지. 포트포워딩을 통과한 뒤의 트래픽까지 본다.
- **두 레이어에서 동시 탐지 가능** — Nikto 스캔이 WAN·VLAN10 둘 다에 찍혔다. = 심층 방어(여러 겹에서 본다)의 IDS판.

> ⚠️ Snort는 **인터페이스별로 룰이 독립**이다. 각 인터페이스의 Rules 탭에서 따로 켜야 한다(0-1번의 "인터페이스마다 다른 룰" 그대로).

---

### 4) ET Open 룰셋의 한계 — IDS ≠ WAF (중요 개념)

```
ET Open = 알려진 CVE·특정 제품 기반 시그니처 위주
 → ' OR '1'='1 같은 범용 SQLi 패턴은 미포함 (오탐이 너무 많아 의도적으로 제외)
 → 그래서 실험4를 커스텀 룰로 직접 짜야 했던 것

범용 웹 공격(SQLi/XSS 등)은 → WAF(ModSecurity + OWASP CRS)가 담당
```

> 💡 **Snort IDS와 WAF는 역할이 다르다.** IDS는 "네트워크 전반의 알려진 위협을 넓게 탐지", WAF는 "HTTP 트래픽의 웹 공격을 정밀 차단". = `Security-Solutions.md`의 "WAF는 IPS에서 갈라져 나온 HTTP 전용 장비"가 여기서 체감됨. "왜 IDS만으론 SQLi가 안 잡히지?" → 그건 WAF 영역이라서.

## ⚠️ 오늘 막혔던 / 짚어둘 것

- **SquidGuard는 Save+Apply** — Apply를 따로 안 누르면 룰이 적용 안 됨("분명 저장했는데 안 막힌다"의 범인).
- **투명 HTTPS(SSL Bump)는 SNI 소실로 ERR_DNS_FAIL** — 투명 intercept는 까다롭다. 명시적 프록시로 가는 게 정석.
- **"TP모드" 두 개 혼동 금지** — Squid 투명 프록시(L7) vs 방화벽 TP모드(L2 브릿지)는 다른 것.
- **DNS는 80/443 불필요** — DNS는 53번만. DNS≠게이트웨이(이름안내 ≠ 트래픽운반).
- **Snort GUI 초록불 ≠ 프로세스 실행** — 바이너리 미설치인데 등록만 된 상태. `find`로 바이너리, `-T`로 설정 파싱 확인. 재설치로 해결.
- **데몬 죽은 사유는 alert 로그(`/var/log/snort/`)가 아니라 stderr** — 과정1 Suricata fast.log 삽질과 동일 구조.
- **threshold 탓에 같은 IP 재테스트 안 됨** — `by_src count 1`이면 다른 소스로, `count 5`면 5회 이상 쳐야 뜬다.
- **pfSense 셸은 sh(ash), bash 아님** — `2>&1` 안 먹어서 `2>/tmp/file`로 stderr 분리. 한글 명령어 입력 시 터미널 깨짐 → BPF 필터 오류로 번짐.
- **ET Open엔 범용 SQLi 룰 없음** — 커스텀 룰 직접 작성. 범용 웹 공격은 WAF 영역(IDS≠WAF).
