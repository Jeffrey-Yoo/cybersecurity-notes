# 2026-06-12 — 내부 서비스 채우기 (삼바·내부DNS·NAT 포트포워딩·Alias)

과정2 9일차. 6/11에 VLAN 10/20(Web·Web_Dev) 두 칸만 있던 망에 **VLAN 30(삼바 파일서버)·VLAN 40(내부 DNS)을 더 올려 4칸으로 채운** 날. 단순히 서버 수를 늘린 게 아니라, **"이 망에서 저 망으로 갈 때 왜 열고 왜 닫나"**를 방화벽 정책으로 직접 박아본 날이다. 오늘의 진짜 핵심은 마지막 **Alias** — 같은 IP를 룰 6개에 박아두고 "본컴 IP 바뀌면 6개 다 고쳐야 하네?"에서 출발해 관리 포인트를 1개로 줄인 것.

> FIMS 연결: 과정1 FIMS에선 파일 공유·내부 이름풀이를 따로 세울 일이 없었다(단일 분석 인프라). 오늘은 **운영망 안에 파일서버(삼바)와 내부 DNS를 별도 VLAN으로 분리**해서, "서비스마다 망을 가르고 그 사이 길목을 방화벽으로 통제"하는 EA 설계의 축소판을 만들었다. 6/11이 "망을 가르는" 단계였다면, 오늘은 "가른 망에 서비스를 채우고 길을 통제"하는 단계.

## 0️⃣ 워밍업 — 보안담당자의 관점 (편함 ≠ 적)

개발자는 개발이 본업이라 점점 편한 도구(코파일럿·GitHub류)로 빨려간다. 보안담당자가 이걸 "편함이 보안을 약화하니 쓰지 마"로 막으면 방향이 틀렸다.

```
틀린 방향: 편함이 보안을 약화한다 → 쓰지 마라 (개발 생산성을 적으로 둠)
맞는 방향: 그 소스가 "어떤 환경에서 어떻게" 다뤄지는지 파악
          → 더 강화된 보안으로 유출을 최소화하며 편함을 살린다
```

> 비유: 칼이 위험하다고 식당에서 칼을 뺏는 게 아니라, **안전한 도마와 칼집**을 주는 것. 도구를 없애는 게 아니라 안전하게 쓰게 만드는 게 보안의 일.

> 💡 오늘 실습 전체를 관통하는 태도. 서비스(삼바·DNS·웹)를 다 열어주되, **길목마다 방화벽으로 통제**해서 "편하게 쓰되 새지 않게" 만드는 게 목표다.

### 오늘의 목표 — 방화벽 정책을 "왜"로 설계

```
어떤 망 → 어떤 망 갈 때, 어떤 경로로, 어떤 이유로 열리고 닫혀야 하는가
```

- 인터넷으로 나가는 트래픽도 "왜 나가나"를 따져야 한다. **목적이 패치(업데이트)라면** 서버가 직접 인터넷에 나가는 게 아니라 → **PMS 서버(내부 레포지토리)를 두고 거기서 업데이트를 받아오는 구조**가 가장 적합.

> 🔗 연결: `IT-Infrastructure.md` 망분리·PMS, `Security-Solutions.md` 방화벽 정책 철칙(단방향·Default Deny). 오늘은 그 이론을 pfSense 룰로 실제로 박는 날.

## 📁 삼바(Samba) — 파일서버(VLAN 30)와 마운트(VLAN 20)

운영 파일을 한 곳(VLAN 30 삼바)에 모으고, 개발서버(VLAN 20 webdev)가 그걸 **네트워크 드라이브로 마운트**해 쓰는 구조.

> 비유: 삼바 서버 = 회사 공용 **자료실**. webdev가 자기 책상(`/mnt/samba`)에 자료실로 통하는 **문을 하나 뚫어(mount)** 자기 폴더처럼 쓰는 것. 문은 회사 출입증(삼바 계정)으로만 열린다.

### 서버 측 (VLAN 30, 10.10.30.10)

```bash
apt install samba samba-client
groupadd samba
useradd -M -s /sbin/nologin sambauser   # -M 홈 없음, 로그인 셸 없음(파일공유 전용 계정)
usermod -aG samba sambauser
smbpasswd -a sambauser    # 삼바 전용 비번 (리눅스 로그인 비번과 별개)
smbpasswd -e sambauser    # 계정 활성화

mkdir -p /samba
chown sambauser:samba /samba
chmod 770 /samba
```

`/etc/samba/smb.conf` 맨 아래:

```ini
[samba]
   path = /samba
   browseable = yes      # ⚠️ browsable 아님
   writeable = yes
   guest ok = no
   valid users = @samba  # ⚠️ valid user 아님 — 복수형
```

```bash
testparm                         # 문법 검사 (Unknown parameter 안 뜨면 OK)
systemctl restart smbd && systemctl enable smbd
smbclient //localhost/samba -U sambauser   # 서버 자체 접속 테스트
```

> ⚠️ **오늘 밟은 두 함정 (둘 다 testparm이 잡아줌)**
> 1. `browsable` → **`browseable`** (e 들어감)
> 2. `valid user` → **`valid users`** (복수형)
> → `smb.conf`는 오타가 나도 서비스는 뜨는데 동작만 이상하다. **반드시 `testparm`으로 Unknown parameter를 거른 뒤** restart. (netplan의 `netplan generate`, BIND의 `named-checkconf`와 똑같은 "적용 전 문법검사" 습관)

### 클라이언트 측 (VLAN 20 webdev, 10.10.20.10)

```bash
apt install cifs-utils          # 마운트 전용 (탐색용 samba-client와 다름)
mkdir -p /mnt/samba
mount -t cifs //10.10.30.10/samba /mnt/samba -o username=sambauser

# 비번을 명령행에 안 남기려고 credentials 파일로 분리
mkdir -p /etc/samba
vi /etc/samba/credentials
#   username=sambauser
#   password=비번
chmod 600 /etc/samba/credentials   # 비번 파일이니 600 필수

# 재부팅에도 살아남게 fstab 자동 마운트
vi /etc/fstab
#   //10.10.30.10/samba  /mnt/samba  cifs  credentials=/etc/samba/credentials,uid=1000  0  0
mount -a    # ⚠️ 재부팅 전 fstab 검증 필수
```

> ⚠️ **fstab 줄 형식 헷갈림** — fstab 안에선 `-o`를 쓰지 않는다. **필드 사이는 공백, 옵션들끼리는 콤마.** `mount` 명령행의 `-o username=...`이 fstab에선 그냥 옵션 필드(콤마 구분)로 들어간다.

> 🔗 **세 가지 설정파일 들여쓰기 규칙 비교** (오늘 한 번에 정리됨):
> - netplan(YAML): **들여쓰기 = 문법** — 한 칸 틀리면 에러
> - smb.conf(INI): 들여쓰기 = 가독성용 — 안 해도 됨
> - fstab: **공백으로 필드 구분, 칸 수 무관**
> → 같은 "설정파일"이라도 파서가 다르니 규칙이 다르다. 이걸 모르면 "왜 YAML은 되는데 fstab은 안 되지?"에서 한참 헤맨다.

> Samba 포트: **445(SMB Direct)**, 139(NetBIOS 구버전). 드로우의 `445`가 이것.

## 🌐 내부 DNS (BIND9, VLAN 40, 10.10.40.10)

내부 서버끼리 IP 대신 이름(`web.lab.internal`)으로 부르게 하는 **사내 전화번호부**.

> 비유: 내부 DNS = 회사 **내선 전화번호부**. "회계팀(=web)" 하면 내선번호(10.10.10.10)로 연결. 외부 번호는 모르면 **교환원(forwarders 8.8.8.8)**한테 넘긴다.

### 설계

```
도메인:  lab.internal  /  maxoverpro.org
DNS IP:  10.10.40.10  (web/webdev/samba 모두 이걸 바라봄)
레코드:
  web.lab.internal    → 10.10.10.10
  webdev.lab.internal → 10.10.20.10
  samba.lab.internal  → 10.10.30.10
  dns.lab.internal    → 10.10.40.10
  www.maxoverpro.org  → 10.10.10.10   (web의 외부용 도메인)
```

### 설정 (이전 BIND 노트보다 깊어진 부분)

```bash
apt install bind9 bind9utils bind9-dnsutils
```

`/etc/bind/named.conf.options` — **여기서 오픈 리졸버를 막는 게 보안 포인트**:

```
options {
    directory "/var/cache/bind";
    allow-query     { any; };                      # 누구나 질의는 허용
    allow-recursion { 10.10.0.0/16; localhost; };  # ★ 재귀는 내부망만 (오픈 리졸버 차단)
    forwarders { 8.8.8.8; 1.1.1.1; };              # 모르는 도메인은 구글/클플로 넘김
    forward only;
    dnssec-validation no;
    listen-on { 10.10.40.10; 127.0.0.1; };
    listen-on-v6 { none; };
};
```

> ⚠️ **`allow-recursion`을 any로 열면 "오픈 리졸버"가 되어 DNS 증폭 DDoS의 발판으로 악용된다.** 그래서 질의(allow-query)는 열어도 **재귀(남 대신 외부까지 찾아주는 것)는 내부망(10.10.0.0/16)만** 허용한다. = `Security-Solutions.md` 방화벽 "단방향·최소허용" 원칙의 DNS판.

`/etc/bind/named.conf.local` — 존 선언, 그리고 존 파일(db.lab.internal 등) 작성.

```bash
named-checkconf                                          # 설정 문법
named-checkzone lab.internal /etc/bind/db.lab.internal   # 존 파일 문법
rndc reload                                              # ★ 서비스 무중단 존 재로드 (restart 대신)
dig @localhost web.lab.internal                          # 자체 테스트
```

> ⚠️ **존 파일 두 함정**
> 1. **Serial 번호는 수정 때마다 +1** — 안 올리면 변경이 반영 안 됨(슬레이브/캐시가 "안 바뀐 줄" 앎).
> 2. **FQDN 끝의 점(.)** — 점 있으면 완전한 이름, 없으면 뒤에 `.lab.internal`이 자동으로 붙는다. `ns.maxoverpro.org`에 점을 빼먹으면 `ns.maxoverpro.org.lab.internal`이 되어버림.

> 🧠 **DNS의 본질**: DNS = 이름→IP 변환(전화번호부)**일 뿐**, 통화 허가(방화벽)와는 별개다. 이름이 풀려도(dig 성공) 방화벽이 막으면 접속은 안 된다. "이름풀이 OK ≠ 통신 OK" — 트러블슈팅 때 이 둘을 분리해서 봐야 한다.

## 🔀 pfSense NAT 포트포워딩 (DNAT) — 본컴에서 내부 서버 직접 접근

본컴(192.168.1.83)에서 VLAN 안쪽 서버(10.10.x.x)로 바로 들어가려고, pfSense WAN의 포트를 내부로 꽂아준다.

> 비유: pfSense WAN(192.168.1.82)이 건물 **안내데스크**. "82번 건물 80번 창구로 오세요" 하면 안내데스크가 알아서 안쪽 운영팀(10.10.10.10:80)으로 데려다준다. 방문객은 내부 주소를 몰라도 된다.

| 설명 | WAN 포트 | 실제 목적지 |
|---|---|---|
| 웹 서비스 | 80 | 10.10.10.10:80 |
| web SSH | 1010 | 10.10.10.10:1010 |
| webdev SSH | 2020 | 10.10.20.10:2020 |
| samba SSH | 3030 | 10.10.30.10:3030 |
| dns SSH | 4040 | 10.10.40.10:4040 |

```
Firewall → NAT → Port Forward → Add
  Interface:     WAN
  Dest. Address: 192.168.1.82 (WAN address)   ← 본컴이 치는 IP = pfSense WAN
  Dest. Port:    포워딩할 포트 (80, 1010 …)
  Redirect IP:   실제 서버 IP (10.10.x.10)
  Redirect Port: 실제 서버 포트
```

> ⚠️ **Dest. Address를 본컴 IP(192.168.1.83)로 넣으면 안 됨.** "본컴이 향하는 목적지" = **pfSense WAN IP(192.168.1.82)**다. 여기서 한 번 헷갈렸음 — 포워딩의 Dest는 "들어오는 입구(WAN)" 기준이지 "보낸 사람" 기준이 아니다.
> → NAT 룰을 추가하면 WAN 방화벽 룰은 "자동 생성" 체크로 같이 만들어진다(따로 안 박아도 됨).

> 🔗 6/11 "WAN any 허용" 편의 세팅의 정식 버전. any로 통째로 여는 대신, **포트별로 목적지를 콕 집어** 포워딩한다 = 공격 표면이 훨씬 좁다.

## 🔐 SSH 포트 변경 — Ubuntu 22.04+ `ssh.socket` 함정

서버마다 SSH 포트를 1010/2020/3030/4040으로 바꾸려는데, **`sshd_config`의 Port만 바꿔도 안 먹혔다.**

```
원인: 우분투 최신 버전은 SSH가 ssh.socket(소켓 액티베이션)으로 뜬다.
      → 포트를 소켓이 관리하므로 sshd_config의 Port는 무시됨.
```

해결 — 소켓 오버라이드:

```bash
systemctl edit ssh.socket
#   [Socket]
#   ListenStream=          ← 기존값 비우고
#   ListenStream=원하는포트  ← 새 포트
systemctl daemon-reload    # ⚠️ demon 아님 — daemon 철자
systemctl restart ssh.socket
```

또는 소켓을 아예 끄고 전통 방식으로:

```bash
systemctl disable --now ssh.socket
systemctl enable --now ssh    # 이제 sshd_config의 Port가 먹는다
```

```bash
systemctl status ssh        # "listening on 0.0.0.0 port XXXX" 확인
ss -tlnp | grep ssh         # 실제 리스닝 포트 확인
```

> ⚠️ **"분명 sshd_config 고쳤는데 포트가 안 바뀐다"의 범인이 ssh.socket.** 기존 `Linux-Commands.md`의 SSH 포트 변경(sshd_config만 고치는 법)은 구버전 기준 — 22.04+에선 이 소켓 단계가 추가된다. 포트 바꿨는데 안 되면 `ss -tlnp`로 누가 그 포트를 잡고 있나부터 본다.


접속: `http://192.168.1.82/board/` (위 NAT 80 포워딩을 통해 본컴에서).

> ⚠️ **PHP 미설치면 PHP 코드가 그대로 화면에 출력된다.** 아파치가 `.php`를 실행 안 하고 텍스트로 뱉는 것. `php -v`로 설치 확인. (6/11 "최소설치라 도구가 없다"의 연장 — 웹서버는 apache만으론 안 돌고 php 모듈까지 있어야 한다)

## ⭐ pfSense Alias — 오늘의 핵심 (관리 포인트 줄이기)

### 문제 상황

SSH 허용 IP(본컴 192.168.1.83)를 WAN 룰 **6개에 각각** Source로 박아뒀다. 본컴 IP가 바뀌면 → **6개를 일일이 수정**해야 한다.

> 비유: 출입증 명단을 **검문소 6곳에 따로따로** 붙여놨더니, 명단 한 명 바뀔 때마다 6곳을 다 뛰어다녀야 하는 꼴.

### 해결 — Alias(별칭)

```
Firewall → Aliases → Add
  Name:        ADMIN_IP
  Type:        Host
  Network/IP:  192.168.1.83
```

모든 룰의 Source를 IP 대신 **`ADMIN_IP`**로 지정. IP가 바뀌면 **Alias 한 곳만 고치면 6개 룰에 자동 반영**.

> 🧠 **방화벽 운영의 기본 원칙**: IP·포트 그룹이 여러 룰에 반복되면 **Alias로 묶어 관리 포인트를 하나로** 만든다. 룰 수가 수천 개가 되는 실무에선 이게 곧 운영 가능성. (= 프로그래밍의 "매직넘버 하드코딩 대신 상수 하나로 빼기"와 같은 발상)

```
실무 Alias 예
  ADMIN_HOSTS   : 관리자 IP 묶음 (여러 개)
  WEB_PORTS     : 80, 443
  INTERNAL_NETS : 내부 대역 전체
```

> 🔗 6/12 NAT 포워딩 5개 + WAN 룰 6개를 박으면서 "같은 IP를 반복해 넣는" 비효율을 몸으로 느낀 뒤 Alias로 정리한 흐름. 먼저 노가다를 해보고 그 다음에 추상화를 배우니 "왜 필요한지"가 박혔다.

## ⚠️ 오늘 막혔던 / 짚어둘 것

- **삼바 오타 2종** — `browsable→browseable`, `valid user→valid users`. 서비스는 떠도 동작이 이상 → 무조건 `testparm`으로 거르고 restart.
- **fstab엔 `-o` 없음** — 필드는 공백, 옵션은 콤마. netplan(YAML 들여쓰기=문법)·smb.conf(INI 들여쓰기=가독성)·fstab(공백 구분) 파서별 규칙이 다르다.
- **BIND `allow-recursion`을 any로 열면 오픈 리졸버** — 재귀는 내부망만. Serial +1, FQDN 끝 점(.) 빠뜨리지 말 것.
- **ssh.socket이 sshd_config를 덮어쓴다** — 우분투 22.04+. 포트 안 바뀌면 `ss -tlnp`로 누가 잡았나 확인.
- **NAT 포워딩 Dest.Address = pfSense WAN(192.168.1.82)**, 보낸 사람(본컴 .83) 아님.
- **이름풀이 OK ≠ 통신 OK** — dig 되어도 방화벽이 막으면 접속 안 됨. DNS와 방화벽은 별개 레이어.
