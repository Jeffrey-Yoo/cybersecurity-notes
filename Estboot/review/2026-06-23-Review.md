
Q3. 회선·F/W는 여유로운데 IPS만 CPU·메모리 풀로 차서 뻗었다. 이때 **하드웨어 바이패스와 HA가 왜 못 구하는지** 각각 설명하라. — 서술형

Q4. UDP Flooding을 와이어샤크로 봤을 때 "이 IP만 차단하면 된다"가 안 통하는 이유는? 그리고 이를 원천 차단하는 필터링 기법 이름은? — 실무 적용

Q5. 응용계층(HTTP GET/POST) 공격이 대역폭 경보·블랙리스트로도 안 걸리는 이유는? — 면접 대비

Q6. (FIMS 연결) FIMS에서 로그인 5회 실패 차단(slowapi)을 구현했다. 이건 DDoS 3대 유형 중 어느 것을 막는 발상과 같은 결인가?

<details>
<summary>정답 보기</summary>

A1. 서버 성능이 올라가 **1:1(DoS)로는 안 통하게 됨** → 공격자가 다수의 좀비 PC(봇넷)를 동원해 **n:1 집단폭행(DDoS)** 으로 전환.

A2.
- **대역폭 고갈**: 회선(파이프) 자체 / L3·L4 / **bps**. UDP·ICMP·Fragment Flooding.
- **자원 고갈**: 보안장비 CPU·메모리·세션테이블 / L3·L4 / pps·세션 수. SYN Flood, 단편화, 80·443 인젝션 폭주.
- **응용계층**: 웹 서버·DB·앱 엔진 / L7 / rps. HTTP GET·POST Flooding.

A3.
- **바이패스**: 장비가 "죽은(down)" 게 아니라 자원이 풀로 차서 "지연(delay)"된 상태다. 바이패스는 down을 감지해 우회하는 장치라 안 죽고 느려진 상황은 트리거가 안 된다.
- **HA**: Active가 죽어 Standby로 넘겨도 **그 세션이 그대로 넘어가므로** 넘겨받은 장비도 똑같이 자원 고갈로 뻗는다. 도긴개긴.
→ 그래서 Anti-DDoS와 IPS를 한 라인에 몰면 안 되고, 고스루풋 Anti-DDoS가 앞단에서 굵은 트래픽을 먼저 걸러야 한다.

A4. 패킷마다 **출발지 IP가 전부 위조(제각각)** 되어 있어 차단할 단일 IP가 없다. UDP라 3-way도 안 맺어 정상 연결로도 안 보인다. → **Ingress(들어오는 출발지 검사) / Egress(나가는 출발지 검사) 필터링 = BCP38** 로 위조 IP 자체를 원천 차단. 추가로 **1:1 방화벽 정책**으로 허용 범위를 좁힌다.

A5. 공격자가 **Real IP**로 정상 3-way handshake까지 정상 수행한 뒤 HTTP 요청을 퍼붓기 때문에 ① 위조 IP가 아니라 블랙리스트·출발지 필터링으로 못 거르고, ② **트래픽 양 자체는 크지 않아** 대역폭 경보에도 안 걸린다. → L7 읽는 Anti-DDoS/WAF, 요청 빈도 Rate Limiting, CAPTCHA, 행위 기반 탐지 필요.

A6. **응용계층(L7)**. slowapi의 요청 빈도 제한 = 한 클라이언트가 짧은 시간에 너무 많은 요청을 못 하게 막는 **Rate Limiting** → HTTP Flooding 류 L7 공격 완화와 같은 발상. (브루트포스 방어와도 한 뿌리)

</details>

## 예상 면접 질문

**Q1. "DDoS는 어떻게 막나요?"**
✅ 좋은 답변: "사실상 '완전 차단'은 불가능하다는 전제로 접근합니다. 좀비 PC가 무한히 많아 캐파 경쟁으로는 못 이기기 때문입니다. 그래서 ① **흡수** — 클라우드 오토스케일링·AWS Shield·CDN으로 급증분을 받아내고, ② **분산** — Anycast+BGP로 한 IP를 여러 지역 노드에 뿌려 트래픽을 나누며, ③ **우회/거름** — 앞단 고스루풋 Anti-DDoS로 굵은 플러딩을 흘려보내고 Ingress/Egress 필터링(BCP38)으로 위조 IP를 차단합니다. 핵심은 **골든타임** — 회선·CPU가 맥스를 치기 전 구간을 모니터링으로 잡아 끊는 겁니다."
❌ 주의: "방화벽/IPS로 다 막습니다" — IPS에 DDoS를 떠넘기면 자원 고갈로 같이 뻗는다. Anti-DDoS와 IPS는 라인을 분리해야 한다.

**Q2. "대역폭 고갈 공격과 응용계층 공격의 방어가 왜 다른가요?"**
✅ 좋은 답변: "노리는 계층이 다릅니다. 대역폭 고갈은 L3/L4에서 **bps 단위**로 회선 파이프를 터뜨리니, 앞단 고스루풋 장비로 흘려보내고 위조 출발지 IP를 필터링하면 됩니다. 반면 응용계층은 공격자가 **Real IP로 정상 핸드셰이크 후** HTTP 요청을 퍼붓기 때문에 출발지 차단·대역폭 경보가 안 통합니다. L7까지 읽는 WAF/Anti-DDoS, 요청 빈도 기반 Rate Limiting, 행위 기반 탐지가 필요합니다."
❌ 주의: "둘 다 트래픽 많으니 회선만 늘리면 된다" — 응용계층은 트래픽 양이 크지 않아 회선 증설로 안 잡힌다.

**Q3. "출발지 IP가 위조된 DDoS는 어떻게 대응하나요?"**
✅ 좋은 답변: "단일 IP 차단이 무의미하므로 **Ingress/Egress 필터링(BCP38)** 으로 위조 IP 자체를 못 쓰게 만듭니다. 라우터·방화벽 ACL에서 들어오는 패킷의 출발지가 정상 대역인지, 나가는 패킷의 출발지가 내부 대역이 맞는지 검사합니다. 위조가 막히면 대역폭·자원 고갈류 공격을 대거 예방할 수 있습니다. 또 DNS는 위조 UDP가 노리기 쉬우니 방화벽 안쪽이 아니라 외부 존/별도에 두는 배치도 같이 봅니다."
❌ 주의: "BGP를 BGT로", "BCP38을 그냥 ACL이라고만" — 명칭 정확히. BGP=라우팅 경로 선택 프로토콜, Ingress/Egress 필터링=위조 출발지 차단.
