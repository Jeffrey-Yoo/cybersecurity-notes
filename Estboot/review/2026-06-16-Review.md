# 2026-06-16 복습 — 과정2 11일차: Snort IDS→IPS 전환 + 룰 튜닝 + Kali + VPN

> 어제 "보고만 있던" Snort를 오늘 **걸리면 막는 IPS(인라인)**로 바꿨다. 핵심: ① **Block Offenders** 체크 = IDS→IPS, **Legacy(사후 차단) vs Inline(실시간 drop)**, ② 인라인 전엔 **Hardware Checksum Offloading OFF**(VM은 거의 필수), ③ 차단 켜면 오탐=장애라 **Pass List(차단 면제)·Suppress(알람 끄기)**가 즉시 필요(목적 다름), ④ 커스텀 **SID는 3000000+**, content는 **hex/hash**로(평문 문자열은 인코딩에 빗나감), ⑤ **Kali 공격기**를 WAN(192.168.1.250 static)에 투입, ⑥ **VPN**은 터널+정책 2단, OTP(MFA)·SSL VPN 인증서, 입구에서 **SSL Inspection**.

## 오늘의 복습 퀴즈

**Q1.** [단답형] pfSense Snort에서 어떤 체크박스 하나로 IDS가 IPS가 되나? 그리고 그때 차단 방식 두 모드(Legacy / Inline)의 가장 큰 차이는?

**Q2.** [흐름 이해] 인라인 모드를 켜기 전에 "Hardware Checksum Offloading"을 꺼야 한다. 왜 끄지 않으면 인라인 검사가 꼬이나? VirtualBox에서 왜 더 문제가 되나?

**Q3.** [개념 구분] Suppress와 Pass List는 둘 다 "오탐 대응" 같은데 뭐가 다른가? "이 룰이 알람을 너무 뱉는다"와 "이 서버는 절대 끊기면 안 된다"는 각각 무엇으로 처리하나?

**Q4.** [실무 적용] 커스텀 룰의 content를 평문 문자열로 넣었더니 잘 안 맞는다. 왜 그런가? hex로 넣는 것과 해시(SHA256/MD5)로 넣는 것은 각각 무엇을 잡고 한계는?

**Q5.** [면접/연결] VPN으로 터널만 뚫리면 내부가 다 보이나? 직원이 SSL-VPN으로 들어온 뒤 뭘 하는지는 어떻게 보나? 이게 ZTNA와 무슨 상관인가?

<summary>정답 보기</summary>

**A1.** **Block Offenders**("Checking this option will automatically block hosts that generate a Snort alert"). 켜면 alert를 일으킨 호스트를 자동 차단 → IPS가 됨. 모드 차이: **Legacy**는 패킷 복제본을 보고 alert IP를 pf 차단테이블(snort2c)에 등록해 "이후" 트래픽을 끊는 **사후 차단**(방아쇠 패킷은 이미 통과). **Inline**은 netmap으로 라인 위에서 그 패킷 자체를 **실시간 drop**. (Legacy=미러링, Inline=인라인의 실제 구현)

**A2.** 오프로딩이 켜져 있으면 OS는 체크섬을 비워둔 채 NIC에 넘기고 **NIC이 "랜선 나가기 직전"에 체크섬을 채운다.** 인라인 Snort(netmap)는 그 직전보다 **앞에서** 패킷을 가로채니, 체크섬이 아직 비어있어 "깨진 패킷"으로 오인 → 검사 꼬임·재주입 깨짐. 끄면 **CPU가 미리 체크섬을 채워** Snort가 완성된 패킷을 본다. VirtualBox는 실제 랜선을 안 타고 **호스트가 체크섬을 대신 처리**해 게스트 안에선 빈 체크섬 패킷이 더 흔해서 더 잘 깨진다(VM 인라인엔 거의 필수). + 같은 이유로 TSO/LRO도 끈다.

**A3.** **Suppress = 알람(로그)을 끄는 것**(룰·차단은 살아있음), **Pass List = 차단을 면제하는 것**(가용성 보호). "이 룰이 알람을 너무 뱉는다" → **Suppress**(`suppress gen_id 1, sig_id …` SID 통째로 / 특정 IP면 `track by_src/by_dst`). "이 서버는 절대 끊기면 안 된다" → **Pass List**(등재 IP는 alert 떠도 block 안 함). 둘 다 IP 직접입력은 하수 → **Alias**로.

**A4.** 문자열은 **"문자 그대로" 해석되지 않는 경우가 많다** — URL/Base64 인코딩, 이스케이프, 대소문자, 비출력 바이트로 같은 글자라도 실제 바이트가 달라져 평문 매칭이 빗나간다. **hex**(`content:"|68 70 …|"`)는 바이트를 있는 그대로 매칭 → 인코딩 무관 정확(단 짧으면 오탐). **해시**(`file_sha256/file_md5`)는 **파일 자체를 지문으로** 매칭 → 이름 바뀌어도 잡지만 **1바이트만 바뀌어도 못 잡음(폴리모픽 변종 회피)**. 그래서 변종엔 행위기반(EDR·애노멀리)이 필요.

**A5.** 아니다. VPN은 **터널 + 정책 2단**이라, 터널로 묶인 뒤 서버쪽 VPN 정책을 통과해야 서버에 닿는다. 들어온 뒤의 행위는 **입구에서 SSL Inspection(복호화)**으로 평문화해 관제 상황판에서 본다. 즉 "터널을 깔아 들여보내는 것"과 "들어온 뒤 행위 가시성"은 별개 — 터널만 믿지 않고 안에서 또 본다. 이게 **ZTNA**(터널 통째로 신뢰 X, 요청마다 검증)로 가는 발상. (VPN 게이트웨이 자체가 공격표면=위험 수용이라 더욱 그렇다)

## 예상 면접 질문

**Q. IDS를 IPS로 전환할 때 무엇을 고려해야 하나요?**
✅ 좋은 답변: "탐지(alert)를 차단(drop)으로 바꾸는 순간 **오탐이 곧 서비스 장애**가 됩니다. 그래서 ① 차단 모드(라인 위 실시간 drop인 인라인이냐, 차단테이블 기반 사후 차단인 legacy냐)를 정하고, ② 인라인이면 **NIC 오프로딩(체크섬·TSO/LRO)을 꺼서** 검사 엔진이 원본 패킷을 정확히 보게 하고, ③ **Pass List로 핵심 노드(게이트웨이·DNS·관리 PC)를 차단에서 면제**하고, ④ 시끄러운 룰은 **Suppress**로 노이즈를 줄입니다. 핵심은 '차단을 켜기 전에 안전장치(Pass List)부터 깐다'입니다."
❌ 주의: "차단 옵션 켜면 된다"로 끝내면 위험. **오탐=장애**라는 점과 Pass List/Suppress/오프로딩이라는 동반 작업을 같이 말해야 함.

**Q. 악성코드 탐지 룰을 만들 때 content를 어떻게 넣는 게 좋나요?**
✅ 좋은 답변: "평문 문자열은 인코딩·이스케이프·비출력 바이트 때문에 잘 빗나가서, **바이트를 그대로 박는 hex 매칭**이나, 알려진 검체면 **파일 해시(SHA256/MD5) 기반 매칭**을 씁니다. hex는 인코딩에 안 흔들리되 너무 짧으면 정상 트래픽까지 긁어 오탐이 나고, 해시는 파일 단위로 정확하지만 **1바이트만 바뀌어도 못 잡아** 변종(폴리모픽)에 약합니다. 그래서 시그니처(알려진 것)는 hex/해시로 잡고, 변종은 행위기반(EDR·이상탐지)으로 보완하는 이중 구조로 갑니다."
❌ 주의: "문자열 그대로 넣는다"는 약함. **왜 hex/해시인지**와 **각 한계(짧은 hex 오탐 / 해시 변종 회피)**를 짚어야 함.

**Q. VPN만 깔면 원격 접속이 안전한가요?**
✅ 좋은 답변: "아닙니다. VPN 게이트웨이는 **인터넷에 노출돼야** 터널을 받을 수 있어 그 자체가 공격 표면(위험 수용)이고, 터널이 뚫리면 내부가 통째로 노출될 위험도 있습니다. 그래서 ① 접속 인증을 **OTP(MFA)**로 강화하고, ② 들어온 트래픽을 **입구에서 SSL Inspection으로 복호화해 관제**하며, ③ 장비 펌웨어를 **PMS로 패치**합니다. 더 나아가 '터널을 깔아 망을 통째로 신뢰'하는 대신 요청마다 검증하는 **ZTNA**로 옮겨가는 추세입니다."
❌ 주의: "VPN으로 암호화하니 안전"은 함정. **VPN=공격표면**, **MFA·SSL Inspection·패치·ZTNA**까지 엮어야 함.

## 연결되는 기존 개념

| 오늘 개념 | 연결되는 기존 개념 | 파일 | 어떻게 연결되나 |
|---|---|---|---|
| Block Offenders(IDS→IPS) | IDS=탐지만 / IPS=탐지+차단 | Security-Solutions | 체크박스 하나로 성격이 갈림 |
| Legacy vs Inline 모드 | 인라인 vs 미러링 | Security-Solutions | Legacy=미러링(사후), Inline=인라인(실시간)의 구현 |
| 체크섬 오프로딩 OFF | (신규) 인라인 netmap 전제 | Security-Solutions | NIC이 뒤늦게 손대는 패킷 ↔ 인라인 검사 충돌 |
| Suppress / Pass List | IDS 룰 튜닝(설치 직후 오탐 폭주) | Security-Solutions | 6/8 "튜닝이 핵심"의 실제 도구 |
| Pass List도 Alias로 | Alias / "ip로 치는 건 하수" | Security-Solutions / Lab(6/12) | 관리 포인트 하나로 |
| SID 3000000+ | ET Open SID 2xxxxxx | Security-Solutions | 안 겹치는 빈 번호대 |
| content hex/hash | content는 바이너리(헥스)도 본다 | Security-Solutions | 평문 매칭의 한계 → 바이트/지문 매칭 |
| SKT 공문 룰(outbound) | 역방향(트로이목마) 룰 | Security-Solutions / Attack-Defense | $HOME_NET→$EXTERNAL_NET 실제 사례 |
| 해시는 변종에 무력 | 시그니처 vs 애노멀리 | Security-Solutions | 알려진 것은 시그니처, 변종은 행위기반 |
| VPN 터널+정책, OTP, SSL VPN | VPN(IPsec/SSL-VPN/Site-to-Site/위험수용) | Security-Solutions | 어떻게 연결되나로 한 겹 더 |
| 입구 SSL Inspection | SSL Inspection / 관제(SIEM) | Security-Solutions | 들어온 뒤 행위 가시성 = 복호화+관제 |
| Kali 공격기 투입 | (신규) 방어장비 검증 | Lab | IPS를 만들었으면 같은 사거리 공격기로 검증 |

## 오늘 헷갈렸던 것 / 짚어둘 것

- **인라인 IPS 전 체크섬 오프로딩 OFF** — VirtualBox는 거의 필수. 안 끄면 인라인 검사 꼬임. (TSO/LRO도 같이)
- **Block Offenders = IDS→IPS** — Legacy(사후 차단) vs Inline(실시간 drop).
- **IPS 켜면 오탐=장애** — Pass List(차단 면제) 먼저 깔고, 시끄러운 룰은 Suppress(알람 끄기). 목적이 다르다.
- **Pass List/룰 IP 직접입력은 하수 → Alias.**
- **커스텀 SID는 3000000+** (ET=2xxxxxx 회피).
- **content는 hex/hash** — 평문은 인코딩에 빗나감. 해시는 파일단위지만 변종엔 무력.
- **Kali·공격기는 고정 IP**(192.168.1.250) — 룰/로그 출발지 기준 일관성.
- **VPN은 터널+정책 2단** — 터널만 믿지 말 것. 입구 SSL Inspection + OTP + 패치(PMS) + ZTNA.
