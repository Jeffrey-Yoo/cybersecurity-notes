## 오늘의 복습 퀴즈

Q1. SYN Flooding이 OS를 마비시키는 원리를 backlog queue 개념으로 설명하라. 그리고 왜 "CPS가 빡세야" 효과가 나는가? — 서술형

Q2. 공격자가 출발지 IP를 **실제 존재하는 IP(2.2.2.2)** 로 위조하면 SYN Flood가 오히려 잘 안 먹힌다. 왜인가? — 흐름 이해

Q3. Snort 룰 `flags:S; threshold: type both, track by_src, count 20, seconds 2;` 에서 각 옵션이 뜻하는 바는? 정상 SYN과 공격 SYN을 무엇으로 구분하나? — 단답형

Q4. Anti-DDoS의 **syncookies** 가 정상 사용자와 위조 IP 공격자를 구분하는 메커니즘을 3-way 흐름으로 설명하라. — 서술형

Q5. **SNMP/NMS** 를 도입하는 핵심 이유를 6-23의 "골든타임"과 엮어 설명하라. NMS와 ESM은 어떻게 다른가? — 실무 적용

Q6. Zabbix 설치에서 `apt install zabbix-...` 전에 반드시 `wget …deb → dpkg -i → apt update` 를 먼저 하는 이유는? — 면접 대비

Q7. (FIMS 연결) FIMS에서 서비스가 살아있는지 직접 확인하던 걸, 이번 Zabbix는 어떻게 한 단계 끌어올렸나?

<details>
<summary>정답 보기</summary>

A1. OS는 반쪽 연결(half-open)을 담는 **backlog queue**(보통 300~400개)를 둔다. SYN만 보내고 마지막 ACK를 안 주면 이 큐가 미완성 연결로 가득 차 **정상 SYN도 큐에 못 들어가 거부**된다. 큐를 늘려도 한계가 있다. OS는 타임아웃으로 큐를 비워내므로, **CPS(초당 연결 시도)가 비우는 속도보다 빨라야** 큐가 마를 틈 없이 가득 차서 터진다.

A2. 서버가 2.2.2.2로 **SYN-ACK** 를 보내는데, 2.2.2.2는 SYN을 보낸 적이 없으니 "이게 뭐지?" 하고 **RST** 로 끊어버린다. RST를 받은 서버는 backlog에서 그 연결을 삭제 → **자원이 회복**된다. 즉 SYN Flood 성공 조건은 "SYN 이후 아무 응답도 안 오는 것" → 보통 라우팅 불가한 위조 IP를 쓴다.

A3.
- `flags:S` = SYN 플래그만 세팅된 패킷 매칭(S=SYN, A=ACK, R=RST, P=PSH).
- `threshold type both` = limit+threshold 결합(임계치 넘을 때 발동하되 알림 폭주 억제).
- `track by_src` = 출발지 IP 기준 카운트.
- `count 20 / seconds 2` = "2초에 같은 소스 SYN 20개 초과 = 공격".
- 정상 SYN과 공격 SYN은 패킷 **모양이 같다** → **개수(임계치)** 로 구분한다.

A4.
- 정상: 사용자→AD SYN, AD→사용자 SYN-ACK(seq=cookie) ← **AD가 서버 대신 응답**, 사용자→AD ACK(cookie+1) ← 정상이면 정확히 응답 → 검증 통과 → 그제서야 AD가 서버와 진짜 3-way를 맺고 데이터 push.
- 공격(위조 IP): SYN-ACK를 위조 IP로 보내도 **ACK가 안 옴**(응답 못 받음) → 3-way 미완성 → 서버로 절대 안 넘김 → 서버 backlog는 깨끗.
- 핵심: **서버는 검증 끝난 손님하고만 진짜 연결을 맺는다.**

A5. DDoS가 오면 회선·CPU가 들쑥날쑥하다 풀로 차는데, 풀로 차기 직전(**골든타임**)에 끊어야 한다. 그러려면 **평소 baseline(평균 트래픽·자원)** 을 알고 있어야 이탈을 감지한다 → 그게 SNMP로 수집해 NMS로 보는 것. **NMS ≠ ESM**: NMS=가용성/성능(장비 살아있나·자원 정상인가) 모니터링, ESM=보안 로그 상관분석(위협 있나). 결은 달라도 "흩어진 걸 한 화면에 모아 평소와 다른 걸 잡는다"는 철학은 같다.

A6. 그 `…deb`는 프로그램 본체가 아니라 **apt에 zabbix 공식 출처(저장소 주소+서명키)를 등록**하는 주소록이다. `dpkg -i`로 출처를 등록하고 `apt update`로 그 목록을 apt가 다시 읽어야, 그 다음 `apt install zabbix-...` 가 정품·최신 패키지를 신뢰된 경로로 당겨온다. 이 셋을 안 하면 install이 "패키지 없음"으로 실패한다.

A7. FIMS는 사람이 직접/스크립트로 살아있나 확인하는 수준이었다. Zabbix는 **SNMP/에이전트로 자동 수집해 한 화면(대시보드)에 시계열로 쌓고**, 임계치를 넘으면 알람까지 줄 수 있는 **상시 모니터링 인프라**로 끌어올린 것 — baseline이 그래프로 누적돼 골든타임 감지의 토대가 된다.

</details>

## 예상 면접 질문

**Q1. "SYN Flood는 어떻게 동작하고 어떻게 막나요?"**
✅ 좋은 답변: "3-way의 마지막 ACK를 일부러 안 줘서 반쪽 연결(half-open)로 OS의 backlog 큐를 가득 채우는 공격입니다. 큐(보통 300~400)가 차면 정상 SYN도 거부됩니다. 1차로는 IPS/Snort에 `flags:S` + 임계치(예: 2초에 소스당 20개 초과) 룰을 걸어 끊고, 본격적으로는 앞단 **Anti-DDoS의 syncookies** 로 막습니다. Anti-DDoS가 서버 대신 3-way를 먼저 맺어 cookie로 검증하고, 통과한 손님만 서버로 넘기기 때문에 서버 backlog 자체가 고갈되지 않습니다."
❌ 주의: "방화벽으로 IP 차단하면 된다" — 출발지가 위조라 단일 IP 차단이 무의미하다. 임계치/세션 검증으로 접근해야 한다.

**Q2. "syncookies와 First SYN Drop의 차이는?"**
✅ 좋은 답변: "둘 다 '미완성 연결을 서버 앞에서 검증'하는 발상입니다. syncookies는 Anti-DDoS가 **서버 대신 SYN-ACK(seq=cookie)를 응답해 정상 ACK(cookie+1)가 오는지** 확인합니다. First SYN Drop은 **첫 SYN을 일부러 버리고 TCP 재전송이 오는지** 봅니다(정상 클라이언트는 재전송함). 다만 First SYN Drop은 최근 OS의 재전송 타임아웃이 3초쯤으로 길어 정상 사용자의 첫 접속이 지연될 수 있어, syncookies를 기본으로 둡니다."
❌ 주의: "syncookies는 서버가 처리한다" — 핵심은 **Anti-DDoS(앞단)가 대리로** 맺어 서버 자원을 보호하는 것.

**Q3. "DDoS 골든타임을 잡으려면 평소에 뭘 해둬야 하나요?"**
✅ 좋은 답변: "SNMP 기반 NMS(예: Zabbix)로 회선·CPU·세션을 상시 모니터링해 **평소 baseline** 을 그래프로 쌓아둡니다. 공격이 오면 그 평균에서 급격히 벗어나는 순간을 임계치 알람으로 잡아, 회선·CPU가 100%를 치기 전(골든타임)에 대응합니다. NMS는 가용성/성능을 보는 것이고, 보안 위협 상관분석은 ESM/SIEM이 담당해 둘을 함께 운영합니다."
❌ 주의: "NMS랑 ESM은 같은 거다" — NMS=성능/가용성, ESM=보안 로그 상관분석. 역할이 다르다.
