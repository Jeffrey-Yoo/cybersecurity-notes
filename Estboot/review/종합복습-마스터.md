## 2. 네트워크 기초

### 2.1 TCP/IP 4계층 & 레이어 포함 관계

```
L7 응용   HTTP·DNS·FTP·SMTP   (OSI 5/6/7 묶음, 만들어진 데이터=페이로드)
L4 전송   TCP·UDP             (포트로 앱 구분, 신뢰성)
L3 네트워크 IP·ICMP·ARP        (IP로 경로 결정)
L2 데이터링크 Ethernet·Wi-Fi   (MAC으로 같은 망 안 전달)
```

- **상위 레이어 장비는 하위를 전부 포함**: `L4 ⊃ L3 ⊃ L2`. 그래서 가격도 `L2 100원 < L3 1,000원 < L4 10,000원`.
- **NAT·VLAN 분리는 L3 이상부터.** L2 깡통 허브는 매니지먼트가 없어 VLAN·ACL 불가. L2 매니지먼트 스위치는 VLAN은 되고, 상단 L3에서 ACL을 걸어야 제어된다.

> 비유: 레이어는 **택배 포장 박스가 겹겹이 든 것**. 보낼 땐 L7 데이터에 L4→L3→L2 헤더를 차례로 씌우고(**캡슐화**), 받을 땐 L1부터 벗긴다(**역캡슐화**). 와이어샤크 한 줄(Frame/Ethernet/IP/TCP/Data)이 이 박스를 펼쳐 보여주는 것.

> 🔗 FIMS: FastAPI 라우터(`/api/login`)=L7, Apache Let's Encrypt HTTPS 암호화=L6, JWT 8시간 세션=L5 발상.

### 2.2 L4 — TCP vs UDP

| | TCP | UDP |
|---|---|---|
| 신뢰성 | 시퀀스 번호로 재전송 → 유실 복구 | 재전송 없음, 헤더 심플·빠름 |
| 연결 | 3-way로 맺고 4-way로 끊음 | 비연결 |
| 용도 | 금융·거래·메시지(유실 치명적) | 게임·스트리밍(실시간 > 유실) |

- **L4의 정체성 = IP + 포트 둘 다.** "어느 장비(IP)의 어느 서비스(포트)냐". 둘 중 하나만으론 L4가 아니다.
- **3-way handshake**: `SYN → SYN+ACK → ACK` → ESTABLISHED. 시퀀스 번호 임의 결정 후 +1 응답.
- **TCP 연결 상태**: LISTEN → SYN_SENT → SYN_RECEIVED → ESTABLISHED → (FIN_WAIT/CLOSE_WAIT/TIME_WAIT) → CLOSED. `netstat -an`으로 확인.

> ⚠️ **SYN만 계속 보내고 ACK 안 줌 = half-open connection 폭증 = SYN Flood.** 서버 연결 테이블이 고갈돼 정상 요청을 못 받는다. Suricata 룰 `flags:S`로 탐지.

**TCP 플래그**: SYN(연결요청)·ACK(수신응답)·FIN(정상종료)·RST(강제종료)·PSH(버퍼 안 기다리고 바로 L7로, SSH/텔넷)·URG(긴급).

**세션 테이블(Stateful)**: 2천만 연결을 계속 물면 서버가 터지니, 잠깐 끊어도 세션 값을 기억해 재연결 시 그 서버로 붙여준다.
> 비유: 단골 식당이 잠깐 비운 내 자리를 기억해두는 것. → **Sticky 세션** = 로그인 세션 안 날아가게 늘 같은 서버로.

**DNS 라운드로빈 vs L4 LB (핵심 차이)**: DNS RR은 헬스체크가 없어 죽은 서버에도 보냄. L4 LB는 서버를 "자원"으로 등록해 죽으면 자동 제외. → 그래서 DNS RR은 단독으로 못 쓰고 LB와 함께.

**VIP(가상 IP)**: 사용자는 VIP로 접속 → 방화벽 DNAT(1:1)으로 실제 서버 IP에 매핑 → L4가 VIP 하위 서버들에 분배. `nslookup 도메인` 했을 때 나오는 IP가 보통 이 VIP(실제 서버 IP 아님).

### 2.3 L3 — IP / ICMP / ARP

```
데이터 보내기:  목적지 IP 확인(IP) → 그 IP의 MAC 알아내기(ARP) → 경로 살았나(ICMP) → 배달(IP)
```

- **ICMP는 우선순위 낮음**: ping 느리다고 서비스 느린 게 아니고, ping 막혀도(방화벽이 ICMP만 drop) 서비스는 동작. **ping 안 된다고 서버 죽은 게 아님.**
- **ARP**: ① ARP 테이블 검사 → ② 없으면 브로드캐스트("이 IP 누구야?") → ③ 해당 장비 유니캐스트 응답("나야") → ④ MAC 획득.

> 비유: IP는 "서울 강남구 OO동"(주소), MAC은 "302호 김철수"(받는 사람). 동네까진 IP로, 마지막 문 앞엔 MAC으로.

> ⚠️ **ARP는 Reply의 MAC을 검증하지 않는다** → ARP 스푸핑의 근본 원인(8장 참고).

### 2.4 L2 / L1

- **MAC** = 장비 고유주소(하드웨어 고정이지만 SW로 변경 가능 = **MAC 스푸핑**). 퇴사자 PC의 IP+MAC을 폐기하는 이유.
- **스위치 vs 허브**: 허브=깡통(모든 포트로 뿌림), L2 스위치=MAC을 포트에 매핑 → 미등록 MAC이면 포트 DOWN = **PORT SECURITY**.
- **NAC**: 접속 기기의 보안 상태(백신·패치·인가 여부) 검사 + MAC 충돌 탐지. MAC 필터링만으론 약함 → **802.1X 인증**과 함께 써야 완결. NAC 예산 부족하면 PORT SECURITY로 대체.
- **Full Duplex vs Half Duplex**: 전이중=송수신 동시(왕복 2차선), 반이중=한 번에 한 방향(외길 1차선, 무전기). "1기가 통신"=보통 Full Duplex 1Gbps.
- **ThroughPut**: 단위 시간당 실제 처리량. 이론 대역폭 ≠ 실제 처리량. 검사 레이어 깊어질수록(L3→L7) 떨어짐 → 고성능 보안장비 핵심 스펙.

> 비유(ThroughPut): 10차선 고속도로여도 톨게이트 검문이 빡세면 실제 통과 차량은 준다.

### 2.5 DNS

```
로컬 DNS → (없으면) Root → TLD(.com/.org) → 도메인 DNS → IP 반환 → TTL 동안 캐시
```

- **Split DNS**: 내부만 알아야 할 IP는 내부 DNS, 외부 공개는 외부 DNS로 분리.
- **포트**: 보통 **UDP 53**, 응답 512바이트 초과·Zone Transfer 시 **TCP 53**.
- **DNS 스푸핑**: 캐시 변조/`hosts` 파일(`C:\Windows\System32\drivers\etc\hosts`) 변조로 가짜 사이트 유도.

> 🧠 **DNS = 이름→IP 변환(전화번호부)일 뿐, 통화 허가(방화벽)와 별개.** 이름 풀려도(dig 성공) 방화벽 막으면 접속 안 됨 — 트러블슈팅 때 분리해서 본다.

### 2.6 VLAN

**세그멘테이션(FIMS) vs VLAN**
```
세그멘테이션 → VirtualBox 내부 네트워크 이름 기반 = 소프트웨어 격리
              하이퍼바이저 취약점 뚫리면 격리 붕괴 가능
VLAN(802.1Q) → 물리 스위치 하드웨어가 태그로 분리 = SW 취약점과 무관
              다른 VLAN끼리 통신하려면 반드시 L3(라우터/방화벽) 경유
```
> ⚠️ 면접에서 FIMS를 "VLAN 망분리"라 하면 틀림. **"VirtualBox 내부 네트워크로 세그멘테이션 했다"**가 정확.

- **액세스 포트 vs 트렁크 포트**: 액세스=한 VLAN만(단말 끝단, 태그 없음), 트렁크=여러 VLAN을 태그 달아 한 선에(스위치↔스위치/라우터). 다른 VLAN 통신은 트렁크 타고 L3까지 가야 라우팅.
- **ESXi VLAN — 포트그룹 VLAN ID로 모드 결정**:

| VLAN ID | 모드 | 누가 태그를 다루나 |
|---|---|---|
| 0 | EST | 외부 물리 스위치 |
| 1~4094 | VST | ESXi vSwitch가 태깅(포트그룹마다 고정) |
| **4095** | **VGT(트렁크)** | **게스트 VM이 직접** (모든 태그 통과) |

> 🧠 **4095 = VGT = 트렁크.** pfSense를 VLAN 라우터로 쓰려면 그 포트그룹을 4095로 열어 모든 태그를 통과시켜야 pfSense가 10/20 태그를 직접 보고 라우팅한다. VST로 고정하면 vSwitch가 태그를 벗겨 게스트가 VLAN 구분 못 함.

- **ACL**: VLAN=망을 나누는 것, ACL=나눠진 망끼리 이동 통제. 표준 ACL(출발지 IP만), 확장 ACL(출발지+목적지+포트, 실무 주력).

## 3. 방화벽

### 3.1 세대별 발전 (왜 1→2→3으로 갔나)

**1세대 패킷필터링(L3~L4)** — IP+포트만 본다.
- "L3+ACL로도 되는 거 아냐?" → 순수 차단은 흉내 가능, 하지만 **망 제어(망분리)를 못 해서** 방화벽 대체 불가. **ACL은 접근제어지 망분리가 아니다.**
- 진짜 한계: 세션 개념이 없어 응답을 받으려면 **역방향 ACL을 수동으로 한 줄 더** 넣어야 함. 응답 출발지 포트가 매번 임의(ephemeral)라 어쩔 수 없이 **출발지 포트를 any로** 열게 됨 → 그 틈으로 무분별 유입.

**2세대 Stateful Inspection(L4)** — 세션을 기억한다.
- 내가 시작한 세션으로 돌아오는 패킷(그 임의 포트)만 자동 허용, 세션 닫히면 같이 닫힘(휘발성).
> 비유: **가제트 형사 팔.** 내가 팔(세션)을 뻗어 문을 따고 나가면 응답은 그 팔을 타고만 돌아온다. 팔 거두면 통로도 사라짐.
- ⚠️ 강사 홈 방화벽 강조: 기본은 전부 **Deny**, "양방향 체크"를 함부로 누르면 1세대의 무분별 유입으로 회귀.

**3세대 NGFW(L7)** — 페이로드 안 내용까지 본다.
- IP가 수시로 바뀌는 서비스(N드라이브 등)도 **애플리케이션 단위로 식별·차단**(Palo Alto의 `daum-cloud`/`daum-mail`).
> 한 줄: 1세대=봉투 주소·포트 → 2세대=누가 먼저 말 걸었나(세션) → 3세대=봉투 뜯어 내용까지. **점점 깊이 본다.**

**레이어별**: L2(TP모드, IP 없이 투명) / L3(IP 기준) / L4(IP+포트만 = IP 변조 못 잡음) / L7(페이로드 문자열 필터, pfSense가 대표 L7 UTM).

### 3.2 정책 운영

- **블랙리스트(Default Allow, 차단할 것만)** vs **화이트리스트(Default Deny, 허용할 것만)**. 실무 인바운드=화이트리스트, 보안 중요 서버망=양방향 화이트리스트.
- **정책 철칙**: 양방향 한 번에 체크 ❌. **출발지/목적지/포트 명시한 단방향 룰을 양쪽 따로** ✅.
- **룰 처리 순서 = Top-Down, first match**: 처음 매칭되는 룰에서 멈춤.
  - 특정 **차단룰은 최상단**(허용룰 아래 있으면 무력화).
  - **트래픽 많은 룰일수록 위로**(매번 끝까지 안 훑음 → 성능). 룰 수천 개면 순서가 곧 처리 속도.
- **운영 실무(금융권)**: 신청→검토→반영 프로세스. 신청 시간대 고정(10/13/17시), 보통 2주 전 신청~3일 전 마감. ⚠️ **"이럴 것 같으니 미리 열자"는 금지** — 정확한 출발지/목적지/포트 확인 후 개방.
- **Deny 로그**: 방화벽은 L3에서 패킷을 직접 제어 → 차단된 트래픽은 **방화벽에서만** 보임. DB 무단 접근 시도 추적의 단서.

**UFW 핵심**:
```bash
ufw status numbered / verbose
ufw allow 포트/프로토콜  ·  ufw deny 포트/프로토콜  ·  ufw delete 번호
ufw allow from IP대역 to any port 번호
ufw default deny incoming / outgoing
```
> ⚠️ 위→아래 순서. **좁은 대역 ALLOW를 넓은 대역 DENY보다 위에** 둬야 함.

### 3.3 방화벽 디자인 — A/B/C/D 코스

방화벽 한 대가 인터넷/DMZ/내부망을 인터페이스로 가를 때, 구간별 방향으로 정책 설계.

| 코스 | 방향 | 설명 |
|---|---|---|
| A | 인터넷 → DMZ | 외부 사용자가 서비스(웹)로 들어옴 |
| C | DMZ 서버 → 인터넷 | 서버가 밖으로(업데이트 등) |
| D | 내부망 → 인터넷 | 직원이 밖으로 |

- **DMZ는 공인 IP와 1:1(SNAT/DNAT) 매칭** → 외부가 직접 때릴 수 있음 → SSH(22) 같은 인바운드 경로를 **deny** 필수. 반면 직원 내부망은 DNAT 매칭이 없어 밖에서 못 찾아오니 인바운드 차단을 따로 안 박아도 됨.
- **C코스**: 기본 all deny + 업데이트 목적지만 L7으로 콕(리눅스 repo·패치 서버). **Squid 포워드 프록시**로 "repo permit, google/naver deny" 화이트리스트.
- **D코스**: 화이트리스트가 최선. 경우에 따라 아웃바운드에 IPS·SSL Inspection. **감염 PC의 C2 통신을 나가는 길에서 차단.**
> 💡 인바운드만 막으면 절반. **나가는 길(아웃바운드)도 설계 대상.**
> 💡 "퍼지(purge)를 날린다" = CDN(CloudFlare)에서 오리진(DMZ) 내용이 바뀌면 엣지 캐시를 비워 다음 요청 때 최신본을 다시 받게 하는 것.

### 3.4 NAT — DNAT / SNAT

```
DNAT = 들어오는 방향. 공인 IP → 내부 사설 서버로 꽂아줌 (외부 진입용)
SNAT = 나가는 방향. 내부 사설 IP → 공인 IP로 바꿔 내보냄 (외부 egress용)
```
- **DNAT**: 노출 서버 N대면 공인 IP도 N개 → 사용자가 공인 IP로 들어오면 VIP로 떨구고 그 밑 서버들로 LB.
  - 물리 L4 스위치 LB(경로 위에 실제로 있음) vs **LVS/AWS ALB**(방화벽 옆 선상에 논리적으로, 바인딩 목록 보고 던짐).
- **SNAT**: 다 하나의 공인 IP로 나가면 관리는 편한데 **사고 시 "누가 나갔나" 추적 불가**(전부 같은 IP로 찍힘) → 위험 구간은 SNAT를 쪼갠다.
> 🔗 이 SNAT 추적 불가 문제가 **IDS 음영지역(ISP→TAP→WAN에서 NAT가 출처를 지움)**에서 그대로 발목을 잡는다(4장).

### 3.5 pfSense 실전 (NAT 포워딩 / Alias)

- **NAT 포트포워딩(DNAT)**: 외부가 방화벽 WAN 포트를 치면 내부 서버 IP·포트로 목적지를 바꿔 전달.
  > ⚠️ **Dest. Address = 방화벽 WAN IP**(들어오는 입구 기준), 보낸 사람 IP 아님. 출처 거르려면 Source 쪽에. NAT(주소변환)와 Filter(통과허용)는 **별개 단계** — 포워딩만 하고 허용룰 빠뜨리면 막힌다(NAT 룰 추가 시 WAN 허용룰 자동 생성 체크).
- **Alias**: 반복되는 IP·포트를 별칭 하나로 묶어 **한 곳만 고쳐 모든 룰 반영**.
  > 🧠 프로그래밍의 **매직넘버를 상수로 빼기**와 같은 발상. 룰 수천 개 실무에선 이게 곧 유지보수성. (ADMIN_HOSTS / WEB_PORTS / INTERNAL_NETS)

### 3.6 L2 브릿지 방화벽 (TP모드)

NAT 없이 두 장비 사이에 IP 없이 투명하게 끼어 모든 트래픽 제어. 기존 구조·설정을 안 바꾸고 끼울 수 있음.
- 언제: 1차 방화벽으로 외부는 막았는데 **내부망 길목에 이중 보안**을 덧댈 때(DMZ→내부).
- **한계**: IP가 없으니 L3 라우팅·NAT·망확장·내부망 망제어 불가. 장애 시 **하드웨어 바이패스 필수**(라인 위에 투명하게 끼어 죽으면 통신 단절). → **1차 방화벽 대체 불가.**
> 배치 방식(어떻게 끼냐)과 검사 기능(뭘 보냐)은 별개. TP모드도 안에선 L4/L7 검사.

- **심층 방어(Defense in Depth)**: 1차 F/W(UFW) → TP 브릿지 F/W → VLAN+ACL. 한 겹 뚫려도 다음 겹.

## 4. 탐지 장비

### 4.1 IDS vs IPS & 인라인 vs 미러링

| | IDS (탐지만) | IPS (탐지+차단) |
|---|---|---|
| 기능 | 패킷 감시·탐지·로그(차단 X) | 차단(인라인), DDoS·시그니처 차단 |
| 배치 | **미러링**(옆에서 복제만) | **인라인**(라인 위, 반드시 통과) |
| 운용 | **러프하게**(놓침 없이 넓게 탐지) | **보수적**(오탐=장애니 신중) |
| 대표 | Suricata, Snort | — |

> 비유: IPS=검문소(차 세워 검사), IDS=길가 CCTV(지나가는 거 녹화). CCTV 고장났다고 도로 안 막듯, IDS는 죽어도 서비스 영향 없어야.

- **왜 IDS를 라인에서 빼나**: 로깅 장비라 인라인에 둘 이유 없고, 인라인이면 장애 시 라인 절체=다운타임. 복제(미러링)면 넣었다 뺐다 자유.
- **다 인라인에 욱여넣으면 안 되는 이유**: 차단·제어를 다 라인 위에서 하면 **오탐 한 번에 정상 트래픽까지 끊김=재앙.** → 차단 장비(FW/IPS)만 인라인, 탐지 장비(IDS)는 미러링.
- **IPS+IDS 같이 쓰는 이유**: 회사마다 시그니처 DB가 달라 서로 못 잡은 걸 잡음(알약 이중 엔진). ⚠️ 강사 지론: **IPS·IDS는 서로 다른 벤더**를 써야 시그니처 베이스가 안 겹쳐 미탐이 준다.
- **IPS 멀티 세그먼트**: 3대 사는 게 아니라 1대로 여러 구간 동시 감시(망별 세그먼트를 받아 각각 정책).

### 4.2 TAP / SPAN — 트래픽 복제

- **TAP(Test Access Point)**: 본선은 그대로 흘리고 복제만 뜨는 **수동** 장비(전기적 개입 X → 죽어도 통신 멀쩡). ⚠️ 단, 라인에 끼우는 **설치 순간엔 단선 1회**.
- **SPAN/포트 미러링**: 스위치 자체 기능으로 복제(시스코=SPAN, 델/주니퍼=포트미러링). 출발지 포트들 → 목적지 포트 하나(g6)로 미러 → 거기 IDS. **Tx and Rx = 양방향 복제.** IDS가 수집하는 포트 = **센서**(계속 늘릴 수 있음).
- **트레이드오프**: 스팬=간편하지만 스위치가 복제 일을 더 해 **발열·성능 저하·rx/tx 에러** → 그래서 TAP. / TAP=본선 무영향이지만 설치 시 단선.
- **음영지역**: TAP·스팬은 **트래픽이 지나가는 길목**이 있어야 복제. 무선은 공유기에 직접 붙어 유선 길목을 안 거침 → 안 보임.
  - 무선까지 보려고 ISP→TAP→WAN으로 떠도 **NAT(SNAT)로 출처가 공인 IP 하나로 뭉개짐.**
  - 강사 해법: 라우터에서 무선을 떼어 **별도 L3로 빼서 유선 길목을 인위로 만들고** 거기 TAP.
> ⚠️ 진짜 필요한 능력: 어디 음영이고 길목이 어딘지는 **회사 내부 서비스·네트워크 흐름을 정확히 알아야** 나온다. 보안만 잘해선 안 됨.

### 4.3 NIDS vs HIDS

| | NIDS | HIDS |
|---|---|---|
| 위치 | 네트워크(미러링) | 각 호스트에 에이전트 |
| 장점 | 전체 흐름 한 군데서 | 호스트 상세 + 사용자 단위 분석 |
| 단점 | 경유한 공격만 보임 | 개별 관리 부담 + ⚠️**호스트 뚫리면 해커가 IDS를 꺼버림** |

> ⚠️ SKT처럼 서버 4만 대면 HIDS 전수 설치는 비현실 → NIDS로 통으로 보고 핵심 서버만 HIDS 선별.
> 🔗 FIMS: Wazuh=HIDS 성격(에이전트 로그 수집), Suricata=NIDS 성격(네트워크 트래픽).

### 4.4 탐지의 두 패러다임 — 시그니처 vs 애노멀리

**① 시그니처 기반** — 몽타주랑 대조.
- 알려진 공격 패턴(코드 문자열·바이트·공격 순서)을 DB와 대조. 장점=아는 공격 빠르고 정확(오탐 적음). 한계=처음 보는/변종·난독화는 못 잡음.

**② 애노멀리 디텍션** — 평소랑 다르면 의심.
```
학습 모드(baseline) → 통계 모델링 → 오탐 관리 → 패턴 재정의
```
- 장점=제로데이도 "평소와 다름"으로 잡음. 한계=정상 폭 넓거나 학습 부족하면 **오탐 폭발**.
- **Threshold(임계치)** = 방아쇠. 낮으면 오탐, 높으면 미탐. (IPS DDoS 설정의 SYN 500/s, CPS 100/s, Connection 25가 바로 이 Threshold)

> 한 줄: **시그니처=아는 공격을 정확히 / 애노멀리=모르는 공격을 평소와의 차이로.** 대체가 아니라 **보완**.

**Snort/Suricata 룰 한 줄**:
```
alert tcp any any -> any any (msg:"..."; content:"testmyids.com"; sid:100001; rev:1;)
액션  프로토콜 출발지  방향 목적지   옵션(content=페이로드에서 찾을 문자열이 핵심)
```
- content는 문자열뿐 아니라 **바이너리(헥스)** 도: `content:"|00 0C 23 45 53|"`. ⚠️ 짧은 바이너리는 정상 패킷에도 우연히 매칭 → **오탐**.
- **방향으로 잡기**: 웹 공격은 `$EXTERNAL_NET -> $HOME_NET`(밖→안), **트로이목마 C2는 `$HOME_NET -> $EXTERNAL_NET`(안→밖)** — 감염 PC가 밖으로 거꾸로 말 거는 게 시그니처.
- 패턴 매칭 장점=정확히 탐지, 단점=오탐 → **룰 튜닝(필요 시그니처만 남기고 오탐 대상 제거)이 운영의 핵심.** (설치 직후 Snorby에서 MEDIUM 61만 건 폭발이 정상)

### 4.5 Anti-APT (샌드박스) & Darktrace

- **Anti-APT**: IPS가 **알려진** 공격을 막는다면, Anti-APT는 **알려지지 않은** APT를 잡음. 의심 파일을 **격리된 VM(샌드박스)에서 실제 실행**해 행동(파일 생성·레지스트리 변조·C2 통신) 관찰 = **동적 분석**.
  - 정적 분석(실행 X, 코드·해시·구조) vs 동적 분석(실행 O, 행동). 난독화·패킹하면 정적은 안 보임 → 동적으로.
  > 비유: 폭발물 처리반이 의심 가방을 방폭 컨테이너(VM)에서 열어봄. 끝나면 컨테이너(스냅샷) 비움. → EDR 행위 기반 탐지와 같은 발상.
  > 🔗 아키텍처 그림에서 라인 옆에 빠져있던 "**분석**" 박스의 정체.
- **Darktrace (NDR, AI 자가학습)**: 시그니처를 안 쓰고 **내부 전체를 통째로 학습**해 "self(정상)"를 모델링, 거기서 벗어나는 미묘한 이상을 잡음(내부자·제로데이). 인체 면역체계 비유. 탐지 모드(실시간 차단) + 분석 모드(사후 수사). **"앞으로 보안이 가야 할 방향."**
  > 🔗 FIMS Wazuh의 연관분석이 "분석"의 초기형. 거기에 자가학습 AI가 붙으면 Darktrace 방향.

### 4.6 관제 (SIEM)

각 장비 로그를 따로 보면 의미 없음 → 한 곳(관제 서버)에 모아 **연관 분석(Correlation)**.
```
IPS에서 IP 하나 → 별 의미 없음.  근데 WAF·내부 IDS에서도 같은 IP → "이상하다" → 추적
```
> 🔗 FIMS **Wazuh가 바로 이 관제 서버 역할.**

## 5. 그 외 보안 솔루션

### 5.1 UTM & 오픈소스 스택

방화벽+IDS/IPS+안티바이러스+DLP를 장비 하나에 통합.
```
스타트업 → pfSense(무료)   중소 → FortiGate 중저가(100~300만)
중견 → FortiGate 고급/Palo Alto   대기업/금융 → Palo Alto+Cisco 풀스택+물리 망분리
```
- **오픈소스 스택**: pfSense(BSD 방화벽/UTM) / OPNsense(pfSense fork, UI·업데이트 활발) / **PacketFence**(오픈소스 NAC) / **ClamAV**(오픈소스 백신).
- **다운로드 무결성**: 공식 사이트 + 받은 파일 해시를 게시값과 대조(`sha256sum`, `Get-FileHash`). ⚠️ **MD5는 충돌 위험 → SHA-256 이상.**

### 5.2 WAF

- 웹 공격 특화(HTTP 전용): SQLi·XSS 등 탐지·차단. **IPS에서 갈라져 나옴**(IPS=전체 프로토콜, WAF=HTTP 전용). 웹 서버만 있으면 WAF로 충분, 소켓 서버(메신저) 같이 있으면 IPS도.
- ⚠️ 요즘은 **차단 페이지 안 띄움** — WAF 존재·벤더가 노출되니까(공격자에게 정보 안 줌).

### 5.3 VPN

공용 인터넷 위에 **암호화 터널**을 깔아 사설망처럼 연결.
```
IPsec  → L3(패킷 자체) 암호화, 게이트웨이↔게이트웨이로 망째 묶음 → Site-to-Site(본사↔지사)
SSL-VPN → L7(TLS) 암호화, 개인 단말이 클라이언트/브라우저로 접속 → Remote Access(재택)
```
- **Site-to-Site = 허브-스포크**: VPN 본점(본사 허브)에 VPN 지점(지사 스포크)들이 모임. 중앙 우체국 비유.
- ⚠️ **VPN은 그 자체가 공격 표면.** 게이트웨이가 인터넷에 얼굴을 내놓아야 터널을 걸 수 있음 → **VPN 결함(CVE) 익스플로잇 → 터널 타고 내부 침투**(Fortinet/Ivanti/Pulse가 초기 침투 단골). 그래서 "노출 위험을 **수용(Risk Acceptance)**하고 쓰는 것" → 펌웨어 패치(PMS)+MFA 필수.
- 한계(터널만 뚫리면 내부망 통째 노출)를 보완하는 게 **ZTNA**(요청마다 검증).
> 🔗 FIMS WireGuard 터널이 정확히 이 발상. WAS는 VPN 없어 AWS에서 직접 못 갔음.

### 5.4 EDR / 백신

기존 백신은 시그니처 기반이라 신규 악성코드 못 잡음 → EDR은 **행위 기반 탐지** + TI(Threat Intelligence) 점수화로 AI 자동 분류.
```
EDR 행위 탐지 → 시그니처 추출 → 백신에 전달 → 이후 동일 패턴 자동 차단 (이중 구조)
```

### 5.5 DLP (정보 유출 방지)

- **NDLP(Network)**: 네트워크 길목(F/W→백본→메일서버)에서 탐지.
- **HDLP(Host)**: 사용자 단말 에이전트로 중요 정보 탐지(담당자 무단 반출 방지).
> 엔드포인트만으론 부족(에이전트 삭제 가능) → 둘 결합해야 완결.
- **NDLP는 보통 관찰용**: 메일서버 사례에서 코어 상단 TAP으로 복제만 받음(발신에 담당자 승인이 있어 차단까지 불필요, 인라인이면 오탐 시 정상 메일 끊김). → IDS를 미러링으로 빼는 논리와 동일.

### 5.6 나머지 솔루션 빠른 정리

- **DRM**: Agent로 파일 암호화 + 라이센스 키 있어야 열람. 라이선스 서버 다르면 키가 달라 타 기업과 공유 불가.
- **ECM**: 문서 중앙 집중·강제 저장·접근권한 관리. DRM과 같이 쓰면 효율↑.
- **PMS(패치 관리)**: 제로데이·APT가 미패치 환경을 노리니 수백~수천 대를 원격 패치. 폐쇄망은 `[인터넷]→[패치수집(DMZ)]→망연계→[내부 배포]→엔드포인트`.
- **WIPS**: 비인가 무선장치 차단, 인가 Wi-Fi만 허용, Jamming으로 SSID 식별 방해.
- **Anti-DDoS**: 맨 앞에 둠(대량 트래픽 먼저 걸러야 뒤 장비 안 죽음, 고성능 필요). L7까지 읽어 HTTP Flood도. 없으면 IPS Threshold로 임시 대체(단, IPS 뻗을 위험).
- **SSL Inspection**: NGFW가 암호화 트래픽을 복호화→검사→재암호화.
- **허니팟**: 공격 유도용 미끼 서버, 공격자 패턴 파악.
- **인라인/바이패스**: 내장 바이패스(전원 꺼지면 자동 직결, 사고 대비) / 외부 바이패스(계획된 교체·점검·펌웨어 업데이트 때 수동 전환 → 무중단 작업).

**보안 분야 지도**: 모의해킹·침해대응·보안컨설팅·악성코드분석·보안솔루션개발·디지털포렌식·보안감사(ISMS-P).

## 6. 아키텍처 & 고가용성

### 6.1 아키텍처 종류 & 설계

- **EA**(기업 IT 전략, ISP 수립, AS-IS→TO-BE) / **AA**(앱 배치) / **SA**(솔루션 선정) / **DA**(데이터 흐름·저장·권한) / **TA**(기술 구현·회선·이중화).
- **설계 6단계**: 서비스 정의 → 요건 정의(외부/내부) → 구성요소 도출 → 데이터 흐름 → 보안 레이어 → 고가용성.
> 서비스 특성이 인프라를 결정: 게임→CacheDB, 금융→EAI, 검색→캐시+검색엔진.

### 6.2 3-Tier & 장비 순서

```
인터넷 → 보안장비(IPS/WAF/DDoS/WIPS) → DMZ(Web, 외부 접점) → 내부망(WAS 로직) → DB(외부 직접 접근 X)
```
- **장비 순서**: F/W(L4) → L3 라우터 → [TP F/W] → LB → L2 스위치 → 서버.
- **L3 → LB 순서인 이유**: L3=어느 네트워크로(서울 어느 구), LB=그 안에서 어느 서버로(그 구 안 어느 집). LB가 먼저면 경로도 모르고 분배.
- **보안 장비 배치 = 큰 것부터 거르고 정밀하게**: Anti-DDoS(굵은 체) → 방화벽(중간 체) → IPS(미세 체) → Anti-APT → WAF(거름종이, HTTP) → L3 → [IDS는 미러링으로 옆에] → 서버.

### 6.3 요건 기반 배치 (정답 배치는 없다)

세 질문: ① 가장 필요한 보안 시스템? ② 가장 위협되는 대상? ③ 보호할 자산?
- 서비스마다 받는 공격이 다름 → 장비도 다름. 웹서버=WAF지만 **정적 페이지/CDN이면 과설계**일 수 있음. 개발팀이 진짜 표적일 수도(소스코드·내부 권한).
- **교집합으로 묶기**: 개발팀+그룹웨어 둘 다 인트라넷 → IPS 1대로 묶음(멀티 세그먼트).
- **같이 둘까 분리할까 = 스루풋이 결정**: 게임 2G + 웹 100G면 분리(병목), 둘 다 1G 안이면 같이 둬도 OK.
- **직원발 위협 → 네트워크냐 엔드포인트냐**: 피싱→EDR, 메일첨부→안티스팸, 악성코드→EDR/백신, RAT→IPS/Darktrace.
- **스루풋은 데이터시트로 계산**: 64Byte 성능 vs 최대 성능을 같이 봄(작은 패킷이 진짜 부하=플러딩에 가까움). 구간 트래픽 합 < 장비 스루풋.
> 면접 포인트: **모든 곳에 L7을 다 갖다 놓으면 오히려 이상하게 본다.** 역할에 맞는 레이어 배치가 현실 감각. **"틀린 아키텍처는 없다, 개선·최적화할 게 있을 뿐."**

### 6.4 고가용성 (HA)

- **왜 방화벽을 1순위로 이중화**: 모든 트래픽을 받고·제어하고·로깅하고·넘겨주니 가장 많은 일 → 가장 잘 죽음. (HA = High Availability라는 **설계 개념**이지 L3 구간 이름이 아님 — 면접 함정)
- **Active-Standby**: 주(Active)가 일하고 보조(Standby)는 세션 동기화받으며 대기. 주가 죽으면 **VIP를 보조로 넘김(failover)** → 사용자는 같은 VIP라 모름(무중단). ⚠️ 주/보조는 **동일 성능**이어야(세션 이어받을 때 스루풋 못 버티면 보조도 뻗음).
  - ⚠️ **언제 안 넘기나**: DDoS로 스루풋 100% 초과는 장비 고장이 아니라 공격 → 넘겨도 보조도 똑같이 뻗음 → 넘기면 안 됨. 그 외 정상 장애는 다 넘김.
- **Active-Active**: 두 대가 둘 다 일하며 세션 상호 동기화(양쪽 라인 다 깔아야 = 풀 메쉬). 장점=부하 분산, 단점=돈 많이 듦.
- **본딩/티밍**: 한 서버의 여러 NIC을 논리적으로 하나로 묶음(한 회선 끊겨도 지속 = 가용성, 대역폭 합쳐 처리량↑). 리눅스=본딩, 윈도우=티밍.

### 6.5 로드밸런서 & 파일공유

- **L4 LB**(IP+포트, 순수 분배·빠름) vs **L7 LB**(URL·쿠키·헤더, nginx/ALB). 라운드로빈, VIP, Session Table.
- **파일 공유 서버**: 글(메타데이터)→DB, 파일 자체→별도 스토리지(NAS). SMB/CIFS(윈도/리눅스 혼용), NFS(리눅스 전용). ⚠️ 주요 포트 **139(NetBIOS), 445(SMB)** 불필요하면 닫고 SMB 최신 유지.

### 6.6 메일 서버 구성도 (⭐ 2차 프로젝트 참고)

```
좌측(본선): 외부메일 → 스팸서버 → Mail Proxy(A-S) → L4 LB → Mail Main+백신(A-S) → Mail-DB → SAN → 스토리지
우측(DLP):  코어 상단 TAP → 복제라인 → 메일 모니터링 수집서버 → 로그서버(=DLP)
```
- 본선 핵심 장비는 Active-Standby 이중화. DLP는 본선에 안 끼우고 **TAP 복제로 관찰**(승인 흐름이 있어 차단 불필요).
- 로그서버 데이터는 내장 용량이 작아 **PTL → LTO(테이프)로 소산**.
- **엣지 메일서버**: 앞단에서 이미지·코드를 먼저 분석(안전 변환본만 담당자에 보여주고 승인) = CDR/콘텐츠 무해화 발상.
- **바이패스 Inline/Out-of-Band 구성도**: 차단 장비(FW/IPS/WAF)=인라인+바이패스, 분석 장비(포렌식/APT)=아웃오브밴드(미러링).

> ⚠️ **2차 프로젝트에서 보여야 할 것(설계 철학)**: 장비를 무자비하게 인라인으로 욱여넣지 말고, **"이 장비가 죽으면 무슨 문제가 오나"를 장비마다 가상으로 시뮬레이션**하고 그 고민의 흔적(이중화·바이패스·미러링 선택 근거)이 구성도에 드러나게. **배치는 정답이 아니라 장애 시나리오를 돌려본 의사결정.**

> ⚠️ 프로젝트 도식 주의: 클라우드 서버를 내부망 안쪽에 그리지 마라(클라우드는 외부). 내부→내부도 방화벽 타면 트래픽(내부 스루풋으로 캐파 산정). Stateful이니 S3 업로드 회선은 나가는 것뿐 아니라 **돌아오는 응답까지** 산정.

## 7. IT 인프라 & 망분리

- **인프라 유형**: 컨버지드(온프레미스) / 클라우드(빌림) / **HCI**(4대를 1개처럼, 슈퍼컴 개념).
- **데이터센터(IDC)**: Public/Hybrid/Owned. 보안통제 — 출입통제, 반입/반출 무결성, **Disk 폐기 Degaussing(반자기화)**, USB/CD 반출입 통제.
- **인터넷 회선**: ISP(=회선, 두 개 써서 가용성), **HCX**(클라우드↔IDC↔내 DC 물리 케이블 직결, 스니핑 불가), 국제망(해저 케이블), **DWDM**(광케이블 하나에 여러 파장).
- **OS**: Windows(사용자 중심, Close Source, EOL) / Linux(GNU 오픈소스, 서버 주력) / Unix(안정·금융, IA-64, 최근 U→L 전환).
- **가상화**: Hypervisor(한 머신을 여러 OS), HCI(VM들을 클러스터로 묶어 하나의 물리처럼).

**망분리** (법적 의무 — 전자금융감독·정보통신망법):
- **물리적**: 업무망/인터넷망을 물리 하드웨어로 분리. KVM(모니터·키보드만 공유, 데이터 안 넘어감).
- **논리적 SBC(서버 기반)**: 가상화로 서버 분리(VLAN), HCI 서버에 가상 데스크탑 화면 송출.
- **논리적 CBC(클라이언트 기반)**: 사용자 PC에 가상화 프로그램. 이식성 좋지만 서버 기반보다 보안 약함.

**망연계** (보안↔비보안 구간 데이터 이동):
- **데이터 다이오드**: 반대 통로가 물리적으로 없는 단방향(군·원자력).
- **CDG 연계서버**: 백신+콘텐츠검사+승인+로그(inbound=악성코드 검사, outbound=개인정보 유출 검사=DLP 역할). 가장 흔함. (벤더: 한싹, 휴네시온)
- **스토리지 중계**: 중간 창고 경유.

**백업**:
- **RTO**(얼마나 빨리 복구=복구 시간 목표) / **RPO**(얼마나 과거로=데이터 손실 허용 범위).
> **정책 수립 → 문제 발생 → 빠른 대처** (복구→문제→정책 ❌).
- **LTO**(장기 보관 테이프, 대용량·저렴·오프라인) + **PTL**(로봇 암으로 자동 적재). **Tape-Out / AIR Gap**(테이프 물리 분리 = 네트워크 원천 차단).

- **GitLab(사내, 소스 유출 방지, CI/CD)** vs **GitHub(외부, 유출·권한제어 어려움)**. **Docker Hub** 이미지는 프라이빗 레지스트리 + Pull 전 취약점 스캔.
- **CDN**: 정적 콘텐츠를 가까운 Edge에서 전달(낮은 레이턴시) + DDoS 완화. AWS CloudFront.

## 8. 공격 기법

- **Malware**: Ransomware(암호화 후 몸값), Spyware, Virus, **RAT(원격 트로이)**.
  - **RAT**: 화면을 스크린샷으로 계속 찍어 전송(OS 인터럽트=움직임 있을 때만 → 효율적, 들키기 어려움). → 화면에 띄운 순간 다 털린다는 전제로 **망분리·접근통제·로그** 필요.
  - 크랙 버전(MS오피스 등)에 악성코드 심김: **Updater, OrcusRAT, XMRig(코인 채굴)**. ⚠️ 악성코드만 지워도 **작업 스케줄러 재설치 루틴**이 남으면 또 깔림.
  - **CDK 추출**: 레지스트리 평문 키를 읽어 C2(CNC)로 전송. **키로깅**: 입력 전부 가로채 외부로.
- **웹 공격**:
  - Directory Listing(접근제한 없는 파일 무작위 침투), **SQLi**(`' or 1=1 --`로 인증 우회), **XSS**(`<script>` 삽입→쿠키 탈취), **Webshell**(.php 업로드→서버 조작, 리버스셸).
- **계정 탈취**:
  - **브루트포스**: 무차별 대입, 로그 많아 탐지 쉬움(access.log 인증실패 급증 + 세션 급증 = 상관관계).
  - **크리덴셜 스터핑**: 유출된 ID/PW 쌍을 다른 서비스에 1회씩 → 로그 적어 탐지 어려움(VPN IP 변조, 시간 분산, User-Agent 변경).
- **네트워크 공격**: MAC 스푸핑(NAC가 막음), DNS 스푸핑(hosts 변조), **ARP 스푸핑**(Reply MAC 미검증 악용 → 중간자), **Lateral Movement(수평 이동)**.
  - **수평 이동 시나리오**: DMZ 진입(ID/PW 확보) → 세팅값 들고 서버→서버 이동 → DB 도달 → C2로 반출. ⚠️ 평문 저장된 PW·경로가 RAT 한 번에 통째로 털림. **아웃바운드 차단으로 마지막 반출을 막는다.** 과정1에서 L3를 잘게 나눈 이유.
- **DDoS**: 집단 폭행(자원 고갈). Anti-DDoS로 맨 앞에서 차단. L7(HTTP Flood)은 L7 읽는 Anti-DDoS 필요.
- **시그니처 난독화**: "위키"→"위 키" 띄어쓰기·인코딩으로 시그니처 우회. 방어=문자열 조합 능력으로 오탐 없이 잡는 커스텀 시그니처.
- **방어 기법**: 포트 변경(자동 스캔 우회), Rate Limiting(FIMS slowapi 5분 5회), 세션 관리(JWT 8h), CORS 제한(`*` 아님), PrivateRoute. **SOAR**(보안 작업 자동화), **FDS**(이상 거래 감지).

## 10. 실습 랩 (ESXi · pfSense · VLAN · 삼바 · DNS)

### 10.1 환경 (6-9 ~ )

- Workstation Pro(타입2) 위에 **ESXi 8.0(타입1) 올리는 중첩 가상화**. 그 위에 pfSense·Ubuntu·Kubuntu 게스트.
- 내 홈: 공유기(192.168.1.1) — 본컴(.83) — NFS(.220, 본컴→서버컴 용량 추가) — 서브컴(.63) → ESXi(.200).
- ⚠️ **물리 NIC은 ESXi에서 못 더함** → 한 층 아래 Workstation에서 LAN Segment로 추가(ESXi는 그걸 '물리'로 인식). vSwitch=ESXi가 그 위에 만드는 가상 스위치.

### 10.2 pfSense WAN/LAN (6-10)

- NIC1=WAN(em0, 브릿지 → 192.168.1.82), NIC2=LAN(em1 → 192.168.2.1). WAN은 MAC 대조로 식별, LAN은 스킵해도 자동 할당.
- ⚠️ **대역 충돌 삽질**: pfSense LAN 기본값(192.168.1.1)이 공유기 G/W와 똑같음 → **WAN/LAN은 반드시 다른 대역** → 콘솔 2번으로 LAN을 192.168.2.1/24 + DHCP(2.100~200).
- **DHCP = DORA**(Discover→Offer→Request→Acknowledge). 기기가 먼저 외치고 pfSense가 응답. ⚠️ **같은 포트그룹(브로드캐스트 도메인)**에 있어야 Discover가 닿음 → DHCP 안 받으면 포트그룹부터 의심.
- 트러블슈팅은 **아래에서 위로**: 내부통신 → ping → **pfSense 8번 Shell에서 직접 ping**(방화벽 자신이 닿나 = 라우팅 문제 / 룰 문제 갈림).

### 10.3 VLAN (6-11)

- **ESXi LAN 포트그룹 VLAN ID = 4095(VGT 트렁크)** → 모든 태그를 pfSense로 통과 → pfSense가 VLAN 10(운영 Web, G/W 10.10.10.1)·20(개발 Web_Dev, 10.10.20.1) 라우팅. ⚠️ 특정 ID로 고정(VST)하면 vSwitch가 태그 벗겨 VLAN 라우팅 불가.
- 운영/개발 VLAN 분리 이유: 개발 사고가 운영으로 전파되는 걸 끊음. 다른 VLAN은 pfSense 경유라 그 길목에 방화벽 룰 통제 가능.
- pfSense 트래픽 그래프: WAN(파랑)↑인데 LAN(빨강) 그대로면 **막힌 것**, 둘 다↑면 통과(패킷 캡처 없이 룰 동작 1차 검증).
- 최소설치 서버 필수 유틸: net-tools, iputils-ping, telnet(포트 도달성), dnsutils, bmon/nmon/htop, openssh-server.

### 10.4 삼바 · 내부 DNS · NAT (6-12)

**삼바(VLAN 30 파일서버 ↔ VLAN 20 webdev 마운트)**:
```bash
useradd -M -s /sbin/nologin sambauser   # 파일공유 전용(홈 없음·셸 없음)
smbpasswd -a/-e sambauser                # 삼바 전용 비번(리눅스 로그인과 별개)
chmod 770 /samba
# smb.conf: [samba] path/browseable=yes/writeable=yes/guest ok=no/valid users=@samba
testparm                                 # ★ 적용 전 문법검사
# 클라: cifs-utils, mount -t cifs, credentials 파일(chmod 600), fstab 자동마운트
```
- ⚠️ **오타 2종**: `browsable`→**`browseable`**(e), `valid user`→**`valid users`**(복수). 서비스는 떠도 동작만 이상 → `testparm`으로 거르고 restart.
- ⚠️ **fstab엔 `-o` 없음**: 필드=공백, 옵션=콤마. (netplan YAML 들여쓰기=문법 / smb.conf INI 들여쓰기=가독성 / fstab 공백 구분 — 파서별로 규칙 다름)

**내부 DNS (BIND9, VLAN 40)**:
```
named.conf.options:
  allow-query { any; };                       # 질의는 열어도 됨
  allow-recursion { 10.10.0.0/16; localhost; }; # ★ 재귀는 내부망만 (오픈 리졸버 차단)
  forwarders { 8.8.8.8; 1.1.1.1; };
```
- ⚠️ **`allow-recursion`을 any로 열면 "오픈 리졸버"** → DNS 증폭 DDoS 발판. 질의는 열어도 재귀는 내부만(= 방화벽 "최소허용"의 DNS판).
- ⚠️ 존 파일 함정: **Serial은 수정 때마다 +1**(안 올리면 반영 X), **FQDN 끝 점(.)** 빠뜨리면 존 이름이 자동으로 또 붙음. 검증=`named-checkconf`/`named-checkzone`, 무중단 반영=`rndc reload`.

**NAT 포트포워딩 + Alias**: 위 3.5 참고. Dest.Address=WAN IP, 반복 IP는 ADMIN_IP Alias로.

## 11. FIMS 프로젝트 회고 (과정1)

> **FIMS** = 소규모 프랜차이즈에 대기업급 보안·운영 인프라를 통째로 주는 SaaS. `https://9haejo.duckdns.org`

```
인터넷 → AWS EC2(공인, Apache 리버스프록시+SSL · Wazuh · ETL · WG서버 · S3)
       ↓ WireGuard 터널
어플라이언스(Suricata·Wazuh Agent) / DB(MariaDB·socat 중계)
── IDC(VirtualBox 6대): 어플라이언스─30─DMZ─40─WAS─70─DB─80─Storage (관리망 56 공통)
```

**핵심 설계 포인트(면접에서 설명할 것)**:
- **VirtualBox 어댑터 3종**: NAT(인터넷 egress=apt), 내부 네트워크(서비스망 30/40/70/80, VM끼리만 = 망분리), 호스트 전용(관리망 56, 윈도우에서 SSH).
- **socat 중계 = 설계의 핵심**: WAS는 VPN 없어 AWS에서 직접 못 감 → DB만 내부망(70)+WireGuard(10.8.0.3) 양쪽을 아는 유일 서버 → DB에 socat으로 "DB:8000→WAS:8000" 중계. = **망연계 게이트웨이 발상**(단, 검사·승인·로그는 없음).
- **Apache 리버스프록시 + Let's Encrypt HTTPS**(구글 OAuth가 HTTPS 요구 → 도메인 → DuckDNS).
- **Suricata(IDS)**: `flags:S` + threshold로 포트스캔 탐지. **Wazuh(SIEM)**: Suricata→Agent→Manager 레벨 판정→webhook→앱 알림, 위험 IP는 blacklist 폴링 차단.
- **백업 3중화**: mysqldump→sha256sum→rsync(Storage 폐쇄망)→scp(AWS)→Groq AI 체크섬 판정→S3.
- **웹앱 보안 9가지**: HTTPS / JWT(64자,8h) / Rate Limit(slowapi) / 로그인 IP 기록 / CORS 제한 / 망분리 / WireGuard / S3 퍼블릭 차단 / PrivateRoute.

**대표 삽질**:
| 문제 | 원인 → 해결 |
|---|---|
| DB WireGuard 불통 | wg0.conf에 AWS 개인키 잘못 → DB 개인키로 ("개인키=신원") |
| socat exit-code 1 | 8000 좀비 점유 → pkill 후 재시작 |
| Suricata fast.log 빔 | 룰 경로 불일치 → /var/lib/suricata/rules로 복사 |
| 앱 알림 안 옴 | forwarder API키 ≠ WAS 키 → 일치 |
| Wazuh OOM | CVE DB 10GB+·tmp 누적 → tmp 삭제+vuln 비활성+Swap 2GB (97%→36%) |
| WAS 구글로그인 실패 | NAT 제거 후 oauth2.googleapis.com 접근불가 → NAT 어댑터 재추가 |

> **실무 연결 한 줄**: "오픈소스로 IDS-SIEM-차단 파이프라인을 직접 엮어봐서, 상용 NGFW/DLP 콘솔을 봐도 박스 안에서 무슨 일이 일어나는지 안다."

> 🔗 **FIMS ↔ 과정2 매핑**: 어플라이언스(Suricata)=NGFW/UTM SW판 / AWS 보안그룹+UFW=1차 방화벽(L4) / VirtualBox 내부망=세그멘테이션(VLAN 약한 버전) / WireGuard=VPN / socat=망연계 게이트웨이 / Wazuh=관제(SIEM)=Darktrace의 초기형 / Groq 체크섬=무결성 자동화.

<summary>정답 보기</summary>

**A1.** 1세대는 세션 개념이 없어 나간 요청의 응답을 받으려면 역방향 ACL을 수동으로 한 줄 더 넣어야 했다. 응답 출발지 포트가 매번 임의라 어쩔 수 없이 any로 열게 되고 그 틈이 위험. 2세대 Stateful은 **세션을 기억**해 내가 시작한 세션의 응답만 자동 허용하고 세션 닫히면 같이 닫는다(가제트 팔).

**A2.** ① 차단 안 하는 로깅 장비라 라인 위에 둘 이유 없고, 인라인이면 장애 시 라인 절체=다운타임. ② 다 인라인에 욱여넣으면 오탐 한 번에 정상 트래픽까지 끊김=재앙. 복제(미러링)면 죽어도 서비스 영향 없고 넣었다 뺐다 자유.

**A3.** 시그니처=알려진 공격 패턴을 DB와 대조(처음 보는/변종 못 잡음), 애노멀리=정상 baseline 학습 후 벗어나면 의심(제로데이는 잡지만 오탐이 약점). 대체가 아니라 보완.

**A4.** 트로이목마는 감염 내부 PC가 밖의 C2로 거꾸로(outbound) 말 거는 게 특징 → "안→밖" 방향 자체가 시그니처. 웹 공격(밖→안)과 방향이 반대.

**A5.** 세션을 동기화(클러스터)해 주가 들고 있던 세션 상태를 그대로 이어받아야 하는데, 성능이 다르면 넘겨받는 순간 스루풋을 못 버텨 보조도 뻗음. DDoS는 장비 고장이 아니라 공격이라 넘겨도 보조도 똑같이 뻗으니 failover 대상 아님.

**A6.** 4095=VGT(트렁크)=모든 VLAN 태그를 안 지우고 게스트로 통과. pfSense가 10/20 태그를 직접 봐야 VLAN 10/20으로 라우팅 가능. VST(특정 ID 고정)면 vSwitch가 태그를 벗겨 게스트가 VLAN 구분 못 함.

**A7.** 재귀를 누구에게나 열면 "오픈 리졸버"가 되어 DNS 증폭 DDoS의 발판으로 악용됨. 질의(우리 존 알려줘)는 열어도, 재귀(남 대신 외부까지 찾아줘)는 외부에 열면 공격 도구가 됨.

**A8.** **방화벽 WAN IP**(들어오는 입구 기준). 흔한 실수는 보낸 사람(관리 PC) IP를 넣는 것. 출처를 거르려면 Source 쪽에. NAT(변환)와 Filter(허용)는 별개 단계.

**A9.** WAN으로 나갈 때 SNAT 처리로 출발지 IP가 공인 IP 하나로 뭉개져 내부 출처가 사라짐(SNAT 추적 불가 문제가 탭 가시성에서도 발목).

**A10.** "웹 공격 표면이 있느냐"는 요건으로 판단. 입력받는 동적 서비스가 없는 정적 페이지거나 CDN에 올려 오리진을 직접 못 때리면 WAF가 과설계일 수 있음. 장비가 있으니 두는 게 아니라 요건이 생겼는지로 접근.

**A11.** 통신 구간 기밀성은 올라가지만 VPN 게이트웨이 자체가 인터넷에 노출된 공격 표면. CVE가 초기 침투 경로로 자주 쓰임(Fortinet/Ivanti/Pulse). "노출 위험을 수용(Risk Acceptance)하고 쓰는 것" → 펌웨어 패치(PMS)+MFA가 전제. 그래서 ZTNA로 가는 흐름.

**A12.** DMZ는 SNAT/DNAT으로 공인 IP와 1:1 매칭돼 외부가 직접 때릴 수 있어 SSH 같은 진입 경로를 deny해야 함. 직원 내부망은 DNAT 매칭이 없어 밖에서 못 찾아오므로 인바운드 차단을 따로 안 박아도 됨.

**A13.** Wazuh=관제(SIEM)/Darktrace 초기형, Suricata=NIDS/IPS(인라인), WireGuard=VPN 암호화 터널, socat=망연계 게이트웨이(검사·승인·로그 없는 약한 버전).

</details>

## 13. 종합 면접 질문

> 각 질문에 ✅모범 / ❌함정. 입으로 한 번 말해보고 비교.

**Q. 보안 장비 배치 순서와 근거는?**
✅ Anti-DDoS를 맨 앞에 — 대량 트래픽을 먼저 걸러야 뒤 방화벽·IPS가 안 죽음. 이후 방화벽(L4 접근제어)→IPS(시그니처 차단)→Anti-APT(샌드박스)→WAF(HTTP 정밀)로 점점 정밀. IDS는 차단 안 하니 인라인 대신 미러링.
❌ "방화벽이 제일 앞" — DDoS가 방화벽을 먼저 죽이므로 Anti-DDoS가 더 앞.

**Q. 패킷 필터링(1세대)과 Stateful Inspection(2세대) 차이는?**
✅ 핵심은 **세션 상태 추적**. 1세대는 세션 개념이 없어 응답을 받으려면 역방향을 수동으로 열어야 하고 출발지 포트를 any로 열게 됨. 2세대는 세션을 기억해 내가 시작한 세션의 응답만 자동 허용.
❌ "2세대는 포트를 더 잘 막는다" — 본질은 세션 추적.

**Q. NGFW(3세대)가 다른 점은?**
✅ L7 페이로드까지 검사해 IP가 수시로 바뀌는 서비스도 **애플리케이션 단위로 식별·차단**.
❌ "방화벽인데 IPS·백신도 된다" 나열 — 본질은 L7 애플리케이션 식별.

**Q. IDS를 이중화해야 하나요? 방화벽과 비교하면?**
✅ IDS는 차단 안 하니 다운돼도 서비스 영향 없어 싱글로 시작 가능(로그 수집만 잠깐 멈춤). 방화벽은 죽으면 통신 끊겨 HA 1순위. IDS가 관제 중심이 되면 이중화(공유 볼륨 클러스터).
❌ "보안 장비는 다 이중화" — 차단 여부가 우선순위를 가름.

**Q. NIDS vs HIDS, 대규모에선?**
✅ NIDS=네트워크 전체를 한 군데서(경유한 공격만), HIDS=호스트 상세·사용자 단위(개별 관리 부담 + 뚫리면 꺼짐). 서버 수만 대면 HIDS 전수는 비현실 → NIDS로 통으로 + 핵심 서버만 HIDS.
❌ "HIDS가 더 정밀하니 좋다" — 운영 규모·비용·한계를 같이.

**Q. SPAN과 TAP, 어떤 기준으로 고르나?**
✅ 구간 상황으로. 트래픽 많아 스위치가 발열·rx/tx 에러 낼 수 있으면 본선 무영향인 TAP, 라이트하면 간편한 스팬. 단 TAP은 설치 시 단선 1회, 어느 쪽이든 길목이 있어야 함.
❌ "TAP이 무조건 좋다" — 설치 시 단선 트레이드오프.

**Q. 시그니처 기반의 한계와 보완은?**
✅ 빠르고 오탐 적지만 신규·난독화는 못 잡음 → 애노멀리(정상 학습 후 벗어남 탐지)로 보완, 오탐 관리가 과제, Darktrace류 AI NDR로 발전.
❌ "시그니처는 구식, 애노멀리가 우월" — 대체가 아니라 보완(아는 공격은 시그니처가 빠르고 정확).

**Q. 샌드박스(Anti-APT)는 어떻게 잡나?**
✅ 동적 분석 — 의심 파일을 격리 VM에서 실제 실행해 파일 생성·레지스트리 변조·외부 통신 같은 악성 행위 관찰. 잡으면 시그니처 추출해 배포. EDR 행위 탐지와 같은 발상.
❌ "파일을 검사한다" 뭉뚱그림 — 실행해서 행동을 본다(동적) vs 코드만 본다(정적) 구분이 핵심.

**Q. Active-Standby vs Active-Active?**
✅ A-S는 주만 일하고 보조는 세션 동기화받다 failover로 VIP 인계(무중단). A-A는 둘 다 부하 분산하며 상호 동기화(양쪽 라인 다 깔아 비용↑). 공통으로 세션 동기화 때문에 동일 성능.
❌ "A-A가 항상 좋다" — 비용 트레이드오프.

**Q. VPN을 도입하면 보안이 강해지나요?**
✅ 구간 기밀성은 올라가지만 게이트웨이 자체가 공격 표면. CVE가 초기 침투 단골(Fortinet/Ivanti/Pulse). 노출 위험을 수용하고 쓰는 것 → 펌웨어 패치+MFA 전제, ZTNA로 가는 흐름.
❌ "VPN = 안전" 단정 — 양면을 말해야.

**Q. 보안 장비 구성도를 그릴 때 가장 신경 쓰는 것은?**
✅ "이 장비가 죽으면 서비스에 어떤 영향이 오나"를 장비마다 먼저 따짐. 차단 장비는 인라인+이중화+바이패스, 탐지·분석 장비는 미러링으로 본선 밖. 장애 시나리오를 돌려본 흔적이 드러나야.
❌ "성능 좋은 장비를 본선에 다 넣는다" — 장애 가정→인라인이냐 미러링이냐의 의사결정이 핵심.

**Q. 내부 DNS 운영 시 보안상 반드시 챙기는 설정은?**
✅ 오픈 리졸버가 안 되게 **재귀를 내부망으로 제한**(allow-recursion). 존 변경 시 Serial +1, named-checkconf/zone으로 검증, rndc reload로 무중단 반영.
❌ "DNS 깔고 레코드 넣으면 끝" — 오픈 리졸버 차단 + 이름풀이≠통신을 짚어야.

## 14. 심화

> 내가 복습본마다 **"다음에 깊게 볼 것"**으로 남긴 항목들 + 그걸 이해하는 데 받쳐주는 배경지식. 수업에서 한 발 더 나간 레이어다.

### 14.1 DPI 원리 — NGFW는 페이로드의 뭘 보고 `daum-mail`/`daum-cloud`를 구분하나

NGFW의 애플리케이션 식별 = **DPI(Deep Packet Inspection)**. IP·포트가 아니라 패킷 안 내용으로 앱을 식별한다.
- **단계**: ① 포트/프로토콜 휴리스틱 → ② **패턴/시그니처**(HTTP `Host:` 헤더, TLS **SNI**(접속하려는 도메인이 핸드셰이크 평문에 노출), 인증서 CN, 앱 고유 바이트 시퀀스) → ③ **행위·통계**(패킷 크기·주기 패턴)로 암호화돼도 추정.
- 그래서 같은 443 위에서도 SNI가 `mail.daum.net`이냐 `cloud.daum.net`이냐로 갈라 정책을 건다.
> 🔗 Snort `content:`가 한 문자열을 찾는 것의 "앱 단위 종합판". 한계: TLS 1.3 **ESNI/ECH**로 SNI까지 암호화되면 이 식별이 무력화 → SSL Inspection이나 행위 기반으로 넘어가야 함.

### 14.2 SSL Inspection을 걸 때 인증서가 어떻게 처리되나

복호화하려면 방화벽이 **중간자(MITM) 역할**을 해야 한다.
- 방화벽이 **사설 CA(루트 인증서)**를 들고, 클라이언트↔방화벽 구간은 방화벽이 그 CA로 **즉석 발급한 가짜 인증서**로 맺고, 방화벽↔서버 구간은 별도 TLS로 맺는다(두 개의 TLS를 방화벽이 잇는다).
- 그래서 **사내 단말에 그 사설 루트 CA를 신뢰 목록에 미리 깔아둬야** 인증서 경고가 안 뜬다(안 깔면 전 사이트가 경고).
- ⚠️ 한계: 인증서 핀닝(certificate pinning)을 쓰는 앱(은행·일부 모바일)은 가짜 인증서를 거부 → 예외 처리 필요. 개인정보·법적 이슈(직원 통신 복호화)도 동반.

### 14.3 시그니처 vs 애노멀리 vs 알파벳 수프 (EDR/XDR/NDR/SIEM/SOAR) 한 장 정리

| 약어 | 보는 대상 | 한 줄 |
|---|---|---|
| **IDS/IPS** | 네트워크 패킷 | 시그니처로 탐지(IDS)/차단(IPS) |
| **NDR** | 네트워크 흐름 | 애노멀리·AI로 이상 탐지(Darktrace) |
| **EDR** | 엔드포인트(단말) | 행위 기반 탐지·대응 |
| **XDR** | 엔드포인트+네트워크+클라우드+메일 통합 | EDR을 여러 소스로 확장 |
| **SIEM** | 모든 로그 | 수집·연관분석(Wazuh) |
| **SOAR** | 대응 워크플로 | 탐지 후 조치를 자동화(플레이북) |
> 흐름: 점 단위 탐지(IDS/EDR) → 로그 통합·연관(SIEM) → 멀티소스 통합(XDR) → 자동 대응(SOAR). FIMS는 Suricata(IDS)+Wazuh(SIEM)까지 했고, 여기에 자동 차단(blacklist 폴링)이 SOAR의 맹아.

### 14.4 샌드박스 회피 (sandbox evasion)

악성코드가 "지금 VM(분석 환경)이네"를 눈치채면 잠잠해진다.
- **탐지 단서**: VM 특유의 아티팩트(드라이버명·MAC OUI·레지스트리 키), CPU 코어 1개·RAM 적음, 마우스 안 움직임, `sleep`으로 분석 타임아웃 넘기기, 특정 도메인이 살아있는지(인터넷 연결) 확인.
- **대응**: 샌드박스를 진짜 사용자 환경처럼 위장(마우스 움직임 시뮬레이션, 충분한 자원·실제 같은 아티팩트), 베어메탈 샌드박스, 실행 시간 연장, **정적+동적 병행**(동적이 잠잠해도 정적으로 의심 구조 포착).

### 14.5 ZTNA — VPN을 대체하는 흐름

VPN은 "터널만 뚫리면 내부망을 통째로 믿는" 구조(과신뢰). **Zero Trust = "절대 믿지 말고 항상 검증"**.
- **ZTNA**: 망에 붙였다고 통과시키는 게 아니라 **요청(앱·리소스)마다** 사용자·단말 상태·맥락을 검증. 사용자는 "망"이 아니라 "특정 앱"에만 연결됨(횡적 이동 차단).
- 구성 요소: IdP(인증)+MFA, 단말 상태(EDR 연동), 정책 엔진, 브로커. **마이크로세그멘테이션**(워크로드 단위 분리)과 짝.
> 🔗 과정2의 "VLAN+ACL로 망을 잘게 가르고, 다른 망은 방화벽 경유"가 ZTNA 사고의 물리 버전. FIMS가 L3를 잘게 나눠 수평이동을 막은 것도 같은 방향.

### 14.6 인증/암호 기초 — HTTPS·VPN·DRM이 다 여기서 나온다

- **대칭키**(같은 키로 암복호, 빠름) vs **비대칭키/공개키**(공개키로 암호화→개인키로만 복호, 느림). 실무는 **하이브리드**: 비대칭으로 대칭키를 안전하게 교환한 뒤 본 통신은 대칭으로(TLS가 이 방식).
- **TLS 핸드셰이크 요지**: 클라가 서버 인증서를 받아 **CA 체인으로 신뢰 검증** → 키 교환(요즘 ECDHE로 forward secrecy) → 세션 대칭키 합의 → 암호화 통신.
- **PKI / 인증서 체인**: 루트 CA → 중간 CA → 서버 인증서. 브라우저가 루트 CA를 신뢰하니 그 밑 체인이 검증됨. (FIMS Let's Encrypt가 이 체인을 자동 발급)
- **JWT vs 세션**: JWT=서명된 토큰(서버가 상태 안 들고 검증, FIMS 8h) / 세션=서버가 상태 보관. **mTLS**=서버뿐 아니라 클라이언트도 인증서로 증명(서버↔서버, 강한 내부 인증).
> 🔗 FIMS의 HTTPS·WireGuard·JWT·OAuth가 전부 이 PKI/대칭·비대칭 뿌리에서 나온 가지들.

### 14.7 Cyber Kill Chain & MITRE ATT&CK — 공격을 단계로 본다

지금까지 배운 공격(정찰→침투→수평이동→반출)을 프레임워크로 정리하면 면접·실무에서 말이 정돈된다.
- **Lockheed Martin Kill Chain**: 정찰 → 무기화 → 전달 → 익스플로잇 → 설치 → C2 → 목표 행동. (각 단계를 끊으면 공격 실패)
- **MITRE ATT&CK**: 실제 공격 전술(Tactic)·기법(Technique)을 매트릭스로 카탈로그화. 탐지 룰·레드팀·갭 분석의 공통 언어.
> 🔗 매핑: 피싱=전달, RAT=설치+C2, Lateral Movement=ATT&CK의 Lateral Movement 전술, 데이터 반출=Exfiltration. **방화벽 아웃바운드 차단·D코스·트로이목마 역방향 룰**이 다 "C2/Exfiltration 단계를 끊는" 방어.

### 14.8 방화벽 룰 실제 동작 (iptables 체인) — UFW 밑단

UFW는 iptables의 래퍼다. 한 단계 내려가면:
- 패킷이 **체인**(INPUT/OUTPUT/FORWARD)을 통과하며 룰을 위→아래로 매칭(first match), 타깃은 ACCEPT/DROP/REJECT.
- **DROP vs REJECT**: DROP=조용히 버림(공격자에게 정보 안 줌, 스캔 느리게), REJECT=거부 응답을 줌(빠른 실패). 보안은 보통 DROP.
- **conntrack**(연결 추적 모듈)이 곧 Stateful의 구현체 — `ESTABLISHED,RELATED`를 허용하면 응답이 자동으로 돌아온다(2세대 방화벽의 정체).
> 🔗 "Top-Down first match"·"좁은 ALLOW를 넓은 DENY 위에"가 iptables 체인에서 실제로 이렇게 돈다. NAT도 iptables의 nat 테이블(DNAT/SNAT 타깃)에서 처리.

### 14.9 PMS 내부 레포 경유 — 서버의 직접 인터넷 차단

6-12 "다음에 볼 것"에 적은 것. 서버가 `apt update`로 직접 인터넷에 나가면 아웃바운드를 열어야 하고 그게 공격 표면.
- **해법**: 내부에 **미러 레포지토리(APT mirror / Nexus / Artifactory)**를 두고, 서버는 그 내부 레포에서만 패치를 받음 → 서버의 직접 인터넷 아웃바운드를 0으로.
- 폐쇄망 PMS와 같은 구조: 인터넷에서 패치를 받는 건 DMZ의 수집 서버 하나뿐, 내부는 망연계/내부 배포로.
> 🔗 C코스(아웃바운드 화이트리스트)의 최종 형태 — "repo만 허용"에서 "외부 직접 차단 + 내부 레포 경유"로.

### 14.10 Snort/Suricata 룰 옵션 더 (flow/depth/offset/metadata)

6-8 "다음에 볼 것". content 외 자주 쓰는 옵션:
- `flow:established,to_server;` — 핸드셰이크가 맺힌 뒤 서버로 가는 방향만(오탐↓, 성능↑).
- `depth:` / `offset:` — content를 페이로드의 **어디서부터 몇 바이트 안에서** 찾을지(범위 좁혀 오탐↓·속도↑).
- `nocase;` — 대소문자 무시. `pcre:` — 정규식 매칭(유연하지만 무거움).
- `threshold:`/`detection_filter:` — N초에 M회 같은 빈도 조건(FIMS 포트스캔 룰의 `threshold:type both,count 5,seconds 10`).
- `metadata:` — 룰 분류·관리용 태그.
> 핵심 원리는 그대로: **범위를 좁힐수록(flow·depth·offset) 오탐이 줄고 빨라진다.** 4.4의 "짧은 바이너리 오탐" 문제를 이 옵션들로 다스린다.

## 15. 헷갈린 것 / 음차 교정 총정리

> 노트 곳곳에 흩어진 ⚠️·💡·음차 오류를 한 곳에 모았다. 시험·실무 직전 빠른 점검용.

**음차 교정**
- 웹스키드/스키드 프록시 → **Squid 프록시**(포워드·캐싱, 아웃바운드 화이트리스트)
- Wmware → **VMware**, ESXi **8.0**, 인수사 **Broadcom**
- ovnsense → **OPNsense**, clamvnet → **ClamAV**
- 피티엘/엘티오 → **PTL**(테이프 라이브러리) / **LTO**(테이프 매체)
- OrcusRAT, XMRig(코인 채굴), Deny 로그 — 표기 정정
- "DOA 위험 수용" → 위험 처리 4전략(회피/전가/완화/**수용**)의 '수용'으로 이해(약어 원본 흐릿 → 수업 자료 재확인)

**자주 틀리는 표현(면접 함정)**
- FIMS는 "VLAN 망분리" ❌ → **"VirtualBox 내부 네트워크 세그멘테이션"** ✅
- HA는 L3 구간 이름 ❌ → **고가용성(High Availability) 설계 개념** ✅
- "방화벽이 맨 앞" ❌ → **Anti-DDoS가 맨 앞** ✅
- "보안 장비는 다 이중화/다 인라인" ❌ → **차단=인라인+이중화, 탐지=미러링** ✅
- ping 안 되면 서버 죽음 ❌ → **ICMP만 막혔을 수 있음**(서비스는 동작) ✅
- 로그인 5회 차단 = Fail2Ban ❌ → FIMS는 **slowapi** ✅

**실수하기 쉬운 설정 디테일**
- smb.conf: `browseable`(e), `valid users`(복수) — `testparm`으로 거름
- fstab: `-o` 없음, 필드=공백·옵션=콤마
- BIND: `allow-recursion` 내부망만, Serial +1, FQDN 끝 점(.)
- ssh.socket(22.04+)이 sshd_config Port를 덮어씀 → `ss -tlnp`로 확인
- NAT 포워딩 Dest.Address = WAN IP(보낸 사람 아님), NAT≠Filter(허용룰 별도)
- ESXi 트렁크 = VLAN ID **4095**(VGT), WAN/LAN 대역 충돌 금지, DHCP는 같은 포트그룹
- MD5 < SHA-256(다운로드 무결성)
- 설정파일 검증 습관: `testparm` / `named-checkconf` / `netplan generate` / `mount -a`(fstab) — **적용 전 문법검사**
