# 2026-06-11 복습 — 과정2 8일차: VLAN으로 웹서버 망 가르기 (ESXi VGT 4095 · pfSense VLAN 라우팅)

> 오늘은 6/10에 잡은 LAN(192.168.2.0/24) 위에 **VLAN으로 운영 Web(VLAN 10, 10.10.10.1)과 개발 Web_Dev(VLAN 20, 10.10.20.1)를 갈라** 서버를 올린 날. 핵심은 **ESXi 포트그룹의 VLAN ID 4095(VGT=트렁크)** — 태그를 안 지우고 pfSense로 다 넘겨야 pfSense가 VLAN 라우터 노릇을 한다. 곁들여 ① 다운로드 해시 검증 습관, ② 트러블슈팅 순서(내부통신→ping→pfSense 8번 쉘), ③ pfSense 트래픽 그래프(WAN 파랑/LAN 빨강)로 룰 동작 읽기, ④ Full/Half Duplex, ⑤ 최소설치 후 필수 유틸 세트까지.

## 오늘의 복습 퀴즈

**Q1.** [단답형] ESXi 포트그룹에서 VLAN ID를 **4095**로 주면 무슨 모드이고, 누가 VLAN 태그를 처리하게 되나?

**Q2.** [흐름 이해] pfSense LAN 포트그룹을 특정 VLAN ID(예: 10)로 고정하면 VLAN 10/20 라우팅이 왜 안 되나? 4095여야 하는 이유는?

**Q3.** [실무 적용] 새 웹서버를 올렸는데 패킷이 안 간다. 방화벽 룰부터 뒤지지 않는다면 어떤 순서로 좁혀가나?

**Q4.** [FIMS 연결] FIMS의 "망 분리"와 오늘 VLAN 분리는 격리 방식이 어떻게 다른가? 면접에서 FIMS를 뭐라고 표현해야 정확한가?

**Q5.** [그래프 읽기] pfSense 트래픽 그래프에서 WAN(파랑)은 오르는데 LAN(빨강)은 안 오른다. 무슨 상황인가?

<details>
<summary>정답 보기</summary>

**A1.** **VGT(Virtual Guest Tagging) = 트렁크 모드.** vSwitch가 태그를 건드리지 않고 **게스트(pfSense)에게 모든 VLAN 태그를 그대로 넘긴다.** 그래서 게스트가 직접 태그를 보고 VLAN을 구분·라우팅한다. (1~4094=VST는 vSwitch가 태깅/언태깅, 0=EST는 외부 스위치가 처리)

**A2.** 특정 VLAN ID로 고정(VST)하면 **vSwitch가 그 포트로 나가기 전에 태그를 벗겨버린다** → pfSense는 들어온 트래픽이 10번인지 20번인지 구분 못 함 → VLAN 라우팅 불가. **4095(VGT)** 로 열어야 태그가 안 지워진 채 pfSense까지 와서 pfSense가 VLAN 10(10.10.10.1)·VLAN 20(10.10.20.1)으로 분류·라우팅할 수 있다. "모두를 포함해야 하니 4095"가 이 뜻.

**A3.** **아래에서 위로**: ① 내부 통신은 되나(같은 망 안) → ② ping으로 L3 도달성 확인 → ③ **pfSense 콘솔 8번(Shell)에서 직접 ping.** 방화벽 자신이 닿는지를 봐야 "라우팅/대역 문제"인지 "방화벽 룰/포트그룹 문제"인지 갈린다. pfSense는 되는데 클라이언트는 안 되면 룰·포트그룹 문제, pfSense조차 안 되면 라우팅·대역 문제.

**A4.** FIMS = **VirtualBox 내부 네트워크 이름 기반 세그멘테이션**(소프트웨어 격리, 하이퍼바이저 뚫리면 격리 붕괴 가능). 오늘 = **802.1Q VLAN 태그 기반 격리**(스위치가 태그로 나눔, 다른 VLAN끼리는 반드시 L3 pfSense 경유). 면접에선 FIMS를 "VLAN 망분리"라 하면 **틀림** → "**VirtualBox 내부 네트워크로 세그멘테이션 했다**"가 정확.

**A5.** WAN으로 트래픽이 들어왔지만(파랑↑) **방화벽이 차단**해서 내부(LAN)로 안 나간 상황(빨강 평평). 통과됐다면 LAN 빨강도 같이 올랐을 것. → 패킷 캡처 없이 그래프만으로 룰이 막았는지 1차 판단 가능.

## 예상 면접 질문

**Q. 가상화(ESXi) 환경에서 VLAN을 어떻게 구성하나요? 방화벽 VM이 VLAN 라우터가 되게 하려면?**
✅ 좋은 답변: "ESXi는 **포트그룹의 VLAN ID**로 모드가 갈립니다. 1~4094는 VST로 vSwitch가 태깅/언태깅하고, **4095는 VGT(트렁크)로 태그를 안 지우고 게스트에게 다 넘깁니다.** pfSense 같은 방화벽 VM을 VLAN 라우터로 쓰려면, 그 VM이 붙은 포트그룹을 **4095로 열어 모든 태그를 통과**시키고, pfSense 안에서 VLAN 10·20 인터페이스를 만들어 각 게이트웨이로 라우팅합니다. 특정 ID로 고정하면 vSwitch가 태그를 벗겨 게스트가 VLAN을 구분 못 합니다."
❌ 주의: "포트그룹에 VLAN 번호만 주면 된다"로 끝내면 약함. **트렁크(4095/VGT)와 액세스(고정 VLAN)의 차이**, 그리고 **다른 VLAN끼리는 L3 경유**라는 걸 짚어야 함.

**Q. 운영 서버와 개발 서버를 같은 물리 인프라에서 분리하려면?**
✅ 좋은 답변: "**VLAN으로 대역을 갈라** 운영(VLAN 10)·개발(VLAN 20)을 다른 브로드캐스트 도메인에 둡니다. 케이블을 새로 안 깔아도 논리적으로 분리되고, **다른 VLAN끼리는 L3(방화벽)를 거치므로 그 길목에 방화벽 룰로 통제**까지 겁니다. 목적은 개발 중 사고(설정 실수·악성코드 테스트)가 운영으로 전파되는 걸 끊는 것입니다."
❌ 주의: "서버를 따로 둔다" 수준이면 약함. **VLAN(L2 격리) + 다른 VLAN은 L3 경유 + 그 지점에서 룰 통제**의 3단을 말해야 함.

**Q. 소프트웨어/ISO를 다운로드할 때 보안담당자로서 챙기는 건?**
✅ 좋은 답변: "**검증된 공식 사이트에서 받고, 받은 파일의 해시를 공식 게시값과 대조**합니다. 무결성 검증으로 중간자 변조·바꿔치기를 거르는 거죠. MD5는 충돌 위험이 있어 가능하면 **SHA-256 이상**을 씁니다(`sha256sum`, `Get-FileHash`)."
❌ 주의: "공식 사이트에서 받는다"만 말하면 절반. **해시 대조(무결성)** 와 **MD5→SHA256** 까지 가야 완결.

## 연결되는 기존 개념

| 오늘 개념 | 연결되는 기존 개념 | 파일 | 어떻게 연결되나 |
|---|---|---|---|
| VLAN 태그 격리 | 세그멘테이션 vs VLAN, FIMS 망분리 | OSI-7Layer / FIMS | "이름 격리(FIMS)" → "태그 격리 + L3 경유(오늘)" |
| ESXi VGT 4095 트렁크 | 액세스/트렁크 포트, 802.1Q | OSI-7Layer | 트렁크=여러 VLAN 태그 통과, 4095가 그 가상화판 |
| 다른 VLAN은 pfSense 경유 | L3에서만 VLAN 간 통신, ACL | OSI-7Layer | VLAN=나누기 / 라우터·룰=이동 통제 |
| pfSense 트래픽 그래프로 룰 검증 | 인라인/미러링, 가시성 | Security-Solutions | 통과/차단을 트래픽 관찰로 확인(미러링의 가벼운 버전) |
| 오픈소스 UTM(OPNsense/PacketFence/ClamAV) | UTM, NAC | Security-Solutions | 상용 말고 오픈소스로 방화벽+NAC+백신 스택 |
| 다운로드 해시 검증 | 무결성, 공급망 변조 | Security-Solutions | 무결성 검증으로 변조 파일 거르기 |
| Full/Half Duplex | L1 물리 계층, ThroughPut | OSI-7Layer | 양방향 동시 전송 = 1Gbps 양방향 |
| 운영/개발 VLAN 분리 | 세그먼트 분리, 장애 전파 차단 | Architecture / FIMS | 사고 전파 차단 = FIMS 세그멘테이션 목적과 동일 |

## 오늘 헷갈렸던 것 / 짚어둘 것

- **VLAN ID 4095가 왜 LAN 포트그룹에만?** — 정리: 4095=VGT=트렁크. pfSense가 VLAN 라우터가 되려면 태그가 안 지워진 채 들어와야 해서, 그 인터페이스만 "모든 VLAN 통과(4095)"로 연다. 특정 ID 고정(VST)은 vSwitch가 태그를 벗겨 VLAN 구분 불가.
- **트러블슈팅은 아래에서 위로** — 패킷 안 가면 룰부터 의심 말고 `내부통신 → ping → pfSense 8번 쉘 ping`. 방화벽 자신이 닿는지가 라우팅 vs 룰 갈림길.
- **`any` inbound는 랩 한정** — 호스트 크롬 편의로 WAN any→wansubnet을 열었지만, 실무에선 출처를 관리 IP로 좁혀야 한다(any inbound=공격 표면).
- **MD5 < SHA256** — 무결성 검증 습관은 좋지만 MD5는 충돌 위험. 가능하면 SHA256.
- **telnet은 로그인용 아님** — 여기선 `telnet IP PORT`로 **포트 도달성 확인용**. 평문이라 실제 원격 셸은 SSH.
- ⚠️ 음차 메모 정정: "ovnsense"=**OPNsense**, "clamvnet"=문맥상 **ClamAV**로 봄 → 강사 자료로 한 번 더 확인.
