# 2026-06-17 복습 — OpenVPN 구축 · VPN 운영함정 · Security Onion(NSM)

> 오늘 핵심: ① pfSense OpenVPN을 **실제로** 깔았다(CA·인증서·터널·redirect GW·키배포) ② SSL-VPN을 왜 any로 여나 / VPN≠포트포워딩 / 재택 "닌자게이트" ③ Security Onion = **미러 기반 NSM 센서**(보는 쪽) vs Snort 인라인(막는 쪽).

## 오늘의 복습 퀴즈

Q1. (단답) SSL-VPN에서 접속자의 **출발지(source)** 정책을 보통 `any`로 열 수밖에 없는 이유는?

Q2. (흐름) **VPN과 포트포워딩(DNAT)의 차이**는? "VPN으로 사설 IP를 받으면 내부 어디든 갈 수 있다"가 왜 틀린가?

Q3. (개념) 재택근무자가 **개발망 + 인터넷망을 동시에** 라우팅으로 붙여 쓸 때 생기는 "닌자게이트" 위험을 RAT 시나리오로 설명하라.

Q4. (실무) Security Onion 센서를 설치한 직후, "이 센서가 실제로 트래픽을 보고 있나"를 확인하는 **점검 2가지**는?

Q5. (면접/FIMS) Security Onion은 IPS인가? 그리고 과정1 FIMS의 어떤 구성과 같은 발상인가?

<details>
<summary>정답 보기</summary>

A1. 재택·외근 직원이 스벅·집·LTE로 **출발지가 계속 바뀌고(잦은 이동) 수가 많아**, 사용자마다 1:1로 출발지 정책을 짜는 걸 사람이 감당할 수 없기 때문. 그래서 입구(출발지)는 `any`로 열되, **들어오는 건 SSL 인증(로그인+OTP)으로 막는다.** 보안 책임이 "출발지 제한 → 인증"으로 이동한 것. (원래 폐쇄형 → 공개형 전환)

A2. **포트포워딩(DNAT)** = 외부 포트를 내부 IP:포트로 "꽂는" 것(서비스 하나 노출). **VPN** = 접속자에게 **사설 IP를 발급**해 내부망의 일원처럼 만드는 것. 단, **사설 IP를 받았다고 다 들어가는 게 아니다** — 그 IP에 대한 **정책(어떤 목적지로 갈 수 있나)이 따로 생성돼야** 한다. 즉 "사원증 발급 ≠ 전 층 프리패스", 카드로 열리는 문(정책)만 열린다. = 터널 + 정책 2단.

A3. 두 망을 동시에 붙이면 그 노트북이 **두 망을 잇는 다리**가 된다. 이 노트북이 RAT(원격제어 악성코드)로 장악되면, 해커가 **인터넷 쪽 → 노트북 경유 → 개발망**으로 피벗해 들어갈 수 있다. 망분리가 노트북 한 대로 무력화되는 것 = "닌자게이트"(눈에 안 띄는 우회 게이트웨이). 바람직한 운영은 **VPN 끊고 인터넷 쓰고 다시 연결**(동시 연결 금지). 코로나 이후 **ISMS 심사가 이 환경부터 점검**한다.

A4. ① `so-status` — 모든 서비스(도커 컨테이너)가 **running**인지(수집/탐지 파이프라인이 살아있나). ② `ifconfig` — **모니터링 NIC의 RX 패킷이 늘어나는지**(SPAN 미러가 실제로 들어오나). **RX가 0이면 룰·콘솔이 멀쩡해도 탐지 0 = 장님 센서** → SPAN/케이블/포트그룹부터 의심.

A5. **아니다.** Security Onion은 기본이 **미러(SPAN/TAP) 기반 NSM/IDS** — 탐지·가시성 플랫폼이지 인라인 차단 장비가 아니다(차단은 Snort 인라인 IPS의 몫). FIMS에서 **Suricata(탐지) + Wazuh(관제)를 따로 깔아 손으로 엮은** 그 NSM을, Security Onion은 **Suricata+Zeek+Elastic+SOC 콘솔을 한 박스로 제품화**한 것 = 같은 발상의 완제품 버전.

## 예상 면접 질문

**Q. "VPN을 쓰면 내부망이 안전한가요?"**
✅ 좋은 답변: "VPN은 **암호화 터널을 깔아 들여보내는 것**까지입니다. 터널만 뚫는다고 다 되는 게 아니라 **터널 + 서버측 접근 정책 2단**을 통과해야 하고, 들어온 뒤의 행위는 **입구 SSL Inspection + 관제**로 따로 봅니다. 게다가 **VPN 게이트웨이 자체가 인터넷에 노출된 공격 표면**이라 펌웨어 패치(PMS)와 MFA가 같이 가야 하며, 그 한계 때문에 요청마다 검증하는 **ZTNA**로 가는 추세입니다."
❌ 주의: "VPN으로 암호화하니 안전합니다" — 터널=프리패스로 오해하는 전형적 오답. VPN 장비 CVE가 초기 침투 단골 경로라는 점을 빠뜨리면 감점.

**Q. "IDS와 IPS, 그리고 Security Onion은 어떻게 다른가요?"**
✅ 좋은 답변: "IPS는 **라인 위(인라인)에서 실시간 차단**, IDS는 **미러(SPAN/TAP) 복제본을 보고 탐지만** 합니다. Security Onion은 **미러 기반 NSM 플랫폼**으로, 센서가 복제 트래픽을 Suricata(탐지+PCAP)·Zeek(메타)로 보고 Elastic에 적재해 SOC 콘솔에서 분석합니다. 즉 **Snort 인라인 = 막는 쪽 / Security Onion = 보는 쪽**으로 역할이 다르고, 같이 둬야 막으면서 놓친 것도 봅니다."
❌ 주의: "Security Onion으로 공격을 차단한다" — 기본은 탐지·가시성. 차단 장비로 말하면 틀림.

**Q. "Security Onion에서 경보가 떴습니다. 어떻게 분석하나요?"**
✅ 좋은 답변: "SOC 콘솔에서 **Alerts**로 시작해 수상한 경보를 드릴다운하고, **Hunt**로 관련 Zeek 메타·로그를 더 팝니다. 필요하면 **PCAP**로 실제 패킷(증거)을 확인하고, 진짜 사고면 **Cases**로 승격해 분석가끼리 협업·종결합니다. 즉 **경보 → 헌팅 → 증거(PCAP) → 사건화** 흐름입니다."
❌ 주의: "로그를 grep 한다" 수준으로 답하면 NSM 콘솔의 분석 동선(피벗)을 모른다는 인상.

## 아직 애매하거나 보충할 것

- ⚠️ Security Onion **실설치를 못 따라감**(PC 문제) → ISO 재다운로드 후 so-status·ifconfig RX·SPAN 미러를 직접 확인해 이 정리 검증 필요.
- OpenVPN **client-export로 실제 원격 접속** + 사용자별 목적지 정책(사설 IP ≠ 프리패스) 아직 검증 전.
- "닌자게이트"는 강사님 표현 — 표준 용어로는 **피벗(pivoting)/우회 게이트웨이**. 면접에선 표준 용어로 바꿔 말할 것.
- redirect gateway의 "게이트웨이는 하나" 제약과 split-tunnel(부분 라우팅) 옵션의 트레이드오프는 다음에 더 파볼 것.
