## 오늘의 복습 퀴즈

Q1. (단답) Zabbix가 기본 템플릿에 없는 임의의 값(예: SYN_RECV 개수)을 수집할 때 에이전트 설정파일에 정의하는 것은?

Q2. (흐름) `zabbix_agentd -t`로는 값이 나오는데 `zabbix_get`으로는 빈값이 나온다. 원인과 해결은?

Q3. (실무) 파일 무결성(FIM) 트리거를 만들었는데, 변조 직후 떴던 알람이 10초 뒤 자동으로 사라진다. 왜이고, 어떻게 막나?

Q4. (개념) snoopy가 `~/.bash_history`보다 신뢰할 수 있는 이유 3가지는? 그리고 snoopy로도 부족할 때 쓰는 도구는?

Q5. (FIMS 연결) 6/19에 "SSH 실패 로그 무더기 = Hydra"라고 정리했다. 오늘 Zabbix는 그걸 어떻게 자동화했나? 그리고 FIMS의 어떤 장치와 같은 발상인가?

Q6. (면접) "journald가 있는데 rsyslog를 왜 같이 쓰나?"

Q7. (개념) AD에서 Golden Ticket이 위험한 이유와, 그래서 DC가 공격자의 최종 목표가 되는 이유는?

<details>
<summary>정답 보기</summary>

A1. **UserParameter** (`/etc/zabbix/zabbix_agentd.conf`에 `UserParameter=키,쉘명령`). 키 호출 → 쉘명령 실행 → 표준출력 1줄이 값. "키↔쉘명령 매핑".

A2. 원인 = **권한을 데몬(zabbix 유저) 기준으로 봐야 하는데 `-t`는 root로 즉석 실행이라 됐던 것**. 대상 파일(예: `authorized_keys`, `drwx------`)을 zabbix 유저가 못 읽음. 해결 = 파일 권한을 풀지 말고 **`/etc/sudoers.d/zabbix`로 `md5sum`만 한정 sudo 허용**(NOPASSWD) + UserParameter에 `sudo` 붙이기(최소권한).

A3. 원인 = 변조 후 다음 수집 때 `#1=#2`(최신=직전)가 되면 Zabbix가 **정상 복귀로 보고 OK 이벤트를 자동 생성**해 알람 해제. 해결 = 트리거에서 **OK event generation: None**(자동해제 끔) + **Allow manual close**(담당자가 확인 후 수동 닫기) → 보안 알람 누락 방지.

A4. ① 사용자가 **못 지움**(커널 레벨 개입 불가, history는 `history -c`로 삭제 가능) ② **실시간** 기록(history는 세션 종료 시에만) ③ **모든 execve 포착**(쉘 종류 무관·스크립트 내부 명령·sudo 전후까지 / history는 bash만). 부족할 때 = **auditd**(커널 audit, 우회 어렵고 파일/네트워크/시스템콜 전부). 실무는 둘 다.

A5. 6/19엔 사람이 `auth.log`(또는 `journalctl`)를 보고 실패 로그를 셌다. 오늘은 `UserParameter=ssh.failed,journalctl -u ssh | grep -c "Failed password"`로 **Zabbix가 자동으로 세고 트리거(`>20`)로 알람**. = **FIMS의 slowapi(요청 속도 제한)** 와 같은 "임계치로 무차별 대입을 끊는다"는 발상의 모니터링판. (차단까지 가면 fail2ban)

A6. journald는 **바이너리 DB**라 빠른 인덱싱 조회엔 좋지만 grep/tail이 안 되고 **외부 전송이 어렵다**. rsyslog는 journald 로그를 받아 **facility로 분류해 텍스트 파일**(auth.log/syslog…)로 떨궈서 grep도 되고 **외부 SIEM 전송이 쉽다**. 그래서 실무는 둘 다 — 바이너리(빠른 조회)+텍스트(외부 전송). 즉 **SIEM 연동의 입구가 rsyslog**.

A7. Golden Ticket = DC의 **krbtgt 해시**(KDC가 TGT를 서명하는 마스터 키)를 탈취하면 **위조 TGT를 마음대로 발급** → 도메인 전체에 무제한·장기 접근. 인증 체계 자체가 위조되는 것. AD는 계정·권한·정책을 DC에 중앙집중하므로 **DC 하나 = 도메인 전체**, 그래서 DC가 최종 표적.

</details>

## 예상 면접 질문

**Q. "서버에서 SYN Flood를 어떻게 탐지하고 대응하나요?"**
✅ 좋은 답변: "공격 시 `SYN_RECV`(half-open) 상태가 커널 연결테이블에 적체됩니다. 저는 Zabbix UserParameter로 `ss -ant | grep -c SYN-RECV`를 수집해 임계치 트리거(예: >50)로 탐지했습니다. **CPU는 거의 안 오릅니다** — 자원 고갈이 아니라 backlog 큐(`tcp_max_syn_backlog`)가 차서 정상 연결을 못 받는 게 본질이라서요. 대응은 IP 차단이 아니라 `tcp_syncookies=1`로 half-open을 메모리에 안 올리는 SYN Cookie, 길목에선 IPS 임계치 룰입니다."
❌ 주의: "SYN Flood가 들어오면 CPU/메모리가 터진다"는 부정확. SYN Flood의 본질은 자원 고갈이 아니라 **연결 큐 고갈**.

**Q. "파일 무결성 모니터링(FIM)을 직접 구성해봤나요? 한계는요?"**
✅ 좋은 답변: "Zabbix로 `authorized_keys`·`passwd`·`sshd_config`의 md5 해시를 수집해 `#1<>#2`(직전 해시와 다름)로 변조를 잡았습니다. 보안 알람은 자동 해제되면 누락되니 **OK event generation을 None으로 두고 수동 close**하게 했습니다. 다만 해시 폴링이라 수집 간격 사이 변조→원복은 놓치고 '누가/언제'를 모릅니다. 그래서 **FIM은 상태 확인, auditd/Wazuh는 행위·주체 추적**으로 보완해야 한다고 봅니다."
❌ 주의: "FIM이면 변조를 다 잡는다"는 과신. 폴링의 한계를 함께 말해야 함.

**Q. "리눅스 로그가 auth.log에 없던데 어떻게 보나요?" (Ubuntu 24.04)**
✅ 좋은 답변: "24.04는 systemd-journald가 로그를 바이너리 DB로 잡아서 `journalctl -u ssh`로 봐야 합니다. 텍스트 파일(auth.log)이 필요하면 rsyslog가 facility 기준으로 분류 저장하고, 그게 SIEM으로 보내기도 쉬워서 실무에선 journald+rsyslog를 같이 씁니다."
❌ 주의: "auth.log를 grep하면 된다"고만 하면 최신 배포판을 모른다는 인상.
