## 포트 포워딩

### 문제 상황
내부 서버(192.168.1.10)에서 웹서버를 돌리고 있음
같은 네트워크(LAN) → 접속 가능 ✅
외부 인터넷       → 접속 불가 ❌ (사설 IP는 내부에서만 의미)
### 해결: 라우터에서 포트 포워딩 설정
"공인 IP(82.62.51.70)의 80번으로 오는 요청
→ 내부 192.168.1.10의 80번으로 연결해"
친구 → 82.62.51.70:80 → [라우터가 변환] → 192.168.1.10:80 ✅
### 아파트 비유
친구: "82.62.51.70 아파트 80호 찾아가자"
↓
경비실(라우터): "80호요? 1동 10호로 가세요"
↓
내부 서버(1동 10호)에 도착!
### NAT (Network Address Translation)
- 경비실이 외부 주소를 내부 주소로 번역해주는 것
- 외부에서는 내부 구조를 모르고, 경비실(라우터)만 알면 됨
- 포트 포워딩은 방화벽이 아니라 **라우터**에서 설정

## Q&A: 포트 포워딩과 보안

### Q: 포트를 열면 해킹당하는 거야?
**포트가 열려있다 ≠ 해킹 성공. 포트는 문일 뿐이고, 문 안의 서비스를 뚫어야 진짜 해킹이다.**
포트 열려있음 = 문이 열려있음
→ 아직 해킹 아님
→ 문 안 서비스의 취약점을 뚫어야 해킹
실제 공격 시나리오:
약한 비밀번호 → SSH(22번) 브루트포스로 뚫림
악성코드가 몰래 4444번 포트 열어놓음 (백도어)
라우터 비밀번호가 기본값 → 포트 포워딩 규칙 조작
### Q: 그럼 포트를 안 열면 되는 거 아니야?
**포트를 안 열면 서비스 자체가 불가능하다. 서비스를 하려면 포트를 열 수밖에 없다.**
포트 닫혀있음 = 안전하지만 서비스 불가 (가게 문 잠금)
포트 열려있음 = 서비스 가능하지만 보안 필요 (가게 영업 중)
→ "편의성과 보안은 반비례"
→ 보안 전문가의 역할 = 이 균형을 잡는 것
### 열린 포트를 지키는 방법
암호화 (HTTPS/SSH) → 엿봐도 내용 못 읽음
인증 (비밀번호/키/2단계 인증) → 아무나 못 들어옴
방화벽 규칙 → 특정 IP만 허용
최소한의 포트만 열기 → 안 쓰는 포트 닫기
서비스 업데이트 → 알려진 취약점 패치

## 방화벽

### 역할: 네트워크의 보안 검문소
방화벽 없으면:
외부 → [라우터] → 내부 서버 (누구든 다 들어옴 😱)
방화벽 있으면:
외부 → [방화벽 검문소] → [라우터] → 내부 서버 (검사 후 통과/차단)
### Stateless — 규칙표만 따르는 경비원
┌──────────────────────────────┐
│ 규칙표:                       │
│ 198.51.100.34 → 차단         │
│ 80번 포트 → 허용              │
│ 22번 포트 → 차단              │
└──────────────────────────────┘
→ 패킷 하나하나 규칙표와 대조
→ 빠르지만 맥락을 모름
→ 3계층(IP), 4계층(포트) 정보만 봄
### Stateful — 기억력 좋은 경비원
┌──────────────────────────────────┐
│ 기억:                             │
│ "아까 내부 직원이 구글에 접속했음"    │
│ → 구글에서 돌아오는 응답은 통과!     │
│                                   │
│ "이 패킷은 아무도 요청 안 했는데?"   │
│ → 차단!                          │
└──────────────────────────────────┘
→ 연결 전체를 기억하고 판단
→ 똑똑하지만 리소스를 많이 씀
→ 3~7계층까지 추적
### 실습: DoS 공격 방어
공격:
해커(198.51.100.34) → 서버(203.0.110.1):80 초당 수만 개 요청 → 서버 다운 💀
방화벽 규칙 추가:
┌────────────────────────────────┐
│ Source:      198.51.100.34      │
│ Destination: 203.0.110.1       │
│ Port:        80                │
│ Action:      DROP ⛔           │
└────────────────────────────────┘
결과:
해커 → [방화벽: "너 차단"] → ❌
정상 사용자 → [방화벽: "너 통과"] → ✅

## VPN

### VPN = 인터넷 위에 만든 암호화된 비밀 터널
VPN 없이:
집 PC → [인터넷 🌐] → 회사 서버
↑
누구나 엿볼 수 있음 😱
= 유리관 속 택배 (다 보임)
VPN 사용:
집 PC → [===암호화 터널===] → 회사 서버
↑
인터넷을 지나지만 안 보임 🔒
= 강철관 속 택배 (안 보임)
### 장점 세 가지
프라이버시: 트래픽 내용을 아무도 못 봄
익명성: 내 실제 IP 대신 VPN 서버 IP가 보임
원격 접속: 집에서 회사 내부 네트워크 접속 가능
### VPN 기술 세 가지
PPP:    암호화 + 인증의 기본 뼈대 (다른 기술의 재료)
PPTP:   PPP로 터널을 만듦, 설정 쉬움, 암호화 약함 (유리관에 색칠)
IPSec:  IP 자체를 암호화, 강력함, 현재 가장 많이 쓰임 (진짜 강철관)

## 라우터와 스위치

### 아파트 비유
스위치 = 동 안의 복도
→ 같은 네트워크 안에서 기기를 연결
→ MAC 주소로 "누구한테?" 판단
라우터 = 아파트 단지 정문
→ 다른 네트워크나 인터넷으로 연결
→ IP 주소로 "어디로?" 판단
### 스위치의 두 종류
Layer 2 스위치:
→ MAC 주소만 봄
→ 같은 네트워크 안에서만 작동
→ "이 프레임은 3번 포트 기기 거"
Layer 3 스위치:
→ MAC + IP 둘 다 봄
→ 기본적인 라우팅도 가능
→ 스위치 + 라우터 하이브리드
### 실제 데이터 흐름
내 PC → [스위치: 같은 네트워크 안 전달]
↓
[라우터: 다른 네트워크로 보냄]
↓
[인터넷]
↓
[상대 라우터 → 스위치 → 상대 서버]

## 네트워크 시뮬레이터 실습

### Computer1 → Computer3 TCP 전송

### Q: 왜 핸드셰이크 로그가 5개야?
TCP는 열 때만 핸드셰이크가 있는 게 아니라 닫을 때도 있다.
열 때 3개 + 닫을 때 2개 = 총 5개.
[연결 열기: 3-way handshake]
Computer1 → Computer3:  SYN        "연결하자"
Computer3 → Computer1:  SYN/ACK    "좋아"
Computer1 → Computer3:  ACK        "시작"
← 데이터 전송 →
[연결 닫기]
4. Computer1 → Computer3:  FIN        "나 끝났어"
5. Computer3 → Computer1:  FIN/ACK    "알겠어, 나도 끝"
닫을 때 원래 4단계(FIN→ACK→FIN→ACK)인데
시뮬레이션에서는 서버가 ACK+FIN을 합쳐서 보내니까 2단계로 줄어듦

## 새로 알게 된 용어
| 용어 | 뜻 |
|------|-----|
| Port Forwarding | 외부 트래픽을 내부 특정 기기로 연결 |
| NAT | Network Address Translation, IP 주소 변환 |
| Firewall | 트래픽을 규칙에 따라 허용/차단하는 장치 |
| Stateful | 연결 전체를 추적하는 방화벽 (똑똑, 무거움) |
| Stateless | 개별 패킷만 보는 방화벽 (빠름, 단순) |
| DoS | Denial of Service, 서버 과부하 공격 |
| VPN | 암호화된 가상 사설 네트워크 터널 |
| PPP | 암호화 + 인증 기본 프로토콜 |
| PPTP | PPP 기반 터널링 (설정 쉬움, 암호화 약함) |
| IPSec | IP 기반 강력한 암호화 VPN (현재 주류) |
| Layer 2 Switch | MAC 주소 기반 스위치 |
| Layer 3 Switch | MAC + IP, 기본 라우팅 가능 하이브리드 |
| Routing | 네트워크 간 최적 경로 찾기 |
| Defense in Depth | 다층 방어 원칙 — 여러 겹으로 보안 |
