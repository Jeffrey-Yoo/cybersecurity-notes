## Task 1 — 패킷과 프레임의 차이

### 핵심 구분
Packet (패킷) = IP 주소가 있는 데이터 → 3계층 (Network)
Frame (프레임) = IP 주소가 없는 데이터 → 2계층 (Data Link)
### 봉투 비유
바깥 봉투 = 패킷 (IP 주소 적혀있음)
안쪽 봉투 = 프레임 (MAC 주소만 있음)
바깥 봉투를 뜯으면(IP 정보 벗기면) 안에 프레임이 남아있음
### 왜 작은 조각으로 나눠 보내?
- 큰 데이터를 한 번에 보내면 네트워크가 막힘 (병목현상)
- 작은 조각으로 나눠 보내면 효율적
- 예: 이미지를 조각조각 보낸 뒤 받는 쪽에서 재조립

### 패킷의 구조
┌──────────────────────────────┐
│         Header (헤더)          │ ← 주소, 규칙 등 관리 정보
├──────────────────────────────┤
│         Data (데이터)          │ ← 실제 내용
└──────────────────────────────┘
주요 헤더 정보:
TTL: 패킷 수명, 라우터 지날 때마다 -1
Checksum: 데이터 무결성 확인 값
Source Address: 보내는 사람 주소
Destination Address: 받는 사람 주소

## Task 2 — TCP/IP 모델

### OSI 7계층 vs TCP/IP 4계층
OSI 모델 (이론)          TCP/IP 모델 (실제 인터넷)
7. Application    ─┐
6. Presentation    ├→  4. Application
5. Session        ─┘
4. Transport      ───→  3. Transport
3. Network        ───→  2. Internet
2. Data Link      ─┐
Physical       ─┘→   1. Network Interface
OSI = 이론적 표준, TCP/IP = 실제 인터넷이 사용하는 모델

## Task 3 — TCP와 3-Way Handshake

### TCP 패킷의 주요 헤더
Source Port       → 보내는 쪽 포트 번호
Destination Port  → 받는 쪽 포트 번호
Sequence Number   → 데이터 순서 번호
Acknowledgement   → "여기까지 잘 받았어" 확인 번호
Checksum          → 데이터 무결성 확인
Flag              → SYN, ACK, FIN 같은 제어 신호
### 3-Way Handshake — 처음 한 번만 수행
나                            서버
|── SYN ─────────────────────>|  ① "연결하자" + 내 ISN 전달
|<── SYN/ACK ─────────────────|  ② "좋아" + 서버 ISN 전달
|── ACK ─────────────────────>|  ③ "시작하자"
|                              |
← 연결 완료, 데이터 전송 시작 →
### 연결 종료 과정 — 끝날 때 한 번만
나                            서버
|── FIN ─────────────────────>|  "나 끝났어"
|<── ACK ─────────────────────|  "알겠어"
|<── FIN ─────────────────────|  "나도 끝났어"
|── ACK ─────────────────────>|  "오케이 종료"
### Checksum의 작동
보내는 쪽: 데이터로 체크섬 값 계산 → 헤더에 포함
받는 쪽:   같은 계산 → 값 비교
일치 → 정상 / 불일치 → 재전송 요청

## 심화: Sequence Number 완전 정복

### SEQ = 패킷의 순서 번호표
택배를 여러 박스로 나눠 보낼 때 각 박스에 붙이는 번호 스티커.
인터넷에서 패킷이 다른 경로로 가면 도착 순서가 뒤죽박죽이 되는데,
SEQ 번호가 있으니까 올바른 순서로 재조립할 수 있다.
도착:      [3] [1] [5] [2] [4]  ← 뒤죽박죽
번호 보고:  [1] [2] [3] [4] [5]  ← 올바르게 재조립!
### SEQ는 핸드셰이크 + 데이터 전송 모두에서 사용
[핸드셰이크] → SEQ 시작 번호(ISN) 교환 = "출발선 정하기"
[데이터 전송] → SEQ 계속 증가 = "몇 번째 데이터인지 추적"
핸드셰이크에서만 쓰이는 게 아니다!
데이터 전송 중에 SEQ가 있어야 유실 감지, 순서 보장, 재전송이 가능.

### 실제 작동 과정
핸드셰이크:
나: "내 시작 번호는 100" (ISN)
서버: "내 시작 번호는 300" (ISN)
데이터 전송:
나 → 서버: SEQ=101, 100바이트
서버 → 나: ACK=201  "201번부터 줘"
나 → 서버: SEQ=201, 100바이트
서버 → 나: ACK=301  "301번부터 줘"
### SEQ와 ACK의 관계
하나의 패킷 안에 SEQ + ACK가 동시에 들어있음:
나 → 서버: SEQ=101, ACK=301
의미: "내 101번 데이터 보낸다 + 너의 301번까지 잘 받았어"
SEQ = 내가 보내는 데이터의 번호
ACK = 상대방 데이터 잘 받았다는 확인 번호
### 유실 감지와 재전송
SEQ=101 ✅ 도착
SEQ=201 ❌ 유실!
SEQ=301 ✅ 도착
서버: "201번이 안 왔어!"
서버 → 나: ACK=201  "201번부터 다시 보내줘"
나 → 서버: SEQ=201  ✅ 재전송!
### 왜 양쪽이 서로 다른 ISN을 가지는가?

**이유 1: 양방향 통신이니까 각자의 번호표가 필요**
나도 보내고 서버도 보내니까 번호가 같으면:
"이 200번이 내 데이터? 서버 데이터?" → 구분 불가!
각자 다른 번호 체계를 쓰면:
나의 SEQ=101 → 내 데이터
서버의 SEQ=301 → 서버 데이터
→ 절대 헷갈리지 않음
**이유 2 (핵심): 보안 — 예측 방지**
❌ 둘 다 0부터 시작하면:
공격자: "다음 SEQ는 100이겠지?" → 가짜 패킷 끼워넣기 가능
✅ 랜덤 시작:
나: ISN=48291 / 서버: ISN=739105
공격자: 시작점을 모르니까 예측 어려움
### SEQ 예측 공격과 다층 방어
ISN이 랜덤이어도 데이터 크기만큼 증가하니까,
패킷을 엿보면 다음 번호를 예측할 수 있다.
그래서 방어가 여러 겹으로 되어있다:
1차 방어: ISN 랜덤 → 시작점 예측 방지
2차 방어: 같은 네트워크에 침입해야 엿볼 수 있음 → 네트워크 보안
3차 방어: HTTPS 암호화 → 엿보더라도 조작 불가
= 보안의 다층 방어 (Defense in Depth) 원칙
한 겹이 뚫려도 다음 겹이 막아주는 구조

## 심화: 통신의 전체 구조

### 핸드셰이크 vs 데이터 전송의 차이

**핸드셰이크 (SYN/ACK):**
- 보낼 데이터(7계층 내용)가 없음
- 4→3→2→1 하위 계층만 사용
- TCP 플래그만 담아서 전기신호로 보냄
- 빈 봉투에 "연결하자"만 적어서 보내는 것

**데이터 전송:**
- 실제 내용이 있음
- 7→6→5→4→3→2→1 전체 계층 사용
- 매 요청/응답마다 7계층 전체를 통과
- 봉투 안에 편지(데이터)를 넣어서 보내는 것
[핸드셰이크] SYN → SYN/ACK → ACK (1번만, 하위 계층만)
↓
검색어 보내기:   7→1 전체 포장 (전기신호로 출발)
검색 결과 받기:  1→7 전체 개봉 (화면에 표시)
이미지 요청:     7→1 전체 포장 (또 전기신호로 출발)
이미지 받기:     1→7 전체 개봉 (또 화면에 표시)
↓
[연결 종료] FIN → ACK → FIN → ACK (1번만)
### SYN vs ACK — 전송 중에는 ACK만
SYN (Synchronize) = 처음 연결할 때 1번만
ACK (Acknowledge) = 데이터 전송 중 계속 반복
전송 중에 SYN은 없음!
"연결하자"를 매번 하는 게 아니라 "잘 받았어"만 확인
### 매번 ACK를 보내지는 않음 — 윈도우(Window)
❌ 비효율 (하나씩 확인):
보내고 → 기다리고 → 보내고 → 기다리고 (느림)
✅ 실제 방식 (여러 개 보내고 한번에 확인):
SEQ=101 →
SEQ=201 →
SEQ=301 →
← ACK=401 "301번까지 다 받았어" (빠름!)
윈도우 = 한번에 ACK 없이 보낼 수 있는 패킷 수
### ACK vs 체크포인트의 차이
ACK (4계층 Transport):
→ "이 패킷 잘 받았어" 작은 단위 확인
체크포인트 (5계층 Session):
→ "여기까지 전체적으로 완료" 큰 단위 기록
→ 파일 다운로드 "이어받기" 기능
비유:
ACK = 택배 기사가 "박스 하나 전달 완료" 사인
체크포인트 = "전체 이사짐 50% 완료" 기록
### 모든 통신은 결국 전기신호
핸드셰이크든 데이터 전송이든
최종적으로 1계층(전기신호/와이파이 전파)으로 나감
그 안에 계층별로 포장된 정보가 들어있는 것!

## Task 4 — UDP

### TCP vs UDP
TCP                    UDP
연결 방식    3-way 핸드셰이크 필요      연결 없이 바로 보냄
신뢰성      도착 보장                  도착 보장 안 함
순서        순서대로 재조립             순서 보장 안 함
속도        느림 (확인 과정)            빠름 (그냥 보냄)
헤더        복잡하고 큼                 단순하고 작음
용도        웹, 이메일, 파일            스트리밍, 게임, DNS
비유        카톡 (읽음 확인 ✓✓)        확성기 (들었는지 모름)
### UDP 패킷 구조 (TCP보다 훨씬 단순)
┌─────────────────┐
│ Source Port      │
│ Destination Port │
│ Length           │
│ Checksum         │
├─────────────────┤
│ Data             │
└─────────────────┘
### UDP를 쓰는 이유
유튜브: 픽셀 하나 깨져도 괜찮음, 끊기지 않는 게 중요 → UDP
카톡: 글자 하나라도 빠지면 안 됨, 정확해야 함 → TCP

## Task 5 — 포트 (Ports 101)

### 포트 범위
0 ~ 1024:    공통 포트 (잘 알려진 서비스)
1025 ~ 65535: 임시 포트 (사용자 연결에 자동 배정)
### 반드시 외울 포트 번호
21    FTP      파일 전송
22    SSH      보안 원격 접속
23    Telnet   원격 접속 (보안 없음, 위험)
25    SMTP     이메일 보내기
53    DNS      도메인 → IP 변환
80    HTTP     웹사이트 (암호화 없음)
110   POP3     이메일 받기
143   IMAP     이메일 받기 (서버 유지)
443   HTTPS    웹사이트 (암호화)
3389  RDP      윈도우 원격 데스크톱
### 포트가 작동하는 방식
내 컴퓨터(192.168.1.77:52301) → 구글 서버(142.250.196.110:443)
52301 = 내 컴퓨터가 자동 배정한 임시 포트
443   = 구글 서버의 HTTPS 서비스 포트
응답: 구글(443) → 내 컴퓨터(52301)로 돌려보냄

## 보안관제 연결 포인트

### 공격 탐지 예시
SYN Flood:
→ SYN만 대량 전송, ACK를 안 보냄
→ 서버가 자원 낭비하며 다운
→ 로그: "SYN만 비정상적으로 많다" → 공격!
포트 스캔 (Nmap):
→ 모든 포트에 SYN을 보내서 열린 포트 탐색
→ 로그: "짧은 시간에 수천 포트에 SYN" → 스캔 탐지!
SEQ 예측 공격:
→ 패킷 엿보고 다음 SEQ 예측 → 가짜 패킷 삽입
→ 방어: ISN 랜덤 + 네트워크 보안 + HTTPS 암호화

## 새로 알게 된 용어
| 용어 | 뜻 |
|------|-----|
| Packet | IP 주소 포함 데이터 단위 (3계층) |
| Frame | IP 없이 MAC만 있는 데이터 단위 (2계층) |
| TCP/IP Model | 실제 인터넷의 4계층 모델 |
| Checksum | 데이터 무결성 확인 값 |
| Sequence Number | 데이터 순서 추적 번호 (핸드셰이크 + 전송 중 계속 사용) |
| ISN | Initial Sequence Number, 랜덤 시작 번호 |
| ACK | 수신 확인 ("잘 받았어") |
| SYN | 연결 시작 요청 (처음 1번만) |
| FIN | 연결 종료 요청 |
| Window | ACK 없이 한번에 보낼 수 있는 패킷 수 |
| Defense in Depth | 다층 방어 — 보안의 핵심 원칙 |
| SYN Flood | SYN만 대량 전송하는 DoS 공격 |
| Common Ports | 0~1024, 잘 알려진 서비스용 포트 |
