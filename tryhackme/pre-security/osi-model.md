## 데이터 전송의 두 단계 (가장 중요한 개념!)

실제 통신은 두 단계로 나뉜다:

### 1단계: 핸드셰이크 (연결 수립)
- 실제 데이터 없이 "악수"만 하는 과정
- TCP 3-way Handshake: SYN → SYN-ACK → ACK
- 이때 세션 + TCP 연결이 양쪽 동시에 만들어짐
- 하위 계층(4→3→2→1)만 사용, 상위 계층 데이터는 비어있음
보내는이                   받는이
|                         |
|-- SYN ----------------->|  "연결하자"
|<- SYN-ACK --------------|  "좋아, 준비됐어"
|-- ACK ----------------->|  "오케이 시작!"
|                         |
← TCP 연결 + 세션 완성 →
### 2단계: 데이터 전송
- 세션이 만들어진 후에 실제 데이터가 오감
- 이때 7→1 전체 계층을 사용
보내는이: 7→6→5→4→3→2→1 (포장해서 보냄)
받는이:   1→2→3→4→5→6→7 (개봉해서 확인)

## 7계층 — Application (응용) 🖥️
> 사용자와 네트워크가 만나는 유일한 계층

### 핵심 개념
- 7계층 = 앱 자체가 아니라, 앱이 네트워크와 대화할 때 쓰는 규칙(프로토콜)
- 크롬 브라우저 = 7계층 프로토콜을 사용하는 도구
- GUI(마우스 클릭)와 CLI(터미널 명령어) 두 가지 방식으로 사용

### 핵심 프로토콜과 포트 번호
HTTP    (80)   → 웹 브라우징 (암호화 없음, 엽서)
HTTPS   (443)  → 보안 웹 브라우징 (암호화, 봉투 편지)
DNS     (53)   → 도메인 → IP 변환 (전화번호부)
FTP     (21)   → 파일 전송 (택배)
SSH     (22)   → 원격 접속 - 암호화 (보안관제 필수!)
Telnet  (23)   → 원격 접속 - 암호화 없음 (구식, 안 씀)
SMTP    (25)   → 이메일 보내기 (우체통)
POP3    (110)  → 이메일 받기 (다운로드 후 서버 삭제)
IMAP    (143)  → 이메일 받기 (서버에 원본 유지)
### 포트의 작동 방식
- 포트는 "페이지"에 붙는 게 아니라 "서버의 서비스"에 붙는 것
- 서버 관리자가 필요한 포트만 열고 나머지는 닫아놓음
- 브라우저가 프로토콜에 맞는 포트를 자동 선택
https://google.com → 자동으로 443번
http://google.com  → 자동으로 80번
### 한 페이지에서 여러 포트를 동시에 사용
쇼핑몰 페이지 하나를 열면:
브라우저 → DNS 서버:53       "도메인 IP가 뭐야?"
브라우저 → 쇼핑몰 서버:443   "페이지 주세요"
브라우저 → 이미지 서버:443   "상품 사진 주세요"
브라우저 → 결제 서버:443     "결제 모듈 불러오기"
같은 443 포트여도 출발지IP:포트 + 도착지IP:포트 조합으로 구분:
192.168.1.77:52301 → 쇼핑몰:443   (세션1)
192.168.1.77:52302 → 이미지서버:443 (세션2)
192.168.1.77:52303 → 결제서버:443  (세션3)
### 서버에 포트를 설정하는 과정 (예: 웹사이트 개설)
서버에 웹서버 프로그램(Nginx) 설치
설정 파일에서 443번 포트에 웹서비스 연결
방화벽으로 필요한 포트만 열기
443 (HTTPS) → 사용자용
22 (SSH) → 관리자 원격접속용
나머지 전부 닫기 → 보안!
DNS 등록: mysite.com → 서버 IP 주소
### 보안 포인트
- 대부분의 공격이 7계층에서 발생
- SQL 인젝션, XSS, 피싱, 브루트포스 등
- 닫혀있어야 할 포트가 열려있으면 → 취약점/백도어 의심!
"외부 IP에서 22번 포트로 로그인 시도 500회"
→ 브루트포스 공격 → IP 차단
"열려있으면 안 되는 4444번 포트가 열려있음"
→ 악성코드 백도어 의심 → 즉시 조사

## 6계층 — Presentation (표현) 🔄
> 데이터의 번역가

- 서로 다른 프로그램이 같은 데이터를 이해할 수 있게 변환
- 예: Gmail로 보내도 상대방 Outlook에서 똑같이 보임
- 암호화(HTTPS)도 이 계층에서 처리
- 비유: 한국어 편지를 영어로 번역하는 단계

---

## 5계층 — Session (세션) 🤝
> 연결을 시작하고 유지하고 끝내는 관리자

### 핵심 개념
- 세션은 데이터 보내기 전에 핸드셰이크로 먼저 만들어짐
- 데이터가 먼저 가고 세션이 나중에 생기는 게 아님!
- 세션이 있어야 데이터가 갈 수 있음

### 보내는이 vs 받는이의 세션 역할
보내는이 세션: "세션 ABC123 잘 살아있지? 여기에 실어서 보낸다"
받는이 세션:   "이건 세션 ABC123에서 온 거네, 유저A한테 전달하고 응답도 여기로"
### 동시 여러 연결 관리
브라우저 탭1: 구글 검색 → 세션 A
브라우저 탭2: 유튜브    → 세션 B
카카오톡: 메시지        → 세션 C
→ 세션이 없으면 데이터가 뒤섞임!
### 세션의 세 가지 역할
1. 연결 관리: 세션 ID로 각 연결 구분
2. 체크포인트: 끊겼을 때 이어받기 가능
3. 순서 관리: 누가 먼저 말할 차례인지 정리

### 체크포인트 비유
체크포인트 없으면: 50번에서 끊김 → 1번부터 다시 😩
체크포인트 있으면: 50번에서 끊김 → 45번부터 이어서 😊
= 파일 다운로드 "이어받기" 기능
### 4계층(TCP)과의 차이
5계층(세션): 누구와 연결할지, 연결 상태 관리 → 전화 연결 관리자
4계층(전송): 데이터 쪼개기, 정확한 전달, 에러 체크 → 실제 배달 기사
---

## 4계층 — Transport (전송) 🚚
> 데이터를 쪼개서 어떤 방식으로 보낼지 결정

### TCP (Transmission Control Protocol) — 등기 택배
장점: 데이터 정확하게 도착 보장, 순서대로 도착
단점: 느림, 연결 유지해야 함
용도: 이메일, 웹 브라우징, 파일 전송
비유: 카톡 메시지 (읽음 확인 ✓✓ 있음)
### UDP (User Datagram Protocol) — 일반 우편
장점: 빠름
단점: 도착 보장 안 함, 순서 뒤바뀔 수 있음
용도: 영상 스트리밍, 온라인 게임, ARP, DHCP
비유: 확성기로 외치기 (들었는지 모름)
---

## 3계층 — Network (네트워크) 🗺️
> 목적지까지 가는 길을 찾아주는 계층

### 핵심 역할 세 가지
1. IP 주소로 "어디에 있는지" 식별
2. 라우팅으로 최적 경로 결정 (최단/신뢰성/속도)
3. 패킷 단위로 데이터 관리

### 패킷 구조
┌─────────────────────────┐
│  출발지 IP: 192.168.1.77  │
│  도착지 IP: 8.8.8.8       │
│  TTL: 64                  │ ← 라우터 지날 때마다 -1, 0이면 폐기
│  데이터: "검색 요청"        │
└─────────────────────────┘
### MAC vs IP 차이
MAC 주소 → 주민등록번호 (바뀌지 않음, 누구인지)
IP 주소  → 집 주소 (이사하면 바뀜, 어디에 있는지)
### ARP는 3계층과 2계층의 다리
- 데이터를 보내려는 순간 상대방 MAC을 모르면 작동
- 처음 연결할 때 자동으로 작동하는 게 아님
- ARP 캐시에 저장 후 재사용, 일정 시간 지나면 만료
3계층: "목적지 IP는 아는데..."
→ ARP: "이 IP의 MAC이 뭐야?"
2계층: "MAC 알았어, 전달한다"
### 핵심 프로토콜
IP   → 주소 체계와 패킷 전달 규칙
ICMP → ping처럼 "살아있어?" 확인
ARP  → IP → MAC 찾기
### 보안 포인트
"외부 IP에서 내부 서버로 접근 시도"
→ 출발지 IP 국가 확인 → 정상/비정상 판단
---

## 2계층 — Data Link (데이터 링크) 🔗
> 바로 옆 장비에 전달하는 계층

- MAC 주소로 같은 네트워크 안에서 정확한 기기로 전달
- 3계층 패킷에 MAC 주소를 붙여서 프레임으로 만듦
- 스위치가 이 계층에서 작동
- 비유: 아파트 동 안에서 정확한 호수로 택배 전달

---

## 1계층 — Physical (물리) ⚡
> 실제 전기 신호로 데이터를 보내는 계층

- 이더넷 케이블, 광케이블, 와이파이 신호 등
- 데이터를 1과 0의 전기 신호로 변환
- 비유: 실제로 택배 트럭이 도로 위를 달리는 단계

---

## 캡슐화 (Encapsulation)

### 보낼 때: 위 → 아래 (포장)
7계층: 데이터 생성              [데이터]
6계층: 변환/암호화
5계층: 세션 확인
4계층: TCP/UDP 헤더 추가    →  [세그먼트]
3계층: IP 주소 추가         →  [패킷]
2계층: MAC 주소 추가        →  [프레임]
1계층: 전기 신호 변환       →  [비트]
### 받을 때: 아래 → 위 (개봉)
1계층: 전기 신호 수신
2계층: "도착지 MAC이 내 거 맞네?" → 수락
3계층: "도착지 IP가 내 거 맞네?" → 수락
4계층: 세그먼트 재조립
5계층: "세션 ABC123 거네" → 해당 연결로 전달
6계층: 복호화/변환
7계층: 사용자에게 데이터 표시
각 계층에서 "이거 내 거 맞아?"를 확인하고,
맞으면 위로 올려보냄. 하나라도 내 거 아니면 거기서 버림.

## 각 계층별 데이터 이름
| 계층 | 데이터 이름 | 비유 |
|------|-----------|------|
| 7~5계층 | Data (데이터) | 편지 내용 |
| 4계층 | Segment (세그먼트) | 편지를 접음 |
| 3계층 | Packet (패킷) | 봉투에 주소 적음 |
| 2계층 | Frame (프레임) | 택배 상자에 포장 |
| 1계층 | Bit (비트) | 트럭에 실어서 출발 |

## 보안관제에서 OSI 모델이 중요한 이유

### 계층별 공격 유형
1계층: 케이블 절단, 물리적 침입
2계층: MAC 스푸핑, ARP 스푸핑
3계층: IP 스푸핑, ICMP 플러딩(ping 공격)
4계층: SYN 플러딩(TCP 연결 공격)
7계층: SQL 인젝션, XSS, 피싱, 브루트포스
공격이 몇 계층에서 발생했는지 판단해야 적절한 대응 가능!

## 전체 흐름 한판 정리
"google.com 검색하고 싶다"
[1단계: 연결]
TCP 핸드셰이크 (SYN→SYN-ACK→ACK) → 세션 생성
[2단계: 데이터 전송]
7. Application:  DNS로 IP 알아내고, HTTPS로 요청 생성
6. Presentation: 데이터 암호화/변환
5. Session:      세션 ABC123에 실어서 보내기
4. Transport:    TCP로 세그먼트 쪼갬
3. Network:      출발지/도착지 IP 붙이고 경로 결정
└ ARP:        "게이트웨이 MAC이 뭐지?" (필요 시)
2. Data Link:    MAC 주소 붙여서 프레임 완성
Physical:     전기 신호로 출발!
→ 구글 서버에서 1→7 순서로 개봉
→ 응답을 다시 7→1로 포장해서 돌려보냄

## 새로 알게 된 용어
| 용어 | 뜻 |
|------|-----|
| OSI | Open Systems Interconnection |
| Encapsulation | 각 계층에서 정보를 추가하며 포장하는 과정 |
| TCP | 신뢰성 있는 전송 프로토콜 (등기 택배) |
| UDP | 빠르지만 보장 없는 전송 프로토콜 (확성기) |
| Handshake | 데이터 전송 전 연결을 수립하는 악수 과정 |
| Port | 서버에서 서비스를 구분하는 문 번호 |
| DNS | 도메인 → IP 변환 (전화번호부) |
| SSH | 암호화된 원격 접속 프로토콜 |
| Segment | 4계층에서의 데이터 단위 |
| Packet | 3계층에서의 데이터 단위 |
| Frame | 2계층에서의 데이터 단위 |
| Bit | 1계층에서의 데이터 단위 (0과 1) |
| TTL | Time To Live, 패킷의 수명 |
| OSPF/RIP | 최적 경로 찾는 라우팅 프로토콜 |
| Nginx | 웹서버 프로그램 |
| 방화벽(Firewall) | 포트를 열고 닫아 트래픽을 제어하는 보안 장치 |
